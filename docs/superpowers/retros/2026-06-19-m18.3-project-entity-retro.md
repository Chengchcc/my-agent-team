## 踩到的坑

### 1. `bun:sqlite` `db.run()` 参数传递

**现象**: 用 `db.run(sql, ...params)` 展开数组报 `TS2345`。

**原因**: `params` 类型为 `unknown[]`，bun:sqlite 期望 `SQLQueryBindings[]`。

**解决**: 对照 Agent adapter 发现它传数组不展开 —— `db.run(sql, params)`。

### 2. `@hookform/resolvers` v5 + zod v3 类型不兼容

**现象**: `useForm<FormValues>({ resolver: zodResolver(formSchema) })` 报 `TS2322`，resolver 产出类型（input type，含 optional）与 `z.infer` 类型（output type，含 default）不匹配。

**临时手段**: 去掉 `useForm` 的显式泛型，靠 TS 推断绕过。

**最终方案**: 将 `@hookform/resolvers` 从 v5.4.0 降到 v4.1.3（与 zod v3 类型兼容），恢复 `useForm<FormValues>` 显式泛型。manypkg 已将 zod 统一到 `^3.23.0`，不应为前端一个表单引入 zod v4 的 breaking change。

### 3. `@base-ui/react/dialog` 不支持 `asChild`

**现象**: shadcn Dialog 基于 `@base-ui/react`（非 Radix），`DialogTrigger` 不支持 `asChild` prop。

**解决**: 去掉 `asChild`，直接在 `DialogTrigger` 内放 Button —— 功能不变，`Trigger` 会透传点击事件。

### 4. `z.string().optional().default("")` 在 zod v3 中的语义

**现象**: 同时使用 `.optional()` 和 `.default("")` 在 zod v3 中产生冗余的类型层。

**解决**: `.default("")` 在 v3 中已隐含 optional 输入行为 —— 去掉 `.optional()`，只保留 `.default("")`。

### 5. biome 路径相对性

**现象**: `bunx biome check --write apps/web/src/...` 报 `No such file or directory`，路径被重复拼接。

**解决**: 用绝对路径 `/Users/chen/Workspace/my-agent-team/apps/web/src/...`。

## 设计哲学对照

| 原则 | 实施情况 |
|------|---------|
| **统一本体，不复制语义** | Project 严格定位为“近乎纯机制”——没有 status、不参与转移表、不引入新工作语义。`ProjectRow` 只存归属元数据。 |
| **暴露业务，隐藏机制** | 前端暴露“Project 下拉选”和“Project 管理页”，内部 adapter/service 对用户透明。 |
| **边界要硬，概念要少** | 新增了一个实体（Project），但未引入新领域本体（文档明确 Project 不是本体）。唯一跨边界点是 `countIssuesByProject`（project adapter 读 issue 表）——两害取轻。 |

## 遗留项

| 项 | 严重度 | 说明 |
|----|--------|------|
| `countIssuesByProject` 跨 feature 读表 | 低 | Project adapter 直接 `SELECT COUNT(*) FROM issue`，将来 issue 表迁移需同步关注 |
| `UNIQUE constraint failed` 字符串匹配 | 低 | 依赖 bun:sqlite 错误消息格式，跨版本可能断裂 |
| `ProjectList` 死代码 `message.includes("409")` | 低 | 应改为 `(err as ApiError).status === 409` |
| AgentForm 仍用自定义 modal | 中 | 已改造为 `useForm` + zod，但弹窗容器仍是手写 div（未像 Issue/Project 那样切到 shadcn Dialog） |
