## 2. What Could Be Improved

### Frontend form uses raw `<input>` instead of shadcn/ui `<Input>`
The IssuesPage create form uses plain HTML `<input>` elements with raw Tailwind classes. All other forms in the app use shadcn/ui `Input` from `@/components/ui/input`. This is a cosmetic inconsistency. Fix: swap to `<Input>` component in a follow-up.

### Type duplication between backend and frontend
`IssueStatus` and `IssueRow` are defined in both `apps/backend/src/features/issue/entities.ts` and `apps/web/src/lib/api.ts`. This is intentional per the architecture (no shared package for wire types yet), but it's a known maintenance tax — changing a field means updating both copies.

### `Date.now()` granularity in tests
The "applyTransition moves planned → in_progress" test initially asserted `updatedAt > createdAt` but failed because both timestamps fell in the same millisecond. Fixed by using `>=`. A mockable `now` injection was available (`IssueServiceDeps.now`) but the simpler assertion change sufficed for the test.

### Router transition line missing `async` wrapper
Line 530 in router.ts: `withAuth((r) => issues.transition(r, issueTransitionMatch[1]!), token)(req)` — the callback is NOT marked `async`, unlike the other 4 Issue routes. This works because `issues.transition` already returns `Promise<Response>` (the callback return type satisfies the union), but it's inconsistent with the pattern on adjacent lines. Should be `async (r)` for consistency.

### Webs without typecheck caught the sync-vs-async `withAuth` issue
The initial implementation used `withAuth((r) => issues.list(r), token)` (no `async`), which caused TS2739 errors because `Response` doesn't satisfy `Promise<Response>`. Adding `async` to the callbacks fixed it. This was a type-level issue only — runtime would have worked fine (sync functions are auto-wrapped by the runtime), but tsc correctly enforced the interface contract.

## 4. Lessons for M18.2 (Orchestrator)

1. **`applyTransition` is the integration point.** The Orchestrator's backfill listener calls `issueSvc.applyTransition(issueId, to)` — the same function the frontend will call on drag (M18.3). This single-writer invariant is verified working.
2. **`ISSUE_STATUSES` / `LEGAL_TRANSITIONS` are consumed via index barrel.** Orchestrator can import them from `@/features/issue` without touching `entities.ts` internals.
3. **The router has `issues?:` optional field.** Orchestrator routes can be added as a separate optional field `orchestrator?:` in FeatureSet, following the same pattern.
4. **Test pattern for service:** Orchestrator tests should follow the same pattern — `openDb('/tmp/test-orchestrator-...db')`, inject via deps, test against real SQLite.
