## 2. 架构决策记录

### Decision A: runId 注入方式 — A1 采用 ✅

选择了 `AgentRunOptions.runId?` 显式注入：runner-daemon 在 `#iteratorFor` 传入 `runId: run.runId`，framework 回退 `thread.id`。未采用自生成 ID（会与全局 runId 体系分裂）和 thread.id 替代（跨 run 会碰撞）。

**实际效果**：正确。`run/continue/resume` 三入口均设 `rt.runId = opts.runId ?? thread.id`，fallback 测试通过。

### Decision B: packages/message 已存在 ✅

基线确认 `@my-agent-team/message` 在 M17.1 已创建，无需前置补做。

### Decision C: prefix 移除 — 延期至 M17.3 ⚠️

`projectForMember` 仍拼接 `[name]:`/`[系统]` 前缀。spec §3.10 和 §6.6 目标"喂模型的 content 不含渲染前缀"未在 M17.2 完成。理由：改变 agent checkpoint 格式会影响多 agent 对话质量（agent 靠前缀区分 speaker），需在 M17.3 配合结构化 speaker 归属一起做。

**风险**：如果 M17.3 不做，模型上下文永远带着 `[name]:` 前缀，违反"边界要硬、概念要少"。

## 6. Lessons

### 做得好的

1. **TDD 顺序正确** — 先改 framework 词表 → runner 透传 → backend 去组装，每一跳都独立可测
2. **delta 删除干净** — wire type → daemon → supervisor → HTTP route，一整条死代码链全清（~65 行）
3. **framework 单测为新 envelope 提供回归网** — 6 个新测试覆盖 messageId/state/terminal/legacy-type-absence
4. **边界的 type narrowing** — supervisor.onRunEvent 从 `{type: string}` 收为 `AgentEvent`，编译期保证

### 可以改进的

1. **Turbo cache 误判** — `parseMessageRevision` 缺 import 但 typecheck 首次通过（缓存了旧 dist）；改完后应用 `--force` 重跑
2. **Commit 颗粒可更细** — P0 fix (40a034f) 是 review 后补的，应该被 review 在 commit 前拦住
3. **@mention 假设错误** — "framework emit text" 的假设未验证就被写进 commit message，正确做法是 trace `buildAssistantRevision` 的实际字段
4. **Spec 未落全** — 写进文件的 spec 只有 Summary + 实施顺序，section 3.10/6.6/10 在 PR 正文里但未落盘
