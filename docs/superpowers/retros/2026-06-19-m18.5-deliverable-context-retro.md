## 什么做对了

1. **Deliverable 模块独立、接口窄**。`features/deliverable/` 照 `column-config/` 粒度：domain/ports/adapter-sqlite/service/index 五个文件，每个职责单一。与 issue 模块之间只靠 `issueRoutes(svc, opsStore, deliverableSvc, opts?)` 签名耦合——未来换存储实现只需换 adapter。

2. **`renderPrompt` 只长出路径取值，不长出 DSL**。正则从 `\w+` 扩到 `[\w.]+`，加 `split(".").reduce()` 逐层取值——三行代码完成。`{{title}}` 单层照旧命中，`{{deliverables.plan.summary}}` 嵌套命中，缺失回退空串。无条件/循环/过滤器语法引入。测试从 4 个扩到 9 个，覆盖嵌套命中、缺失路径、中间对象回退空串、空格不匹配回归。

3. **`issueRoutes` 签名从例外回到主流**。旧签 `issueRoutes(svc, opts?)` 把回调塞 opts 是例外形态；新签 `issueRoutes(svc, opsStore, deliverableSvc, opts?)` 把必选协作者走位置参数、可选回调走尾部 opts，与 `agentRoutes`/`runRoutes` 一致。类型系统消解传参顺序风险（`RuntimeOpsStore` 与 `DeliverableService` 是不同命名类型，传反编译失败）。

4. **`surfaceContext` 注入不绑 surface 字面值**。runner-daemon 注入条件写 `sc?.capabilities.includes("submit_deliverable") && sc.issue && spec.mode !== "reflect"`，不判 `sc.surface === "orchestrator"`。未来若有其他 surface 也需要交付物工具（如 CLI 手动触发 issue 步骤），不用改注入判定。

5. **`idempotency_key` 一次建表，无 ALTER**。交付物表是本棒全新引入——不存在「先建基础表再补列」的平滑迁移需求。`idempotency_key TEXT` + `CREATE UNIQUE INDEX ... WHERE idempotency_key IS NOT NULL` 直接写进 5012 的 CREATE TABLE，零存量数据、零两步迁移。

6. **`submit_deliverable` 工具走 `{isError:true}` 不抛**。与 `start_new_conversation` 的抛错策略不同——交付提交是最佳努力，失败不该炸掉整个 run。`isError:true` 让模型可见错误信息并重试，但不中断执行流。

## 遇到的问题

### 1. runner-protocol `dist/` 类型缓存导致 supervisor type error

`packages/runner-protocol/package.json` 的 `exports` 字段把 `types` 指向 `dist/index.d.ts`。改了 `src/messages.ts` 的 zod schema 后，backend typecheck 仍报 `"orchestrator"` 不兼容 `"lark" | "web" | "cli"`——因为 `tsc` 解析的是旧的 `.d.ts` 而非源文件。需手动 `cd packages/runner-protocol && bun run build` 重建 dist 才消。

**教训**: 改 protocol schema 后必须 rebuild dist，typecheck 才会反映新类型。可以在 `bun run typecheck` 的 turbo pipeline 里把 `runner-protocol#build` 设为 `backend#typecheck` 的 dependsOn，但当前 turbo.json 未配此依赖——手动 rebuild 是目前的约定。

### 2. commitlint scope-enum 限制

提交信息写 `feat(protocol): ...` 被拒——`protocol` 不在 scope 白名单里。白名单含 `runner-protocol`，用全名才过。

**教训**: commitlint scope 必须用 `package.json` 里的确切包名，不能用缩写。

### 3. `typeof fetch` 严格类型导致 mock 类型断言

`bun:test` 的 `mock()` 返回 `Mock<...>` 类型，不全等于 `typeof fetch`（缺少 `preconnect` 等方法）。测试里 `globalThis.fetch = mock(...)` 需 `as unknown as typeof fetch`。这是 bun 类型定义的已知限制，`start-new-conversation-tool` 没有测试所以此前未暴露。

### 4. biome formatter 对长类型断言的换行偏好

`mock(...) as unknown as typeof fetch` 写在一行时 biome 要断成多行。`npx biome check --write` 可直接修复——比手动改快。

## 余下项

1. **`POST /api/issues/:id/deliverables` 端点零测试覆盖**。endpoint 的路由分发、zod 校验、run_origin 反查、幂等逻辑全未测试。当前靠 service 层真库测试间接覆盖了数据路径，但 HTTP 层的 400/404/409 分支无覆盖。应补充——至少 201 落库、400 validation、404 issue、409 run/issue mismatch、幂等重复提交五个用例。

2. **`reactor.startStep` 未测试 deliverable → prompt 的端到端渲染**。`mockDeliverableSvc()` 返回空数组，`buildPromptVars` 的效果仅靠 render.test.ts 的单元测试验证，而非 reactor 集成测试。应加一条：给 `mockDeliverableSvc` 喂一条 plan 交付物，断言 `buildSpec` 收到的 `input` 参数含交付物字段值。

3. **`fromStatus` 空字符串边界**。`submitDeliverable` 端点里，若 `runId` 未传且 `fromStatus` 也未传，`fromStatus` 被设为 `""`——落库仍成功（列是 `NOT NULL`，但空字符串是合法非 NULL 值）。语义上空 `fromStatus` 表示「归属不明」，但不阻断写入。当前工具总是同时发送 `runId` + `fromStatus`，触发不到此边界——但端点独立接收 HTTP 请求时应更坚固。可以在 `fromStatus` 解析后加 `if (!fromStatus) return json({ error: "fromStatus is required" }, 400)`。

4. **`buildPromptVars` 里 `ref` 可能与 `fields.ref` 冲突**。如果 Agent 在 `fields` 里提交了 `{ ref: "xxx" }`，`byKind[d.kind] = { ...d.fields, ...(d.ref ? { ref: d.ref } : {}) }` 会用列值覆盖字段值。当前文档说 `ref` 是独立参数，Agent 不应在 `fields` 里用 `ref` 键——但无强制校验。低概率，可接受。

## 设计哲学对齐

- **统一本体、不复制语义**: Deliverable 是新增领域本体——不是 Issue/Conversation/Run 的变体。它有自己的生命周期（追加式落库、不可变），与 Issue 通过 `issueId` 外键关联而非嵌套。`DeliverableRow` 只有 9 个字段，不发明子类型。
- **暴露业务、隐藏机制**: `createSubmitDeliverableTool` 对外暴露的是「提交交付物」这个业务动作；`surfaceContext → extraTools` 注入通道、HTTP 回调、幂等键这些机制对 Agent 完全不可见——Agent 只看到一个普通的 `submit_deliverable` 工具。
- **边界要硬、概念要少**: Deliverable 模块与 orchestrator 的耦合只有一个方法签名 `listByIssue(issueId): DeliverableRow[]`；与 issue/http 的耦合只有 `submit(input) → DeliverableRow` + `getByIdempotencyKey(key)`。模块内部可自由重构 adapter/service，不影响调用方。
