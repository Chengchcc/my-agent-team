## 二、设计决策回顾

### 做对的

1. **判定式用框架终止信号，不发明新字段**。`content.some(b => b.type === "tool_use")` 就是框架 ReAct loop 的退出条件本身（`toolUses.length === 0 → return`）。零依赖、零新协议字段、不靠 runId（ledger 没有）。

2. **grouping 放渲染层不放 reducer**。`groupTurns` 是 `messages → segments` 的幂等纯函数。放渲染层（`useMemo`）保持 reducer 纯净，不影响 optimistic replace / draft clear / self echo 等现有逻辑。

3. **跨消息配对修 tool_result 消失 bug**。这正是 M14.5 的核心附加值——不只是视觉折叠，而是让之前不可见的数据变得可见。

4. **复用 Collapsible 基座**。`@base-ui/react/collapsible` 已封装，不自己写动画 → 零 CSS bug 风险。

5. **ToolStep 去三层嵌套**。旧 `ToolCallCard`+`ToolResultCard` 各自带边框+各自可折叠 → card-in-card-in-collapsible。新 `ToolStep` 无边框、单行摘要+展开，终端日志风 → 视觉干净。

### 做错的 / 待改进

1. **锚点逻辑与 renderItems 重复 sender-change 检测**。`extractAnchors` 和 `renderItems` 都计算"相邻 segment sender 是否变化"→ 重复逻辑。可抽一个 `segmentBoundaries(segments)` helper 两处复用。非 bug，维护债。

2. **`segId` 未使用变量**。`renderItems` 里最初计算了 `segId` 但从未使用。已在 post-review 清理（`1124eff`）。

3. **web 缺少 `lint`/`format` 脚本**。`apps/web/package.json` 只有 `test`+`typecheck`，无 lint。不影响功能，但代码风格靠手工。

## 四、经验教训

1. **"判定式即框架语义"的威力**。不需要后端新字段、不需要 runId 进 ledger——框架已经给了足够信号。前端只负责投影这个信号到正确的视觉形态。

2. **"消息级判定"是对的**。模型输出 text+tool_use 本就属于同一条 assistant 消息（framework 把它们装进同一个 `content[]`），不存在"text 和 tool_use 是两条消息"的错觉。判定按消息级做（本条含不含 tool_use），100% 可靠。

3. **跨消息配对是前端数据就位但渲染错配的典型案例**。tool_use ↔ tool_result 的配对关系已存在于数据中（`tool_use_id`），只是 `renderContentBlocks` 单条内建 Map 永远查不到下一条消息里的 tool_result。把 Map 的作用域从"单条消息"扩到"整个 turn 段"就自然修复——不需要改数据、不需要后端参与。

4. **Collapsible 基座复用极大降低风险**。不自己写动画、不碰 grid-rows，直接复用已验证的 `@base-ui/react/collapsible` → 折叠/展开行为零 bug。
