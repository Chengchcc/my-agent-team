## 七、教训

1. **spec 中的 "forkRun 传 specJson" 是架构缝，不是实现细节。** 第一次实现把它当成 "传空字符串就行" 的琐事，但实际它是 backend→runner 的唯一通道——spec 不对，整个 @ 触发链不工作。这条应该在设计阶段就明确 "forkRun 是同步 builder，需构造完整 AgentSpecV1 JSON"。

2. **"锁" 必须配 "解锁"。** activeConversations 只 add 没 delete——在设计文档（D18）里说了 "ConversationService 维护 Set"，但没写删除时机。review 一眼就发现，但如果 reviewer 也假设 "肯定在某处删了" 就会漏。

3. **写前校验 vs 写后校验——顺序决定语义。** 409 在 appendAndBroadcast 之后抛，产生的不仅是 "浪费一次 DB 写入"，更严重的是 "客户端收到失败响应但副作用已经发生"——这在分布式系统里是标准的 correctness bug。规则：副作用之前必须完成所有可判定拒绝的检查。

4. **测试不能靠手工 INSERT 绕过架构空窗。** service.test.ts 的 `setupConv` 手工 INSERT threads 行——这意味着测试在 "thread 已存在" 的前提下通过，但生产代码从未创建 thread。这种模式在 adapter 层测试里正确（测试 adapter 本身），在 service 层测试里危险（掩盖了 service 没有创建 thread 的事实）。

5. **"物化" 决策要在数据库物理层面验证。** D7 说 "广播经 checkpointer 物化进 thread.messages"，设计上正确，但没检查 M9 生产代码里子进程 checkpointer 实际写哪个 DB。结果就是投影写到 backend.db、runner 读 workspace file db——同一种抽象的两种物理实例，语义对但连不上。

6. **SSE 长轮询的 "空转" 陷阱。** `subscribeConversation` 最初的 `while(true){break}` 是故意的——CLI 的 EventSource 会自动重连，每次重连触发一次 poll。但改成真正的长轮询后，测试必须在 `pollMs=0` 模式运行，否则 500ms × 120 = 60s 超时。生成器里的无限循环 + sleep = 测试杀手——需要显式退出条件。

7. **spec 的 "显式不做" 清单值得每次 review 时过一遍。** M10 的 8 条 "不做" 全部正确遵守，但 triggerMode='all' 的 schema 口子和 HTTP 层校验不一致（schema 接受但行为忽略）。教训：留口子的字段要么在 schema 层面拒收（做验证门），要么在行为层面降级（明确拒绝并告知），"接受但忽略" 是最差的中间态。
