## 二、实现中发现的 Bug

### B1: `runOneCollect` 与 batch loop 双重 push（已修复）

**文件**: `packages/framework/src/execute-one.ts`, `packages/framework/src/run-loop.ts`
**根因**: `runOneCollect` 在正常完成路径和 skip 路径都 `rt.thread.messages.push(resultBlock)`，而 batch loop 在 `Promise.all` 后又按原序 push 一次→tool_result 重复。
**修复**: 从 `runOneCollect` 移除内部 push+save，让 batch loop 统一拥有写回责任（保证顺序）。串行路径（`executeOne` generator）保持不变。
**教训**: 提取函数时，共享可变状态（thread.messages）的"谁写"责任边界必须明确。spec 只说了"batch loop 按原序写回"，没说 `runOneCollect` 不应写。

### B2: 自然 stop 跳过 follow-up drain（已修复）

**文件**: `packages/framework/src/run-loop.ts`
**根因**: 当模型返回零 `tool_use` blocks（自然结束）时，代码执行 `return` 直接退出整个 `runLoop` 函数，外层 `while(true)` 的 follow-up drain 永远不可达。
**修复**: 引入 `doneNormally` flag——自然结束时设 flag 并 `break` 内层 for 循环，让外层 while 的 follow-up drain 运行。`return` 仅用于 abort/interrupt/maxSteps 等真正的终止。
**教训**: 从"单层 for + return"重构到"外层 while + 内层 for + break"时，每一条 `return` 语句都要审计：它是"本任务结束（应 break）"还是"整个 run 结束（应 return）"。

### B3: shape→beforeModel 调序破坏现有测试（已修复）

**文件**: `packages/framework/src/create-agent.test.ts`
**根因**: 现有测试 `"ContextManager.shape called before plugin.beforeModel"` 断言旧顺序。调序后测试名和 expected order 均需更新。
**修复**: 测试名改为 `"plugin.beforeModel called before ContextManager.shape"`，expected order 从 `["cm", "plugin"]` 改为 `["plugin", "cm"]`。钩子签名从 `(msgs)` 修正为 `(_ctx, msgs)`（beforeModel 接收 `(ctx, messages)`）。
**额外修复**: 钩子返回 `readonly Message[]` 导致类型错误，改为 `return [...msgs]` 产出可变数组。

### B4: 测试 model 的 `tool_use` input 需 `input_json_delta`（已修复）

**文件**: `packages/framework/src/run-loop.test.ts`
**根因**: 测试用 `toolUseModel` 直接在 delta 里传 `input` 字段，但 stream 中 tool_use input 应通过 `input_json_delta` 增量传输。`mergeChunkIntoBlocks` 不认识直接的 `input` 字段。
**修复**: `{ type: "tool_use", id, name, input_json_delta: JSON.stringify(input) }`

## 三、设计决策记录

### D1: `PreserveHint` 标记全量注入为保留区

P2.2 调序后将所有 `beforeModel` 产出标记为 `preserve: { ranges: [{ start: 0, end: injected.length }] }`——即整个注入后的 messages 数组都是保留区。这意味着当前实现下 shaper 不会裁剪任何内容（passthrough 行为不变，summarizing/sliding-window/token-budget 需要适配 preserve 才能真正压缩）。这是一个**两步走的妥协**：先修顺序（让 shaper 能"看见"全量），再让各 shaper 实现认 `preserve` 做智能裁剪。第二步留给后续。

### D2: `structuredSummarize` 是 prompt 约束而非 schema 校验

模型可能不严格遵守五槽位格式。当前实现容忍缺槽位——不做解析、不做重试。这是有意为之：prompt 是强提示但非合约，摘要质量下降（自由文本）优于摘要失败（报错中断 run）。

### D3: `SteeringQueue` 是进程内、单 run 生命周期

这与 spec §5 的非目标一致：不做跨 run 持久化 steering。当前 `drain()` 语义是消费性的——取走即清空，不保留回放能力。如果后续要支持"steering 历史"，需要在 drain 侧做持久化而非改接口。

## 四、做得好的

1. **Spec 准确性高**: 6 个"当前代码事实"（spec §2）在实现中全部验证无误，没有"spec 说 A、代码实际是 B"的情况。spec 作为实现蓝本的价值被确认。
2. **Subagent 抓 bug 有效**: 5 个 subagent 中有 3 个发现了 spec/实现间的不一致（B1/B2/B3），且都在 commit 前修复，没有遗留到 review 阶段。
3. **P1→P4 顺序正确**: 每个 phase 只依赖前一个 phase 的产物，没有"做到一半发现需要回头改"的情况。P2.2 改 run-loop.ts 时 P1 的 batch 逻辑已稳定，merge 无冲突。
4. **测试不降级**: 每 phase 结束时 `bun test` 全绿。25 commits 中没有一个 commit 导致测试失败——每个 subagent 在 commit 前都跑了全套测试。
5. **向后兼容守住了**: 新 `executionMode` 默认 `"serial"`、`steering`/`followUp` 可选、`root` 选项仍可用——所有现有调用方零改动。
6. **文档与代码同步落地**: 4 页架构文档在代码完成后立即更新，没有"文档留到后面"的尾债。

## 五、如果重来一次

1. **`runOneCollect` 的责任边界应在 spec 里明确**: spec 说了"batch loop 按原序写回"，但没说 `runOneCollect` 不应写。如果 spec 明确"`runOneCollect` 只收集结果、不修改 thread，写回由 batch loop 统一执行"，B1 不会出现。
2. **重构 runLoop 结构应先用伪代码验证所有退出路径**: B2（自然 stop 跳过 follow-up）的根因是"for+return"到"while+for+break"的退出语义变换。如果在 spec 里列出所有退出路径（正常结束、工具中断、abort、maxSteps）并逐一验证外层 while 行为，bug 可以避免。
3. **P4 skill-load 的 `findMatchingRoot` 逻辑应更早暴露**: skill-load.ts 的 `${SKILL_DIR}` 解析在单域时简单（一个 root→一个映射），双域后需要知道"这个 skill 来自哪个 root"才能正确映射。这个复杂度被低估了，实现中 `findMatchingRoot` 是临场设计的。
4. **分 phase commit 而非分 task commit**: 25 commits 对于此规模的工作偏多——有些是 biome format fix、type fix 等机械 commit。如果按 phase 聚合（P1 5→2 commits，P2 6→2 commits，P3 3→2 commits，P4 3→1 commit，docs 4→1 commit），~10 commits 更可读。
