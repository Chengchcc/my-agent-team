3. **两种 SSE 解析分离**（§4.0.1）。run 级包层 vs 会话级扁平，两种解析路径独立。

## 五、经验教训

1. **"保存相对量、存储绝对量"的设计威力**。`isSelf` 踢到渲染层后，所有身份问题变成 `senderMemberId === viewerMemberId`。
2. **纯函数 reducer 的迁移成本极低**。M13.2 收敛到单一 reducer，M14 仅在类型上泛化——结构完全不变。
3. **Biome 全局格式化的副作用**。格式修改应单独 PR，不混入 feature commit。
4. **hook 复杂度偏高**。`useConversation` 有 4 个 `useEffect`，后续可抽独立 hook。

## 六、Post-Review Bug 修复

> 2026-06-10 | 代码审查发现 + 测试验证 | 9 个追加 commits

### P0 — Agent 消息渲染为空白

**根因**：后端 `main.ts` D19 把 agent 回复存成 `JSON.stringify({ text })`（对象），用户消息存裸 string。前端 `norm()` 对非 string/array 直接 `as ContentBlock[]` 误判为数组 → `extractText` 返回 `""` → 消息空白。

**修复** (`174e65f`)：
- 后端：`JSON.stringify({ text })` → `JSON.stringify(text)`（与 user 对齐）
- 前端：`norm()` 加 `{text}` 解包兜底 + 测试用例

**教训**：ledger content 约定必须统一。前后端 content 序列化不一致是隐式契约 bug，单元测试只有 string/array 用例没覆盖对象形态。

### P0 — 移除成员必失败 (400)

**根因**：`api.removeConversationMember` 用 `method: "POST"`，router 把 `POST /members` 路由到 `addMember`（需要 `kind` 字段）。body 只有 `{memberId}` → schema validation 失败 → 400。

**修复** (`174e65f`)：`method: "POST"` → `"DELETE"`。

**教训**：前后端 HTTP method 约定不一致。review 只检查了前端代码、没 cross-check router 的 method 匹配。

### P1 — Resume 后不再流式

**根因**：新 hook 的 `resumeRun` mutation 无 `onSuccess`。`approve()` 后 phase 停在 `"interrupted"` → SSE effects 门控 `phase==="running"` 不通过 → stream/events 永不重连。

**修复** (`174e65f`)：`resumeRun.onSuccess` dispatch `run/started`，phase 拉回 running。

**教训**：mutation 的 onSuccess 是副作用入口，重构时容易漏。

### P2 — D19 只持久化 text，丢弃 tool_use/tool_result

**根因**：`main.ts` D19 只提取 `text` 块 join 成字符串，`tool_use`/`tool_result` 被过滤掉。刷新后工具调用永久消失。

**修复** (`d10e31a`)：存完整 `lastAssistant.content`（string 或 ContentBlock[]），不再过滤。

### P2 — `stream/toolStart` 先于 `text_delta` 被丢弃

**根因**：reducer `stream/toolStart` 守卫 `s.draft ? ... : s`，首个 SSE 事件是 tool_start 时 draft 为 null → 直接跳过。

**修复** (`d10e31a`)：draft null 时从 `s.run` 种子创建最小 draft（`text: ""`）。

### P0 — 导航链接未更新（404）

**根因**：`df45d60` 把路由从 `/threads/[id]` 改为 `/conversations/[id]`，但 NavRail + ThreadList 的 6 处 `router.push` 仍指向旧路径。Review 只核了 diff 内文件，没跨组件查引用。

**修复** (`6df3f48`)：6 处 `/threads/` → `/conversations/`。

**教训**：删除/重命名路由时，必须全局 grep 跳转引用。diff 视野不等于完整影响面。

### P0 — agent memberId ≠ agentId（全局唯一 PK）

**根因**：member 表 `member_id TEXT PRIMARY KEY`（全局唯一）+ backfill 用 `agent-${thread.id}` 做 memberId。同一 agent 在不同 conversation 中 memberId 不同，但无法加入第二个 conversation（UNIQUE 冲突）。

**修复** (`d7c3f60` + `35f0752`)：
- member PK 改为 `(conversation_id, member_id)`
- agent memberId 永远等于 agentId
- `INSERT OR IGNORE` 幂等

**教训**：schema 设计时 PRIMARY KEY 语义要区分"实体全局唯一"和"关系内唯一"。member 是 conversation 内的关系，不是全局实体。

### P0 — 新建 thread 不创建 conversation（orphan）

**根因**：backfill 只在启动时跑一轮，运行时通过 NavRail/ThreadList 创建 thread 不建 conversation → `/conversations/:id` snapshot 返回 404 → roster 空。

**修复** (`22fd59d` + `030c8d8` + `825b4f8`)：
- NavRail/ThreadList 创建 thread 后同步调 `api.ensureConversation(thread)`
- 抽出共享函数避免两处重复

**教训**：backfill 是启动时一次性补偿，不能替代正常的创建路径。

### P0 — 多 agent 触发后锁提前释放，后续 agent 消息丢失

**根因**（两环）：
1. `postMessage` 只 fork `targets[0]!`，其他 addressed agent 静默跳过 (`c82d8c9`)
2. `forkRun` 无 try/finally → 失败后锁永不释放 (`c82d8c9`)
3. 修复后 fork 所有 targets，但第一个 run 完成时 `completeRun` 释放了共享锁 → 第二个 run 完成时 `onRunComplete` 找不到 conversation → D19 跳过 → 消息永不被持久化 (`8ca9d51`)

**修复**：
- `c82d8c9`：遍历所有 targets fork + try/finally 失败时释放锁
- `8ca9d51`：`pendingRuns` 计数器，最后完成的 run 才释放锁

**教训**：多 agent 并发时，"一把锁 + 多个异步完成回调"是竞态条件的温床。计数器模型比布尔锁更适合 N 个并发任务。

### P1 — 已加入的 agent 在 AddMember 弹窗中应置灰不可选

**根因**：AddMemberButton 只过滤掉已加入的 agent（隐藏），用户看不到"为什么这个 agent 不能加"。

**修复** (`4b62f1d`)：显示全部 agent，已加入的排底部、置灰、打 ✓，hover 提示。

## 八、Thread 概念消除 — 第一性原理重构 (`02ad1a6`)

### 问题

经过多轮 bug 修复后，thread↔conversation 的映射在各种边缘情况下持续断裂：
- 导航到 conversation thread `C:agentId` 返回 404
- 同一 agent 在多个 conversation 中时 thread 列表被 conversation thread 污染
- `ensureConversation` 补丁式地给每个新 thread 建 conversation，但仍无法解决"conversation thread 出现在 agent thread 列表"的问题

### 第一性分析

**Thread 不是用户概念。** Thread 是 agent 在 conversation 里的执行沙箱——checkpointer 需要 threadId 做 key，runner 需要 threadId 加载上下文。但 threadId 可以从 `conversationId:memberId` 推导，不需要持久化表。

**前端只应该知道 conversation。** "列出 agent 的会话"本质是"列出 agent 参与的 conversations"，应该 JOIN member 表而非查 threads 表。"创建会话"应该直接调 `POST /conversations`，而非 `createThread` + `ensureConversation`。

### 变更

| 删除 | 替换为 |
|---|---|
| `ThreadList` 组件 | `ConversationList` 组件 |
| `api.listThreads` / `createThread` / `getThread` / `updateThread` / `ensureConversation` | `api.listConversations` + `api.createConversation` |
| NavRail "Threads" 区 + `agentThreads` 过滤 | NavRail "Conversations" 区 + 直接 `api.listConversations` |
| NavRail "New Thread" → `createThread` + `ensureConversation` | "New Conversation" → `createConversation` 一步 |
| `toConversationPath` / `isThreadActive` 的 thread→conv ID 映射逻辑 | 直接用 `conv.conversationId`，无需映射 |

**后端新增**：`GET /api/conversations?agentId=` — JOIN member 表按 agent_id 过滤

**死代码删除**：
- `apps/web/src/components/ThreadList.tsx` — 文件删除
- `api.listThreads`, `api.createThread`, `api.getThread`, `api.updateThread` — 移除
- `api.ensureConversation` — 移除（"建 thread 同时建 conversation"的补丁不再需要）
- NavRail 中的 `agentThreads` 过滤、`toConversationPath`、`isThreadActive` 等 thread 映射逻辑 — 移除

### 残留

- `api.deleteThread` / `ThreadRow` 类型保留 —— `ConversationList` 的删除按钮仍用它。应替换为 `DELETE /api/conversations/:id`
- `threads` 表仍在后端（runner/checkpointer/forkRun 需要），但前端永不感知
