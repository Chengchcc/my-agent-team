## 二、关键架构决策

### 1. ContentBlock 随 Message 一起迁入 message 包

**决策**：`TextBlock | ToolUseBlock | ToolResultBlock` 从 `@my-agent-team/core` 迁入 `@my-agent-team/message/src/content-block.ts`，core 通过 re-export 保持兼容。

**理由**：ContentBlock 是 Message 的内容原子——Message.blocks 的类型。如果 ContentBlock 留在 core，message 包必须依赖 core，形成 message→core 依赖。反过来：ContentBlock 在 message，core 依赖 message，语义更正确——"内容格式属于消息模型"。

**结果**：`@my-agent-team/message` 零外部依赖，成为系统最底层类型包。`core` 的 `message.ts` 退化为 re-export 薄层。依赖方向干净：`message ← core ← framework ← backend/web/lark`。

### 2. Message 字段全 optional（除 role）

**决策**：`id?`、`state?`、`text?`、`blocks?` 等字段设计为 optional，而非拆成 `FrameworkMessage` / `DomainMessage` 两个类型。

**理由**：框架 run loop 构造 assistant message 时只知道 `role + blocks`（不知道 messageId 和 lifecycle）；backend projection 赋予 `messageId + state + updatedAt`。同一类型不同层填不同字段子集，好过又拆出两个"同一语义对象的不同名字"——那正是设计哲学要消灭的。

**代价**：`id` 是 optional 意味着 Web reducer 和 Lark watcher 需要防御 `message.id ?? ""`。这在 `getApprovalTarget` 和 `upsertAuthoritative` 中各出现一次。可以接受。

### 3. 原子替换，不保留旧 Message

**决策**：一个 commit 删 core 的 `Message` 和 `Role`，所有 ~40 个文件同步改为 import `@my-agent-team/message`。

**理由**：设计哲学第三原则——"不兼容旧技术债"。让新旧 Message 共存一个过渡期只会让系统继续存在多个消息本体，违背 M17.1 的根本目的。

**代价**：commit 很大（73 files changed），但每个文件的改动是机械的——`content: "str"` → `text: "str"`，`content: [...]` → `blocks: [...]`。typecheck 能精确捕获遗漏。

### 4. Revision 是 snapshot，不是 patch

**决策**：`MessageRevision` 携带完整字段集合，`mergeMessageRevision` 只是 field-level `?? base.field` 覆盖。

**理由**：spec 明确 "M17.1 选择 snapshot-only，避免引入 patch merge 复杂度"。流式输出场景每次 revision 携带完整 text（后端 `buildAssistantRevision` 总是构造全量），不需要 delta merge。

**结果**：`mergeMessageRevision` 实现简单——30 行的 spread + nullish coalescing。测试 3 个 case 覆盖所有场景。

## 四、与 spec 的主要偏差

### 4.1 ContentBlock 迁入 message 包（spec 未明确）

**spec 说**："ContentBlock 继续来自底层协议层，Message 复用它，不重新定义 RevisionContentBlock"。

**实际**：ContentBlock 直接迁入 message 包，core 退化为 re-export。这是 spec 的合理延伸——ContentBlock 是 Message 的内容类型，放一起更符合"统一本体"原则。这样做还避免了 message→core 的依赖方向，让 message 成为最底层零依赖包。

### 4.2 parseMessage（非 revision）未实现

**spec 6.3 节列了** `parseMessage(input: unknown): Message`。

**实际**：只实现了 `parseMessageRevision`。`parseMessage` 的语义不明确——Message 的字段几乎全是 optional，解析一个只有 `{ role: "user" }` 的对象没有验证价值。surface 层通过 `mergeMessageRevision(null, revision)` 创建 Message，不需要单独 parser。**有意省略，非遗漏。**

### 4.3 `messageId()` 命名函数未实现

**spec 6.3 节列了** `messageId(parts: MessageIdParts): string`。

**实际**：只实现了 `assistantMessageId(runId)`。通用 `messageId(parts)` 缺乏实际调用场景——目前所有 messageId 都是 `run:${runId}:assistant:0` 或系统事件 ID（`s-${seq}`）。M17.2 如果需要通用工厂再加。**YAGNI。**

### 4.4 Web action type 名 `ledger/message` 未改

**spec 7.4 节暗示** surface 不应理解 ledger 机制。

**实际**：`apps/web/src/lib/conversation-reducer.ts:40` action type 仍为 `"ledger/message"`——名字泄露了 ledger 机制。但这是 SSE transport 层面的 action 名称，影响前后端协议。改动需要同步改 backend `broadcastMessage` 的事件格式。**留给 M17.2（conversation API 命名全面转向 message-facing）。**

### 4.5 `projectForMember` 的 `formatContent` 仍兼容旧 shape

**spec 5 节明确**："不做旧 ledger content 的兼容解析"。

**实际**：`packages/conversation/src/index.ts:87-93` 的 `formatContent` 仍处理 `string | { text }` 两种 shape。但这是 **thread projection 路径**（给 agent checkpoint 喂消息），非 surface 路径。它的输入是 `projectForMember` 返回的 `{ role, text }` 对象——text 已经是 string。`formatContent` 的旧 shape 兼容实际上是死代码路径。**M17.2 清理。**

## 六、问题和教训

### 6.1 背景 agent 进程退出导致工作丢失

**问题**：3 个并行 background agent 在 Claude Code 进程退出后全部丢失——虽然它们实际上完成了工作（文件已写入磁盘），但无法返回结果通知。

**教训**：background agent 适合短任务。大规模迁移任务应使用 foreground agent 或 Workflow，避免进程退出损失协调逻辑。不过 TypeScript 类型系统作为安全网——即使 agent 不返回，磁盘上的文件改动也会被 tsc 验证。

### 6.2 测试文件 `content` 引用被遗漏最多

**问题**：typecheck 修复循环中，80% 的残留错误来自测试文件（`echo-model.test.ts`、`repair-tool-pairs.test.ts`、`progressive-skill.test.ts`、`lifecycle.test.ts`、`event-log/index.test.ts`）。agent 优先修了源码文件的 import 和类型引用，测试文件的 `{ role, content }` 字面量被批量遗漏。

**教训**：迁移指令应明确列出测试文件路径，或将测试文件的 `content`→`text`/`blocks` 替换独立成一步。

### 6.3 `s-${seq}` 命名的语义残留

**问题**：`apps/backend/src/main.ts:576` 的 `messageId: 's-${runId}-${role}'` 和 `apps/web/src/lib/conversation-reducer.ts:209` 的 `const id = 's-${a.seq}'` 仍使用 `s-` 前缀。

**分析**：这两处不是 parser fallback（spec 要消灭的目标），而是 ID 生成——backend 为 non-assistant 消息生成 synthetic ID，web 为系统事件生成唯一 ID。`s-` 前缀是历史惯例而非概念债。**M17.2 可统一成 `msg:`/`sys:` 前缀。**

### 6.4 turbo 缓存遮蔽了部分 typecheck 错误

**问题**：第一次 `bun run typecheck` 跑了 30+ 个任务但只报了一个错误（`echo-model.test.ts`）。修复后第二次跑报了 `repair-tool-pairs.test.ts` 的错误——第一次被 turbo cache 命中了部分包的旧结果。需要多次 `typecheck` 才能清空所有缓存层。

**教训**：大规模重构后应该用 `turbo run typecheck --force` 清缓存全量跑，避免被缓存遮蔽。
