## 二、按文件审查

### 2.1 `events-db-migrations.ts` — issue_event 表 ✅

`events_v15`（id=3014）建 `issue_event` 表，schema `{seq AUTOINCREMENT, issue_id, kind, payload, ts}` + 索引 `(issue_id, seq)`。

- `seq AUTOINCREMENT` 做全局排序键、`ts` 仅展示——不依赖时间戳排序，避免同毫秒乱序。
- 索引 `(issue_id, seq)` 同时支撑「按 issue 取所有事件」+ SSE 的 `seq > lastSeq` 增量。
- 迁移按 name 去重执行（`_migrations` 表已有），零风险。

**潜在问题：无。**

### 2.2 `types.ts` — IssueEventKind + IssueEvent ✅

```
IssueEventKind = "created" | "started" | "run.started" | "run.ended"
               | "deliverable.submitted" | "status.advanced" | "human.decided"
```

七种 kind 与设计 ⑦ 一一对应。`payload: Record<string, unknown>` 是宽 JSON 面——兼容未来的字段扩展和新的 kind，无需改 schema。

**潜在问题：无。**

### 2.3 `store.ts` — 5 个新方法 ✅

**`appendIssueEvent`**：`INSERT INTO issue_event` + `last_insert_rowid()` 返回 seq。`payload` 经 `JSON.stringify` 落库，与时序键分离。`ts` 总是 `Date.now()`（不收入参，杜绝调用方伪造时间戳）。

**`getIssueEvents(issueId, afterSeq=0)`**：`WHERE issue_id = ? AND seq > ? ORDER BY seq`。一个查询两用——`afterSeq=0` 是全量快照、`afterSeq=lastSeq` 是增量 SSE。`payload` 经 `JSON.parse` 还原。

**`getRunOriginsByIssueId`**：`WHERE issue_id = ? ORDER BY created_at`，用现成 `idx_run_origin_issue`。这是「issue→所有 run」的唯一缺失方法——⑧ 的核心接缝。

**`getRuns(runIds[])`**：`WHERE run_id IN (?)` 批量查，占位符拼接 `runIds.map(() => "?").join(", ")` + positional params——纯语法拼接、零注入风险。detail 端点用此法一次性补全所有 run 的终态，避免 N+1（Q2→A 规则）。

**潜在问题：无。**

### 2.4 `emit-issue-event.ts` — best-effort 旁路包装 ✅

```
try { opsStore.appendIssueEvent(...) }
catch (e) { console.error(...) }
```

- 返回 `void`，永不冒泡。所有 19 处调用方 fire-and-forget。
- reactor 测试显式验证「事件写入失败不阻塞起棒」——不变量成立。
- 吞错后只记 `console.error`——Timeline 偶有洞是接受的设计权衡（观测不能拖垮被观测的业务）。

**Observation 1（低）**：`emitIssueEvent` 捕获所有 Error 但吞掉了——没有区分「表不存在」（致命配置错）和「disk full」（临时故障）。前者会静默丢失所有事件而无人察觉。缓解：Timeline 只是观测旁路，不影响业务；`console.error` 至少会出现在日志里。未来可加 `stderr` 级别的首次失败告警（后置）。

### 2.5 `subscribe-issue-timeline.ts` — SSE 生成器 ✅

与 `subscribeIssues`（service.ts:80-122）一比一复刻：

- catch-up 阶段先 yield 所有已有事件，记录 `lastSeq`
- 长轮询：`getIssueEvents(issueId, lastSeq)` 增量取
- 30 轮无声 → `{ _heartbeat: true }`
- `pollMs === 0` → oneshot（给测试用）
- `signal.aborted` → 停止

**Observation 2（低）**：若未传 `opts` 或未传 `opts.signal`，`!opts?.signal?.aborted` 永远是 `!undefined === true`——无限循环。HTTP 上下文一定传 `{ signal: req.signal }`，生产不触发。但若未来复用此函数于非 HTTP 场景，记得防御。**不需要本棒修。**

### 2.7 `http.ts` — 8 处排放 + 3 个新端点 + 补偿回滚 ✅

**排放覆盖**（全部 19 处逐一核对）：

| kind | 调用方可数 | by 来源 |
|------|-----------|---------|
| `created` | http create:1 | — |
| `started` | http transition draft→planned:1 | — |
| `run.started` | reactor startStep:1 | — |
| `run.ended` | reactor onRunComplete:1 | — |
| `deliverable.submitted` | http submitDeliverable:1 + reject rework_feedback:1 | — |
| `status.advanced` | http transition:1 + http approve:1 + http reject:1 + reactor auto:1 + http rollback:1 | human/human/rework/reactor/revert |
| `human.decided` | http approve:1 + http reject:1 | — |

- `deliverable.submitted` 仅在 `!replay` 时记——幂等重放不刷重复事件。
- `status.advanced` 的 `by` 枚举完整覆盖：reactor/human/rework/revert——零遗漏。

**Timeline 端点**：
- `GET /timeline`：404 守卫 → 全量快照。数据同源 `getIssueEvents(issueId)`。
- `GET /timeline/events`：404 守卫 → `subscribeIssueTimeline` async generator → `sseResponse` 包装。SSE 与 M18.4 的 issue SSE 同款范式。

**detail 端点**：一次返回 `{issue, timeline, runs}`。`runs` 用 `getRunOriginsByIssueId`→`getRuns` 批量补 run 终态；缺失 run 行回退 `status: "unknown"`——优雅降级。

**P7-c 补偿回滚**：`deliverableSvc.submit` + `emitIssueEvent` + `onReviewRejected` 都在同一个 try 块里——任一步失败触发 CAS 回滚 `in_progress→in_review` + emit `status.advanced(by:"revert")` + 502。回滚也失败 → 500。对称补偿覆盖全 reject 路径。

**Potential edge case（极低）**：若 `submit` 成功但 `onReviewRejected` 失败，回滚后留下一条 `rework_feedback` 交付物（已被写入，不被删除）。但此交付物在 `in_review` 状态下存在——无害。下次真 reject 时，`deliverableSvc.submit` 的 upsert（`INSERT OR IGNORE (run_id, kind)`，注意 rework_feedback 的 `run_id: null` 不参与唯一约束）会再写一条新的。前一条 orphan 交付物是「观测噪声」——不违反不变量。**不建议修**——删除孤儿交付物需要补偿事务，而低频人工操作不值得。

## 三、设计评估

### 做对了什么

1. **三套事实物理隔离**。`issue_event`（按 issue 聚合工作事件）、`run_ops_event`（按 run 聚合执行细节）、`ledger`（按 conversation 聚合对话内容）——三张表、三种查询方法、三组类型。互不混入、无 JOIN、无共享列。边界清晰。

2. **埋点是旁路，不进控制流**。所有 19 处 `emitIssueEvent` 调用都在「业务动作已成功」之后——绝不在事件发射前提交业务，绝不让事件失败回滚业务。`emitIssueEvent` 返回 void、永不冒泡。

3. **分散埋点 + `by` 来源标注**。`status.advanced` 的 5 个调用点各自标注 `reactor`/`human`/`rework`/`revert`——不在 service 层埋点（拿不到触发来源语义），在各调用方旁路标注。零新抽象、零 AOP 框架。

4. **窄 schema + 宽 payload**。七种 kind 共享同一张表的 5 列——加新 kind 或扩 payload 字段零迁移。演化性压在 JSON 里，schema 稳定。

5. **⑧ 零新采集**。`getRunOriginsByIssueId` 只是反查已有数据——`run_origin.issue_id` 列在 M18.2 就已就位、M18.6 把 key 切为纯 `runId`。本棒未给 run 级观测加一行采集代码。复用 M16 存量，不发明新机制。

6. **P7-c 对称补偿**。把 `deliverableSvc.submit` 和 `onReviewRejected` 纳入同一 try/catch——reject 路径任一步失败都回滚到 `in_review`。M18.6 的补偿从「只覆盖 onReviewRejected」扩到「覆盖全部 reject 后置步骤」。

### 可改进

1. **`subscribeIssueTimeline` 无信号时的无限循环**（见 §2.5）。当前 HTTP 路径必有 signal，生产无害。但若未来复用，需防御 `opts?.signal ? !opts.signal.aborted : false`。

2. **前端降级模式缺 detail 交互**（见 §2.9）。`statuses.length === 0` 的平铺展示未传 `onOpenDetail`——简单补。
