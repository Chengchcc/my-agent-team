## 3. 可改进的

### 3.1 （P1）projection Phase 1/2 对 issue-driven run 写入多余 ledger 数据

**发现**：`projection.ts` 的 `opsStore` guard 放在 Phase 3 之前（line 241）——Phase 1 和 Phase 2 对 issue-driven run **仍然执行**。对于没有 accumulator 的 issue-driven run，Phase 1 会创建一个合成 terminal revision 写入 conversation ledger。

**根因**：spec P6 明确要求"放在 Phase 1 和 Phase 2 之后"，理由写的是"issue-driven run 的消息仍要正常落 ledger"。但实际上 issue-driven run **不走** onRunMessage 回调（main.ts 的 onRunMessage 里 `parseThreadId` 对 issue threadId 解析不到 accumulator），所以没有消息要落 ledger。Phase 1 创建的是一个**没有实际消息内容**的合成 empty terminal revision——这是噪音数据。

**影响**：对 issue threadId 映射到的 conversation 写入无意义的 terminal revision。不会导致崩溃（锁在 finally 里照常释放），但 conversation 的历史里会多出一行空消息。

**修复思路**：把 opsStore guard 移到 Phase 1 之前（line 168 之后），让 issue-driven run 彻底跳过整个 onRunComplete。Phase 1 和 Phase 2 对 issue-driven run 都不需要——没有 accumulator、没有 lock 要释放（issue run 不走 `lock.acquire`）。

```ts
// 移到 line 168 之后（parseThreadId 之后、Phase 1 之前）
const origin = opsStore.getRunOrigin(runId);
if (origin?.issueId) return;
```

这个 fix 需要同步修改 spec P6 的描述（去掉"Phase 1/2 仍执行"的说法）。

### 3.2 （P2）idempotencyKey 的 `[2]` 解析依赖格式约定

**发现**：`reactor.ts:86` 用 `origin.idempotencyKey?.split(":")[2]` 解析 fromStatus。这依赖 `if (fromStatus && issue.status !== fromStatus) return` 的短路——如果格式意外变化（如未来改 key 命名），`fromStatus` 会是乱值，但 guard 只会误跳过（不会错误推进），再加上后续 CAS 做最后防线。

**缓解**：issue-driven 的 idempotencyKey 格式是 `issue:<ulid>:<IssueStatus>:run`，其中 ulid 不含 `:`。在 IssueStatus 只有 4 个固定值的约束下，`[2]` 是确定性的。但它是隐式约定——没有类型层保证。

**建议**：不修（M18.2 不引入过度抽象），但加一行注释说明格式约定。M19 若转移表可配置，可考虑把 fromStatus 写入 run_origin 的独立列（而不是编码在 key 里）。

### 3.3 （P2）`_threadId` 未使用

**发现**：`reactor.ts:65` onRunComplete 的 threadId 参数前缀 `_` 标记未使用。supervisor 传了这个参数，但 orchestrator 通过 run_origin ≈ runId 反查一切，确实不需要它。

**建议**：保留（M18.3 拖拽可能用 threadId 做关联校验）。不加 `_` 前缀也无所谓，只是 style。

### 3.4 （P3）web typecheck 预存失败

**发现**：`IssueBoard.tsx` 的 `"unknown" as IssueStatus` 和 `s === "unknown"` 比较导致 TS2367。这是 M18.1 的遗留问题（"Other" 兜底列），与 M18.2 无关。

**建议**：M18.3 修，可以在 COLUMN_LABEL 里加 `unknown` key 或改成 string 索引类型。

## 6. M18.3 教训

1. **spec 的"Phase 1/2 仍执行"应重新审视**：issue-driven run 不走 accumulator，Phase 1 是噪音。设计 review 时应把每条"数据怎么来"画出来，而不只画"调用关系"。

2. **idempotencyKey 的语义编码**：`issue:<id>:<status>:run` 是隐式约定，没有类型层校验。M19 应在 run_origin 加 `from_status` 独立列。

3. **转移表 agentId 硬编码了 planner/developer/reviewer**：这些 agent 在真实部署里可能不存在。M18.2 校验了引用完整性（抛错），但没有"创建这些 agent"的工具——留给运维手工。M18.3 可以做个 seed/migration 自动建这三个 agent。

4. **web typecheck 预存失败不应跨里程碑携带**：M18.1 遗留的 `IssueBoard.tsx` TS2367 应该在本里程碑顺手修掉（改动一行）。下次 review checklist 加一条"检查 typecheck 是否全绿"。
