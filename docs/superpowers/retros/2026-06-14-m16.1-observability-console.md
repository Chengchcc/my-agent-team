## 代码审查发现的 bug（已修复）

1. **[P0]** `[object Object]` — `norm()` 不认 `{blocks, runId}` 信封，ContentBlock 数组被 `String()` 成字面量
2. **[P0]** 重试 run 用最老 attempt 误诊 — `getRunDetail` ASC 排序与 `diagnoseRun` 假设 `[0]`=最新冲突
3. **[P0]** Surface 计数全 "not instrumented" — 后端嵌套 payload `{watchers:{...}}` 与前端 `Record<string,number>` 类型错配
4. **[P1]** `isUnhealthyAgent` 漏判 `degraded` 状态
5. **[P1]** Cancel/recover 后 Overview 不刷新 — query key `["ops","agentRuntime",agentId]` 无法匹配 `["ops","agentRuntime",[...ids]]`
6. **[P2]** 历史 attempt transport 全显示 detached — live session 套用到所有 attempt
7. **[P2]** `listRuns` 端点静默丢弃 transport/heartbeat/traceId 过滤参数

## 经验教训

1. **前后端契约错配是最高发 bug 源**。Bug 1（`{blocks}` 信封）、Bug 3（嵌套 payload）、Bug 5（query key 形状）、Bug 7（参数未透传）本质上都是契约在某一端变了但另一端没跟。
2. **`[0]` 语义必须文档化**。Bug 2 的根因是列表页 DESC、详情页 ASC，而诊断函数假设 `[0]`=最新。这种跨多个函数/文件的排序约定很容易漂移。统一为 DESC 后用注释标注语义。
3. **类型撒谎比无类型更危险**。Bug 3 的 `counters: Record<string, number>` 类型出现在前端 `SurfaceOpsItem`，但后端实际产出的值是嵌套对象。TypeScript 强制了调用方的 `number` 访问方式，却掩盖了运行时 `[object Object]`。
4. **Query key 形状一致性**。React Query 的 prefix match 依赖数组段相等，Bug 5 说明了当 aggregation 查询和 per-item 查询使用不同 key 形状时如何导致失效遗漏。
