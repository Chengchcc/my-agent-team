## 二、关键决策

### 2.1 P7 架构决策：onRunMessage 新回调 vs 复用 onRunEvent

**选择**：新增 `supervisor.onRunMessage` 回调，与 `onRunEvent` 物理分离。

**理由**：
- `onRunMessage` 承载的是业务事实（critical，必须 await，失败上抛）
- `onRunEvent` 承载的是可观测副作用（best-effort，fire-and-forget）
- 两者的一致性等级不同——放在同一个回调数组里，调用约定无法区分

**代价**：supervisor 多了一个回调数组和一个注册方法。但这是正确的代价——把不同等级的东西混在一起才是债。

### 2.2 P6 分层决策：AgentFsLike 在 core 还是 tools-common

**选择**：下沉 `core`（L1 原语层），与 `Tool`/`Message`/`ChatModel` 并列。

**理由**：
- `AgentFsLike` 描述的是"Agent 能读写的文件系统长什么样"——这是 L1 领域原语
- `agent-fs`（L1 文件系统实现）和 `tools-common`（L2 工具层）都是它的消费者
- 如果留在 `tools-common`，`agent-fs` 依赖 `tools-common` → L1 反向依赖 L2 → 分层倒置

### 2.3 P8 字段归属：runId 进 LedgerEntry 本体？

**选择**：进本体，`z.string().optional()`。

**理由**：`runId` 是消息→运行的追溯链——这是领域概念，不是存储行附属。如果留在 backend 局部 `&` 扩展，lark-bot 和 web 永远拿不到这个字段。

### 2.4 P1 折中：run_done 路径的 onRunComplete 改为 fire-and-forget

**选择**：`run_finalized` 先发，业务 listener fire-and-forget，各自 catch。

**权衡**：`completeRun`（锁释放）在 listener 内部——如果 listener 整体抛错，锁由 `projection.ts` 的 `finally` 保证释放。但如果进程在 `finally` 之前崩溃，锁泄漏只能靠 reaper 兜底。这是已知的进程崩溃语义——不在本里程碑修复。

## 五、架构演进轨迹

```
M17.1–M17.3: 消息本体下压，revision 流 + wire codec 收口
M17.4:       机制层泄漏收口（EventLog 折叠、checkpoint 改名、assistantMessageId ordinal 预留）
M17.5:       链路一致性定级 + 包边界纠偏 + 事实归一 ← 当前
```

M17.5 是"机制边界 → 业务边界"的转折点：
- **之前**：修的是机制层泄漏（事件日志不该独立成包、checkpoint 表名被借用）
- **M17.5**：修的是业务层边界（谁写事实、谁发控制信号、哪个包拥有哪个契约）

## 七、原则验证

对照 `docs/architecture/design-philosophy.md` 三条铁律：

1. **统一本体，不复制语义** → P8（LedgerEntry/Member 单一本体）、P6（AgentFsLike 唯一规范定义）
2. **暴露业务，隐藏机制** → P7（event_log 退入机制层，ledger 是唯一业务事实载体）
3. **边界要硬，概念要少** → P5（删除单一消费者薄包）、P4+P11（并发锁收敛为单一闸门）

**无违反。**
