## 二、按文件审查

### 2.1 `dispatcher.ts`（新）— 单一调度入口 ✅

```ts
export function createRunDispatcher(deps: { supervisor; opsStore; now? }) {
  return {
    async dispatch(cause: DispatchCause): Promise<{ runId; attemptId }> {
      const { attemptId } = await deps.supervisor.startMainRun(…);
      deps.opsStore.insertRunOrigin({ …cause.origin, runId, originKind, createdAt });
      return { runId, attemptId };
    },
  };
}
```

核心决策正确：

- **薄**——只统一起 run + 写 origin 两步，不构 spec、不跑业务守卫（各域差异大）。正确的边界：调度层永远不知道「为什么起这一棒」的领域语义，只知道「起因枚举」。
- **`RunOriginInput = Omit<RunOriginRow, "originKind" | "createdAt" | "runId">`**——三个字段由 dispatcher 自填，调用方传的 origin 不带这些。这消除了「调用方在 origin 里写错 originKind」的可能性——起因永远是 kind 的一等字段，与 origin 分开传。
- **依赖方向干净**：`run/dispatcher.ts` 只 import `supervisor.ts` + `store.ts`，不依赖 Issue / conversation。三方引用它，反向依赖不会出现。
- **`opts` 类型 `Record<string, unknown>`**：弱类型但有意识——`opts` 是传给 `supervisor.startMainRun` 的第三个参数，其类型随 surface 不同（preloadedMessages 仅 mention 需要，surfaceContext 仅 orchestrator/mention 需要）。不强约束 opts 子字段避免了 dispatcher 需要了解各 surface 的 opts 差异。

**Observation 1（低）**：`opts` 的 `as Parameters<RunSupervisor["startMainRun"]>[3]` 类型断言在 `startMainRun` 签名变化时会静默漂移——编译期不会报错。缓解：① `startMainRun` 签名是稳定的基础设施，极少变化；② `opts` 最终经 `transport.send({ type:"start" })` 透传，runner-daemon 按需读取，多余字段无害。**不建议修**——为低概率事件引入"dispatcher 知道所有 surface 的 opts 子字段"的依赖违反 §3.3 的「薄」原则。

### 2.4 `main.ts` — 三处 `issue:` 短路移除 ✅

旧代码在 `onRunComplete`（:139）、`onRunMessage`（:160）、`onRunEvent`（:220）三处 `if (threadId.startsWith("issue:")) return`。移除后 issue run 的助手消息经 `onRunMessage` 写入 conversation ledger（Coding Thread 的物理基础），互斥由 Part B 的 `origin_kind` 承担。

**风险评估**：这是本棒最高风险变更。如果 origin_kind 互斥不到位，issue run 完成时 projection 的 Phase 3（@提及级联）会跟 reactor 的状态机同时反应 → 双驱动。缓解链：
1. `projection.ts` 已改判 `originKind === "orchestrator"` → 跳过 Phase 1 + Phase 3（B1/B2）
2. `reactor.ts` 已改判 `originKind !== "orchestrator"` → 跳过状态机（B1）
3. `onRunMessage` 无 guard → issue run 的 assistant message **正常写入** ledger（这是想要的行为——Coding Thread 的数据来源）
4. `broadcastMessage` 对 issue-side conversation 正常投影——agent member 的 thread-projection 会收到消息（这也是想要的行为）

**结论：安全**——两个 onRunComplete listener 互斥，onRunMessage 放行是正确的。

### 2.5 `render.ts` — Handlebars 重写 ✅

```ts
const hb = Handlebars.create();
const cache = new Map<string, Handlebars.TemplateDelegate>();
export function renderPrompt(template: string, vars: PromptVars): string {
  try {
    let tpl = cache.get(template);
    if (!tpl) { tpl = hb.compile(template, { noEscape: true, strict: false, knownHelpersOnly: true }); cache.set(template, tpl); }
    return tpl(vars);
  } catch { return template; }
}
```

沙箱设定合理：
- `noEscape: true`——prompt 是纯文本，HTML 转义会污染模型输入。
- `strict: false`——缺失变量回退空串，保留旧 render 语义。兼容现有模板。
- `knownHelpersOnly: true`——拒绝未知 helper 调用，引擎层堵死 `{{#include}}` / `{{#exec}}` 等。
- 独立 `Handlebars.create()` 实例——不污染全局 `Handlebars` 的 helpers/partials。
- 编译缓存——模板集合小且稳定（per-Project per-status ColumnConfig），LRU 不需要。
- 坏模板 catch 回退裸原文——不崩 run。

**Observation 3（低）**：`{{#each deliverables}}` 对大集合（如 1000+ 交付物）理论上可能使渲染变慢。但：① `{{#each}}` 是受数据规模控制的 O(n)，不是无限循环；② 交付物的数量受 Issue 生命周期限制（一棒=一个交付物），不可能到 1000。**实际风险为零**。

## 三、设计评估

### 做对了什么

1. **dispatcher 薄如纸**。只统一起 run + 写 origin 两步，不越界构 spec、不迁就各域的业务守卫。这是「调度层统一」的正确粒度——一次做一件事，不贪多。

2. **origin_kind 从约定升级到枚举**。旧逻辑靠 `issueId` 非空 / `surface` / `issue:` 前缀三个维度的组合推断「为什么起」——任何一个约定被破坏就分不清。新逻辑一个 `origin_kind` 枚举一步定案，读侧不再组合推断。

3. **互斥从前缀约定升级到结构分派**。三个 `issue:` 前缀 guard 被两个 `origin_kind` guard（reactor + projection 各一）替代。每条路径的「进/不进」由一个显式枚举决定，不被 threadId 的字符串格式影响。

4. **Handlebars 沙箱设计克制**。`noEscape` + `strict:false` + `knownHelpersOnly` + 独立实例 + 编译缓存 + catch 回退。六层防御，每层针对一个具体威胁。没有 over-engineer——prompt 场景不需要 `#partial`/`#layout`，就没注册。

5. **approval_posture 不硬编码闸门状态**。`HUMAN_GATES` 从「唯一判据」退为「无配置时的兜底」。闸门语义从代码级不可配升级为 per-Project per-status 可配——column_config 编辑 UI 新增一个 select 即可，不需要动 reactor。

6. **`auto_orchestrate` 端到端接线链完整**。project 列 → adapter 读写 → service 透传 → http schema → reactor deps → onRunComplete 守卫——六段无一遗漏（review 确认每段都到位）。

### 可改进

1. **前端 `LEGAL_TRANSITIONS` 与后端重复**（见 §2.10 Observation 6）。两份硬编码同步靠人记——应在公共包维护单一事实源。

2. **`IssueService.createIssue` 未建 issue conversation**。spec §9.3 设计的 `convPort` 注入 + 创建 conversation 未落地——Coding Thread 卡片依赖的 `/conversations/${issueId}` 页面能否正常加载取决于 conversation 是否已由其他途径创建（如首次 run 的 `broadcastMessage` 懒建）。实测中 `buildConversation` 返 null 时 `broadcastMessage` 早退——issue 助手消息落 ledger 但不会被投影到 thread-projection，也不影响 Coding Thread。但若 conversation 行一直不存在，`getConversation` API 会 404。**建议 M20 补 F2**——在 createIssue 中同步建 `origin='issue'` 的 conversation + human owner member。

3. **「编辑」按钮未接线**。详情 Sheet 的「编辑」按钮只有 `disabled={transitioning}` 绑定，没有 onClick 调 PATCH/PUT。这是一个**占位按钮**——spec §10.4 提到「编辑按钮（D11 PATCH）」但实现在本棒被搁置。缓解：删除功能可用。**建议 M20 接线编辑表单（Dialog 内嵌 form 或跳转编辑页）。**

4. **Token 用量未聚合**。spec §10.4 要求的详情面 Token 用量（该 issue 所有 run 的 prompt+completion 求和不含 reflect）在当前实现中是**缺失字段**——属性表有「预计完成」「描述」但没有「Token 用量」。数据源需要从 `run_insights` 聚合。**建议 M20 补**。

## 五、总结

1 个 commit，45 个文件，837 行净增。七 Parts 全部落地且独立可测——dispatcher (A) → 互斥 (B) → 富化 (C) → Handlebars + 配置 (D) → 只读工具 (E) → 投影接回 (F) → 前端 (G)。

核心架构决策兑现：
- **单一调度入口**：三入口收口 dispatcher，`origin_kind` 显式枚举
- **结构化互斥**：reactor/projection 各自按 origin_kind 分派，不再靠前缀约定
- **投影接回**：issue run 经 `onRunMessage` 入 ledger，靠 origin_kind 兜底不双驱动
- **有界 DSL**：Handlebars logic-less 引擎，六层沙箱收敛
