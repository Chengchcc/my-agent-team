## 二、设计决策回顾

### 做对的

1. **入站幂等 reserve→POST→confirm 三段式**。spec review 发现原伪代码的「先 POST、后 insert」窗口会漏重复触发 run。改为先 reserve（本地事务）、再 POST（跨进程）、后 confirm。语义上「宁可丢入站，不重复触发 run」。

2. **SSE watcher 复用 CLI conversation mode 的 SSE 行协议解析**。CLI 已有 `id:`/`event:`/`data:` 解析模板，lark-bot 的 sse-watcher 直接复制了这套已验证模式，不重造。

3. **`botDisplayName` 来自 agent 元数据，通过 registry spawn 传参**。不是环境变量、不是配置文件、不是 lark-cli profile 属性。干净：agent 创建时填入 → agentStore 持久化 → registry spawn 时附加 `--bot-display-name`。

4. **`render()` 先 JSON.parse 再分支**。Ledger 存的是 JSON-encoded string。render 覆盖四类：raw string → 直接返回、`{text}` → 取 `.text`、`ContentBlock[]` → join text blocks → fallback `[Unsupported content]`。不向 Lark 用户暴露 JSON 原文。

5. **`withLarkOrchestration` wrapper 保持 AgentService 纯净**。service 只管 agent 领域逻辑（CRUD + workspace），lark profile 初始化和 registry 通知是 L6 surface 副作用，在 composition root wrapper 中集中处理。不做「把 surface 副作用塞进 service」。

6. **lark-bot 与 runner daemon 进程拓扑完全对等**。都是 per-agent、都由 registry 管理、都只经 backend HTTP/SSE。registry 崩溃重启带退避、SIGTERM 优雅退出（不 SIGKILL）。dev 脚本不改——backend shutdown 时 registry.dispose() 自动级联清理。

### 做错的 / 待改进

1. **`ingest.ts` 的 `reserved()` 被调用多次**。最初用 `const reserved = db.transaction(...); if (!reserved().ok) ...; if (reserved().needCreateConv) ...`，每次 `reserved()` 重新执行事务，导致第二次调用时 `inboundExists` 返回 true 短路，`needCreateConv` 永远为 false。修复方式：`const reserveResult = db.transaction(...)()` 一次性调用并缓存结果。

2. **`bindings-sqlite.ts` 的 `getChatBinding` 未做 snake_case→camelCase 映射**。bun:sqlite 返回 DB 原始列名（`pushed_seq`），接口用 camelCase（`pushedSeq`），直接 `as ChatBinding` 类型断言导致 `pushedSeq` 永远是 `undefined`——所以 `updatePushedSeq` 测试失败。修复：显式映射每个字段。

3. **`tsconfig.json` 未 exclude 测试文件**。`tsc` build 把 `*.test.ts` 编译进 `dist/`，导致 `bun test` 同时跑 src 和 dist 的测试、出现双份。修复：`"exclude": ["src/**/*.test.ts"]` + `rm -rf dist` 清理。

4. **`botDisplayName` 迁移遗漏**。初始实现只加了 `lark_enabled/lark_app_id/lark_profile_ref` 三列，web UI 需要 `botDisplayName` 才发现要第四列。追加 migration 5007 + 全链路（domain/adapter/service/http/registry）更新。

## 五、Lark-CLI 能力 Spike 结论

> 2026-06-13 · lark-cli v1.0.53 · Feishu app `cli_aa8fe4d88af8dbdb` · bot name `小开`

| 验证项 | 结论 | 对实现的影响 |
|--------|------|-------------|
| `event consume` 输出格式 | **扁平 NDJSON**，每行一条 JSON，10 个字段：`type/event_id/timestamp/id/message_id/create_time/chat_id/chat_type/message_type/sender_id/content` | parser 接口 `LarkMessageEvent` 字段匹配，无需调整 |
| `mentions[]` 数组 | **不暴露**。lark-cli 的 Process hook (`BuildMentionKeyMap` + `ResolveMentionKeys`) 已把 mention key 转成 `@name` 内联进 `.content` | group @bot 判定**只能**用 `content.includes("@botDisplayName")`，没有 mention 数组可查 |
| stderr ready marker | `[event] ready event_key=im.message.receive_v1` | bootstrap 需 block 等待此 marker 后再宣布 running |
| `im +messages-send` 群聊 | `{"ok":true, "data":{"chat_id":"...","message_id":"om_..."}}` | sender 返回格式确认，`ok: true/false` 判定成功/失败 |
| `im +messages-send` P2P | 同上，P2P chat_id 可用 | 与群聊无差异 |
| `--idempotency-key` | **生效**。相同 key 两次调用返回相同 `message_id` + `create_time` | 出站必须传 `<conversationId>:<seq>` 作为幂等键 |
| `--as bot` | 以 bot 身份发送 | main.ts spawn 已传 `--as bot` |
| `config init --new` | 浏览器 OAuth 流程，输出含 `appId` + `appSecret: ****` | **不应用于 backend 请求生命周期**（阻塞 + 浏览器依赖），只用于人工初始化 |
| `config init --app-secret-stdin` | 非交互模式可用，`echo $secret \| lark-cli config init ...` 格式正确 | backend `larkProfileInit` 用此模式，secret 不进 argv |
| `event consume` 退出行为 | `--max-events N` 正常退出 code=0；SIGTERM 优雅退出；stdin EOF 视为优雅退出 | main.ts 必须保持 stdin 不关闭，父进程用 SIGTERM 停止 |
| interactive card `content` | **JSON string**（非 object），内含 `user_dsl`（二次 JSON-encoded） | parser 不做 JSON.parse，原样传入 `POST /messages` content |
| `senderDisplayName` | **不暴露**。NDJSON 只有 `sender_id`（`ou_xxx`），无 display name | `displayName` fallback 为 `sender_id` |

### 关键发现：`@bot` 判定只能靠文本匹配

lark-cli 的 Process hook 改变了输出形状——raw API 的 `mentions[]` 数组在扁平化过程中被消费，mention key 被替换为 `@name` 内联进 `.content`。这是 spec 与原架构文档（18-im-adapter，用 raw Lark API 描述模型）的关键偏离。M15 的 group @bot 判定设计已基于此发现重写（spec §六）。

## 七、不做什么（已知 Gap）

| 不做 | 原因 |
|------|------|
| 流式回显 | D19 是 run 后一次性写 ledger；流式需额外订 `/runs/:id/events` delta + 占位消息反复 edit |
| 出站 exactly-once | MVP 没有 Lark 发送侧 dedup token（`--idempotency-key` 防正常重放，崩溃窗口按 at-least-once） |
| 富交互卡片 | 出站 `render()` MVP 取纯文本；卡片回调作为新入站事件类型走同一入口 |
| 群内多 agent 协调 | 一个 Lark 群挂多 AgentMember 时需「出站 owner」选举 |
| 成员进出事件投影 | MVP 只消费 `im.message.receive_v1`，惰性建会话 |
| thread_id 投影 | 一个 chat_id = 一个 Conversation |
| 入站附件/图片 | 需 lark-cli 下载媒体 → 上传 AFS `/shared/` |
| 绑定 TTL 回收 | rule of three 未到 |
| webhook 入站模式 | 需监听端口，只取 `event consume` 出站长连接 |
| 自建持久入站队列 | backend 短暂不可达时只内存缓冲 + 退避 |
| E2E 集成测试 | 需要真实 Lark app + lark-cli 常驻环境 |
