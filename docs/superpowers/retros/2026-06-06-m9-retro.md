## 二、Review 发现（code review 12 findings）

| 严重度 | 数量 | 典型案例 |
|---|---|---|
| CRITICAL | 3 | events.db 缺表、线程锁泄漏、resume stub |
| HIGH | 2 | cancel 不标 aborted、rediscover 不重建 #active |
| MEDIUM | 5 | spawn 先于 INSERT、DB 连接泄漏、Zod 丢失、SIGTERM 漏杀 |
| LOW | 2 | EventBus 死链路、require() in ESM |

**根因分析：**
- **backend 重构是最大的风险面** — 12 个 bug 中 9 个在 backend（supervisor + service + http + main），因为同时改了执行模型（进程内→子进程）、实体模型（runs→run+attempt）、API 语义（SSE 绑 POST→202+GET /events）
- **跨文件契约断裂** — supervisor 和 service 之间没有显式契约（onRunComplete、run/attempt 表归属），全凭口头协调
- **mock 掩盖问题** — service 单元测试用 mock supervisor 不触发真实竞态（spawn/DL/exit handler 时序）

## 五、教训

1. **重构执行模型时，先写集成测试再改代码** — backend 从进程内→子进程的改动涉及 supervisor/service/http/main 四层，mock 单元测试只能测单层，真实竞态（spawn 先后、exit handler 时序、DB 连接共享）只在集成层暴露
2. **跨组件契约显式化** — supervisor 暴露的 `onRunComplete` 回调、`run`/`attempt` 表的归属 DB 应该在接口层明确，而非口头约定
3. **review 发现率 12/12 集中在 backend** — 说明 review 在复杂重构时不可省略。如果跳过 review 直接 push，3 个 CRITICAL bug 会导致生产 crash
4. **spec→commit 映射 + TDD 节省了大量返工** — 前置债务 3 commits 独立、EventLog 新包独立、runner-stdio 独立，每个 commit 可独立验证
