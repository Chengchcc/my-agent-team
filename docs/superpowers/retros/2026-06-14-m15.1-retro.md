## 二、关键架构决策

### 1. `surfaceContext` 放在 transport 消息而非 spec 内
**决策**：`surfaceContext` 作为 `HostToRunner.start` 的顶层字段，与 `spec` 并列，不嵌套在 `spec` 内。
**理由**：避免 surfaceContext 被持久化到 checkpointer（spec 会被保存）；`#fireReflect` 可以显式 delete 而非深拷贝后修改。
**结果**：正确。reflect run 不会继承 Lark tool，避免了 agent 自省时意外调用 `start_new_conversation`。

### 2. Daemon 通过环境变量获取 backendUrl
**决策**：`BACKEND_URL` + `BACKEND_AUTH_TOKEN` 环境变量，与 lark-bot 模式一致。
**理由**：daemon 不应知道 backend 路由细节；env var 是最简单的注入方式；`DevRunnerRegistry` 已用 `env: process.env` 透传。
**结果**：正确。无需修改 registry 代码。

### 3. `sendTextOnly` 与 conversation ingest 分离
**决策**：rebind 确认消息 "已开启新的对话" 使用直接 `lark-cli im +messages-send`，不经过 `POST /messages`。
**理由**：避免 bot 自己的系统消息进入 conversation ledger 污染 agent 上下文。
**结果**：正确。`send-text-only.ts` 是纯 Lark 出站通道。

### 4. Setup sessions 存内存态
**决策**：`LarkSetupManager` 用 `Map<string, LarkProfileSetupSession>` 存储，agent DB 只在 setup 完成时写入。
**理由**：spec 明确允许 M15.1 内存态；backend 重启丢失 pending session 安全（无半启用状态）。
**结果**：正确。`onComplete` 回调写入 agent DB + 拉起 lark-bot。

### 5. 卡片流式作为运行时投影
**决策**：`run_stream` 表在 lark-bot 本地 sqlite 中，conversation ledger 仍是最终事实源。
**理由**：卡片是 ephemeral UX；ledger 保证 web/Lark 跨 surface 最终一致。
**结果**：正确。安全阀 `canSkipFinalLedgerText` 只在 sendCard+updateCard 全部成功时才跳过 ledger 文本。

## 四、问题和教训

### 4.1 Spec 初版多处事实错误
**问题**：初版 spec §3.5 声称 lark-cli = 1.0.26、`config init --new` 不可用、PATCH as bot 未验证。
**教训**：写 spec 前必须先 spike 验证，不能在 spec 里推测环境状态。已在 grill 阶段修正。

### 4.2 `render.ts` 未及时更新
**问题**：D19 新格式 `{ blocks: ContentBlock[], runId }` 在实现时未同步更新 `render.ts`，导致 lark-bot 可能将新格式内容渲染为 `[Unsupported content]`。
**教训**：修改数据生产者（D19）时必须同时检查所有消费者。

### 4.3 `runner-protocol` dist 未及时重建
**问题**：修改 `messages.ts` 源码后，backend typecheck 仍因 dist/ 中旧类型而报错。
**教训**：修改跨包类型后必须 `bun run build` 更新 dist/，否则其他包看到的是旧类型。

### 4.4 测试 DB 共享导致 flaky test
**问题**：ingest.test.ts 原来所有 test 共享同一个 DB 文件，前一个 test 的 chat_binding 影响后续 test 的 needCreateConv 路径。
**教训**：每个 test 应使用独立 DB 文件（`testDbPath()` with counter），而非共享。

### 4.5 `idGen` 破坏已有测试
**问题**：新增 `idGen` 为 `ConversationServiceDeps` 的 required 字段，导致 `router.test.ts` 编译失败。
**教训**：新增 required dep 时必须 grep 所有调用点。
