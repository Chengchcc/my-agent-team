## 什么做对了

1. **纯函数不动这条边界守住了**。`deriveStatuses` / `deriveLegalMap` / `nextTransition` 一行没改，reactor 控制流也一行没改——只是把 `import { TRANSITIONS }` 换成 `columnConfigSvc.transitionsForProject()`。Orchestrator 核心功能零回归。

2. **ColumnConfig 模块独立、接口窄**。与 orchestrator 之间只靠 `transitionsForProject(projectId) → Transition[]` 一个出口耦合，测试可以 mock 一行函数。Adapter 用了 `ON CONFLICT ... DO UPDATE`（SQLite upsert），避免手写 CAS。

3. **threadId 派生干净**。`issue/service.ts` 从 `idGen()` 拿 `issueId`、立即拼 `threadId`，不依赖外部状态。`threadExists` 钩子随之一并删除——死代码没留。

4. **agentSvc.exists 比闭包方案好**。加在 AgentService 接口上、走 `port.findById` 单查，而不是在 main.ts 里 `list(false).some(...)` 全量扫。供 ColumnConfig 用，未来其他模块也能用它。

5. **SSE 复用 conversation 外壳**。`subscribeIssues` 的异步生成器签名、心跳 sentinel、`pollMs=0` one-shot 模式与 `subscribeConversation` 完全同构。`sseResponse` 一个字没改。

## 遇到的问题

### 1. `replace_all: true` 只匹配单引号 `'planned'`，漏掉了双引号 `"planned"`

`adapter-sqlite.ts` 的 INSERT VALUES 用单引号 `'planned'`（SQL），return 语句用双引号 `"planned"`（JS）。`replace_all: true` 的匹配串 `'planned'` 只替换了 SQL 里的值，return 语句的 JS 字符串没动。导致插入到 DB 是 `'draft'`，但返回对象仍是 `status: "planned"`——测试报 `Expected: "draft" Received: "planned"`。

**教训**: 用 `Edit` 做字符串替换时，双引号和单引号是不同目标；涉及 SQL 和 JS 混写的文件，最安全的做法是指定更大的上下文（包含双引号变体）作为 `old_string`，或者用 `Write` 全量重写。

### 2. 持久化测试 DB 导致跨运行污染

`reactor.test.ts` 用 `/tmp/test-orchestrator-issue.db` 这个固定路径，不是 `Date.now()` 时间戳。旧跑的 `status='planned'` 数据留存在 DB 里，覆盖迁移已执行过不会再跑，新跑创建的 issue 受旧数据干扰。

**教训**: 测试 DB 路径应加时间戳或 UUID（如 `service.test.ts` 的 `/tmp/test-issue-svc-${Date.now()}.db`）。reactor 测试之所以用固定名可能是因为多测试共享同一个 DB（`beforeAll` 创建一次），但清理不彻底。要么在 `beforeEach` 里清表，要么用时间戳路径每个 describe 独立。

### 3. commitlint header-max-length 卡了提交

`feat(backend): reactor reads Transition[] from ColumnConfig instead of TRANSITIONS constant for M18.4` — 101 字符，超了 100 限制。缩成 `feat(backend): reactor reads Transition[] from ColumnConfig for M18.4` 才过。

**教训**: 英文 commit message 容易写长，尤其涉及技术术语。先在编辑器里数一下字符，或者写完用 `git log -1 --format=%s | wc -c` 验。

### 4. 重复 ORDER 常量（DRY 违反）

`ORDER = ["draft", "planned", "in_progress", "in_review", "done"]` 在两个模块各定义了一次——`orchestrator/transitions.ts` 和 `column-config/service.ts`。Code review 发现这是一个维护风险：如果有人只改一处（比如中间插一个新 status），`LEGAL_TRANSITIONS` 和 `transitionsForProject` 派生的拓扑就会不一致。

**修复**: 让 `column-config/service.ts` 从 `../orchestrator/transitions.js` import `ORDER`，删掉本地重复定义。

**教训**: 跨模块的常量（尤其是“事实来源”级常量）必须只在一处定义。导入比复制定义更安全。

### 5. `ports.ts` 注释过期

`IssuePort.createIssue` 的 JSDoc 写 `初始 status = "planned"`，但 adapter 已改成 `'draft'`。代码是正确的，注释是错的。

**修复**: 改成 `初始 status = "draft"`。

**教训**: 改了实现要扫一眼引用该实现的注释和文档。这种过期注释不会导致类型错误或测试失败——只能靠人工 review 发现。

## 设计哲学对齐

- **统一本体、不复制语义**: ColumnConfig 不是新领域本体——它是一张配置表，无生命周期状态机。`ColumnConfigRow` 只是 `{status, agentId, promptTemplate}` 三个字段，不发明新语义。`draft` 是 `IssueStatus` 的新取值，不是新实体。
- **暴露业务、隐藏机制**: `subscribeIssues` 的轮询 diff 是机制，但暴露给外部的只有 `async *subscribeIssues()` 这一个生成器——消费方（SSE 端点、测试）看到的是 `AsyncIterable<IssueRow>`，不知道内部怎么比的 `updatedAt`。
- **边界要硬、概念要少**: ColumnConfig 模块与 orchestrator 的耦合只有一个函数 `transitionsForProject(projectId) → Transition[]`。模块内部可以自由重构 adapter/service，不影响 orchestrator。
