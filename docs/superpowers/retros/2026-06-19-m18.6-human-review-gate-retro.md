三条新增 + 一处 IIFE 包装。核心决策正确：

## 三、设计评估

### 做对了什么

1. **闸门是「停」，不是「新状态」**。`nextTransition` 一行短路避免状态空间膨胀。`in_review` 既是闸门也是正常列——看板、拖拽、ColumnConfig 全部复用。

2. **回退边与前进边物理隔离**。reactor 永远看不到回退边（`transitionsForProject` 只产前进边），杜绝「reactor 误走回退边」的可能性。这比「reactor 记着别走」的约定更可靠——是结构保证，不是纪律保证。

3. **幂等键改 `runId` 而非加轮次后缀**。轮次后缀需要额外状态（返工计数存在哪？），徒增概念。runId 已在手、已唯一，直接做键是最小增量。

4. **打回反馈复用交付物载体**。零新表、零新迁移、零新服务方法。`deliverableSvc.submit` / `listByIssue` / `buildPromptVars` 全部复用。⑥ 站在 ⑤ 的肩膀上——这是 spec §1 说的「不合并」的正确结果：⑤ 的机制恰好是 ⑥ 的载体。

5. **裁决走专门端点**。没有给通用 `transition` 端点塞「识别回退边 + 落反馈 + 起返工棒」的特例，保持了 transition 端点的「纯状态机 + 拖拽」语义。裁决的复合语义收在一个意图明确的端点里。

### 可改进

1. **端到端测试缺口**（见 §2.6）。「reject → rework_feedback 进 prompt」这条链路目前只有单元/集成覆盖，缺一条穿过 `renderPrompt` + `buildPromptVars` 的真库测试。**建议 M18.7 补**——那时 Timeline 事件源可同时验证整条链路。

2. **`BACKWARD_EDGES` 硬编码单条边**。spec 明确这是本棒范围（§5「不支持多级回退」）。未来多回退边时，`BACKWARD_EDGES` 扩数组即可，`LEGAL_TRANSITIONS` 合并循环自动覆盖——结构已预留扩展性。

# 附录：三轮 review→fix 闭环（首轮验收之后的真实故事）

> **首轮 verdict（`9876c1b`）说「mergeable, 0 critical, 0 high」——但它低估了。**
> 提交这份 retro 之后，又跑了两轮 review，发现并修了 **5 个 medium/high 级问题**。
> 链路：`9876c1b`（首轮 retro）→ `13c73a3`（轮二：4 修）→ `8b19496`（轮三：补偿回滚 + 不变量）。
> 这一节记真实闭环：为什么首轮漏了、后两轮补了什么、教训是什么。

## 六、为什么首轮会低估

首轮聚焦「**这棒新加的代码本身对不对**」——拓扑结构、键设计、裁决端点逻辑，逐文件读下来确实干净，三个 observation 都是真低危。但首轮**没系统问两类问题**：

1. **闸门语义有没有从 transitions 纯函数「漏」到所有消费方**。首轮验证了 `nextTransition` 短路闸门、reactor 不自动出 `in_review`——但没追：**还有谁会让 Issue 走 `in_review→in_progress`**？答案是「拖拽端点」和「列配置派生的前进表」两条旁路，首轮都没查。
2. **reject 路径的失败态是否对称**。首轮 Observation 1 提到「reject 三步非原子」但判「不建议修」——这判断对了一半：孤儿交付物确实无害，但首轮**漏了更严重的一态**——起返工棒失败时 Issue 卡在 `in_progress` 没有回退，用户看到「我点了打回，但它既没回到评审、也没真的重跑」。

教训：**「新代码干净」≠「闸门不变量全局成立」**。闸门是横切语义，验收必须枚举**所有能触发该转移的入口**（端点、reactor、派生表、前端拖拽），逐个确认闸门在每条路径上都成立——而不是只验证引入闸门的那个纯函数。

## 七、轮二（`13c73a3`）：4 处旁路漏洞

| # | 问题 | 旁路在哪 | 修法 |
|---|------|----------|------|
| 1 | **拖拽可绕过闸门反向回退** | `transition` 端点对 `in_review→in_progress` 不拦，前端拖回卡片即触发非法返工（不走裁决、不落反馈） | 端点加 `BACKWARD_EDGES.some(...)` 判定 → 409；前端 `IssueKanban` 反向拖拽 toast 拦截 + 不乐观更新 |
| 2 | **列配置派生表把闸门列也产成前进边** | `column-config/service.ts` 的 `transitionsForProject` 遍历 `ORDER` 相邻对，**没跳过 `HUMAN_GATES`**——`in_review→done` 被当普通前进边产出，reactor 可能自动越过人工闸门 | 派生循环加 `if (HUMAN_GATES.has(from)) continue` |
| 3 | **reject 时序：先转移后落反馈** | 原 reject 先 `submit(rework_feedback)` 再 `applyTransition`——并发双 reject 下，CAS 还没把状态锁住，两条反馈都落库 | 调成 `applyTransition` 先行（CAS 序列化并发 reject）→ 再落反馈 → 再起棒 |
| 4 | **`fromStatus` 守卫的空串语义** | reactor `if (fromStatus && ...)` 把空串当 falsy 一起短路，但语义应是「空串=origin 不可信，保守跳过」 | 显式 `if (fromStatus !== "" && issue.status !== fromStatus)` 区分空串与失配 |

①② 是同一个根因的两个面：**闸门只在 reactor 的 `nextTransition` 上短路了，但「谁能产生转移」不止 reactor**。拖拽走 HTTP `transition` 端点（绕过 reactor），列配置派生表喂给 reactor 的输入本身就含闸门边。两处都要补——这正印证了第六节的教训。

## 八、轮三（`8b19496`）：补偿回滚 + 不变量固化

轮二把 reject 时序理顺了，但轮三才补上**失败态的对称性**——首轮 Observation 1 真正该修的那一半：

1. **reject 起棒失败 → 补偿回滚到 `in_review`**。`onReviewRejected` 从 `void ...catch(log)`（fire-and-forget）改成 `await` + 失败时 `svc.revertReviewReject(issueId)` 把状态退回 `in_review`，返回 502；连回滚都失败才 500。用户语义恢复对称：**要么真的进返工，要么留在评审等重试，不会卡在「打回了但没动」的中间态**。
2. **`revertReviewReject` 走专门补偿方法**。绕过 `LEGAL_TRANSITIONS`（`in_progress→in_review` 是反向边、本就不在合法集），CAS 以 `in_progress` 为前置防并发——补偿语义和正常转移物理隔离，不污染合法转移表。
3. **`main.ts` 的 `onReviewRejected` 把「无 ColumnConfig」显式抛错**。原来 `startStep` 返回 `false`（无配置）被静默吞掉；现在 `if (!started) throw`，让上面的补偿回滚能接住——否则「起棒失败」根本传不到 catch。
4. **`insertRunOrigin` 固化不变量**：`issueId != null && fromStatus === "" → throw`。把「issue 驱动的 run 必带非空 fromStatus」从**约定**升级成**运行时守卫**——配合轮二的 reactor 空串守卫，两端闭合：写入端拒绝空 fromStatus，读取端遇空保守跳过。
5. **reactor 守卫再收紧**：`if (fromStatus === "" || issue.status !== fromStatus) return`——空串直接判不可信跳过（而非轮二的 `!== ""` 继续比较），与写入端不变量呼应。

## 九、闭环教训

1. **闸门/反向边是横切不变量，验收要枚举所有入口**。首轮只验了引入闸门的纯函数；漏了拖拽端点、列配置派生表两条旁路。**修法的可靠性来自结构（`BACKWARD_EDGES` 物理隔离、派生表 `continue` 跳过），但验收的完整性来自枚举入口**——两者缺一不可。

2. **「非原子」的风险评估要分态判**。首轮把 reject 三步非原子整体判「不建议修」，但其实**孤儿交付物（无害，可不修）**和**起棒失败卡死（有害，必须补偿）**是两态。混在一起评估就漏了后者。教训：评估复合操作的失败态，要逐个失败点列「这一步失败后系统停在哪、用户看到什么」，而不是整体下一个「低危」结论。

3. **fire-and-forget 的回调在「有补偿语义」时必须 await**。reject 起棒原本 `void ...catch(log)`——只记日志不回退。一旦这个异步动作有「失败要回滚」的语义，就不能 fire-and-forget，必须 `await` 才能在 catch 里做补偿。

4. **不变量要两端闭合（写入拒绝 + 读取保守）**。`fromStatus` 非空：写入端 `insertRunOrigin` throw、读取端 reactor 空串跳过。单端守卫会留缝——写入端漏写一次，读取端就拿到脏数据。两端都设防才真闭合。

5. **首轮 retro 不是终点**。这棒的真实质量曲线是 `9876c1b`（自评 mergeable）→ 两轮共修 5 个 medium/high。**首轮「mergeable」是过度自信的信号，不是质量证明**——后续 milestone 的验收应默认「首轮会低估横切不变量」，主动安排第二轮专查旁路。

**最终 severity 真实汇总**：首轮报 0H/0C/3L，实际闭环修复 **2 处 high（拖拽绕闸门、reject 卡死无补偿）+ 3 处 medium（派生表闸门、reject 时序、fromStatus 两端不变量）**。HEAD `8b19496` 全绿后方为真 mergeable。
