## 二、按文件审查

### 2.1 `transitions.ts` — 拓扑改造 ✅

三条新增 + 一处 IIFE 包装。核心决策正确：

- **`BACKWARD_EDGES`** 硬编码单条边、与前进边**物理隔离**——reactor 的 `nextTransition` 遍历的是 `transitionsForProject` 派生的前进表，回退边从不出现在那里。不靠「reactor 记着别走回退边」的约定，靠「回退边根本不在 reactor 能看到的表里」的结构保证。
- **`HUMAN_GATES`** 是 `Set<IssueStatus>`——`nextTransition` 短路一行 `if (HUMAN_GATES.has(from)) return undefined` 完成闸门语义。零新概念，零新控制流。
- **`LEGAL_TRANSITIONS` IIFE** 合并回退边到合法集。`if (arr)` 是防御性的（`deriveLegalMap` 保证所有 status 都有空数组占位），不增加运行时路径。

**潜在问题：无。** 这是本棒最干净的一块。

### 2.2 `reactor.ts` — 两行改键 + 切列 ✅

- **`idempotencyKey: runId`**：从 `issue:<id>:<status>:run` 改为纯 `runId`。语义上 run_origin 幂等单元就是「一个 run = 一行 origin」，runId 已经全局唯一。前缀 `issue:<id>:<status>:` 不参与任何索引或查询——纯调试糖，但调试已由权威列 `from_status` 接管。删掉前缀是 YAGNI 的正确应用。
- **`const fromStatus = origin.fromStatus`**：从 `split(":")[2]` 切到 R3 权威列。改键后 key 里不再有 status 可抠，这一步从「建议」变为「必须」，语义更干净。

**潜在问题：无。**

### 2.3 `http.ts` — 裁决端点 ✅

**reviewDecisionSchema**：zod `refine` 实现「reject 必填 note」——比手写 if 更声明式，错误信息自动生成 path 指向 `["note"]`。

**reviewDecision handler**：
- approve 路径：纯 `applyTransition(in_review→done)`，不起棒。正确——done 是终态。
- reject 路径三步：① `deliverableSvc.submit` 落 `rework_feedback`（先于 transition，保证 `startStep` 的 `buildPromptVars` 读到）→ ② `applyTransition(in_review→in_progress)` → ③ `void onReviewRejected?.(updated)` 起返工棒。顺序正确，与 M18.5 §3.7「提交先于读取」时序一致。

## 三、设计评估

### 做对了什么

1. **闸门是「停」，不是「新状态」**。`nextTransition` 一行短路避免状态空间膨胀。`in_review` 既是闸门也是正常列——看板、拖拽、ColumnConfig 全部复用。

2. **回退边与前进边物理隔离**。reactor 永远看不到回退边（`transitionsForProject` 只产前进边），杜绝「reactor 误走回退边」的可能性。这比「reactor 记着别走」的约定更可靠——是结构保证，不是纪律保证。

3. **幂等键改 `runId` 而非加轮次后缀**。轮次后缀需要额外状态（返工计数存在哪？），徒增概念。runId 已在手、已唯一，直接做键是最小增量。

4. **打回反馈复用交付物载体**。零新表、零新迁移、零新服务方法。`deliverableSvc.submit` / `listByIssue` / `buildPromptVars` 全部复用。⑥ 站在 ⑤ 的肩膀上——这是 spec §1 说的「不合并」的正确结果：⑤ 的机制恰好是 ⑥ 的载体。

5. **裁决走专门端点**。没有给通用 `transition` 端点塞「识别回退边 + 落反馈 + 起返工棒」的特例，保持了 transition 端点的「纯状态机 + 拖拽」语义。裁决的复合语义收在一个意图明确的端点里。

### 可改进

1. **端到端测试缺口**（见 §2.6）。「reject → rework_feedback 进 prompt」这条链路目前只有单元/集成覆盖，缺一条穿过 `renderPrompt` + `buildPromptVars` 的真库测试。**建议 M18.7 补**——那时 Timeline 事件源可同时验证整条链路。

2. **`BACKWARD_EDGES` 硬编码单条边**。spec 明确这是本棒范围（§5「不支持多级回退」）。未来多回退边时，`BACKWARD_EDGES` 扩数组即可，`LEGAL_TRANSITIONS` 合并循环自动覆盖——结构已预留扩展性。
