## What went well

1. **复用现有执行层** — CronJob 不发明新的起运行/写账本/记事件机制。dispatcher/supervisor/projection 三个核心文件各改 ~3 行（加 originKind + 隔离 + cronJobId），其余全部是增量 feature 代码。没有双写者漂移，没有第二份执行权威。

2. **Issue 套路直接投影** — threadId/自带会话/convPort 注入全部照搬 Issue，零学习成本。`conversationId = cronJobId` + `threadId = "<cronJobId>:owner"` 的派生模式与 Issue 的 `<issueId>:owner` 完全同构。

3. **事实与投影分离** — `cron_job` 表是事实，`Bun.cron` 句柄是投影。重启后按表重建，简单可靠。明确拒绝 OS crontab 形态，状态全在 DB。

4. **Review 流程守住质量** — 每个 Task 经过 implement → spec review → code quality review 三轮，抓住了一个关键 bug（fireKey 漂移导致 maxRetries 无穷）和若干 pattern 不一致。

## Key decisions

1. **Non-blocking retry**：supervisor 串行 await 监听器（`for...of`），retry 用 `setTimeout` + async IIFE 避免阻塞 ledger write/lock release 等关键监听器。

2. **fireKey 语义**：同一触发点（同一 cron 表达式同一次点火 + 所有重试）共享 `cronJobId:unixSecond` 为 key，retryCount 按此 key 计数。重试通过可选参数传入不重新计算。

3. **Async service**：`agentSvc.exists()` 返回 `Promise<boolean>`，service 方法改为 async。project service 是 sync（无 async 依赖），各按其需。
