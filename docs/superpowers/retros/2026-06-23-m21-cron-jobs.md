## 二、按文件审查

### 2.1 `domain.ts` — 触发型实体 ✅

```ts
export interface CronJobRow {
  cronJobId: string; name: string; agentId: string; cronExpr: string;
  prompt: string; enabled: boolean; timeoutMs: number; maxRetries: number;
  createdAt: number; updatedAt: number;
}
```

核心决策正确：

- **threadId 不存字段**——CronJob 的 threadId 是 `<cronJobId>:owner` 派生值，与 Issue 的 `<issueId>:owner` 同构。Issue 存 `threadId` 列是因为它的 threadId 在创建时由 issueId 组合而成需要落地；CronJob 每次 fire 时动态拼接，不存。JSDoc 已修正为 "derives threadId"。
- **`enabled` boolean-in-domain / integer-in-db**——与 project 的 `autoOrchestrate` 完全同构：adapter 层 `=== 1` 读、`? 1 : 0` 写。
- **`timeoutMs` / `maxRetries` 默认 0**——0 = 不启用对应策略，简单清晰。

### 2.4 `scheduler.ts` — 进程内 Bun.cron + 超时 + 重试 ✅

```ts
import type { CronJob } from "bun";
const handles = new Map<string, CronJob>();
```

核心约束全部满足：

- **仅 2 参形态**：`Bun.cron(job.cronExpr, handler)`，绝无 3 参 OS crontab 形态。
- **不重叠保证**：Bun.cron 自带「handler 落定后算下次」，与重试 fireKey 透传正交。
- **Non-blocking retry**：supervisor 串行 `await` 监听器（`for...of`），retry 用 `setTimeout` + async IIFE 避免阻塞 ledger write / lock release 等关键监听器。
- **fireKey 穿透**：`fire(job, fireKey?)`——cron tick 不传由 `Date.now()` 计算，retry 传入原 key 保持计数连续。ID 格式 `"<cronJobId>:<firedAtUnix>:run:<attempt>"`——带 attempt 后缀防 `idx_run_origin_idem` 唯一约束冲突。
- **per-job 看门狗**：`timeoutMs > 0` 时 `setTimeout` 调 `supervisor.cancel(runId)`，run 完成清 timer。`dispose()` 清 watchdogs + retryTimers + retryCounts 三个 Map/Set。
- **retry 事件复用**：`retry_requested` / `retry_started`（`RunOpsEventKind` 已有，M18.2），不需新增事件类型。

## 四、设计评估

### 做对了什么

1. **不发明执行机制**。dispatcher/supervisor/projection 三个核心文件各改 ~3 行，其余全部增量。CronJob 只新增「定时触发规则」一层语义，执行层、会话层、可观测层全部复用。没有双写者漂移。

2. **事实与投影分离**。`cron_job` 表是事实，`Bun.cron` 句柄是投影。重启后按表重建，绝不用 OS crontab 形态。`start()` / `register()` / `unregister()` / `dispose()` 四个生命周期操作清晰：事实先落表，投影后更新。

3. **threadId 派生不存储**。`<cronJobId>:owner` 与 Issue 的 `<issueId>:owner` 同构但 CronJob 不存 threadId 字段——每次 fire 时动态拼接。避免了 Issue 的 threadId 列（创建时组合）与 CronJob 的 threadId（触发时组合）的时间语义差异造成的数据不一致。

4. **超时分两层，各司其职**。per-job 看门狗回答「这条 job 不该跑超过 timeoutMs」，全局 reaper 回答「执行端是不是死了」。两者不替代——per-job 看门狗不能用 reaper 顶替（粒度不对），reaper 也不能删（看门狗依赖 daemon 能收 abort）。

5. **Non-blocking retry**。发现 supervisor 串行 await 监听器后，retry 退避改为 `setTimeout` 调度而非 `await sleep`——避免阻塞 ledger write 和 lock release 等关键监听器。这是实施中基于代码事实的正确设计决策。

### 可改进

1. **进程内定时器不跨重启**。后端宕机期间错过的触发不补跑，at-most-once 语义。如需 misfire 补偿，后续 milestone 可加 `lastFiredAt` + 启动追赶。

2. **retry 事件用原 runId**。`retry_started` 事件记录的 runId 是触发重试的那一棒（失败的），不是重试产生的新棒。对可观测性来说，用新 runId 更直接——当前需要在 idempotencyKey 上关联。但 ops 事件是追溯路径，不是主心智模型，不阻塞。

3. **cron 表达式 5 段限制**。当前仅支持标准 5 段 cron（分/时/日/月/周），不支持秒级精度。M21 接受此限制，若后续需要 6 段表达式只需改 zod regex。
