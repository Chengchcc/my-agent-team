## 审查总结

从 M18.2 的 250 tests/31 files 长到 333 tests/35 files，新增了 `project`、`column-config`、`deliverable` 三个 feature，前端看板从 `IssueBoard` 升级为 `IssueKanban`（含拖拽状态推进 + 详情 Sheet + SSE 实时刷新）。整体质量 **solid**，设计哲学 §3"统一本体、不复制语义"和 §4"边界要硬、概念要少"都守住了。以下是逐项发现。

## 正面确认

### 设计哲学合规
- **无新领域本体**：project/column-config/deliverable 都是机制层。Project 是归属标识（近乎纯机制，`domain.ts` 只有 20 行），column-config 是转移表配置投影，deliverable 是 agent↔agent 交接物——都不是像 Issue 那样的领域本体。
- **单一写者不破**：`applyTransition` 仍是 status 唯一写入口。`revertReviewReject` 绕过了 LEGAL_TRANSITIONS 但调的是同一个 `port.setStatus` CAS 物理入口。
- **from_status 独立列**（P2 提升）: run_origin 加了 `from_status TEXT NOT NULL DEFAULT ''` 列，根治了我 M18.2 retro 里指出的"idempotencyKey 隐式格式依赖"问题。`store.ts:102-107` 还有 invariant 校验——issue-driven run 必须带非空 fromStatus。这个防御做得比我建议的更好。

### 架构演进健康
- **columnConfigSvc 解耦转移表**：`transitionsForProject(projectId)` 让每个 project 可配自己的转移表（替代硬编码 TRANSITIONS），orchestrator 的 `startStep`/`onRunComplete` 签名几乎不动——这就是 M19 需要的扩展点。
- **O→S surfaceContext**：`startMainRun` 的 opts 里传了 `surfaceContext: { surface: "orchestrator", capabilities: ["submit_deliverable"], issue: {...} }`——让 runner 知道这是编排启动的 run，后续可据此做工具权限控制。干净。
- **`threadId = issue:<issueId>`**：M18.1 的 `threadId` 是用户输入字符串，M18.3+ 改成 `issue:<id>` 格式，issue service 自己推导。这个把 threadId 从"外部传入"收口到"内部派生"是正确方向——它降低了调用方的概念负担。
- **`Object.create(null)`**：`buildPromptVars` 用 `Object.create(null)` 构造 `byKind` 字典，防 prototype pollution（agent 可控的 kind 字符串注入 `__proto__`）。这个小细节体现安全意识。

### 测试密度
- **e2e-issue-lifecycle.test.ts**（253 lines）：覆盖了 planned→done 全链、rework 回退、agent 缺失、CAS 幂等、conversation 隔离——是真正的集成测试，跑在真实 DB 上。
- **issue/http.test.ts**（588 lines）：对每个端点都做了 request→response 的完整测试（含 400/404/409 错误路径）。测试量是 M18.2 我写的 17 tests 的 34 倍——远超预期。
- **reactor.test.ts 从 253 lines 长到 575 lines**：新增了 column-config 解析、deliverable 上下文注入、rework 裁决拒绝、review-decision 推进、status.advanced 事件发射、buildPromptVars 嵌套变量等覆盖。

## 发现的问题

### P1 — `subscribeIssues` 位置不当（设计债务，非 bug）

`issue/service.ts:80-122` 的 `subscribeIssues()` 是 500ms 轮询 + heartbeat 的 SSE generator。按设计哲学 §3"暴露业务，隐藏机制"，轮询是纯机制——它不表达 Issue 的业务语义。放在 service 里让它成了"域服务的一部分"，但实际上它是 HTTP 层的读模型刷新策略。更合适的位置是 HTTP handler 里（inline generator）或 runtime-ops 的 projection。

不过这是设计品味问题，不是 bug——功能正确、测试覆盖，aesthetic debt。

### P2 — `revertReviewReject` 绕过 LEGAL_TRANSITIONS

`issue/service.ts:68-76` 的 `revertReviewReject` 硬编码 `port.setStatus(issueId, "in_progress", "in_review", ts)`——绕过了 `LEGAL_TRANSITIONS` 校验。代码注释写清了是补偿性回滚、只用于"reject 起棒失败的回退"。但这个设计的隐患是：一旦有人误调这个方法（非 rework 回退场景），状态机就被旁路。

**建议**：加一个调用方校验（如要求 caller 传入的上一个 status 必须是 in_review），或者把 `revertReviewReject` 收进 orchestrator 内部（不暴露给 HTTP 层）。

### P3 — `subscribeIssues` 的 pollMs=0 退化行为不一致

`issue/service.ts:112`: `if (pollMs === 0) break; // one-shot for tests`。这个 mode 只在 `pollMs === 0` 时触发，但 HTTP handler `events()` 传入的是 `{ signal: req.signal }`（不设 pollMs），所以走默认 500ms 轮询。`pollMs=0` 的 one-shot 模式只被测试用到，但生产代码里有一条专为测试设计的路径——这是 code smell。

**建议**：把 pollMs=0 的 one-shot 行为移到测试 helper 里，或把 `subscribeIssues` 拆成 `listIssues()` + `pollIssues()` 两个方法。

### P4 — `review-decision` 端点的 opsStore 依赖未在 deps 声明

`issue/http.ts` 的 `reviewDecision` 和 `detail` handler 分别用了 `opsStore.getIssueEvents` 和 `opsStore.getRunOriginsByIssueId`/`getRuns`。这些 opsStore 方法是读操作（安全），但 http.ts 之前只依赖 `IssueService`，现在隐式依赖了 `RuntimeOpsStore`。`issueRoutes` 的签名没有体现这个依赖。

**建议**：把 `opsStore` 作为 `issueRoutes` 的第二个参数 opts 字段传入（类似 `onIssueCreated`），让依赖显式化。

### P5 — `IssueBoard.tsx` 被删但 TS2367 未回归

`IssueBoard.tsx` 整个文件被删，替换为 `IssueKanban.tsx`。`IssueKanban.tsx` 是全新实现（211 lines，含拖拽 + 列渲染），没有 TS2367 问题。`api.ts` 的 `IssueStatus` 类型可能也需要同步确认——如果 kanban 不再用 `"unknown"` 兜底列，那 M18.1 的 `"other"` 列需求可能丢失。

**建议**：确认未知状态的 Issue 卡片在 kanban 里的处理——是被静默丢弃了还是有 visible fallback。

### Observations（非阻塞）

1. **`renderPrompt` 的类型收窄**：`render.ts` 现在导出了 `PromptVars` 类型和 `RENDER_VARS` 正则，render 签名从 `Record<string, string>` 收窄到了结构化类型——比 M18.2 的裸 `Record` 更好。
2. **`columnConfigSvc` 无 singleton/缓存**：每次 `startStep` 都调 `transitionsForProject(projectId)` 从 DB 读，小数据量无影响，但高频下应该加缓存。不过这是 M19 的事。
3. **`submit-deliverable-tool.ts`**：runner 侧的新工具，让 agent 在编排跑时提交交接物（kind + fields + ref）。对 runner 的侵入是最小化的（仅仅多一个 tool），符合"机制低调"原则。
