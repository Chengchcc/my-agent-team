## Lessons

1. **"Render-into-context" 是 React 的定时炸弹。** 把 JSX 通过 `setState` 塞进 context——组件渲染 → 新 JSX 引用 → setState → provider re-render → 组件 re-render → ∞。用了 useMemo + useCallback + useEffect 三层防护仍然每渲染都炸（因为有 RQ mutation 对象这个隐藏脏依赖）。**最终解决是删除整个抽屉架构——比任何 workaround 都干净。** 教训：如果你发现自己在给一个 pattern 连续加 memo/ref/callback 仍兜不住，这个 pattern 本身就是错的。

2. **RQ v5 mutation 对象不稳定，文档不够醒目。** `useMutation()` 每渲染返回新对象——这是 RQ v5 的设计，但文档没有在"和 useEffect/useCallback 一起用"的场景里大声警告。`startRun`（对象）进依赖 → callback 每次重建 → 所有下游 memo 失效。修法：依赖 `startRun.mutate`（`.mutate` 是 stable function identity）。**第三方库的"对象 identity 不保证稳定"应该在接入评审时就标注。**

3. **死代码在 greenfield 项目里积累速度远超预期。** Patch[] 系统设计时预想的增量 DOM patch 从未实现；三个 export 函数从未被 import；BlockLike 是 ContentBlock 的手动副本。这些都不是"写错了"——是"写了但没被用"。TypeScript 不警告"exported but never imported"，ESLint 的 `no-unused-vars` 只覆盖局部变量。**每月跑一次 `grep -r <export-name> src/ | grep -v "export\|test"` 可以抓住这些。**

4. **埋在设计里的产品决策不可 review。** "Cream/brass/charcoal 暖手作"是 M12 的视觉方向——但它只是 CSS 变量，没有文档解释"为什么是暖色调、给谁看"。产品 review 判断受众（开发者/agent 作者）需要终端母语时才翻盘。**主题 token 不是设计文档——设计文档是说清"为谁、什么场景、什么感觉"的决策记录。** M13.1 把 VoltAgent DESIGN.md 落盘为视觉契约，此后任何改色必须先改契约。

5. **信息密度是入口页面最便宜的 ROI。** Agent 卡片从"名字 + 三个装饰点"变成"名字 + permission badge + 创建日期"（3 行 JSX），ThreadList 的项目从"Untitled Thread"变"带最后活跃时间 + 活跃脉冲点"（5 行 JSX）——两个改动加起来不到 10 行，但产品感受从"demo gallery"变成"可用工具"。**用户不需要更多页面，需要更多信号在已访问的页面上。**

6. **纯数据变换活过所有 UI 重构。** stream-ast（markdown 块分拆、buffer 管理）是 `(StreamAst, delta) → StreamAst` 的纯函数——不依赖 React、不依赖 SSE、不依赖 DOM。所以 cream→dark 换主题时没动它，drawer→center 改布局时没动它，inline style→CSS class 改动画时也没动它。**框架层的纯逻辑天然免疫 UI 层的 churn。** 反过来，useDeltaStream（EventSource + rAF + React state）在同一个周期里被大改了 3 次。
