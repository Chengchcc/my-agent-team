# M14.5 Spec — ReAct 中间过程折叠（reasoning trace as collapsed terminal log）

> M14.5 解决一个**前端语义错配**：一个 agent turn 在 ReAct loop 下会产生多轮消息——`assistant(带 tool_use)` → `user(tool_result)` → `assistant(带 tool_use)` → … → `assistant(纯 text，无 tool_use)`。其中**只有最后一条无 tool_use 的 assistant 消息是「结论」**，中间所有轮次都是「思考过程」（调工具、看结果、再决策）。当前前端把这些消息**平铺成一串独立气泡 + 一堆 ToolCallCard**，用户被迫滚过大量中间噪音才看到结论；更糟的是——**完成态的 turn 里 tool_result 根本不渲染**（`tool_use` 在 assistant 消息、`tool_result` 在紧跟的 `user` 消息，而 `renderContentBlocks` 只在单条消息内配对，跨不到下一条）。
>
> 根因：**前端按「消息」渲染，但用户认知按「turn」**。turn 是 ReAct loop 的一个完整推理单元，中间轮次是实现细节。把每条 assistant/tool_result 消息当成对等的顶层气泡，等于把 loop 的内部状态泄漏成 UI 噪音。
>
> 判定信号（框架原生，非前端发明）：**`assistant` 消息是否带 `tool_use` block** = ReAct loop 是否继续的退出条件本身（`framework/create-agent.ts` 里 `blocks.filter(b=>b.type==="tool_use")` 为空才 `return` 结束 loop）。所以「最后一条无 tool_use 的 assistant = 结论」不是前端猜测，是框架的终止语义。判定式 `Array.isArray(m.content) && m.content.some(b => b.type === "tool_use")` 100% 可靠。
>
> 修法（右解）：前端把**同一 turn 的连续消息聚合成一个气泡**——中间轮次（带 tool_use 的 assistant + 其 tool_result）塞进一个**默认折叠的 Collapsible「推理轨迹」**（VoltAgent 终端日志风），结论（最后一条纯 text assistant）**零容器全宽**渲染在折叠块下方。流式中（draft）自动展开看推理，完成后自动折叠留结论。
>
> 关联：[Timeline.tsx](../../../apps/web/src/components/Timeline.tsx)（turn 渲染入口）· [render-blocks.tsx](../../../apps/web/src/lib/render-blocks.tsx)（tool_use/tool_result 配对）· [conversation-reducer.ts](../../../apps/web/src/lib/conversation-reducer.ts)（UiMessage 模型）· [collapsible.tsx](../../../apps/web/src/components/ui/collapsible.tsx)（复用基座）· [M14 spec](./2026-06-09-m14-web-conversation.md)（D19 ledger 投影 / SSE 现状）· [M14.3 spec](./2026-06-10-m14.3-reflection-post-run.md)（runId/turn 语义对齐先例）。基线 commit：**`d12e0a5`**（origin/next，M14.3 + M14.4 已落地）。
>
> ⚠️ **编号说明**：`M14.4` 已被「agent-to-agent @mention 触发」(commit `a75fa97`，`service.test.ts:424` `describe("M14.4: triggerMentionedAgents")`) 占用。本前端折叠特性顺延为 **M14.5**，避免与已上线标签冲突。

## 一、定位

M14.5 是 **纯前端（apps/web）的渲染层重构**，不动后端、不动协议、不动数据流：

- 改 `apps/web/src/components/Timeline.tsx` — 渲染前先把 `messages` **按连续同发送者聚合成 turn 段**；agent turn 段交给新的 `<ReasoningTrace>`，human/system 维持原样。
- 新增 `apps/web/src/components/ReasoningTrace.tsx` — 一个 turn 段的渲染器：中间轮次（带 tool_use 的 assistant + 对应 tool_result）折进默认折叠的 Collapsible；结论（末条纯 text assistant）全宽渲染在下方。**跨消息**配对 tool_use↔tool_result（修当前 bug）。
- 新增 `apps/web/src/components/ToolStep.tsx` — 无边框的「单步」渲染（tool 名 + input + result 同一行块），替代 trace 内部的 ToolCallCard+ToolResultCard 三层嵌套。
- 改 `apps/web/src/lib/render-blocks.tsx` — 抽出 `collectToolResults(blocks[])` 工具函数供跨消息配对复用；`renderContentBlocks` 保留给非 turn 场景（防御性兜底）。
- 改 `apps/web/src/lib/conversation-reducer.ts` — `UiMessage` 不加字段（grouping 在渲染层做，保持 reducer 纯净）；仅导出一个 `groupTurns(messages)` 选择器。
- 改 `apps/web/src/components/DraftMessage.tsx` — 流式中也用 trace 外观（展开态），让「流式展开 → 完成折叠」视觉连续。

**核心不变量**：

- **判定唯一来源是框架终止信号**——`Array.isArray(m.content) && m.content.some(b => b.type === "tool_use")` 为 true = 中间轮次；turn 内最后一条「无 tool_use 的 assistant」= 结论。不引入任何后端新字段、不依赖 runId（ledger 无 runId）。
- **数据已就位，零新增订阅**——D19 已把一个 run 的每一轮 assistant/user 消息**逐条**广播进 conversation ledger SSE，全部落进 `state.messages`。M14.5 只重排渲染，不碰 `useConversation.ts` 的 SSE 订阅。
- **聚合是纯展示层**——reducer / ledger / 消息 id 不变；`groupTurns` 是 `messages → 渲染段` 的幂等纯函数，刷新后从 ledger 重建结果一致。
- **结论始终可见、不可折叠**——折叠的只有中间过程；结论永远全宽常驻。
- **修复 tool_result 渲染**——完成态 turn 里 tool_use 必须能配到下一条 user 消息里的 tool_result（当前跨不到，等于完成后工具结果消失）。

**M14.5 显式不做（边界）**：

- **不改后端 / D19 / ledger schema**——不给 `LedgerRow` 加 `runId`/`turn`/`step`（grouping 在前端按连续同发送者推断，见 §3.3）。
- **不改 SSE 订阅**——`useConversation.ts` 三条 EventSource（ledger / runs stream / runs events）一字不动。
- **不引入虚拟列表 / 分页**——沿用现有 `contentVisibility:auto` 虚拟化；turn 聚合不改变 DOM 顺序，IntersectionObserver 锚点逻辑兼容（见 §3.5）。
- **不折叠结论、不折叠 human/system 消息**——只折中间推理轮次。
- **不做「记住每个 turn 的展开状态」持久化**——展开态是组件本地 state，刷新回默认（完成=折叠）。奥卡姆，先上最小可用。
- **不动 reflect run 的渲染**——反思走独立 reflect runId、不进 conversation ledger（M14.3 已物理隔离），前端本就扫不到，无需特殊处理。

## 二、问题根因（M14.5 要消除的）

一个 agent turn 在 ledger 里的真实形态（D19 逐条广播，`senderMemberId` 全 = 该 agent）：

```text
ledger SSE 投影到 state.messages（一个 turn = 连续同发送者段）：

  seq=10  human    "帮我查下天气然后总结"          ← 用户消息（turn 边界）
  ┌─ seq=11  agent   assistant [text:"我查一下", tool_use:get_weather]   ← 中间轮次 1
  │  seq=12  agent   user      [tool_result: {...天气数据...}]            ← 工具结果 1
  │  seq=13  agent   assistant [tool_use:summarize]                       ← 中间轮次 2
  │  seq=14  agent   user      [tool_result: {...摘要...}]                ← 工具结果 2
  └─ seq=15  agent   assistant [text:"北京今天晴，建议……"]              ← 结论（无 tool_use）

当前前端渲染（Timeline 逐条 map）：
  气泡: "我查一下"
  ToolCallCard get_weather  ← 但 tool_result 在 seq=12 那条 user 消息里，配不到 → 完成后工具结果消失
  (seq=12 tool_result 这条：extractText 为空、renderContentBlocks 只渲 tool_use → 整条几乎空白)
  ToolCallCard summarize     ← 同样配不到 result
  气泡: "北京今天晴，建议……"

崩溃点 1  ← 用户要滚过 N 个中间气泡/卡片才看到结论；turn 越长噪音越多
崩溃点 2  ← 完成态 tool_result 跨不到 → 工具结果在历史里彻底不可见（流式时 tool_start/end 反而能看到，完成后消失，反直觉）
崩溃点 3  ← tool_result 那条 role=user 消息渲染成半空白气泡（extractText 空 + 只渲 tool_use=无）
共同根因  ← 前端按「消息」平铺，但 turn 才是用户的认知单元；中间轮次是 ReAct loop 实现细节
```

修复后：

```text
  seq=10  human    "帮我查下天气然后总结"              ← 原样
  ┌──────────────────────────────────────────────┐
  │ ▶ 推理轨迹 · 2 步 · get_weather, summarize     │  ← 默认折叠，单行 hairline 摘要
  └──────────────────────────────────────────────┘
  北京今天晴，建议……                                ← 结论，零容器全宽，常驻

  展开后（点击 ▶ 或流式中自动展开）：
  ┌ 推理轨迹 ────────────────────────────────────┐
  │ ▎ 我查一下                                     │  ← 中间轮次 text
  │ ▎ → get_weather   {input}                      │  ← ToolStep：调用
  │ ▎   ⤷ {…天气数据…}                            │  ← ToolStep：结果（跨消息配对）
  │ ▎ → summarize     {input}                      │
  │ ▎   ⤷ {…摘要…}                                │
  └────────────────────────────────────────────────┘
  北京今天晴，建议……
```

**turn 与渲染单元第一次对齐**：一个 ReAct turn = 一个折叠块（中间过程）+ 一段全宽结论。用户默认只看结论，需要时一键展开看 agent 怎么想的。

## 三、设计决策

### 3.1 判定式为何可靠：tool_use 是框架的 loop 终止信号本身

`framework/create-agent.ts` 的 runLoop：每轮把模型输出 `blocks` 组装成 `assistant` 消息 yield 出去，然后

```ts
const toolUses = blocks.filter((b): b is ToolUseBlock => b.type === "tool_use");
if (toolUses.length === 0) { /* run_end "complete" */ return; }  // ← loop 退出条件
// 否则执行工具 → 产生 tool_result（role:"user"）→ 下一轮
```

即「assistant 无 tool_use」**就是** ReAct loop 的收敛点。前端判定 `m.content.some(b => b.type === "tool_use")`：

- **有 tool_use** → 这是中间轮次（loop 还要继续）→ 折进 trace。
- **无 tool_use 的 assistant** → loop 终止 → 这是结论 → 全宽常驻。

这不是 UI 启发式，是把框架已有的终止语义投影到前端。流式下同样成立：`text_delta` 是临时打字机，最终落库的 `type:"message"` 事件带完整 `ContentBlock[]`（含 tool_use），D19 据此投影；ledger 里每条 assistant 消息的 content 数组是否含 tool_use 是确定的。

### 3.2 数据来源：复用现有 conversation ledger SSE，零新增订阅

> **纠正一个直觉误区**：曾设想「中间轮次要单独订阅 `/runs/:id/events`」。**不需要**。D19（`main.ts` onRunComplete）已把一个 run 的**每一条** assistant/user 消息**逐条** `appendLedgerEntry` + `broadcastMessage`，全部经 conversation ledger SSE（`/conversations/:id/events` 的 `message` 事件）进 `state.messages`。所以一个 turn 的所有中间轮次**早就在前端内存里**，只是被平铺了。

M14.5 只需在**渲染层**把这些已有消息重新分组。`useConversation.ts` 的三条 EventSource 一律不动。

### 3.3 grouping 策略：连续同发送者段（ledger 无 runId，只能这么分）

`LedgerRow` 字段 = `{seq, conversationId, senderMemberId, addressedTo, kind, content, ts}`，**没有 runId / turn / step**。且 D19 把一个 run 的 assistant 和 tool_result(role=user) 消息**全部**以 `senderMemberId = 该 agent` 写入。所以：

> **一个 turn = `messages` 中一段「连续的、同一 agent senderMemberId 的、kind=message 的」消息**。被 human 消息或 system notice 或换一个 sender 打断即为段边界。

`groupTurns(messages)` 算法（纯函数，渲染层）：

```ts
type Segment =
  | { kind: "single"; message: UiMessage }                 // human / system / 单条 agent
  | { kind: "turn"; sender: SenderRef; rounds: UiMessage[]; conclusion: UiMessage | null };

export function groupTurns(messages: UiMessage[]): Segment[] {
  const out: Segment[] = [];
  let i = 0;
  while (i < messages.length) {
    const m = messages[i]!;
    const isAgent = m.sender.kind === "agent";
    if (!isAgent) { out.push({ kind: "single", message: m }); i++; continue; }
    // 收集连续同 agent 段
    const start = i;
    while (i < messages.length &&
           messages[i]!.sender.kind === "agent" &&
           messages[i]!.sender.memberId === m.sender.memberId) i++;
    const block = messages.slice(start, i);
    // 段内最后一条「无 tool_use 的 assistant 文本」= 结论；其余 = rounds
    const lastIdx = findLastConclusionIdx(block);   // 见下
    const conclusion = lastIdx >= 0 ? block[lastIdx]! : null;
    const rounds = block.filter((_, k) => k !== lastIdx);
    out.push({ kind: "turn", sender: m.sender, rounds, conclusion });
  }
  return out;
}
```

`findLastConclusionIdx`：从段尾向前找第一条「array content 且不含 tool_use，或纯 string content」的消息（= 末条 assistant 文本结论）。注意 tool_result 那条是 `role` 语义上的 user、但 ledger 里 sender 仍是 agent——它的 content 是 `[{type:"tool_result"}]`，`extractText` 为空、不含 tool_use 但也无 text，不应被当结论；用「含 `type:"text"` 文本块」做正向判据更稳：

```ts
function isConclusion(m: UiMessage): boolean {
  if (typeof m.content === "string") return m.content.trim().length > 0;
  const blocks = m.content;
  const hasToolUse = blocks.some((b) => b.type === "tool_use");
  const hasText = blocks.some((b) => b.type === "text" && !!(b as {text?:string}).text?.trim());
  return !hasToolUse && hasText;
}
```

**边界**：若一个段没有任何 conclusion（极少：loop 被 interrupt / 末轮还带 tool_use），`conclusion=null`，全部进 rounds、trace 展示但无独立结论行——退化为「只有推理过程」，可接受。

### 3.4 跨消息配对 tool_use ↔ tool_result（修当前 bug）

当前 `renderContentBlocks` 只在**单条**消息内建 toolResults Map——但 tool_use 在 assistant 消息、tool_result 在**下一条** user 消息，单条内永远配不到 → 完成态工具结果不渲染。

修法：trace 渲染时把**整个 turn 段的所有消息**摊平，先全局收集 tool_result，再渲染每个 tool_use：

```ts
// ReasoningTrace 内：先聚合整段的 tool_result
const resultMap = new Map<string, {content: string; isError?: boolean}>();
for (const m of rounds) {
  if (Array.isArray(m.content)) collectToolResults(m.content, resultMap); // 抽自 render-blocks
}
// 再按消息顺序渲染：text 块 → ToolStep（tool_use + 查 resultMap 配 result）；tool_result 块本身跳过（已被上游 tool_use 消费）
```

`collectToolResults(blocks, map)` 从 `render-blocks.tsx` 现有逻辑抽出，单一职责、两处复用。

### 3.5 渲染：折叠用现有 Collapsible，trace 用 ToolStep 去三层嵌套

- 复用 `components/ui/collapsible.tsx`（已封装 `@base-ui/react/collapsible`），不自己写 grid-rows 动画。
- trace 内**不**用 `ToolCallCard`+`ToolResultCard`（各自带边框/各自可折叠 → card-in-card-in-collapsible 三层嵌套，丑且重）。新增**无边框** `ToolStep`：左侧一根 `--primary` 竖线（`▎`），`→ {name}` + input、`⤷` + result 紧凑两行，整段就是一条「终端日志」。
- 折叠态摘要单行：`▶ 推理轨迹 · {N} 步 · {tool 名去重 join}`，hairline 描边、`--mute` 文字、mono 字体。
- **流式（draft）**：`DraftMessage` 改为 trace 外观且**默认展开**（用户正盯着 agent 思考）；run 完成、authoritative 消息覆盖 draft 后，turn 段渲染为**默认折叠**。视觉上「展开的实时轨迹 → 收起成一行摘要 + 结论」连续过渡。
- **锚点兼容**：`Timeline` 的 turn 锚点（`extractAnchors` / IntersectionObserver / `id="turn-..."`）基于「相邻消息 sender 变化」。聚合后 DOM 仍按 segment 顺序产出，sender 变化点不变——把 `id="turn-..."` 挂到 segment 容器的第一个元素即可，锚点逻辑无需重写（§4.2 落地确认）。

### 3.6 为什么 grouping 放渲染层、不放 reducer

reducer 当前是「消息流 → 扁平 messages[]」的纯状态机，多处依赖（optimistic 替换、draft 清理、self echo）。turn 聚合是**纯展示派生**（`messages` 的幂等函数），放进 reducer 会污染状态、增加 action。放渲染层（`useMemo(()=>groupTurns(messages),[messages])`）符合「reducer 管事实、组件管呈现」的分层，且 SSR/刷新从 ledger 重建结果一致。

### 3.7 关键边界：一条 assistant 消息**同时**带 text 和 tool_use（边说边调）

> 用户尖锐地指出的核心边界：**一轮长文本（多个 text_delta）后又调了工具，这 text 和 tool_use 不是「一轮输出」吗？** —— 是的，它们**在同一条 assistant 消息里**。这正是判定式必须按「**消息级**」而非「step 级」设计的原因。

核对框架源码 `create-agent.ts`（L343–397）确认模型输出的装配方式：

```text
model.stream(...) 吐出一串 chunk，runLoop 累积进 ONE blocks 数组：
  chunk(text "我先看一下天气")   → text_delta(blockIndex=0)  ┐
  chunk(text "再决定要不要……")  → text_delta(blockIndex=0)  ├─ 累进 blocks[0] = {type:"text", ...}
  chunk(tool_use get_weather)    → mergeChunkIntoBlocks       ┘  blocks[1] = {type:"tool_use", ...}
  done
  ↓
  assistantMsg = { role:"assistant", content: [ {type:"text",...}, {type:"tool_use",...} ] }  ← 一条消息，两个 block
  yield { type:"message", payload: assistantMsg }   ← 整条作为一个 message 事件落库
  ↓
  toolUses = blocks.filter(tool_use) → 非空 → loop 继续（去执行工具）
```

**关键事实**：text_delta 是**临时预览**（流式打字机），它和最终 tool_use **本来就属于同一条 assistant 消息**——框架把一轮模型输出的所有 block（text + tool_use）装进**同一个** `content[]`，作为**一个** `message` 事件 yield、D19 作为**一条** ledger entry 落库。所以前端拿到的就是一条 `content=[text, tool_use]` 的消息，不存在「text 和 tool_use 是两条消息」的情况。

**这条「边说边调」的消息怎么归类？** 按 §3.1 判定式：它**含 tool_use → 是中间轮次 → 整条进 trace（折叠）**。它的 text（agent 边说的「我先看一下天气」）属于**思考过程的旁白**，不是结论——结论是 loop 收敛后那条**纯 text、无 tool_use** 的 assistant 消息。所以折叠完全正确：用户默认看不到「我先看一下天气」这种过程旁白，展开 trace 才看到。

落到 `ReasoningTrace`（§4.3）的渲染：一条中间轮次消息可能**同时**有 text 和 tool_use block，渲染时**两者都渲**——

```text
trace 展开内，一条「边说边调」消息渲染为：
  ▎ 我先看一下天气，再决定要不要……        ← extractText(m.content) 渲出 text 块
  ▎ → get_weather   {input}                  ← 同一条消息里的 tool_use 渲成 ToolStep
```

§4.3 的 `rounds.map` 已正确处理：先 `extractText(m.content)` 渲 text 旁白，再遍历 `content` 把每个 `tool_use` 渲成 ToolStep——**一条消息内 text 与 tool_use 并存，按 block 顺序铺开**，不丢任何一个。

**流式期间的衔接**（draft）：text_delta 实时累进 `draft.text`（用户看着「我先看一下天气」逐字蹦出），同时 tool_start 把工具名塞进 `draft.tools`（显示 `Running get_weather…`）。run 结束 D19 落库后，这条「边说边调」消息变成 authoritative，被归入所属 turn 段的 rounds，trace 折叠收起。**流式看到的 text 不会丢**——它落库进了 message 的 text block，展开 trace 仍在。

> 一句话回答用户：text 和 tool_use **就是同一轮（同一条 assistant 消息）的输出**，框架把它们装进同一个 `content[]`。判定按「这条消息含不含 tool_use」做——含，则整条（连同它的 text 旁白）归为中间轮次折叠；text 旁白在展开的 trace 里和它的 tool_use 一起按块顺序显示，一个都不少。

### 4.3 `apps/web/src/components/ReasoningTrace.tsx` —— 新增（核心）

职责：渲染一个 `turn` 段 = 折叠的中间轨迹 + 全宽结论。

```tsx
"use client";
import { useState } from "react";
import { Collapsible, CollapsibleTrigger, CollapsibleContent } from "./ui/collapsible";
import { ToolStep } from "./ToolStep";
import { MessageBubble } from "./MessageBubble";
import { Markdown } from "./Markdown";
import { collectToolResults } from "@/lib/render-blocks";
import { extractText } from "@/lib/timeline";
import type { TurnSegment } from "@/lib/conversation-reducer";
import type { ContentBlock } from "@/lib/api";

export function ReasoningTrace({
  segment, defaultOpen = false,
}: { segment: Extract<TurnSegment, {kind:"turn"}>; defaultOpen?: boolean }) {
  const [open, setOpen] = useState(defaultOpen);
  const { rounds, conclusion, sender } = segment;

  // 跨消息聚合 tool_result（§3.4，修当前 bug）
  const resultMap = new Map<string, { content: string; isError?: boolean }>();
  for (const m of rounds) if (Array.isArray(m.content)) collectToolResults(m.content as ContentBlock[], resultMap);

  const stepCount = rounds.reduce((n, m) =>
    n + (Array.isArray(m.content) ? m.content.filter(b => b.type === "tool_use").length : 0), 0);
  const toolNames = [...new Set(rounds.flatMap(m =>
    Array.isArray(m.content) ? m.content.filter(b => b.type==="tool_use").map(b => (b as {name?:string}).name ?? "") : []))]
    .filter(Boolean);

  return (
    <div className="my-1">
      {rounds.length > 0 && (
        <Collapsible open={open} onOpenChange={setOpen}>
          <CollapsibleTrigger className="w-full text-left flex items-center gap-2 px-3 py-1.5
            border border-[var(--hairline)] rounded-md hover:bg-[var(--canvas-soft)] transition-colors
            text-[11px] font-[family-name:var(--font-mono)] text-[var(--mute)]">
            <span className="text-[var(--primary)]">{open ? "▼" : "▶"}</span>
            <span>推理轨迹 · {stepCount} 步{toolNames.length ? ` · ${toolNames.join(", ")}` : ""}</span>
          </CollapsibleTrigger>
          <CollapsibleContent>
            <div className="border-l-2 border-[var(--primary)]/30 ml-1.5 pl-3 py-1 my-1 flex flex-col gap-1.5">
              {rounds.map((m) => {
                if (!Array.isArray(m.content)) return null;
                const text = extractText(m.content);
                return (
                  <div key={m.id} className="flex flex-col gap-1">
                    {text && <div className="text-[13px] text-[var(--body)]">{text}</div>}
                    {(m.content as ContentBlock[]).map((b) =>
                      b.type === "tool_use" && b.id
                        ? <ToolStep key={b.id} name={b.name ?? ""} input={b.input}
                            result={resultMap.get(b.id)} />
                        : null)}
                  </div>
                );
              })}
            </div>
          </CollapsibleContent>
        </Collapsible>
      )}
      {/* 结论：零容器全宽，常驻 */}
      {conclusion && (
        <MessageBubble align="left"
          name={sender.displayName ?? sender.memberId} kind="agent"
          content={extractText(conclusion.content) || (typeof conclusion.content === "string" ? conclusion.content : "")} />
      )}
    </div>
  );
}
```

> 注：结论沿用 `MessageBubble` 保持气泡一致性；若产品要「结论零容器」可换 `<Markdown>` 裸渲染（§五不变量 4 只要求「常驻可见」，容器形态留实现微调）。

### 4.5 `apps/web/src/lib/render-blocks.tsx` —— 抽出 `collectToolResults`

```ts
export function collectToolResults(
  blocks: BlockLike[],
  into: Map<string, { content: string; isError?: boolean }> = new Map(),
): Map<string, { content: string; isError?: boolean }> {
  for (const b of blocks) {
    if (b.type === "tool_result" && b.tool_use_id) {
      into.set(b.tool_use_id, { content: normalizeToolResultContent(b.content), isError: b.is_error });
    }
  }
  return into;
}
```

## 五、不变量（invariant）

1. **判定唯一来源**——中间轮次 ⇔ `Array.isArray(content) && content.some(b=>b.type==="tool_use")`；结论 ⇔ 段内末条「含 text 文本块且无 tool_use」的消息。无任何后端新字段参与。
2. **零新增订阅 / 零后端改动**——`useConversation.ts` 三条 SSE、D19、ledger schema 全不动；M14.5 改动文件均在 `apps/web/src`。
3. **聚合幂等纯派生**——`groupTurns(messages)` 对同一 `messages` 恒产出同一 segments；刷新从 ledger 重建一致。
4. **结论常驻可见**——折叠只作用于中间轮次；结论永远渲染、不可折叠。
5. **tool_result 完成态可见**——turn 内 tool_use 跨消息配到后续 user 消息的 tool_result（修当前完成态工具结果消失的 bug）。
6. **流式连续**——draft 展开态 → 完成折叠态，无闪烁、无消息丢失（draft 被 authoritative segment 覆盖，消息 id 不变）。
7. **锚点不回归**——turn 导航锚点数量/定位与改前语义等价（相邻 sender 变化点不变）。
8. **非 agent 消息原样**——human / system notice 渲染零变化。
9. **同消息内 text+tool_use 并存**——一条「边说边调」的 assistant 消息（content 含 text 块和 tool_use 块）被判为中间轮次整条折叠；展开 trace 时其 text 旁白与 tool_use 按 block 顺序都渲染，一个不丢。
