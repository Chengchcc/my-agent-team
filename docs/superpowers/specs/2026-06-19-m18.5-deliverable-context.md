# M18.5 Spec — 交付物结构化 + Issue 上下文累积 + 模板变量提取

> **Status:** Spec → Ready for implementation.
> **Baseline commit:** `027f6a7`（M18.1~M18.4 已落：Issue 本体、Orchestrator 可配置转移、threadId 派生、草稿态、看板拖拽/SSE 实时；M18.4 review 三处修复 P1/P2/P4 已并入）。
> **术语:** **Deliverable（交付物）= 某一棒执行器在干完时显式提交的结构化产物**——`{issueId, kind, fields: Record<string,string>, ref?}`：`kind` 标交付物类型（`plan`/`mr`/`review`/…），`fields` 是供下一棒提取的结构化小字段，`ref` 是**指针**（doc/MR 链接），大产物存引用不存全文。**Issue 上下文 = 创建信息（title/issueId）+ 历棒交付物**，是 `renderPrompt` 变量字典的来源。**指针化** = 交付物里大体量内容只落引用、不把整篇文档塞进后续 prompt 把上下文撑爆。
> **关联:** `docs/architecture/foundations/issue-workflow.md`（design 总图，本棒落地其 ④⑤）· `docs/architecture/foundations/issue.md`（Issue 本体）· `docs/architecture/backend/orchestrator.md`（current，本棒回填）· `docs/architecture/backend/run-supervisor.md`（surfaceContext → runner 注入工具的通道）· `docs/architecture/roadmap/future-work.md`（M18.5 行）· `apps/backend/src/features/orchestrator/`（render/reactor，本棒主战场）· `apps/backend/src/features/issue/`（http 新增交付物端点）· `apps/backend/src/features/runtime-ops/store.ts`（`run_origin` 反查 issueId/fromStatus）· `packages/core/src/tool.ts`（`Tool` 接口）· `packages/runner-daemon/src/`（`createSubmitDeliverableTool` 工厂 + `surfaceContext` 注入）· `packages/runner-protocol/src/messages.ts` + `apps/backend/src/features/run/supervisor.ts`（`surfaceContext` schema 扩展）· `apps/backend/src/main.ts`（接线 `buildIssueSpec`）· `apps/backend/src/infra/sqlite/migrations.ts`（交付物表迁移）。

## 1. 背景

[Issue 协作工作流设计](../../architecture/foundations/issue-workflow.md) 把现状诊断为「建即跑、写死、单向、看不见」。M18.4 之后，「写死」已解（列可配置、绑真实 Agent、转移从 DB 读），「建即跑」已解（草稿态 + 人工入列）。但还剩一处核心缺口：**上一棒的产出只能靠扫 Agent 的自由文本喂给下一棒**——这正是 @提及自动流的老毛病（模型换个说法、少打个标记，链路就断）。本棒摘取设计中**互相咬合、构成「让一棒喂下一棒」最小闭环**的两块：

- **交付物结构化捕获（④）**：给 Agent 一个 `submitDeliverable` 工具，让它在干完时**显式提交**结构化交付物（planner 交付计划、developer 交付 MR、reviewer 交付 review）。交付物落库并**指针化**——`fields` 存结构化小字段、`ref` 存大产物的引用。这把「下一步是什么」从模型的措辞里彻底拿走。
- **Issue 上下文累积 + 模板变量提取（⑤）**：有了结构化交付物，`renderPrompt` 的变量字典就能从**累积的 Issue 上下文**（创建信息 + 历棒交付物）构建，支持 `{{deliverables.plan.summary}}`、`{{deliverables.mr.url}}` 这类按路径提取。`renderPrompt` 仍是**纯 `{{}}` 字符串插值**，不引入模板 DSL——变化只在「变量字典从哪来」。

这两块为什么必须一起做、而非各成一棒：交付物结构化（④）是「产出端」，模板变量提取（⑤）是「消费端」——只落④没有消费方，落了等于没接通；只做⑤没有结构化产出可取，变量字典里除了 `title`/`issueId` 仍是空。两者合起来才完成「planner 交付 → developer 取用」这一条端到端链。它们共同触碰的文件也高度重叠（reactor 起步时既要注入工具又要构建上下文、render 既被④的字段喂养又被⑤的路径解析消费），拆开会反复改同一处。

## 2. 当前代码事实

逐项已与 `027f6a7` 代码核对：

1. **`renderPrompt` 只做单层 `{{var}}` 插值、变量字典只有两个键**：`orchestrator/render.ts` 全文即 `renderPrompt(template, vars: Record<string,string>) = template.replace(/\{\{(\w+)\}\}/g, (_, k) => vars[k] ?? "")`，顶部注释明确「Deliberately no conditionals, loops, or filters — no template DSL」。reactor 调用处只喂 `{ title: issue.title, issueId: issue.issueId }`（`reactor.ts` `startStep` 内 `renderPrompt(t.promptTemplate, { title, issueId })`）。**本棒：正则从 `\w+` 扩成 `[\w.]+`、加按 `.` 路径的嵌套查找，`vars` 类型从 `Record<string,string>` 拓宽为可嵌套；但「纯字符串插值、无 DSL、缺失回退空串」三条不变（[issue-workflow](../../architecture/foundations/issue-workflow.md) 不变量 5）。**

2. **`reactor.startStep` 构建变量字典处是上下文累积的唯一注入点**：`orchestrator/reactor.ts` `startStep(issue)` 里 `const prompt = renderPrompt(t.promptTemplate, { title: issue.title, issueId: issue.issueId })`。**本棒：在此处把字典换成「创建信息 + 该 Issue 历棒交付物」构建的嵌套字典**——`startStep` 需新增一个「读该 Issue 全部交付物」的依赖（注入 `deliverableSvc.listByIssue(issueId)`），纯函数 `nextTransition`/`renderPrompt` 形态不动（仍只在 `startStep` 一处喂数据）。

3. **`run_origin` 已能从 runId 反查 issueId + fromStatus**：`runtime-ops/store.ts:13` 的 `RUN_ORIGIN_COLS` 含 `issue_id AS issueId`；`getRunOrigin(runId)`（`:85`）返回整行含 `issueId` 与 `idempotencyKey`；reactor 起每棒时写 `idempotencyKey: issue:${issueId}:${status}:run`（`reactor.ts` `startStep`），`onRunComplete` 已用 `origin.idempotencyKey?.split(":")[2]` 解出 `fromStatus`。**因此「这次 run 属于哪个 Issue、是哪一棒（fromStatus）」对 backend 完全可反查——交付物提交端点只需收 runId 即可定位 `(issueId, fromStatus)`，无需 Agent 自报。**

## 3. 第一性原则

### 3.1 「下一棒读什么」必须从结构化交付物来，不从自由文本扫

@提及自动流的教训是：靠扫 Agent 输出的自由文本传递「下一步」，模型一换措辞链路就断（[issue-workflow](../../architecture/foundations/issue-workflow.md) ④）。本棒立的契约是：**上一棒的产出经 `submitDeliverable` 显式、结构化地落库；下一棒读的是这份结构化交付物，不是上一棒"说了什么"。** 这条契约把「下一步是什么」从模型的措辞里彻底拿走——这是不变量 4。

### 3.2 交付物指针化——大产物存引用，不塞进后续 prompt

交付物里典型有两类内容：少量供提取的结构化字段（计划摘要、MR 标题/URL、review 结论）和大体量正文（整篇计划文档、完整 diff）。若把大正文也累积进 Issue 上下文、再插进后续 prompt，上下文会被迅速撑爆。所以**`fields` 只存小字段、`ref` 存大产物的指针（doc/MR 链接）**——后续 prompt 取 `{{deliverables.mr.url}}` 这类小字段，需要全文时顺指针去取。这是「指针化」的全部含义，也是上下文不爆的根。

### 3.3 交付物归属 `(issueId, fromStatus)`，且由 backend 反查、不由 Agent 自报

「这份交付物是哪个 Issue、哪一棒产的」是机制事实，不该让 Agent 在工具入参里自报（自报会错、会被模型省略）。M18.2 起每棒的 `run_origin` 已记 `issueId` 与 `idempotencyKey=issue:<id>:<status>:run`（§2.3），故 backend 收到「runId + 交付物内容」即可反查 `(issueId, fromStatus)`。**`submitDeliverable` 工具入参只含交付物本身（`kind`/`fields`/`ref`），归属由 backend 据 runId 补全。** 这与 §2.3 的反查能力对齐，也让工具入参面最小。

> **首版简化（§5 边界）**：工具的 `surfaceContext.issue` 仍携带 `issueId`/`fromStatus` 作为**幂等键与快速定位**之用（避免每次都查 `run_origin`），但权威归属以 backend 反查为准——两者一致。携带它是为让回调端点幂等键稳定（`issue:<id>:<fromStatus>:deliverable`），不是把归属决定权交给 Agent。

### 3.4 交付物工具走 HTTP 回调，复用 surface 工具注入通道

runner-daemon 是独立进程，工具够不到 backend DB（§2.5）。所以 `submitDeliverable` 必须像 `start_new_conversation` 一样是**工厂 + 闭包 + HTTP 回调**：`createSubmitDeliverableTool({ backendUrl, backendAuthToken, issueId, fromStatus, runId })`，`execute` POST 到 backend 新端点。注入路径也复用现成的 `surfaceContext → extraTools` 机制（§2.6），只是判定条件从「lark + start_new_conversation」扩出一条「带 issue + submit_deliverable」。

> **surface 取值**：issue run 不是 lark/web/cli 任一外部端，它是 Orchestrator 驱动的内部执行。给 `surface` enum 加 `"orchestrator"` 一档，语义清晰且与 reflect 剥离逻辑（§2.9，按 `spec.mode==="reflect"` 判，不按 surface）不冲突。注入条件写成 `sc?.capabilities.includes("submit_deliverable") && sc.issue && spec.mode !== "reflect"`，不强绑 surface 字面值，避免脆性。

### 3.5 `renderPrompt` 只长出「路径取值」，不长出 DSL

变量字典从「两个写死键」扩成「创建信息 + 嵌套交付物」后，模板要能写 `{{deliverables.plan.summary}}`。实现只需两点最小扩展：正则 `\w+ → [\w.]+`，匹配到的 key 按 `.` 拆段、在嵌套对象里逐层取值，任一层缺失回退空串。**不引入条件/循环/过滤器/默认值语法**——保持 [issue-workflow](../../architecture/foundations/issue-workflow.md) 不变量 5（仍是字符串插值、无模板 DSL）。这是「变量字典从哪来」的演进，不是模板引擎的演进。

### 3.6 交付物是追加事实，但本棒只读最新——上下文按 kind 聚合

一个 Issue 的同一棒理论上可被返工重跑（M18.6 ⑥，本棒非目标），故交付物表设计成**追加式**（每次提交一行，带 `created_at`）。但本棒的上下文构建只需「每种 kind 的最新一份」：`listByIssue` 后按 `kind` 聚合、同 kind 取最新，拼成 `deliverables.<kind>.<field>` 字典。这样表结构对未来返工友好（不丢历史），而本棒消费逻辑保持简单（最新即有效）。

### 3.7 提交在 run 内、推进在 run 后——天然有序，无需额外同步

交付物提交是 run 期间的一次工具调用（HTTP 同步完成、落库），run 成功后 `onRunComplete` 才推进到下一棒、下一棒 `startStep` 才读交付物。提交必然早于读取，无竞态。**本棒不需要任何「等交付物就绪」的同步原语**——时序由「工具调用先于 run 终态、run 终态先于下一棒起步」自然保证。若 Agent 没调 `submitDeliverable`（最佳努力、非强制），下一棒读到的对应 kind 为空、`renderPrompt` 缺失回退空串，链路降级但不崩。

## 4. 目标

1. 新增 **Deliverable 本体**：域/端口/SQLite 适配器/服务（独立模块 `features/deliverable/`，照 project/column-config 粒度），追加式落库，按 `(issueId, kind)` 可取最新。
2. 新增 **`submitDeliverable` 工具**（`packages/runner-daemon/src/submit-deliverable-tool.ts`，HTTP 回调范式），经 `surfaceContext` 注入 issue run 的 `extraTools`。
3. 扩展 **`surfaceContext` schema**（protocol + supervisor 两处镜像）：`capabilities` 加 `"submit_deliverable"`、新增可选 `issue?: { issueId; fromStatus }`、`surface` enum 加 `"orchestrator"`。
4. 新增 **交付物回调端点** `POST /api/issues/:id/deliverables`：收 `{kind, fields, ref?, fromStatus?, runId?, idempotencyKey?}`，backend 据 runId 反查/校验归属后落库；幂等键 `issue:<id>:<fromStatus>:deliverable`。
5. **`startStep` 注入交付物工具 + 构建累积上下文**：起 run 时传 `surfaceContext.issue`；`renderPrompt` 变量字典从「创建信息 + 历棒交付物（按 kind 聚合最新）」构建。
6. **升级 `renderPrompt`**：正则 `[\w.]+` + 嵌套路径取值，`vars` 类型拓宽；保持纯字符串插值、缺失回退空串、无 DSL。
7. 交付物表迁移（id 5012、`backend_v26_deliverable`）。
8. 测试收口（deliverable service 真库、render 路径解析、reactor 注入 surfaceContext + 构建上下文、http 端点 + 路由顺序、submit-deliverable-tool HTTP 契约）。

## 5. 非目标

- **⑥ 人工验收闸门 + 返工**（`in_review` 人工裁决、`in_review → in_progress` 回退边、转移表线性→图）——M18.6。本棒转移表仍线性、推进仍按 M18.4（run 成功自动接下一棒）。
- **⑦ Issue Timeline**（按 issueId 聚合的追加式工作事件流）——M18.7。本棒交付物落库**不**顺带建 Timeline；交付物表只是「产物事实」，不是「工作事件流」。
- **⑧ Issue 级可观测**（`getRunOriginsByIssueId` 反查 + 详情抽屉 + 诊断聚合）——M18.7。本棒只用 `getRunOrigin(runId)` 正查补全归属，不加 issue 维度反查。
- **前端交付物展示 UI**：看板卡片/详情里展示交付物链接、字段——后置。本棒核心是打通「提交→落库→下一棒取用」后端主链；交付物可视化不阻塞主旨。验证靠端点直查 + 测试。
- **强制提交 / 校验交付物完整性**：`submitDeliverable` 是最佳努力，不强制每棒必提；不校验 `fields` 必含某些键。缺失即降级（§3.7），不阻断流程。
- **模板 DSL**（条件/循环/过滤器/默认值语法）：`renderPrompt` 永远只做 `{{path}}` 字符串插值（不变量 5）。

```ts
    ref            TEXT,            -- 指针：大产物的 doc/MR 链接（可空）
    run_id         TEXT,            -- 产生它的 run（可空，便于追溯）
    created_at     INTEGER NOT NULL
  );
  CREATE INDEX IF NOT EXISTS idx_deliverable_issue ON deliverable(issue_id);
  CREATE INDEX IF NOT EXISTS idx_deliverable_issue_kind ON deliverable(issue_id, kind);`,
},
```

> 追加式（§3.6）：同 `(issue_id, kind)` 可多行，靠 `created_at` 取最新——**不设 `(issue_id,kind)` 唯一约束**（与 column_config 的唯一约束相反，因语义不同：配置是「每列一条」，交付物是「每次提交一行、保留历史」）。

### Patch P2 — `renderPrompt` 升级：路径取值（纯函数，无 DSL）

**目标**：支持 `{{deliverables.plan.summary}}` 嵌套路径，三条不变量（字符串插值/缺失回退空串/无 DSL）保留。

```ts
// render.ts —— 仅扩「key 可带点、按路径在嵌套对象里取值」，不引入任何 DSL
export type PromptVars = { [k: string]: string | PromptVars };

export function renderPrompt(template: string, vars: PromptVars): string {
  return template.replace(/\{\{([\w.]+)\}\}/g, (_, path: string) => {
    const v = path.split(".").reduce<unknown>(
      (acc, seg) => (acc != null && typeof acc === "object" ? (acc as Record<string, unknown>)[seg] : undefined),
      vars,
    );
    return typeof v === "string" ? v : "";   // 缺失 / 非字符串叶子 → 空串（不变量保留）
  });
}
```

> 单层 `{{title}}` 仍照旧命中（`split(".")` 单段）。叶子非字符串（取到中间对象）也回退空串——避免把 `[object Object]` 插进 prompt。正则 `[\w.]+` 不匹配空格，模板里 `{{ x }}` 这类带空格的仍不命中（与现状一致，不放宽）。

**影响文件**：`orchestrator/render.ts`（改）。

### Patch P3 — `startStep` 构建累积上下文 + 注入交付物工具的 surfaceContext

```ts
  // 同 kind 取最新（created_at 最大）；指针化：只放 fields + ref，不放全文（本就无全文列）

async function startStep(issue: IssueRow) {
  const table = columnConfigSvc.transitionsForProject(issue.projectId);
  const t = nextTransition(table, issue.status);
  if (!t) return null;
  const agent = await agentSvc.getById(t.agentId).catch(() => null);
  if (!agent) throw new OrchestratorAgentMissingError(t.agentId, issue.issueId);

  const runId = idGen();
  const vars = buildPromptVars(issue, deliverableSvc.listByIssue(issue.issueId));  // ⑤ 上下文累积

      surface: "orchestrator", conversationId: "", runId,
      capabilities: ["submit_deliverable"],
      issue: { issueId: issue.issueId, fromStatus: issue.status },
    },
  });

  opsStore.insertRunOrigin({ /* 不变，idempotencyKey: issue:<id>:<status>:run */ });
  return { runId };
}
```

### Patch P5 — `createSubmitDeliverableTool`（HTTP 回调工具）

仿 `start-new-conversation-tool.ts`（`packages/runner-daemon/src/submit-deliverable-tool.ts`，新）：

```ts
import type { Tool } from "@my-agent-team/core";

export function createSubmitDeliverableTool(input: {
  backendUrl: string; backendAuthToken: string | null;
  issueId: string; fromStatus: string; runId: string;
}): Tool {
  return {
    name: "submit_deliverable",
    description:
      "Submit your structured deliverable for the current issue step before finishing. " +
      "Put small extractable values in `fields` (e.g. plan summary, MR url/title, review verdict); " +
      "put large artifacts behind `ref` (a doc/MR link), not inline.",
    inputSchema: {
      type: "object",
      properties: {
        kind: { type: "string", description: "Deliverable type, e.g. plan | mr | review." },
        fields: { type: "object", additionalProperties: { type: "string" },
          description: "Structured small fields for the next step to extract." },
        ref: { type: "string", description: "Pointer to a large artifact (doc/MR link). Optional." },
      },
      required: ["kind", "fields"],
    },
    async execute(args: unknown) {
      const a = args as { kind: string; fields: Record<string,string>; ref?: string };
      const url = `${input.backendUrl}/api/issues/${input.issueId}/deliverables`;
      const headers: Record<string,string> = { "Content-Type": "application/json" };
      if (input.backendAuthToken) headers["x-auth-token"] = input.backendAuthToken;
      const resp = await fetch(url, { method: "POST", headers, body: JSON.stringify({
        kind: a.kind, fields: a.fields, ref: a.ref,
        fromStatus: input.fromStatus, runId: input.runId,
        idempotencyKey: `issue:${input.issueId}:${input.fromStatus}:deliverable`,
      }) });
      if (!resp.ok) {
        const body = (await resp.json().catch(() => ({}))) as { error?: string };
        return { content: `submit_deliverable failed: HTTP ${resp.status} - ${body.error ?? "unknown"}`, isError: true };
      }
      return { content: await resp.text() };
    },
  };
}
```

### Patch P7 — 交付物回调端点 + 路由接线

**P7-a 端点**（`issue/http.ts` 新增 `submitDeliverable(req, issueId)`）：

```ts
const deliverableSchema = z.object({
  kind: z.string().trim().min(1),
  fields: z.record(z.string()),
  ref: z.string().optional(),
  fromStatus: z.string().optional(),
  runId: z.string().optional(),
  idempotencyKey: z.string().optional(),
});

async submitDeliverable(req: Request, issueId: string): Promise<Response> {
  const parsed = deliverableSchema.safeParse(await req.json().catch(() => ({})));
  if (!parsed.success) return json({ error: "Validation failed", details: parsed.error.issues }, 400);
  if (!svc.port.getIssue(issueId)) return json({ error: "issue not found" }, 404);

  // 归属反查：runId 在则以 run_origin 为权威，校验/补全 issueId+fromStatus（§3.3）
  let fromStatus = parsed.data.fromStatus ?? "";
  if (parsed.data.runId) {
    const origin = opsStore.getRunOrigin(parsed.data.runId);
    if (origin?.issueId && origin.issueId !== issueId) return json({ error: "run/issue mismatch" }, 409);
    fromStatus = origin?.idempotencyKey?.split(":")[2] ?? fromStatus;
  }

  // 幂等：同 idempotencyKey 已提交过则返回既有（避免重试重复落库）
  const key = parsed.data.idempotencyKey;
  const existing = key ? deliverableSvc.port.getByIdempotencyKey?.(key) : null;
  if (existing) return json({ deliverable: existing });

  const row = deliverableSvc.submit({ issueId, fromStatus, kind: parsed.data.kind,
    fields: parsed.data.fields, ref: parsed.data.ref, runId: parsed.data.runId });
  return json({ deliverable: row }, 201);
}
```

> `issueRoutes` 需新增依赖 `opsStore`（反查）+ `deliverableSvc`（落库）。幂等键落库需在 deliverable 表加一列 `idempotency_key`（或单独的幂等表）——**首版选「交付物表加 `idempotency_key TEXT` 列 + 唯一索引」**，`getByIdempotencyKey` 走它；故 P1-a 迁移同时建该列与 `CREATE UNIQUE INDEX IF NOT EXISTS idx_deliverable_idem ON deliverable(idempotency_key) WHERE idempotency_key IS NOT NULL`（部分唯一索引，允许多个 NULL）。

## 7. 验收标准

1. 某 Project 配好各列 ColumnConfig 后，跑一个 Issue：planner 棒内调用 `submit_deliverable({kind:"plan", fields:{summary:"..."}, ref:"<doc 链接>"})` → 交付物落 `deliverable` 表，`from_status="planned"`、`run_id` 为该 run、`fields` 正确 JSON。
2. planner 棒成功推进到 developer 棒后，developer 棒的 prompt（其 ColumnConfig 模板含 `{{deliverables.plan.summary}}`/`{{deliverables.plan.ref}}`）被渲染成 planner 提交的实际值——**「上一棒喂下一棒」从扫文本升级为结构化提取**。
3. 模板里引用一个不存在的交付物路径（如 `{{deliverables.mr.url}}` 而 mr 尚未提交）→ 渲染为空串，run 正常起、不报错。
4. `submit_deliverable` 工具入参不含 issueId/fromStatus（Agent 不自报归属），归属由 backend 据 runId 反查 `run_origin` 补全（§3.3）；提交端点收到 `run/issue` 不匹配 → 409。
5. 同一棒因重试重复提交（相同 `idempotencyKey`）→ 只落一行，端点返回既有交付物（幂等）。
6. `submit_deliverable` 工具**只在 issue run 注入**：lark/web run 无此工具；reflect run 无此工具（surfaceContext 已剥离）。
7. 大产物走 `ref` 指针、不进 `fields`：后续 prompt 取到的是链接字符串，不是整篇正文（上下文不被撑爆，§3.2）。
8. `renderPrompt` 仍是纯 `{{path}}` 字符串插值：无条件/循环/过滤器；`{{title}}`/`{{issueId}}` 单层照旧命中（不变量 5 不回归）。
9. M18.1~M18.4 不回归：Issue CRUD、Project 选择建 Issue、ColumnConfig 可配置转移、草稿态 + 拖拽入列启动、SSE 实时、threadId 派生、`start_new_conversation` 工具注入全部照旧；Orchestrator 纯函数与 reactor 控制流形态未变（只在 `startStep` 多喂交付物数据 + surfaceContext）。
10. `bun test` 全绿；`turbo run lint typecheck` 通过；`next build` 不受影响。

## 8. 风险与权衡

- **Agent 不调 `submitDeliverable`（最佳努力的代价）**：工具非强制，模型可能忘记调用 → 下一棒对应 kind 为空、prompt 缺失回退空串、链路降级。缓解：(a) ColumnConfig 的 promptTemplate 里显式指示「完成后调用 submit_deliverable 提交计划/MR/review」；(b) 后续（非本棒）可在 `onRunComplete` 加「成功但未提交交付物」的告警。本棒接受降级不崩，强制提交留未来。
- **交付物追加式、同 kind 多份只取最新**：返工重跑（M18.6）会产生同 `(issueId,kind)` 多行，本棒上下文构建按 `created_at` 取最新（§3.6）。风险：若历史版本也需被引用，当前字典取不到。权衡：本棒无返工（§5），「最新即有效」够用；表保留全历史，未来返工时再按 `from_status`/版本扩展字典键，不丢数据。
- **`surfaceContext` 加 `"orchestrator"` surface 档**：扩了 enum，需 protocol + supervisor 两处镜像同步、runner 判定不绑 surface 字面值（§3.4）避免漂移。缓解：注入条件按 capability + issue 判，不按 surface；P8 测试钉住「issue run 注入、lark/reflect 不注入」。
- **幂等用部分唯一索引（`idempotency_key` 可空 + WHERE 子句）**：bun:sqlite 支持部分索引，但需确认其行为；若不稳，退化为「应用层先 `getByIdempotencyKey` 再 insert」（有微小并发窗口，但交付提交是单 run 内串行调用，并发可忽略）。权衡：本棒优先部分唯一索引（DB 层保证），应用层查作冗余防线。
- **工具失败返回 `{isError:true}` 而非抛错**：与 `start_new_conversation` 抛错策略不同（§3.7 论证：交付提交失败不该炸 run）。风险：模型可能忽略 isError 继续完成 → 交付物未落库、下一棒降级。这与「最佳努力」一致，可接受；模型看到 isError 通常会重试。
- **回调端点信任 runId 反查、但 runId 由工具入参传入**：理论上恶意/错误 runId 可被伪造。缓解：端点校验 `origin.issueId === 路径 issueId`（不匹配 409）；issue run 的 backendAuthToken 已是受信通道。本棒接受「runId 来自受信 runner」假设，不做更强的 run 归属密码学校验。
