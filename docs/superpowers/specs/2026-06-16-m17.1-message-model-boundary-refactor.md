# M17.1 Message Model Boundary Refactor — 下压统一 Message 本体

> **Status:** Spec → Ready for implementation.
> **Baseline commit:** `origin/master` @ `9a57a68`.
> **关联:** `docs/architecture/architecture-design-philosophy.md` · `docs/architecture/message-unification-methodology.md` · `docs/superpowers/specs/2026-06-16-conversation-message-revisions.md` · `packages/core/src` · `packages/conversation/src/index.ts` · `apps/backend/src/main.ts` · `apps/web/src/lib/conversation-reducer.ts` · `apps/lark-bot/src/sse-watcher.ts`.

M17.1 的目标是把系统里的消息语义下压成唯一 `Message` 本体，拆掉 `ConversationMessageRevision` 继续作为跨层语义对象的技术债。Conversation、Run、EventLog、Checkpoint、Surface 不再各自解释消息；它们只能保存、引用、记录、恢复或投递同一个 `Message` 及其 `MessageRevision`。

这不是一次兼容性改造，而是一次边界纠偏。旧 shape、旧 parser、旧 fallback、旧 surface-local message lifecycle 都是本次要拆除的技术债，不作为兼容目标保留。

## 1. 背景

`origin/master @ 9a57a68` 已经修复了 M17 前置链路里的几个直接 bug：后端补了 terminal revision fallback，Web 恢复了 pending send / approval UI，Lark 改成按 revision upsert。它们解决的是"消息能不能正确显示、关闭、投递"的局部问题。

但当前代码仍然没有解决更底层的问题：系统继续把 `Message` 语义放在 `packages/conversation` 的 `ConversationMessageRevision` 里，让 backend、Web、Lark 通过 conversation/ledger 语义间接理解消息。这会让 `Conversation` 从业务边界变成消息本体的所有者，也会让 surface 被迫理解 storage/projection 产生的 envelope。

M17.1 要处理的是这个更底层的技术债：**Message 不是 Conversation 的内部格式；Message 是跨 Run、Conversation、Surface、Checkpoint 的领域本体。**

## 2. 当前远程代码事实

基于 `origin/master @ 9a57a68`，当前仍存在以下事实：

1. `packages/conversation/src/index.ts` 定义 `ConversationMessageRevision`、`MessageRevisionState`、`MessageToolRevision`、`parseRevision()`、`assistantMessageId()`。
2. `parseRevision()` 仍把 ledger content 作为输入，并包含 string、blocks、`{ text }`、`{ blocks }`、缺失 `messageId/state` 等旧 shape 的解析路径。
3. `apps/backend/src/main.ts` 的 Conversation Projection 仍构造 `ConversationMessageRevision`，并在 run complete 时写 final done/error revision。
4. `apps/web/src/lib/conversation-reducer.ts` 的 `UiMessage.content` 仍是 `ConversationMessageRevision`，Web reducer 通过 `parseRevision()` 从 ledger entry 得到用户可见消息。
5. `apps/lark-bot/src/sse-watcher.ts` 仍从 `@my-agent-team/conversation` 引入 `parseRevision()`，并基于 ledger entry content 派生 message delivery。
6. `docs/architecture/*` 仍大量使用 ledger、EventLog、Conversation Projection、checkpoint 等实现机制解释用户可见消息链路。

这些事实说明：M17 的第一阶段修复已经把 revision 链路补齐，但 `Message` 语义仍没有下压，机制仍在主链路上浮。

## 3. 第一性原则

### 3.1 Message 是领域本体，不是 ledger content

用户和 Agent 共同世界里存在的是消息：谁发的、内容是什么、现在是否完成、是否等待用户操作、是否需要喂回模型。

ledger entry、event row、checkpoint snapshot、Lark delivery record 都不是新的消息。它们只是保存、记录、恢复或投递 Message 的不同外壳。

### 3.2 Revision 是同一 Message 的变化，不是 Conversation 专属对象

流式输出、等待审批、工具执行状态、完成/失败状态，都是同一条 Message 的状态或内容变化。

因此 revision 应该是：

```text
MessageRevision<Message.id>
```

而不是：

```text
ConversationMessageRevision
LedgerMessage
UiMessageContent
LarkMessage
CheckpointMessage
```

### 3.3 Surface 只消费 Message

Web 和 Lark 是 renderer/delivery layer，不是消息语义解释器。

Surface 可以维护 UI state 或 delivery state，但不能重新定义 message content、message id、message lifecycle，也不能直接消费 RunEvent / EventLog / checkpoint row。

### 3.4 Terminal state 是硬不变量

所有用户可见 run 最终必须关闭它打开的 Message。

`done | error` 必须表达在 `Message.state` 或 `MessageRevision.state` 上，不能靠 SSE close、run_done event、projection chain completion、checkpoint resume state 间接推断。

### 3.5 不兼容旧技术债

M17.1 不保留旧 content shape 的 parser fallback。

旧逻辑看起来像兼容，实质是在系统继续承认多个消息本体：string message、blocks message、ledger message、revision message、UI message。M17.1 要把这些入口收敛成唯一 schema，而不是继续给旧形状补 adapter。

## 4. M17.1 目标

M17.1 完成后，系统应满足：

1. `Message` / `MessageRevision` 定义在底层协议或共享领域包中，不定义在 `packages/conversation`。
2. `ConversationMessageRevision` 被删除或改名为非导出的迁移中间变量；跨包 API 不再暴露它。
3. Conversation 只负责保存和订阅 conversation-visible Message，不拥有消息语义。
4. Run 只 emit Message / MessageRevision，不要求调用方理解 EventLog 或 Projection。
5. Web reducer 只消费 Message / MessageRevision，不消费 ledger-specific parser 输出。
6. Lark watcher 只按 `message.id` 维护 Delivery，不重新解释 ledger content。
7. Checkpoint 只保存 execution state 中的 Message，不定义 checkpoint message。
8. EventLog 只记录 audit event；event payload 可以引用 Message，但 EventLog 不成为用户可见消息事实源。
9. 所有用户可见 Message 都有统一 terminal state 规则。
10. 测试不再覆盖旧 shape fallback，而是覆盖唯一 schema 的严格失败与端到端一致性。

## 5. 非目标

M17.1 不做以下事情：

1. 不做旧 ledger content 的兼容解析。
2. 不保留 `s-${seq}` 这类从 storage seq 反推 message id 的 fallback。
3. 不维护 `_preliminary`、string content、blocks array content 等旧消息入口。
4. 不为了兼容旧 UI 或旧 Lark watcher 保留双 parser。
5. 不把架构文档重写作为第一交付物；M17.1 先改代码，代码验收后再更新文档。
6. 不引入新的 storage 名词作为消息本体。
7. 不把 CQRS/Event Sourcing/Projection 作为 feature 开发者必须理解的主线。

## 6. 目标模型

### 6.0 包位置决策（已定）

- **新建 `@my-agent-team/message`** — 统一 Message / MessageRevision 的唯一家
- **`ContentBlock` 留在 `@my-agent-team/core`** — 它是 L1 协议类型，`@my-agent-team/message` 依赖 core 拿它
- **core 的旧 `Message`（`{ role, content }`）删除** — 不是改名、不是共存、不是渐进过渡。全系统只有 `@my-agent-team/message` 导出的一个 Message
- **`"tool"` role 只加类型定义** — M17.1 不改框架层 tool_result 的 role 赋值逻辑

### 6.1 统一 Message

`id`、`state` 等字段设计为 optional：框架 run loop 构造 assistant message 时只知道 role + blocks/text，不知道 messageId 和 lifecycle；messageId 和 state 由 backend projection 赋予。同一类型，不同层填充不同字段子集。

```ts
export type MessageRole = "system" | "user" | "assistant" | "tool";

export type MessageState = "pending" | "streaming" | "waiting" | "done" | "error";

export interface MessageAuthor {
  kind: "system" | "user" | "agent" | "tool";
  id?: string;
  displayName?: string;
}

export interface MessageToolState {
  id: string;
  name: string;
  state: "running" | "done" | "error";
  isError?: boolean;
}

export interface MessageError {
  code?: string;
  message: string;
}

export interface Message {
  id?: string;
  role: MessageRole;
  author?: MessageAuthor;
  state?: MessageState;
  text?: string;
  blocks?: ContentBlock[];
  tools?: MessageToolState[];
  runId?: string;
  conversationId?: string;
  visibility?: "internal" | "conversation";
  createdAt?: number;
  updatedAt?: number;
  error?: MessageError;
}
```

### 6.2 MessageRevision

Revision 是同一条 Message 的 patch / snapshot。M17.1 可以选择 snapshot-only，避免引入 patch merge 复杂度。

```ts
export interface MessageRevision {
  messageId: string;
  state: MessageState;
  role: MessageRole;
  text?: string;
  blocks?: ContentBlock[];
  tools?: MessageToolState[];
  runId?: string;
  conversationId?: string;
  visibility: "internal" | "conversation";
  updatedAt: number;
  error?: MessageError;
}
```

规则：

1. `messageId` 永远引用同一条逻辑 Message。
2. `state` 是 surface 判断 busy / waiting / terminal 的唯一依据。
3. `runId` 是关联执行，不是 message id 的替代。
4. `conversationId` 是归属上下文，不代表 Message 属于 Conversation 包。
5. `visibility` 控制是否进入 Conversation 可见层。
6. `text` / `blocks` / `tools` 是 Message 内容和状态，不是 surface-specific render payload。

### 6.3 工具函数

统一包至少导出：

```ts
parseMessage(input: unknown): Message
parseMessageRevision(input: unknown): MessageRevision
serializeMessage(message: Message): string
serializeMessageRevision(revision: MessageRevision): string
messageId(parts: MessageIdParts): string
assistantMessageId(runId: string): string
isOpenMessageState(state: MessageState): boolean
isTerminalMessageState(state: MessageState): boolean
mergeMessageRevision(message: Message | null, revision: MessageRevision): Message
```

`parseMessageRevision()` 必须是严格 parser：缺少 `messageId`、`state`、`role` 等必需字段时直接失败，不生成 fallback id。

## 7. 包边界改造

### 7.1 `@my-agent-team/message`（新建）+ `@my-agent-team/core`（旧 Message 删除）

**新包 `@my-agent-team/message`：**

1. 依赖 `@my-agent-team/core`（仅拿 `ContentBlock` 类型）。
2. 导出 `Message`、`MessageRevision`、`MessageState`、`MessageRole`、`MessageAuthor`、`MessageToolState`、`MessageError`。
3. 导出 parser / serializer / state helper / `assistantMessageId()` / `mergeMessageRevision()`。
4. 纯类型 + 纯函数，不依赖 backend、conversation、web、lark。

**`@my-agent-team/core` 变更：**

1. 删除 `Message` 类型（`{ role, content }`）。
2. 删除 `Role` 类型（被 `MessageRole` 替代）。
3. `ContentBlock`、`TextBlock`、`ToolUseBlock`、`ToolResultBlock` 留在 core。
4. 所有构造 `{ role, content }` 的地方改为 `{ role, text }` 或 `{ role, blocks }`。

### 7.2 `@my-agent-team/conversation`

Conversation 包只保留 conversation 边界：

1. `LedgerEntry` 可以继续存在，但 `content` 的语义应改为 serialized `Message` / `MessageRevision`。
2. 删除 `ConversationMessageRevision`、`MessageRevisionState`、`MessageToolRevision`、`RevisionContentBlock`。
3. 删除 `parseRevision()`。
4. 删除或迁移 `assistantMessageId()` 到统一 Message 包。
5. 暴露 API 时使用业务语言：`appendMessageRevision()`、`listMessages()`、`subscribeMessages()`。
6. `appendLedgerEntry()` 只留在 storage adapter 内部，不作为 feature 层主入口。

### 7.3 Backend

Backend 只负责把 Run 产生的 MessageRevision 写入 Conversation 可见层：

1. `buildAssistantRevision()` 返回统一 `MessageRevision`。
2. `RunAccumulator.latestAssistantRevision` 类型改为统一 `MessageRevision | null`。
3. Conversation Projection 内部可以继续存在，但不把 Projection shape 暴露给 surface。
4. run complete 时必须写 terminal `MessageRevision`，且必须等待同 run 的 revision 写入链完成。
5. tool-only run、error run、interrupt/waiting run 必须通过统一 revision 表达。
6. 不再写 `{ text, runId }`、裸 blocks、string content、`_preliminary` 等旧 shape。

### 7.4 Web

Web 只处理统一 Message：

1. `UiMessage.content` 改为 `Message` 或直接把 `UiMessage` 收敛为 `Message + UiState`。
2. reducer 不再 import `parseRevision` from `@my-agent-team/conversation`。
3. reducer 使用统一 `parseMessageRevision()` 和 `mergeMessageRevision()`。
4. `isBusy()` 只看 `message.state in streaming/waiting`。
5. `getApprovalTarget()` 只看 `Message.tools` 和 `Message.state === "waiting"`。
6. Timeline / render-blocks / timeline helpers 不再接受 `string | ContentBlock[] | revision object` 的宽松 union。
7. 发送前 pending state 是 UI state，不得写入 Message 本体。

### 7.5 Lark

Lark 只维护 Delivery：

1. watcher 不再 import `parseRevision` from `@my-agent-team/conversation`。
2. watcher 使用统一 `parseMessageRevision()`。
3. delivery table 只绑定 `conversationId + messageId + larkChatId`。
4. idempotency key 使用 `message.id`，不使用 ledger `seq` 表示同一消息。
5. render 输入为 `Message` / `MessageRevision`，不再直接 render ledger raw content。
6. terminal 后忽略同 message 的后续重复投递，但仍推进 pushed seq。

### 7.6 EventLog

EventLog 退回 Run audit：

1. `RunEvent` 可以包含 `message?: Message | MessageRevision`。
2. EventLog 不提供 surface-visible message parser。
3. EventLog 的 `seq` 只表示 run audit order，不参与 message identity。
4. Troubleshooting 可以读 EventLog，但产品链路不依赖 EventLog 渲染用户消息。

### 7.7 Checkpoint

Checkpoint 只保存 execution snapshot：

1. checkpoint 中保存 `Message[]` 或执行状态引用的 `Message`。
2. 不定义 `CheckpointMessage`。
3. checkpoint resume 后继续使用同一个 `message.id` 和统一 state 规则。
4. waiting / approval 这类用户操作状态必须进入 Conversation-visible Message 或 control，不只存在 checkpoint。

## 8. 代码实施顺序

### Step 1：新建 `@my-agent-team/message` + 删除 core 旧 `Message`

1. 新建 `packages/message/`，依赖 `@my-agent-team/core`（拿 `ContentBlock`）。
2. 定义 `Message` / `MessageRevision` / `MessageState` / `MessageRole` / parser / serializer / helper。
3. 迁移 `assistantMessageId()` 从 conversation 到 message 包。
4. 添加严格 parser 单测：缺 `messageId`/`state` 失败，旧 shape 失败，terminal/open helper 正确。
5. 删除 `@my-agent-team/core` 的 `Message` 和 `Role` 类型。
6. 原子替换所有 27 个 `Message` import：framework、plugins、checkpointers、context-managers、adapter、test-helpers、runner-protocol、runner-daemon、harness、backend。
7. 构造点替换：`{ role, content: "str" }` → `{ role, text: "str" }`，`{ role, content: [...] }` → `{ role, blocks: [...] }`。

### Step 2：迁移 Conversation 包

1. 删除 `ConversationMessageRevision`、`MessageRevisionState`、`MessageToolRevision`、`RevisionContentBlock`、`isRevisionState()`。
2. 删除 `parseRevision()`（旧 shape fallback 全部移除）。
3. 从 `@my-agent-team/message` re-export 或直接让消费者 import message 包。
4. `LedgerEntry.content` 语义明确为 serialized `MessageRevision`。
5. conversation API 逐步转向 `appendMessageRevision()` / `listMessages()` 等 message-facing 命名（完整重命名留给 M17.2）。

### Step 3：迁移 Backend Projection

1. Backend 改用统一 `MessageRevision` 构造 assistant streaming / waiting / done / error。
2. terminal revision 逻辑保留，但类型和 serializer 改为统一包。
3. 删除所有旧 shape 写入路径。
4. 保持 projection chain ordering：terminal revision 必须在同 run streaming revisions 后写入。

### Step 4：迁移 Web

1. reducer 改用统一 parser / merge helper。
2. `UiMessage` 改成 `Message + UiState`。
3. render helper 只接受统一 Message。
4. approval / busy / pending send 测试基于统一 Message state。

### Step 5：迁移 Lark

1. watcher 改用统一 parser。
2. render 输入改为 Message / MessageRevision。
3. delivery 继续以 message id upsert。
4. 删除裸 content render 和旧 preliminary 相关逻辑。

### Step 6：迁移 Checkpoint / EventLog 类型引用

1. EventLog payload 引用统一 Message / MessageRevision。
2. Checkpointer state 中引用统一 Message。
3. 删除任何 `*Message` 但语义重复的类型。

### Step 7：代码验收后更新文档

1. 先跑 typecheck / test / lint。
2. 代码通过后再更新 architecture 文档。
3. 文档主线只讲 Conversation / Run / Message。
4. Ledger、EventLog、Projection、Checkpoint、Delivery 只在 implementation notes / troubleshooting 中出现。

## 9. 测试要求

### 9.1 单元测试

1. `parseMessageRevision()` 对缺少 `messageId` 的输入失败。
2. `parseMessageRevision()` 对缺少 `state` 的输入失败。
3. `parseMessageRevision()` 对旧 string / blocks / `{ text }` content 失败。
4. `mergeMessageRevision()` 对同 `messageId` upsert，不新增消息。
5. `isTerminalMessageState(done/error)` 为 true。
6. `isOpenMessageState(streaming/waiting/pending)` 为 true。

### 9.2 Backend 测试

1. streaming run 写多条同 `messageId` revision，最后写 `done`。
2. tool-only run 即使没有 text，也必须写 terminal `done` 或 `error`。
3. error run 写 `error` revision，surface 不靠 run_done 推断。
4. waiting approval run 写 `waiting` revision，并包含 tools state。
5. terminal revision 在 projection chain 完成后写入。

### 9.3 Web 测试

1. 同 `messageId` 多次 revision collapse 成一条消息。
2. `state=streaming/waiting` 时 busy 为 true。
3. terminal revision 后 busy 为 false。
4. approval card 来自 `Message.state=waiting + tools.running`。
5. 旧 string / blocks ledger content 不再被 reducer 当作 message 接受。

### 9.4 Lark 测试

1. 同 `messageId` 的 streaming revisions 更新同一 delivery record。
2. terminal 后重复 revision 不重复创建消息。
3. idempotency key 不使用 ledger seq 表示 message identity。
4. render 输入为统一 Message / MessageRevision。
5. 旧 `_preliminary` / raw content shape 不再被 watcher 接受。

### 9.5 边界测试

1. `@my-agent-team/conversation` 不导出 `ConversationMessageRevision`。
2. Web / Lark 不从 `@my-agent-team/conversation` import message parser。
3. Backend 不构造 conversation-scoped message type。
4. Checkpoint / EventLog 不定义新的 message 本体。

## 13. 最终准则

M17.1 的判断标准不是"是否修复了当前 Web/Lark bug"，而是：

```text
Message 是否成为唯一跨层语义？
Conversation / Run / Surface 是否只围绕 Message 协作？
Ledger / EventLog / Projection / Checkpoint / Delivery 是否退回实现机制？
旧技术债 shape 是否被删除，而不是被兼容？
```

如果答案不是全部为是，M17.1 就没有完成。
