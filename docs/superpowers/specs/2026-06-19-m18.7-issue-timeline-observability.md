## 1. 背景

[Issue 协作工作流设计](../../architecture/foundations/issue-workflow.md) 把现状诊断为「建即跑、写死、单向、看不见」。M18.3~M18.6 解了前三个：「写死」→列可配置、「建即跑」→草稿态、「单向」→交付物结构化 + 人工闸门 + 返工。**最后一块「看不见」没解**：

- 一件 Issue 除了自身那一行 `status`，**没有「它到底经历了什么」的时间线**——什么时候建的、谁起的、每棒跑了多久、提交过哪些交付物、被打回过几次、谁拍的板，全都散落在 run 级日志里，无法按 issue 串起来。
- M16 的 run 级可观测**已经按 `run_origin.issue_id` 记录了每一棒的 run**（M18 全程写入），但**没有从 Issue 反查 run 的入口**——`store.ts` 只有 `getRunOrigin(runId)` 正查，缺 `getRunOriginsByIssueId`。「这件活的所有 run」目前查不出来。
- 前端 `IssueCard` 只有一张卡片 + M18.6 的裁决按钮，**没有详情视图**——点开一件 Issue 看不到它的历史，更跳不到它名下各 run 的 M16 观测页。

本棒落地设计的 ⑦（Issue Timeline）+ ⑧（Issue 级可观测），把「看不见」补上。两者是一体两面：⑦ 提供「活的视角」（工作事件流），⑧ 把 ⑦ 与既有 M16 run 级观测**经 `run_origin.issue_id` 缝合**成「这件活的完整可观测面」。本棒**不发明新可观测机制**（不变量 8）——run 级观测全部复用 M16，本棒只加「issue 维度的聚合与反查」。

为什么 ⑦⑧ 同棒、且排在 M18 最后：它们是**横切**——要观测的工作事件（created/started/run.*/deliverable.submitted/status.advanced/human.decided）由前面 M18.3~M18.6 各步产生，必须等那些步骤的埋点位都稳定了，才谈得上「把它们记成一条流」。⑧ 又依赖 ⑦ 的事件作为「活的视角」数据源。故收口在最后一棒。

## 3. 第一性原则

### 3.1 Issue Timeline 不是会话账本，也不是 run_ops_event——它是第三套事实

设计 ⑦（`issue-workflow.md:134-154`）已论证：[会话账本](../../architecture/conversation/ledger.md)的不变量是「装**对话可见**内容、按 conversation 聚合」，而 Issue 要记的是**工作事件**、按 issue 聚合——本体/聚合维度/不变量都不同，塞进账本会破账本的硬约束。本棒补充论证第二条边界：**Issue Timeline 也不是 `run_ops_event`**。`run_ops_event` 是 **run 级**（一个 run 内部的 attempt 起止、心跳、工具调用，按 `runId` 聚合，§2.2），回答「这个 run 怎么跑的」；Issue Timeline 是 **issue 级**（这件活的创建/启动/推进/裁决，按 `issueId` 聚合），回答「这件活发生了什么」。

三套事实，三张表，三个聚合维度：

| 事实 | 表 | 聚合键 | 装什么 | 不变量 |
|------|-----|--------|--------|--------|
| 会话账本 | `ledger`（per conversation） | `conversationId` | 对话可见内容 | 追加式、只装对话可见 |
| run 级事件 | `run_ops_event` | `runId` | run 内部执行细节 | 按 run、观测用 |
| **Issue Timeline（本棒）** | **`issue_event`** | **`issueId`** | **工作事件** | **追加式、按 issue、不混对话/run 内部** |

D1 的决定（独立表）就是这条原则的落地——**不复用 `run_ops_event`**，因为复用等于把两个聚合维度强塞进一张表，`issue_event` 里要么 `run_id` 大量为空（创建/裁决没有 run）、要么 `run_ops_event` 里冒出 `issue_id` 污染 run 级查询。独立表是边界清晰的最小代价。

### 3.2 IssueEvent 是「窄事实 + 宽 payload」——schema 稳定，细节进 JSON

七种 `kind` 共用一个窄 schema `{seq, issueId, kind, payload, ts}`（同 `run_ops_event` 的设计）。差异化的内容（哪个 run、哪个 status、哪份交付物、谁裁决）进 `payload`（JSON）。这样加新 `kind` 或扩 payload 字段**不需要迁移表**——schema 是稳定的窄面，演化压在 JSON 里。`seq` 用 `AUTOINCREMENT` 自增主键保证**全局插入序**（Timeline 回放按 seq 升序），`ts` 仅作展示时间戳（不依赖它排序，避免同毫秒乱序）。

```text
IssueEvent {
  seq:     number   // AUTOINCREMENT 全局序，Timeline 回放排序键
  issueId: string   // 聚合键
  kind:    "created" | "started" | "run.started" | "run.ended"
         | "deliverable.submitted" | "status.advanced" | "human.decided"
  payload: object   // kind 相关的细节，JSON；schema 不约束，演化压这里
  ts:      number   // 展示时间戳（非排序键）
}
```

各 kind 的 payload 约定（首版，最小集）：

| kind | 触发点 | payload 关键字段 |
|------|--------|------------------|
| `created` | http `create` | `{projectId, title}` |
| `started` | http `transition` 的 `draft→planned` | `{from:"draft", to:"planned"}` |
| `run.started` | reactor `startStep` 起 run 后 | `{runId, fromStatus, agentId}` |
| `run.ended` | reactor `onRunComplete` 入口 | `{runId, fromStatus, status}`（status=succeeded/failed/…） |
| `deliverable.submitted` | http `submitDeliverable` + reject 的 `rework_feedback` | `{kind, deliverableId, runId|null, ref|null}` |
| `status.advanced` | 各 `applyTransition` 调用方 | `{from, to, by:"reactor"|"human"|"rework"|"revert"}` |
| `human.decided` | http `reviewDecision` | `{decision:"approve"|"reject", note?}` |

### 3.3 ⑧ 不补采集，只补反查 + 聚合 + 导航——run 级观测全是 M16 存量

⑧ 的关键洞察：**采集层零新增**。每一棒的 run 起步时 `insertRunOrigin` 已写 `issue_id`（§2.4），M16 的 run 级观测（run_ops_event、insights、trace、心跳）全程在记。所以「这件活的所有 run 的完整观测」**数据已经在库**，缺的只是三处接缝（`issue-workflow.md:162-166`）：

1. **`getRunOriginsByIssueId` 反查**（store.ts）：`SELECT … WHERE issue_id = ? ORDER BY created_at`，用现成的 `idx_run_origin_issue` 索引。这是把「issue→runs」串起来的唯一缺失方法。
2. **前端详情抽屉**（apps/web）：聚合展示 Timeline（⑦）+ 该 Issue 名下各 run 的列表，每个 run 跳 M16 详情页 `/ops/runs/[runId]`（D3）。
3. **Issue 级诊断聚合**（最小版）：把 run 级诊断/终态按 issue 汇总成一行概览（「N 棒成功、M 棒失败、当前卡在 X」），回答「这件活为什么卡住」。复用 M16 的 run 终态/诊断字段，不发明新诊断机制。

**不内嵌 M16 重组件**（D3）：抽屉不画 TraceWaterfall/RunInsights，只列 run + 跳转。职责是「按 issue 聚合 + 导航」，把 run 详情留给已完备的 M16 页面——避免重写、避免抽屉变成第二个 ops 页。

### 3.4 埋点在调用方，不在领域服务——事件流是观测旁路，不进核心控制流

`status.advanced` 的真相源是 `applyTransition`（§2.6），但**不在 `service.ts` 里埋点**。两个理由：① service 是纯领域逻辑（校验 + CAS + 重读），不应依赖事件流基建（违反分层，且测试要 mock 事件器）；② service 拿不到「为什么转」的语义（自动 vs 人工 vs 返工 vs 补偿）——那只有调用方知道。所以**在各 `applyTransition` 调用方埋 status.advanced**，带上 `by` 来源：

- reactor `onRunComplete` 自动推进 → `by:"reactor"`
- http `transition` 人工拖拽 → `by:"human"`
- http `reviewDecision` reject 的 `in_review→in_progress` → `by:"rework"`；approve 的 `in_review→done` → `by:"human"`
- service `revertReviewReject` 补偿回滚 → `by:"revert"`（这一处例外：补偿是 service 内部方法，调用方是 http reject 的 catch 分支，在 catch 里埋）

**埋点是 best-effort 观测旁路**（同 M17.5 `onRunEvent` 的「best-effort fan-out」范式，supervisor.ts:90-95）：事件写入失败**绝不**影响主流程（status 已转、run 已起）。Timeline 写入包 try/catch 吞错 + warn，不冒泡。这是不变量——**观测不能拖垮被观测的业务**。

> 为什么不集中在一处埋（如 supervisor 统一监听）：status.advanced 的来源语义分散在 5 个调用点，集中监听拿不到 `by`；且 created/started/human.decided 本就不经 reactor/supervisor（纯 HTTP），无统一监听点。分散埋 + 窄写入器（`appendIssueEvent`）是最直接的。

### 3.5 Timeline SSE 复用 issue SSE 范式，按 issueId 过滤

D2 决定本期上 SSE。**不新发明推送机制**——复用 M18.4 的 `subscribeIssues` 长轮询范式（§2.8）：新增 `subscribeIssueTimeline(issueId)`，catch-up 已有事件（按 seq 升序）+ 长轮询增量（`seq > lastSeq` 的新事件）+ 心跳。`GET /api/issues/:id/timeline` 给全量快照（抽屉打开时拉一次）；`GET /api/issues/:id/timeline/events`（SSE）给增量（抽屉打开期间实时追加）。两者数据同源（`issue_event` 表），SSE 只是「快照 + 增量」的推送形态。

> 为什么 Timeline 要实时：返工/推进是异步发生的（reactor 在后台起棒、run 跑完推进）。抽屉打开时若只有快照，用户看不到「返工棒正在跑→跑完→回到 in_review」这条实时链路。SSE 让「活的视角」真的活。

### 3.6 抽屉是「issue 聚合 + 导航中枢」，一个聚合端点喂它

详情抽屉要展示三块：Issue 元信息（已有 `getIssue`）、Timeline（⑦）、run 列表（⑧ 反查）。为减少前端往返，**加一个聚合端点 `GET /api/issues/:id/detail`** 一次返回 `{issue, timeline, runs}`（runs 来自 `getRunOriginsByIssueId` + 每个 run 的终态/诊断概览）。Timeline 的**增量**走 SSE（§3.5），但首屏聚合走这一个端点。每个 run 项带 `runId` → 前端渲染成跳 `/ops/runs/[runId]` 的链接（D3）。

> 为什么聚合端点而非前端拼三个调用：抽屉打开是高频交互，三个串行 fetch 有瀑布延迟；聚合端点一次 round-trip。且「issue 的可观测面」本就是一个语义整体，给它一个端点符合「按 issue 聚合」的设计意图。

### 3.7 债务随观测面一起收口——它们都是「让 Issue 流可信」的同一件事

D5 把四项债务纳入本棒。理由：它们和 ⑦⑧ 同属「让 Issue 协作流真正可用、可信、可维护」的收尾：

- **issue-workflow.md:160 文档同步**：该行仍写 `idempotencyKey = issue:<id>:<status>:run`，M18.6 已改纯 `runId` + `from_status` 独立列。⑧ 的反查正是基于这套 run_origin 结构——文档必须先准，否则 ⑧ 的读者被误导。

- **M18.6 retro:101 e2e 测试债**：「reject → rework_feedback 进 prompt」缺穿过 `renderPrompt`+`buildPromptVars` 的真库 e2e。retro 当时明说「建议 M18.7 补」——本棒补，且**有了 Timeline 后这条 e2e 可同时断言事件流**（reject→human.decided→deliverable.submitted→status.advanced→run.started 一串），一箭双雕。

```ts
// runtime-ops/store.ts —— 新增（payload 存 JSON 字符串，读时 parse）
const ISSUE_EVENT_COLS = `seq, issue_id AS issueId, kind, payload, ts`;

appendIssueEvent(input: {
  issueId: string;
  kind: IssueEventKind;
  payload?: Record<string, unknown>;
}): number {
  this.#db.run(
    `INSERT INTO issue_event (issue_id, kind, payload, ts) VALUES (?, ?, ?, ?)`,
    [input.issueId, input.kind, JSON.stringify(input.payload ?? {}), Date.now()],
  );
  const row = this.#db.query("SELECT last_insert_rowid()").get() as { "last_insert_rowid()": number };
  return row["last_insert_rowid()"];
}

getIssueEvents(issueId: string, afterSeq = 0): IssueEvent[] {
  const rows = this.#db
    .query(`SELECT ${ISSUE_EVENT_COLS} FROM issue_event WHERE issue_id = ? AND seq > ? ORDER BY seq`)
    .all(issueId, afterSeq) as Array<Omit<IssueEvent, "payload"> & { payload: string }>;
  return rows.map((r) => ({ ...r, payload: JSON.parse(r.payload) as Record<string, unknown> }));
}
```

> `afterSeq` 参数同时服务全量（`afterSeq=0`）与 SSE 增量（`afterSeq=lastSeq`）——一个查询两用，无重复 SQL。

const RUN_COLS = `run_id AS runId, thread_id AS threadId, agent_id AS agentId, status, kind, parent_run_id AS parentRunId, started_at AS startedAt, ended_at AS endedAt`;

```ts
// runtime-ops/subscribe-issue-timeline.ts —— 新文件（Q4→C：数据归属 opsStore，不让 issue service 跨域依赖）
// 复刻 subscribeIssues 的 catch-up + 长轮询 + 心跳（§2.8 范式）
export async function* subscribeIssueTimeline(
  opsStore: RuntimeOpsStore,
  issueId: string,
  opts?: { signal?: AbortSignal; pollMs?: number },
): AsyncIterable<IssueEvent | { _heartbeat: true }> {
  const pollMs = opts?.pollMs ?? 500;
  let lastSeq = 0;
  let silentPolls = 0;
  const heartbeatInterval = 30;

  const initial = opsStore.getIssueEvents(issueId, 0);
  for (const e of initial) { yield e; lastSeq = e.seq; }

  while (!opts?.signal?.aborted) {
    const next = opsStore.getIssueEvents(issueId, lastSeq);
    if (next.length) {
      for (const e of next) { yield e; lastSeq = e.seq; }
      silentPolls = 0;
    } else {
      if (pollMs === 0) break;
      silentPolls++;
      if (silentPolls % heartbeatInterval === 0) yield { _heartbeat: true };
      await new Promise((r) => setTimeout(r, pollMs));
    }
  }
}
```

> `http.ts` 从 `runtime-ops/subscribe-issue-timeline.js` 导入 `subscribeIssueTimeline`。数据归属（`issue_event` 归 opsStore）决定代码归属——不让 issue service 跨域依赖 opsStore。

**P7-c reject 对称补偿**（`http.ts` reviewDecision reject）：现状若 `deliverableSvc.submit`（rework_feedback）抛错——它在 `applyTransition(in_progress)` 之后——issue 卡 in_progress 无补偿。改为：submit 也纳入补偿范围，失败时 `revertReviewReject` 回 in_review + 502，与 onReviewRejected 失败同路径。

```ts
// reviewDecision reject —— 把 submit 纳入 try，失败走同一补偿
const updated = svc.applyTransition(issueId, "in_progress");
try {
  deliverableSvc.submit({ issueId, fromStatus: "in_review", kind: "rework_feedback", fields: { note: parsed.data.note! } });
  // …emit events…
  await onReviewRejected?.(updated);
} catch (e) {
  console.error(`[orchestrator] rework reject failed for ${issueId}: ${String(e)}`);
  try {
    const reverted = svc.revertReviewReject(issueId);
    emitIssueEvent(opsStore, issueId, "status.advanced", { from: "in_progress", to: "in_review", by: "revert" });
    return json({ error: "rework failed; issue returned to in_review", issue: reverted }, 502);
  } catch (rollbackErr) {
    console.error(`[orchestrator] rollback failed for ${issueId}: ${String(rollbackErr)}`);
    return json({ error: "rework failed and rollback failed", issue: updated }, 500);
  }
}
```

> 这把 M18.6 的补偿从「只覆盖 onReviewRejected」扩到「覆盖 submit + onReviewRejected」，reject 路径任一步失败都对称回滚到 in_review。

**P7-d reject→prompt e2e 测试债**（M18.6 retro:101）：补一条真库 e2e（见 P8）。

## 7. 验收标准

1. 跑一个 Issue 从创建到完成，`GET /api/issues/:id/timeline` 按 seq 升序返回完整事件流：`created`→`started`→（每棒 `run.started`/`run.ended`/`status.advanced`）→`deliverable.submitted`→`human.decided`，时间线无缺漏。
2. 打回返工一轮，Timeline 串出 `human.decided(reject)`→`status.advanced(in_review→in_progress, by:rework)`→`deliverable.submitted(rework_feedback)`→`run.started`（返工棒）→…→再次 `run.ended` 回到 `in_review`。多轮打回事件全留（追加式）。
3. `GET /api/issues/:id/detail` 返回 `{issue, timeline, runs}`；`runs` 来自 `getRunOriginsByIssueId`、每项带 `runId` + 终态概览。
4. **埋点是 best-effort**：人为让 `appendIssueEvent` 抛错，业务主流程（status 推进、起棒、裁决、交付）**全部照常**——观测失败不拖垮业务（§3.4 不变量）。
5. **Timeline SSE 实时**：抽屉打开时订阅 `GET /api/issues/:id/timeline/events`，catch-up 已有事件后，后台返工/推进产生的新事件实时追加到抽屉（无需刷新）。
6. **前端详情抽屉**：点 Issue 卡片打开抽屉，三段（元信息 + Timeline + run 列表）齐全；每个 run 项点击跳 `/ops/runs/[runId]` 的 M16 详情页（不内嵌 trace/insights）；顶部 issue 级诊断概览（N 成功/M 失败/卡在 X）正确。
7. **三套事实隔离**：`issue_event`（issue 级）、`run_ops_event`（run 级）、`ledger`（对话级）互不混入；`issue_event` 无 `run_id` 列、`run_ops_event` 无 `issue_id` 列、工作事件不进账本（不变量 7）。
8. **⑧ 零新采集**：本棒未给 run 级观测加任何采集代码；「issue 的所有 run」完全来自既有 `run_origin.issue_id`（§3.3）。
9. **P7-c 对称补偿**：reject 路径任一步（submit / onReviewRejected）失败 → issue 回滚到 `in_review` + 502；回滚也失败 → 500。
10. **P7-a/b 文档债清**：`issue-workflow.md:160` 已同步为纯 `runId` + `from_status` 列；4 个重复 spec 草稿已删，canonical 唯一。
11. M18.1~M18.6 不回归：Issue CRUD/Project/ColumnConfig/草稿态拖拽/SSE 实时/交付物/上下文累积/人工闸门/返工/补偿全部照旧；`applyTransition`/`setStatus`/run_origin 采集逻辑零改动（埋点是旁路）。
12. `bun test` 全绿；`turbo run build` 后 `turbo run lint typecheck` 通过（runner-protocol dist 先 build，避免陈旧 d.ts 假阳性）。

## 8. 风险与权衡

- **埋点散落多点 → 漏埋/错埋**：status.advanced 有 5 个调用点（§3.4），人工埋容易漏一处或 `by` 标错。缓解：① 埋点位集中列在 spec §6 / §3.4 表，review 对照；② e2e（验收 1/2）断言完整事件链，漏埋即测试红。权衡：不为「自动埋点」引入 AOP/拦截器（那要 service 依赖事件流，破分层 §3.4）——接受「分散埋 + 测试兜底」。
- **best-effort 埋点 → Timeline 可能有洞**：事件写入失败被吞（§3.4），极端情况下 Timeline 缺事件。缓解：Timeline 是观测视角不是真相源（真相在 issue.status / deliverable / run_origin），缺事件不影响业务，且 SQLite 本地写入失败概率极低。权衡：宁可 Timeline 偶有洞，也不让观测拖垮业务（不变量）。
- **SSE 长轮询的连接成本**：每个打开的抽屉一条 SSE 长连。缓解：复用 `subscribeIssues` 同款长轮询（已在生产用于看板，§2.8），连接数受「同时打开的抽屉数」约束（人工操作，量小）；抽屉关闭即 abort。权衡：实时体验（D2）值这点连接成本。
- **detail 聚合端点的 N+1**：`runs` 概览要为每个 run 查 M16 run 表终态。缓解：单 issue 的 run 数有限（每棒一个，含返工也是个位数）；可一条 `WHERE run_id IN (...)` 批量取，不逐个查。权衡：聚合端点的便利 > 轻微查询成本。
- **payload 无 schema 校验 → 脏数据**：IssueEvent.payload 自由 JSON（§3.2/§5），写错字段不报错。缓解：写入侧按 §3.2 表统一字段、集中在 §6 埋点位；读取侧（前端）容错渲染。权衡：schema 自由换来「加 kind/字段零迁移」的演化性，首版接受。
- **issue_event 表无限增长**：纯追加、不清理。缓解：单 issue 事件数有界（生命周期内几十条），全表随 issue 数线性增长但远小于 run_ops_event/ledger。权衡：观测数据保留有价值，归档/TTL 后置（§5）。
- **抽屉跳 M16 详情页的上下文跳出**：点 run 跳到 `/ops/runs/[runId]` 离开看板上下文（D3）。缓解：M16 详情页有返回导航；新标签打开可选。权衡：不内嵌（D3）省下重写成本，跳转是可接受的导航代价。
