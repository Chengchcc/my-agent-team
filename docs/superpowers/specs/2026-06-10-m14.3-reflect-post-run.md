# M14.3 Spec — 反思副作用移出同步 loop（post-run reflect run，退役 internal-role）

> M14.3 解决一个**架构归属错误**：memory reflection 是「一轮 conversation 结束后的副作用」，但当前它**寄生在主 run 的生命周期里同步执行**——主 run 跑完后，runner-stdio 在退出前 `agent.fork().run(reflectionGuidance())` 又跑一整轮反思，`onRunComplete` 因此**等反思也跑完才触发**。后果：(1) 会话锁（`activeConversations` → `convSvc.completeRun`）被反思尾巴一直占着，前端持续 busy；(2) 用户这一轮根本不关心的反思，把他下一条消息顶成 `ConversationBusyError`；(3) 多 agent 房间里整条 channel 被一个 agent 的私有反思阻塞。
>
> 根因：**反思是 run 的尾巴，但锁跟的是 turn**。turn 在 agent 吐完最后一条对话消息时就该结束；反思属于 run、不属于 turn。把锁绑到「主 run 进程退出」=「等反思跑完」，是把 turn 语义错配到了 run 语义上。
>
> 修法（右解，根治）：把反思**移出主 run 进程**，改成 autoTitle 同构的 **post-run job**——主 run 退出 → `onRunComplete` 立刻放锁 → fire-and-forget 起一个**独立 reflect run**（自己的 runId、自己的 thread），不碰会话锁、不进主 run 的 event_log。一旦反思有了独立 runId，它的输出天然不会混进主对话的 D19 投影，`internal-role` 这个「同 runId 内区分对话/反思」的 hack 就**失去存在理由，整体退役**。
>
> 关联：[run/service.ts autoTitle](../../../apps/backend/src/features/run/service.ts)（被复刻的 post-run 模板） · [M11 agent-lifecycle](./2026-06-08-m11-agent-lifecycle-design.md)（reflect turn 引入处） · [M14 spec](./2026-06-09-m14-web-conversation.md)（会话锁 / D19 投影现状）。基线 commit：**`c31fc9e`**（origin/next，A/B/C1–C4 已落地）。

## 一、定位

M14.3 是 **后端 + runner + 框架/core/plugin 的纯重构**，不动任何对外协议、不动前端：

- 改 `packages/runner-stdio/src/entry.ts` — **删掉主 run 内联的 reflect 段**（166–182 行）；改为：当 `spec.mode === "reflect"` 时跑反思一轮，否则跑正常 run。reflect 与 run 在同一个 runner 入口、互斥分支。
- 改 `apps/backend/src/main.ts` — 在 `supervisor.onRunComplete` 里（放锁 + D19 之后）追加：对**非 reflect、非 genesis、非 resume** 的主 run，fire-and-forget `void (async()=>{...})()` 起一个 `mode:"reflect"` 的独立 fork（新 runId、`reflect:` 前缀 threadId），与 autoTitle 同构。
- 改 `packages/agent-spec/src/index.ts` — `mode` 枚举加 `"reflect"`；新增 genesis 判定字段（见 §3.3，runner 不再自己 `existsSync`）。
- 退役 internal-role（11 处 + 删 1 个 adapter 测试，见 §四）：`core/message.ts` `Role` 去 `"internal"`、`ToolResultBlock.role` 删；`core/tool.ts` `ToolExecuteResult.role` 删；`framework/create-agent.ts` `nextRole` 状态机整段删；`plugin-fs-memory/memory-write.ts` 去掉 `role:"internal"` 返回；`adapter-anthropic` 删第 137 行 + 删对应测试；`backend/main.ts` D19 注释更新。
- 改 `packages/harness/src/reflect.ts` 无需动（`reflectionGuidance()` 文案复用）。

**核心不变量**：

- **锁跟 turn，不跟 run 尾巴**——`onRunComplete`（主 run 退出）= turn 结束 = 立刻放锁；反思在锁外、另一个 run 里跑。
- **反思有独立 runId / 独立 thread**——它的 event_log 自成一支，主对话的 D19（按主 runId 读 event_log）永远扫不到反思消息，**无需 role 过滤**。
- **反思仍是 best-effort**——失败 `try/catch` 吞掉、非致命、不影响主 turn 已落地的结果。
- **反思仍写 fs-memory**（write/edit `memory/*.md`、`SOUL.md`、`USER.md`），落点与现状一致。

**M14.3 显式不做（边界）**：

- **不改反思文案**（`reflectionGuidance()` 一字不动）、不改 fs-memory 落点、不改 reflect 的触发条件语义（仍是非 genesis、非 resume 才反思）。
- **不引入 framework post-run hook**——framework 只有 4 个 hook（beforeModel/afterModel/beforeTool/afterTool），本 spec **不新增第 5 个**；反思编排放在 backend，复用既有 supervisor.fork（奥卡姆）。
- **不动 D19 投影逻辑本身**（它按 runId 读 event_log + 写 ledger 的机制不变；只是因为反思换了 runId，自然不再需要 role 字段）。
- **不动 forkRun 的并发语义**——反思 run 用独立 threadId（`reflect:<cid>:<member>`），与主对话 threadId 不冲突，不需要额外并发阀（见 §3.4）。
- **不保留 internal-role 做向后兼容**——全量退役，无 fallback（线上无依赖该字段的存量数据）。

## 二、问题根因（M14.3 要消除的）

```text
主 run（用户这一轮真正关心的）
  └─ agent 吐完最后一条对话消息  ← turn 在这里就结束了，锁本该这里放
  └─ runner-stdio 继续：agent.fork().run(reflectionGuidance())   ← 反思尾巴
        └─ 又跑一整轮 LLM + write/edit memory（可能数秒～数十秒）
  └─ 进程退出 → supervisor 触发 onRunComplete   ← 锁到这里才放！
        └─ convSvc.completeRun() 放锁 / 前端这才不 busy

崩溃点 1  ← 锁被反思尾巴占住：前端 busy 时长 = 主 turn + 反思，膨胀且不可预期
崩溃点 2  ← 反思期间用户发下一条 → ConversationBusyError 误拒（他根本不关心反思）
崩溃点 3  ← 多 agent 房间：一个 agent 私有反思阻塞整条 channel 的 completeRun 扫描
共同根因  ← 反思是 run 的尾巴，却让 turn 的锁去等 run 退出（turn/run 语义错配）

         ── 连带 hack ──
internal-role 之所以存在，是因为反思与主 run 共用同一个 runId：
  反思的 assistant 输出也进了主 runId 的 event_log，
  D19 按主 runId 扫 event_log 写 ledger 时会把反思消息也投影进对话，
  于是用 role:"internal" 给反思输出打标，让 adapter / D19 / 背书过滤掉它。
  → 一旦反思有独立 runId，D19 根本扫不到它，这个 role 标记彻底多余。
```

修复后：

```text
主 run（turn）
  └─ agent 吐完最后一条对话消息 → 进程退出
  └─ onRunComplete 立刻：completeRun 放锁 + D19 投影对话   ← 前端 busy 在这里结束

  （锁外、异步、另一个 run）
  post-run reflect job  ← backend fire-and-forget
  └─ supervisor.fork(reflectRunId, "reflect:<cid>:<member>", reflectSpec)
        └─ runner-stdio 走 mode:"reflect" 分支：agent.run(reflectionGuidance())
        └─ 写 fs-memory；event_log 落在 reflectRunId（与主对话隔离）
        └─ 失败 try/catch 吞掉，非致命
```

**turn 与 run 的边界第一次对齐**：turn = 主 run 退出 = 放锁点；reflect = 锁外独立 run。锁不再为用户不关心的副作用买单。

## 三、设计决策

### 3.1 为什么是「独立 reflect run」，不是「backend 内联 LLM 调用」（对比 autoTitle）

autoTitle 是**一次性 LLM 调用**（`generateTitle` → 一个 `AnthropicChatModel.stream` → 取标题），不需要工具、不需要 agent loop。反思**不同**：它要 `write`/`edit` memory 文件，必须跑**完整 agent tool loop**（带 fs-memory 插件、带 permissionMode）。

所以反思不能像 title 那样在 backend 进程内直接调 model；它必须复用现成的「起一个带完整工具栈的 agent run」基础设施 = `supervisor.fork(runId, threadId, specJson)`。

**结论**：编排模式抄 autoTitle（`onRunComplete` 里 fire-and-forget），但执行体抄主 run（`supervisor.fork` + runner-stdio）。两者结合 = 「post-run job 起一个独立的、带工具的 reflect run」。

### 3.2 reflect run 复用 runner-stdio 入口，用 `mode:"reflect"` 分支

不为反思单开一个 runner 二进制。`runner-stdio/entry.ts` 已是「parse spec → 建 model/agent → 跑一轮」的通用入口；反思只是「跑的那一轮 input 换成 `reflectionGuidance()`」。

```ts
// entry.ts —— 删掉 166–182 的内联 reflect 段；run/resume/reflect 三分支互斥
const stream =
  spec.mode === "resume"
    ? agent.resume(spec.resumeCommand!, { signal, maxSteps: spec.maxSteps, stream: true })
    : spec.mode === "reflect"
      ? agent.run(reflectionGuidance(), { signal, maxSteps: spec.maxSteps, stream: true })
      : agent.run(spec.input, { signal, maxSteps: spec.maxSteps, stream: true });
```

reflect 分支与 run 共用同一段 `for await` 事件落库逻辑——但因为 reflect spec 带的是**独立 runId**，事件落到独立 event_log 分支，主对话 D19 扫不到。runner 内**不再有** `isGenesis`/`fork()`/第二段 loop。

### 3.3 genesis 判定移交 backend（runner 不再 `existsSync`）

现状 runner 自己 `existsSync(BOOTSTRAP.md)` 判 genesis 来决定跳过反思。移出后，**触发反思的是 backend**，所以「这次主 run 是不是 genesis」要 backend 知道，不能等 runner。

两种选型：

- **(A 推荐)** backend 在起主 run 前已能判定 genesis（它持有 workspace 路径），把结论作为 spec 字段 `isGenesis: boolean` 下发；`onRunComplete` 里据此决定是否起反思。runner 不再判 genesis。
- (B) backend 在 `onRunComplete` 里现场 `existsSync(workspace/BOOTSTRAP.md)`。简单但把文件系统判断散进 onRunComplete，且 workspace 路径要从 agentRow 重查。

选 **A**：genesis 是「主 run 启动那一刻」的快照语义，本就该在起 run 时定，下发给需要它的两端（主 run 跳过反思 + onRunComplete 跳过起反思）。`AgentSpecV1` 加可选 `isGenesis`。

### 3.4 reflect run 的 threadId 与并发——为什么不需要额外锁

主对话 threadId = `<cid>:<memberId>`。reflect run 用 **`reflect:<cid>:<memberId>`** 前缀，与主 thread **不同 key**：

- checkpointer 隔离：反思走自己的 thread，不污染主对话的 `checkpoint_messages`（等价于旧 `agent.fork()` 的隔离目的，但用独立 thread 实现，更直白）。
- **但反思需要主对话的上文**才能反思。旧实现 `agent.fork()` 是 `structuredClone(thread.messages)` 把主线程内存里的消息拷过去。新实现 backend 起独立进程，没有这块内存——所以 reflect spec 要让 runner **从主 thread 的 checkpointer 加载上文**：reflect 的 agent 用**主 threadId 读 checkpointer**（`createAgent` 里 `checkpointer.load(threadId)`），再把反思那一轮**写到一个临时 thread**。

这带来一个 framework 约束冲突需在 spec 落地时定夺（见下 ⚠️）。

> ⚠️ **关键实现约束（落地时必须解决）**：framework 的 `createAgent(threadId)` 把「读上文的 thread」和「写新消息的 thread」绑成同一个 `threadId`——`checkpointer.load(threadId)` 读、`save(threadId)` 写。旧 `agent.fork(msgs, newId)` 之所以能「读主线程内存上文 + 写临时 thread」，是因为它在**同一进程内**把已加载的 `thread.messages` 克隆给新 agent。跨进程的 reflect run 拿不到这块内存，只能从 checkpointer 读，而 `createAgent` 读写同 thread。
>
> 两条落地路径，spec 实现时二选一：
> - **(路径 1 推荐) reflect 用主 threadId 读、临时 thread 写**：给 runner 的 reflect 分支加一步——`createAgent({threadId: spec.threadId})` 加载主上文后，立即 `agent.fork(undefined, "reflect:"+spec.threadId)` 得到「带主上文、写临时 thread」的 agent，再 `.run(reflectionGuidance())`。即 **fork 语义仍在，但移进 runner 的 reflect 分支内部**，且 fork 的 source 是 checkpointer 加载的主上文。锁与触发已在 backend 外移；fork 仅剩「checkpoint 隔离」这一个正当用途。
> - (路径 2) reflect 直接用主 threadId 读写，接受反思那一轮的 user/assistant 落进主 thread 的 checkpoint_messages。简单，但反思污染主对话历史（下一轮主 run 会把反思也当上文），**不可取**。
>
> **采路径 1**：保留 `fork()` 作为「带上文 + 写隔离 thread」的纯机制，但它不再承载任何 role 语义、不再在主 run 尾巴跑、不再被锁等待。

### 3.5 internal-role 整体退役——它的唯一存在理由已消失

internal-role 的全部目的：反思与主 run 共用 runId 时，让反思的 assistant 输出不被 D19 投影进对话、不进模型上下文。链路：`memory_write` 返回 `role:"internal"` → framework `nextRole` 状态机给下一条 assistant 打 `internal` → adapter buildApiMessages 跳过 internal → D19 过滤 `role==="assistant"|"user"` 排除 internal。

反思换独立 runId 后：

- D19 按**主 runId** 读 event_log，反思在**反思 runId**，**物理上扫不到**——无需 role 过滤。
- 反思那一轮的上下文是反思自己的 thread，主 run 的模型上下文里本就没有反思消息——adapter 无需跳过 internal。
- `memory_write` 即便在主 run 中途被调用，它返回的 `role:"internal"` 本意是「别让保存记忆后模型那句话进对话」——但保存记忆是工具调用，工具结果后模型的话本就是正常 assistant 输出，**没有「保存后必须沉默」的产品需求**（这是当初为反思顺带加的耦合）。退役后，主 run 里 memory_write 后模型若再说话，就是正常对话的一部分，符合直觉。

**故 internal-role 11 处全删**，无保留。

### 4.3 `apps/backend/src/main.ts` —— onRunComplete 追加 post-run reflect job

在 §一所列 `onRunComplete`（L224–279）**末尾**、`break` 之前，追加 fire-and-forget 反思编排。判据：只对**主 run**（spec 非 reflect）、**非 genesis**、**非 resume** 起反思。

实现要点：
- 需要知道「刚结束的这个 run 是不是 reflect run」——否则反思 run 结束又触发反思 = 无限递归。用 threadId 前缀判：`if (threadId.startsWith("reflect:")) { threads/锁清理后直接 return，不投影不反思 }`（reflect run 不进 activeConversations 扫描，也不该走 D19）。
- 需要主 run 的 genesis 标记——从 run 行或起 run 时记录。最简：起主 run 时 backend 已 `buildSpecJson` 持有 `isGenesis`，可记进内存 map `runId→{isGenesis, mode, agentId, conversationId, memberId}`，onRunComplete 查表。

```ts
supervisor.onRunComplete((threadId, runId) => {
  // reflect run 自身结束：不放会话锁、不 D19、不再起反思（防递归）
  if (threadId.startsWith("reflect:")) return;

  for (const cid of activeConversations) {
    if (threadId.startsWith(`${cid}:`)) {
      convSvc.completeRun(cid, threadId, runId);   // ← 放锁，前端 busy 在此结束

      void (async () => { /* D19 投影：不变 */ })();

      // M14.3：post-run 反思，fire-and-forget，锁外，独立 run
      const meta = runMeta.get(runId);   // { isGenesis, mode, agentId, agentMemberId }
      if (meta && meta.mode === "run" && !meta.isGenesis) {
        void (async () => {
          try {
            const reflectRunId = ulid();
            const reflectThreadId = `reflect:${threadId}`;
            const specJson = buildSpecJson(/* agentId=meta.agentId */ ..., {
              mode: "reflect",
              threadId,                 // ← 用主 threadId，让 runner 加载主上文后再 fork
              runId: reflectRunId,
              conversationId: cid,
              senderMemberId: meta.agentMemberId,
              input: "",
            });
            supervisor.fork(reflectRunId, reflectThreadId, specJson);
          } catch (err) {
            console.error(`[reflect] failed for ${runId}:`, err instanceof Error ? err.message : String(err));
          }
        })();
      }
      break;
    }
  }
});
```

> `buildSpecJson` 里 spec.threadId 用主 threadId（供 runner `checkpointer.load` 主上文）；`supervisor.fork` 的 threadId 参数用 `reflect:` 前缀（run 表登记 + 防递归识别）。两者职责不同——前者是 agent 读上文的 key，后者是 supervisor/run 表的隔离 key。**落地时确认 supervisor.fork 是否把第二参 threadId 透传进 spec**；若是，需让 spec.threadId（读上文）与 supervisor 登记 threadId（隔离）解耦：给 reflect spec 显式带 `threadId=主threadId`，supervisor 用 `reflect:` 前缀登记。

需新增 `runMeta` map 并在起主 run（`forkRun` / runs 路由起 run 处）写入 `{isGenesis, mode:"run", agentId, agentMemberId}`，`onRunComplete` 用后 `runMeta.delete(runId)`。

## 五、不变量（invariant）

1. **锁在主 run 退出即释放**——`onRunComplete` → `convSvc.completeRun` 不等任何反思；前端 busy 时长 = 主 turn，不含反思。

3. **无递归反思**——reflect run 结束的 `onRunComplete` 命中 `reflect:` 前缀直接 return，不再起反思。
4. **反思失败非致命**——backend fire-and-forget try/catch + runner 内 reflect 分支异常仅记 stderr。
5. **反思触发条件不变**——非 genesis、非 resume、且是主 run（`mode==="run"`）才反思。
6. **`Role` 不再含 `"internal"`**——全代码库无 `role: "internal"` 字面量、无 `role==="internal"` 判断。
7. **反思仍能读到主对话上文**——reflect run 以主 threadId 从 checkpointer 加载上文后再 fork。

### 6.1 backend 反思编排 P1-b 测试（前置：抽出可注入纯函数）

>
> **前置重构（P1-b 的前提）**：把 onRunComplete 里反思那一段抽成**可注入依赖的纯函数** `orchestrateReflection`，放进 `apps/backend/src/features/run/reflect-orchestrator.ts`，main.ts 改为调用它。抽出后既可单测、又让 main.ts 回归「装配」职责。

**新增 `apps/backend/src/features/run/reflect-orchestrator.ts`**：

```ts
export interface ReflectMeta { isGenesis: boolean; agentId: string; agentMemberId: string }

export interface ReflectOrchestratorDeps {
  runMeta: Map<string, ReflectMeta>;
  genId: () => string;                         // ulid（测试可注入确定性桩）
  buildSpecJson: (
    threadId: string, input: string,
    opts: { mode: "reflect"; runId: string; conversationId: string; senderMemberId: string },
  ) => Promise<string>;
  fork: (runId: string, threadId: string, specJson: string) => void;   // supervisor.fork 的窄签名
  onError?: (runId: string, err: unknown) => void;
}

/** 主 run 退出后触发反思。返回是否真的起了反思（供测试断言 / 调用方决策）。
 *  门控：reflect run 自身（reflect: 前缀）不递归；无 meta（resume，P1-a）跳过；genesis 跳过。
 *  调用契约：必须在 completeRun(放锁) 之后调用——锁与反思解耦由调用方保证顺序。 */
export async function orchestrateReflection(
  threadId: string, runId: string, conversationId: string, deps: ReflectOrchestratorDeps,
): Promise<boolean> {
  if (threadId.startsWith("reflect:")) return false;     // 防递归
  const meta = deps.runMeta.get(runId);
  try {
    if (!meta || meta.isGenesis) return false;           // P1-a resume 无 meta / genesis 跳过
    const reflectRunId = deps.genId();
    const specJson = await deps.buildSpecJson(threadId, "", {
      mode: "reflect", runId: reflectRunId,
      conversationId, senderMemberId: meta.agentMemberId,
    });
    deps.fork(reflectRunId, `reflect:${threadId}`, specJson);
    return true;
  } catch (err) {
    deps.onError?.(runId, err);                           // best-effort，非致命
    return false;
  } finally {
    deps.runMeta.delete(runId);                           // 无论起没起反思都清表
  }
}
```

`main.ts` onRunComplete 改为(放锁 + D19 之后)：

> 注：`reflect:` 前缀的早 return 现状在 onRunComplete **最外层**（不放锁、不 D19）。抽出后保留外层早 return 用于「reflect run 不放锁/不投影」；`orchestrateReflection` 内的同款前缀判断是**防御性第二道闸**（即便被误调也不递归）。两处并存,语义不重复。

```ts
import { describe, expect, test } from "bun:test";
import { orchestrateReflection, type ReflectMeta } from "./reflect-orchestrator.js";

function harness(seed: Partial<ReflectMeta> & { runId: string }) {
  const forkLog: Array<{ runId: string; threadId: string; specJson: string }> = [];
  const runMeta = new Map<string, ReflectMeta>();
  runMeta.set(seed.runId, {
    isGenesis: seed.isGenesis ?? false,
    agentId: seed.agentId ?? "ag-x",
    agentMemberId: seed.agentMemberId ?? "mem-x1",
  });
  let n = 0;
  const deps = {
    runMeta,
    genId: () => `reflect-rid-${++n}`,
    buildSpecJson: async (
      _t: string, _i: string,
      o: { mode: "reflect"; runId: string; conversationId: string; senderMemberId: string },
    ) => JSON.stringify(o),
    fork: (runId: string, threadId: string, specJson: string) =>
      forkLog.push({ runId, threadId, specJson }),
  };
  return { forkLog, runMeta, deps };
}

describe("M14.3 P1-b: reflect orchestration", () => {
  // P1-b-1: 正常主 run → 起独立 reflect run（新 runId + reflect: 前缀 thread），并清 runMeta
  test("forks an independent reflect run for a normal main run", async () => {
    const { forkLog, runMeta, deps } = harness({ runId: "run-1" });
    const started = await orchestrateReflection("conv-a:mem-x1", "run-1", "conv-a", deps);
    expect(started).toBe(true);
    expect(forkLog).toHaveLength(1);
    expect(forkLog[0]!.runId).toBe("reflect-rid-1");          // 独立 runId（≠ 主 run-1）
    expect(forkLog[0]!.threadId).toBe("reflect:conv-a:mem-x1"); // reflect: 前缀隔离
    const spec = JSON.parse(forkLog[0]!.specJson);
    expect(spec.mode).toBe("reflect");
    expect(spec.senderMemberId).toBe("mem-x1");
    expect(runMeta.has("run-1")).toBe(false);                 // finally 清表
  });

  // P1-b-2: 防递归 —— reflect run 自身结束（reflect: 前缀）不再起反思
  test("does not recurse when the completed run is itself a reflect run", async () => {
    const { forkLog, deps } = harness({ runId: "reflect-rid-x" });
    const started = await orchestrateReflection("reflect:conv-a:mem-x1", "reflect-rid-x", "conv-a", deps);
    expect(started).toBe(false);
    expect(forkLog).toHaveLength(0);                          // 零 fork = 无限递归被斩断
  });

  // P1-b-3: genesis 跳过 + resume(无 meta)跳过 —— 触发条件门控
  test("skips reflection for genesis runs and for runs without meta (resume)", async () => {
    // genesis：meta 存在但 isGenesis=true
    const g = harness({ runId: "run-genesis", isGenesis: true });
    expect(await orchestrateReflection("conv-a:mem-x1", "run-genesis", "conv-a", g.deps)).toBe(false);
    expect(g.forkLog).toHaveLength(0);
    expect(g.runMeta.has("run-genesis")).toBe(false);        // genesis 也清表

    // resume：runMeta 里根本没有该 runId（forkRun 未登记）
    const r = harness({ runId: "other" });
    expect(await orchestrateReflection("conv-a:mem-x1", "run-resume-unknown", "conv-a", r.deps)).toBe(false);
    expect(r.forkLog).toHaveLength(0);
  });
});
```

**为什么是这三条**：覆盖 §五不变量 3（防递归）、5（触发条件：非 genesis / 非 resume / 主 run）、以及编排的「起独立 run + 清表」主路径——即 d12e0a5 自称却未真正覆盖的反思编排链。「completeRun 先于 fork」的顺序属于 main.ts 装配契约（§4.3），由 `orchestrateReflection` 的「必须在放锁后调用」文档契约 + main.ts 调用点保证；纯函数层不重复测顺序（顺序是调用方职责，已在 §六主表 backend onRunComplete 行覆盖）。
