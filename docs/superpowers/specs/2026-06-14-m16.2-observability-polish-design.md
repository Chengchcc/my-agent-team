# M16.2 — Observability Polish：导航统一、可读身份、产品收口

> 根因：M16.1 把观测面产品化了——动线、诊断、控制、trace、surface 都接上了。但它是**功能正确、表层未收口**的状态：导航语义分裂（一半叫 Observatory、一半叫 Observability，左栏用一套配色、ops 用另一套），所有 ops 视图把 agent 当成裸 `agentId` 字符串甩给运维看。运维进来第一眼看到的不是「哪个 agent 坏了」，而是 `a75b93c63655983fd — Runtime`。
>
> M16.2 不加新业务能力，只做**收口**：把 M16.1 §4.1 早就画好但没落地的「Workspace / Operations」导航分组真正做出来；把两套其实指向同一批颜色的 styling 约定收敛成一套；把 ops 全链路的 `agentId` 换成人类可读的 displayName（后端单一事实源下发，前端不各自查）。外加一轮产品层 gap 盘点，把该做的下一步定下来。
>
> 关联：[M16.1 Observability Console](./2026-06-14-m16.1-observability-console-design.md) · [M16 Runtime Observability](./2026-06-14-m16-runtime-observability-design.md)。
>
> 基线 commit：`537c9f0`（`fix: M16.1 7 bugs from code review`，已含 M16.1 全部页面、ops-diagnosis、QueryState、surfaces 拍平、attempt 排序修复）。

## 一、问题：功能对了，产品没收口

### 1.1 导航语义分裂

左栏（`NavRail.tsx`）现在是：

```text
Observatory                  ← header，链接到 /agents，颜色 var(--ink-strong)
  Observability              ← 一个链接，指向 /ops
  Agents                     ← section header
    <agent.name>
    <agent.name>
  Conversations              ← 选中 agent 后才出现
```

三个问题叠在一起：

1. **命名打架**：产品入口叫 "Observatory"，里面又有个叫 "Observability" 的链接，运维分不清这俩是不是一个东西。
2. **层级混乱**：`Observability` 链接和 `Agents` section 平铺在同一块里，没有「这是工作区、那是运维域」的语义分组。M16.1 §4.1 已经画了目标形态：

   ```text
   Workspace
     Agents
     Conversations
   Operations
     Overview / Runs / Agents / Traces / Surfaces
   ```

   但实现只加了一个孤立的 `Observability` 链接，分组从未落地。
3. **`/ops` 内的二级导航（`OpsTabs`）和左栏完全脱节**：进了 `/ops` 才看到五个 tab，左栏不反映你在 ops 的哪一层。

### 1.2 两套配色，其实是同一批颜色

代码里有两套 styling 约定：

- **裸 CSS 变量**：`var(--canvas)` / `var(--ink)` / `var(--mute)` / `var(--primary)` / `var(--hairline)`……用于 `NavRail`、`AppShell`、所有 chat 组件、agents 页面（26 个文件）。
- **Tailwind 语义 token**：`text-foreground` / `text-muted-foreground` / `bg-muted` / `border-border` / `border-primary`……用于所有 `ops/*` 组件和 ops 页面（33 个文件）。

关键事实：**这两套指向完全相同的颜色**。`globals.css` 的 `@theme inline` 把 Tailwind token 定义成裸变量的别名。

所以这不是「配色不一致」——视觉上颜色是对的。问题是**两套写法并存**：

- 后来人改一个组件不知道该跟哪套，复制粘贴时混入第三种写法。
- 有两个裸变量**没有 Tailwind 对应物**：`--ink-strong`（#ffffff）和 `--body`（#bdbdbd）。
- 长期看，「同一概念两种写法」就是 tech debt。

### 1.3 运维看到的是机器 ID，不是 agent

所有 ops 视图把 agent 渲染成裸 `agentId`：RunOpsTable、AgentRuntimeCard、SurfaceHealthPanel、TraceWaterfall、NeedsAttentionList、agent runtime detail page。

根因是数据形状：ops DTO 只携带 `agentId`，从不带 name。

## 二、目标形态：一套导航、一套配色、一个名字来源

M16.2 收口三件事，外加一轮产品 gap：

```text
导航     Workspace / Operations 两段式分组，命名统一为 Operations
配色     收敛到单一约定（Tailwind 语义 token），补齐缺失的两个 token
身份     后端在 ops DTO 下发 agentName（单一事实源），前端全链路显示 displayName
收口     盘点产品层 gap，6 项（G1~G6）全部纳入本期
```

不变量：

1. **不加新业务能力**：M16.2 不新增 ops 数据维度、不改 run status、不动诊断逻辑。
2. **配色不变值，只换写法**：收敛到 Tailwind token 后，渲染出的颜色必须和现在逐像素一致（因为是别名）。
3. **agentName 是派生展示字段，不是事实源**：agent 的事实源仍是 agent registry。
4. **名字来源单一**：前端不再各页各自 `listAgents` 拼名字；后端在 ops 响应里直接带 `agentName`。
5. **ID 不消失，只降级**：displayName 是主显示，但 agentId 作为唯一标识仍需可见（hover title / 次要灰字）。

## 三、导航统一（Part A）

### 3.1 目标结构

```text
┌─ Header：Operations ──────┐
├─ Workspace ───────────────┤
│  Agents                   │
│    <agent.name>           │
│  Conversations            │
├─ Operations ──────────────┤
│  Overview      /ops       │
│  Runs          /ops/runs  │
│  Agents        /ops/agents│
│  Traces        /ops/traces│
│  Surfaces      /ops/surfaces│
└─ Footer：Sign Out ────────┘
```

方案 A（推荐）：左栏 Operations 段直接列出五个二级入口，删除 `OpsTabs`。导航只有一处事实源——左栏。

### 3.2 折叠态

折叠态两个图标：Workspace（agents 图标）、Operations（四宫格图标）。

## 四、配色收敛（Part B）

### 4.1 收敛方向：Tailwind 语义 token

裸变量保留在 `globals.css` 定义层，但**业务组件不再直接写 `var(--*)`**。

### 4.2 缺失 token 补充

```css
--color-foreground-strong: var(--ink-strong);
--color-foreground-muted:  var(--body);
```

### 4.3 迁移映射

| 裸变量 | 迁移到 |
|---|---|
| `var(--canvas)` | `bg-background` |
| `var(--canvas-soft)` | `bg-muted` / `bg-secondary` |
| `var(--ink)` | `text-foreground` |
| `var(--ink-strong)` | `text-foreground-strong` |
| `var(--body)` | `text-foreground-muted` |
| `var(--mute)` | `text-muted-foreground` |
| `var(--hairline)` | `border-border` |
| `var(--primary)` | `text-primary` / `bg-primary` |

### 4.4 迁移范围

1. **第一批**：`NavRail.tsx` + `AppShell.tsx`（导航重构对象，顺手迁）
2. **第二批**：`ui/button.tsx`、`ui/select.tsx`（已混用）
3. **第三批（可选，可下一期）**：chat 组件群

## 五、可读身份（Part C）

### 5.1 后端下发 agentName

所有 ops DTO 增加 `agentName: string`，从 agent registry join：

- `RunOpsListItem` / `RunOpsDetail.run` / `AgentRuntimeStatus` / `SurfaceOpsItem`

对应前端 `api.ts` 的类型同步加 `agentName: string`。

### 5.2 前端显示规则

displayName 主显示，agentId 降级为可定位的次要信息：

```tsx
<span title={r.agentId}>{r.agentName}</span>
```

各处改造：RunOpsTable、AgentRuntimeCard、SurfaceHealthPanel、TraceWaterfall、NeedsAttentionList、agent runtime detail page。

## 六、产品层 gap（Part D，全部纳入）

### G1. Runs 筛选 UI（兑现 M16.1 §4.3 承诺）

M16.1 fixes Bug 7 移除了 transport/heartbeat/traceId 参数（方案 B）。现在走方案 A：后端 `listRuns` 端到端支持过滤 + 前端筛选 UI。

### G2. 自动刷新

ops 页面 React Query 加 `refetchInterval`（Overview/Runs 30s，run detail running 态 10s）。

### G3. Needs Attention 可操作性

列表项直接挂快捷动作（对 detached run 给 Recover 按钮），复用 `RunControlStrip` 的 mutation。

### G4. 空状态引导

各 ops 页 `QueryState` empty 分支给具体引导文案。

### G5. 诊断依据可展开

诊断头旁加 disclosure，展开显示触发诊断的具体字段值。

### G6. 多 attempt 回归测试

补多 attempt fixture 断言。

## 八、验收标准

1. 左栏统一为 Workspace / Operations 两段式
2. Operations 段直达五个入口，高亮正确
3. 业务组件不再直接写 `var(--*)`（chat 组件群除外）
4. 所有 ops 视图显示 agent displayName，agentId 降级可见
5. agentName 由后端 ops DTO 单一下发
6. G1~G6 全部实现并测试
7. typecheck + 测试全绿

## 九、边界

1. 不加新 ops 数据维度、不改 run status、不动诊断判定逻辑
2. 不强制迁移 chat 组件群配色
3. 不做 agent 改名后实时名字覆盖
4. 不接入真实 OTLP collector（继承 M16.1 边界）
