# M3 Retro — Framework 装配层

## Delivered

- `@my-agent-team/framework`: `createAgent` + `Agent` + `definePlugin` + `slidingWindow` + `consoleLogger` + `fileCheckpointer`
- Thread / Plugin / HookContext / Checkpointer 类型
- L3 自实现 while loop（不调 L2 `run()`），复用 `collectStream`

## Actual vs Spec

### Matched

- createAgent async，checkpointer.load 在构造时
- Agent.run 自动插 systemPrompt（仅首次）
- Agent.fork 复用 config，默认空 messages
- beforeModel 管道链，beforeTool skip 语义
- after* 吞错，before* 传播
- checkpointer.save 在 afterTool + 纯文本 return 前
- signal.throwIfAborted 早抛

### Differences

1. **并发保护未实现** — spec 写了 `#running` guard，当前允许多 run 并发
2. **`createAgentInternal` 重复** — 两处 ~150 行 loop 代码几乎相同
3. **`fileCheckpointer` 忽略 threadId** — 存储格式 `{ messages }`，多 thread 场景用不了
4. **`Agent.run` 签名不跨包复用** — 用 `{ signal, maxSteps }` 内联，未用 core 的 `RunOptions`

## Lessons

- L3 不调 L2 `run()` 是正确选择——L2 的 generator 太封闭，无法在中间 fire hooks
- `collectStream` 复用合理，不需要在 L3 重写 delta 累积
- createAgentInternal 重复是信号——应早点 refactor，不等 Rule of Three
- Checkpointer 独立字段 + Plugin 数组——比 LangChain 的 monolithic config 清晰很多
- fileCheckpointer 的单文件限制是实现简化而非设计限制
- 并发保护只需 4 行代码，应第一天实现
