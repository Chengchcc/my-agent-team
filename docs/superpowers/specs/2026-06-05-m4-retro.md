# M4 Retro — Framework 补齐

## Delivered

- `AgentEvent` envelope (`{ type, payload }`): `Agent.run()` / `Agent.resume()` 返回 `AsyncIterable<AgentEvent>`
- `Interrupt` 独立类型, `ResumeCommand = { approved, message? }`
- `Agent.resume()`: interrupt flow（save → saveInterrupt → yield interrupted → resume → push tool_result → runLoop）
- `#runLoop` / `#executeOne` 三层架构，布尔返回值传 interrupt 信号
- `InterruptSignal` class: `InterruptSignal` 仅在 `tool.execute` 抛时被识别（严格边界）
- Checkpointer: 6 方法 + 成对校验 (`validateCheckpointer` on create)
- `inMemoryCheckpointer()` (默认) + `fileCheckpointer({ dir })` (原子写 tmp+rename, `node:fs/promises`)
- `CheckpointEvent` 重命名 (原 `AgentEvent`——避免与 framework yield 的 `AgentEvent` 撞名)
- ContextManager interface + `ContextManagerContext` (`model` + `logger` injection)
- 5 built-in CMs: `passthrough`, `slidingWindow`, `tokenBudget`, `toolResultTruncator`, `summarizing`
- `pipeContextManagers` with fail-fast error propagation
- summarizingContextManager 3-tier (user summarizer → summarizerModel → ctx.model + builtin prompt)
- tokenBudgetContextManager 3-tier (user countTokens → ctx.model.countTokens → char approx)
- `ChatModel.countTokens?` non-breaking optional method in core
- Logger: `consoleLogger` (level 过滤) + `noopLogger`
- `HookContext` 暴露 `logger` / `checkpointer` / `contextManager` (三字段必需)
- `fork()` 强制新 threadId (同 id throw, 无 id auto-uuid)
- Save timing: 5-point spec
- `plugins/` directory deleted; `slidingWindow` → `context-managers`, `consoleLogger` → `logger.ts`
- `before*` short-circuit / `after*` continue error isolation (by hook type)
- 7 arch docs updated, M4 spec rewritten with 17 grill-session decisions

## Actual vs Spec

### Matched

- AgentEvent envelope `{ type, payload }`, Interrupt independent type
- Resume approved/denied 语义, tool_result is_error 映射
- #runLoop 入口契约, #executeOne 布尔返回值
- InterruptSignal 严格边界 (仅 tool.execute 抛)
- Checkpointer 6 methods, validateCheckpointer, CheckpointEvent rename
- inMemoryCheckpointer 默认, fileCheckpointer({ dir }) atomic write
- threadId validation: no leading dot, no all-dots
- ContextManagerContext.model + logger
- 5 CMs with 3-tier fallback for summarizer and countTokens
- ChatModel.countTokens? optional method
- Logger with level filtering, used by framework internally
- HookContext 3 required fields
- fork forced new threadId
- 5-point save timing
- plugins/ deleted, code migrated
- before* short-circuit / after* continue
- 83 tests (18 M3 regression + 17 M4 new + 48 component)

### Differences

1. **Tool result messages not yielded**. Old code yielded both assistant and user(tool_result). New code yields only assistant (and interrupted). Tool results are pushed to thread.messages but not streamed. This is a behavioral change from M3 — calling code that relied on streaming tool results will need to check agent.thread.messages instead.
2. **maxSteps per-call**. M3 had maxSteps configurable per `run()` call. M4 maintains this but requires explicit type annotations due to TypeScript inference of default parameter types.
3. **`scriptedModel` doesn't consume maxSteps**. The scriptedModel used in tests always returns the same response regardless of step count. This is fine for tests but means maxSteps is the *only* termination condition for infinite tool loops.
4. **slidingWindow turn detection simplified**. The turn-splitting algorithm was rewritten to be simpler but may not handle complex tool_use/tool_result pairing as robustly as the spec's "pairing-aware" promise. Current implementation treats consecutive user(tool_result) messages as part of the same turn as the preceding user message.

## Lessons

- **Grill session before implementation pays off**. 17 design questions surfaced naming collisions (AgentEvent vs CheckpointEvent), control flow boundaries (InterruptSignal strict recognition), and injection patterns (ContextManagerContext.model). Solving these before code avoided rework.
- **TDD wasn't strict enough — but tests came first for critical components**. The create-agent tests were written before debugging, catching type inference issues (maxSteps default, HookContext required fields) immediately. Component tests (logger, CMs, checkpointers) were written after implementation and mostly passed first try.
- **TypeScript inference of default parameters is a footgun**. `opts = {}` silently typed as `{}` rather than the interface type, causing maxSteps to always default to 32. Explicit type annotations are mandatory for destructured opts parameters.
- **`readonly Message[]` vs `Message[]` is pervasive**. The Plugin.beforeModel hook signature returns `Message[]` but receives `readonly Message[]`. Several test patterns returned the input directly, requiring `[...msgs]` to satisfy the mutable return type. This is a design tension — the interface wants to promise non-mutation of input but also allow mutation of output.
- **Biome `noNonNullAssertion` + optional checkpointer methods = constant friction**. Checkpointer's optional methods (saveInterrupt?, etc.) force `!` or `?.` at every call site. The `?.` pattern caused `noUnsafeOptionalChaining` with for-of loops. A better pattern would be capability-checking helpers.
- **`consoleLogger` test difficulty**. Testing that a logger calls `console.log` without mocking is impossible in Bun (no spy on global console). Tests just verify "doesn't throw" — weak coverage.
- **slidingWindow turn detection is the hardest CM to get right**. Multiple rewrites. The current version treats consecutive non-user messages as part of the preceding turn, which is simpler than the spec's pairing-aware approach but may miss edge cases.
