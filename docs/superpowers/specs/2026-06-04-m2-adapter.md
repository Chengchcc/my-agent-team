# M2 Spec — Anthropic Adapter · 通用工具 · CLI

> 基于 M1 core 协议(L1+L2)。
> 目标:用 Anthropic 官方 SDK 接入真 LLM,交付一组可复用的通用 tool,
> 并通过 CLI 实现首次"真 LLM → 真 tool → 真结果"闭环。

---

## 一、设计原则（沿用 M1,不重复展开）

1. **第一性原理** — 从 M1 的事实清单出发:M1 有了协议和运行时,M2 只需补上"真 LLM"和"真 tool"两个缺口
2. **奥卡姆剃刀** — adapter 做纯翻译(SSE → Chunk, Tool → Anthropic Tool),不加一行多余逻辑
3. **状态归调用方** — memory store 是调用方注入的 `Map`,CLI 持有 messages 数组
4. **Rule of Three** — email tool / weather tool / calculator 等零散工具不做,只做 6 个在真实对话中反复出现的通用 tool

---

## 二、四层架构中的位置

```text
┌────────────────────────────────────────────────────────────────┐
│ L4. Harness                                     (M4)           │
├────────────────────────────────────────────────────────────────┤
│ L3. Framework                                   (M3)           │
├────────────────────────────────────────────────────────────────┤
│ L2. Runtime (run loop)                          (M1 core)      │
├────────────────────────────────────────────────────────────────┤
│ L1. Protocols                                   (M1 core)      │
└────────────────────────────────────────────────────────────────┘
```

M2 在 L1-L2 之上、L3 之下,交付三个独立单元:

| 单元 | 层级归属 | 依赖方向 |
|---|---|---|
| `@my-agent-team/adapter-anthropic` | L1 实现（实现 `ChatModel` 协议） | core ← adapter |
| `@my-agent-team/tools-common` | L2 配套（tool 实例,被 `run()` 消费） | core ← tools-common |
| `@my-agent-team/cli`(`apps/cli`) | L2 消费者（调 `run()` 的入口） | core ← adapter ← CLI, core ← tools-common ← CLI |

**依赖方向铁律不变:** 下层不能引用上层。CLI 是顶层消费者,可以引用所有下层包。

## 四、`@my-agent-team/adapter-anthropic`

### 4.1 AnthropicChatModel

```ts
// packages/adapter-anthropic/src/anthropic-chat-model.ts

import { Anthropic } from "@anthropic-ai/sdk";
import type { AIMessageChunk, ChatModel, ChatModelOptions, Message } from "@my-agent-team/core";

export interface AnthropicChatModelConfig {
  /** 模型 ID,默认 "claude-opus-4-7" */
  model?: string;
  /** 自适应 thinking,默认开启 */

  /** 思考深度,默认 "xhigh" */
  effort?: "low" | "medium" | "high" | "xhigh";

  maxTokens?: number;
  /** API key,默认 process.env.ANTHROPIC_API_KEY */
  apiKey?: string;
  /** 自定义 base URL,默认 undefined(用 SDK 默认值) */
  baseUrl?: string;
}

export class AnthropicChatModel implements ChatModel {
  readonly id: string;

  constructor(config?: AnthropicChatModelConfig) { /* ... */ }

  async *stream(
    messages: readonly Message[],
    options?: ChatModelOptions,
  ): AsyncIterable<AIMessageChunk> { /* ... */ }
}
```

### 4.2 stream() 实现要点

**Step 1 — 提取 system prompt**

遍历 `messages`,提取所有 `role === "system"` 的消息。取**最后一条** system 消息的 `content` 作为 API 的 `system` 参数。如果 content 是 `ContentBlock[]`,取第一个 text block 的 text;如果是 `string`,直接用。

没有 system 消息时 `system` 参数为 `undefined`。

**说明:** 取最后一条而不是拼接全部,因为 Anthropic API 的 `system` 可以是 string 或 `TextBlockParam[]`,支持多个 system block。M2 先简化为 string;未来有 cache_control 需求时再升为 `TextBlockParam[]`。

**Step 2 — 转换 messages**

Messages 数组中 system role 已提取。其余 `user` / `assistant` 消息按 role 映射到 Anthropic API 格式。M1 的 `Message.content` 和 `ContentBlock` 字段名已对齐 Anthropic,基本是直通:

- `Message.role` → `MessageParam.role`
- `TextBlock` → `TextBlockParam`（字段名一致）
- `ToolUseBlock` → `ToolUseBlockParam`（`id`/`name`/`input` 一致）
- `ToolResultBlock` → `ToolResultBlockParam`（`tool_use_id`/`content`/`is_error` 一致）

**Step 3 — 调用 SDK stream**

```ts
const stream = client.messages.stream({
  model: this.config.model ?? "claude-opus-4-7",
  max_tokens: this.config.maxTokens ?? 16000,
  system,
  messages: apiMessages,
  tools: toAnthropicTools(tools),
  thinking: this.config.thinking,
});
```

`tools` 参数:adapter 不自动持有 tools。**调用方通过 `ChatModelOptions` 传入 tools 数组**——M1 的 `ChatModelOptions` 需要新增一个 `tools?: readonly Tool[]` 字段。

等等——重新考虑。adapter 的 `stream()` 签名来自 `ChatModel` 接口,当前只有 `signal`。如果 adapter 需要 tools,有两个选择:

**A) 改协议** — 在 `ChatModelOptions` 加 `tools?: readonly Tool[]`,所有 ChatModel 实现都能收到 tools
**B) 构造函数收 tools** — `new AnthropicChatModel({ tools: [...] })`,tools 在构造时固化

选 **A**。tool 列表本来就是每次 `stream()` 调用可能不同的(不同会话可用不同 tool 集),放 options 里更灵活。而且 M3 `createAgent` 装配 model+tools 时,需要把 tools 传给 model stream。

**Step 4 — 遍历 SSE events**

SDK 的 `stream` 返回一个 `MessageStream`,通过 `for await` 遍历 SSE events:

```ts
for await (const event of stream) {
  switch (event.type) {
    case "content_block_start":
      if (event.content_block.type === "text") {
        yield { delta: { type: "text", text: event.content_block.text } };
      } else if (event.content_block.type === "tool_use") {
        yield { delta: { type: "tool_use", id: event.content_block.id, name: event.content_block.name } };
      }
      break;

    case "content_block_delta":
      if (event.delta.type === "text_delta") {
        yield { delta: { type: "text", text: event.delta.text } };
      } else if (event.delta.type === "input_json_delta") {
        yield { delta: { type: "input_json_delta", id: event.delta.id, partial_json: event.delta.partial_json } };
      }
      break;

    case "message_stop":
      // stream 结束后,SDK 把 finalMessage 挂在 stream 实例上
      break;
  }
}

// 拿到 finalMessage 后 yield done chunk
yield {
  done: true,
  stopReason: stream.finalMessage?.stop_reason,
  usage: {
    input: stream.finalMessage?.usage.input_tokens ?? 0,
    output: stream.finalMessage?.usage.output_tokens ?? 0,
  },
};
```

**关键设计点:**

1. **不缓存整个 stream** — 每个 SSE event 即时翻译成 `AIMessageChunk` 并 yield,不积压
2. **done chunk 在 stream 结束后 yield** — `stopReason` 和 `usage` 来自 `stream.finalMessage`,遍历完 events 后才可用
3. **`content_block_start` + `content_block_delta` 都产 chunk** — text 出现时直接给 text 块,tool_use 出现时给 tool_use 块;M1 的 `mergeChunkIntoBlocks` 负责累积
4. **`signal` 透传给 SDK** — `client.messages.stream({ ... }, { signal })`
5. **不处理 abort 后的 cleanup** — SDK 自己管理连接生命周期

### 4.3 toAnthropicTools

```ts
// packages/adapter-anthropic/src/to-anthropic-tools.ts

import type { Anthropic } from "@anthropic-ai/sdk";
import type { Tool } from "@my-agent-team/core";

/**
 * 将 M1 Tool 数组映射为 Anthropic SDK 的 Tool 数组。
 * 纯字段映射:name → name, description → description, inputSchema → input_schema。
 * 不做 JSON Schema 合法性校验;Anthropic API 自己会拒绝非法 schema。
 */
export function toAnthropicTools(tools: readonly Tool[]): Anthropic.Messages.Tool[] {
  return tools.map((tool) => ({
    name: tool.name,
    description: tool.description,
    input_schema: tool.inputSchema as Anthropic.Messages.Tool.InputSchema,
  }));
}
```

**说明:**

- 不校验 `inputSchema` 合法性,tool 作者负责保证 schema 正确
- 不用 deep clone —— `inputSchema` 是 plain object,SDK 序列化时不会修改它
- 返回新数组,不修改原 Tool[]

### 4.4 M1 协议增补

M2 需要在 `ChatModelOptions` 中新增 `tools` 字段,让调用方把 tools 传给 model:

```ts
// packages/core/src/chat-model.ts — 仅在 ChatModelOptions 中加一行

export interface ChatModelOptions {
  signal?: AbortSignal;
  /** Tool 列表,供 model 传递给 LLM API(adapter 需要,echoModel 忽略) */
  tools?: readonly Tool[];
}
```

**说明:** `tools` 是可选的。echoModel 不需要 tools(M1 测试中 `run()` 自己持有 toolMap)。AnthropicChatModel 需要 tools 来构建 API 请求。这个字段不是 adapter 专属——它是协议的合理补充,任何 ChatModel 实现都可以选择使用或忽略。

### 4.5 错误处理

**透传,不包装。** Anthropic SDK 抛出的所有错误(网络错误、401/403、429 rate limit、400 bad request、server 500)直接向上抛。调用方通过 SDK 的 `APIError` 类型(`error.status` / `error.message` / `error.headers`)自行判断和分支。

M1 定了"不定义自有错误类"的原则,adapter 遵守。上层出现 3 次以上需要按错误类型分支的需求时,再考虑分类包装。

### 4.6 不做的事(M2 明确范围外)

- ❌ 不实现 retry / 自动重试(调用方在 adapter 外用 loop 包装即可)
- ❌ 不缓存 token(每次调 API 都实时统计)
- ❌ 不校验 `inputSchema` 合法性
- ❌ 不支持 Anthropic prompt caching(M3 需要时再加)
- ❌ 不实现 compaction(M3 需要时再加)
- ❌ 不自动选择模型(调用方在构造时指定)

## 五、`@my-agent-team/tools-common`

### 5.1 设计决策

**Tool 粒度:** 一个 tool 一个文件,每个 tool 导出常量或工厂函数。

**工厂模式 vs 常量:** 需要外部注入依赖的 tool 用工厂(`createWebSearchTool(apiKey)`,`createMemorySaveTool(store)`),无外部依赖的 tool 用常量(`webFetchTool`,`readTool`,`writeTool`)。

**不提取公共 tool 基类/helper:** M1 的 `Tool` 接口已经足够,6 个 tool 不构成重复模式。Rule of Three 判断:等出现第 4 个需要 HTTP 请求的 tool 时才提取 `createHttpTool` helper。

### 5.2 web_fetch

```ts
// packages/tools-common/src/web-fetch.ts

import type { Tool } from "@my-agent-team/core";

export const webFetchTool: Tool = {
  name: "web_fetch",
  description: "Fetch content from a URL and return as plain text",
  inputSchema: {
    type: "object",
    properties: {
      url: { type: "string", description: "The URL to fetch" },
    },
    required: ["url"],
  },
  async execute(input) {
    const response = await fetch((input as { url: string }).url);
    const text = await response.text();
    const maxLength = 20_000;
    if (text.length <= maxLength) {
      return { content: text };
    }
    return {
      content: `${text.slice(0, maxLength)}\n\n... (truncated, original length: ${text.length})`,
    };
  },
};
```

**说明:**
- 用原生 `fetch`,零外部依赖
- 截断到 20_000 字符,防止大网页撑爆 context
- 不做 HTML → markdown 转换(那是 M4 的职责);现在返回原始文本

### 5.3 web_search(Tavily)

```ts
// packages/tools-common/src/web-search.ts

import type { Tool } from "@my-agent-team/core";

export function createWebSearchTool(apiKey: string): Tool {
  return {
    name: "web_search",
    description: "Search the web using Tavily and return top results as JSON",
    inputSchema: {
      type: "object",
      properties: {
        query: { type: "string", description: "The search query" },
        maxResults: {
          type: "number",
          description: "Maximum number of results (default: 5)",
        },
      },
      required: ["query"],
    },
    async execute(input) {
      const { query, maxResults = 5 } = input as { query: string; maxResults?: number };
      const response = await fetch("https://api.tavily.com/search", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          api_key: apiKey,
          query,
          max_results: maxResults,
        }),
      });
      const data = (await response.json()) as { results?: unknown };
      return { content: JSON.stringify(data.results ?? data) };
    },
  };
}
```

**说明:**
- `apiKey` 由调用方从 `TAVILY_API_KEY` 环境变量注入
- 用 `createWebSearchTool(apiKey)` 工厂模式,不是常量——因为每个实例的 API key 不同
- 返回 JSON string,LLM 自行解读

### 5.4 memory_save / memory_recall

```ts
// packages/tools-common/src/memory-save.ts

import type { Tool } from "@my-agent-team/core";

export function createMemorySaveTool(store: Map<string, string>): Tool {
  return {
    name: "memory_save",
    description: "Save a value to memory under the given key",
    inputSchema: {
      type: "object",
      properties: {
        key: { type: "string", description: "The key to store under" },
        value: { type: "string", description: "The value to store" },
      },
      required: ["key", "value"],
    },
    execute(input) {
      const { key, value } = input as { key: string; value: string };
      store.set(key, value);
      return { content: `Saved: ${key}` };
    },
  };
}
```

```ts
// packages/tools-common/src/memory-recall.ts

import type { Tool } from "@my-agent-team/core";

export function createMemoryRecallTool(store: Map<string, string>): Tool {
  return {
    name: "memory_recall",
    description: "Recall a value from memory by key. Returns an error string if not found.",
    inputSchema: {
      type: "object",
      properties: {
        key: { type: "string", description: "The key to look up" },
      },
      required: ["key"],
    },
    execute(input) {
      const { key } = input as { key: string };
      const value = store.get(key);
      if (value === undefined) {
        return { content: `Key not found: ${key}`, isError: true };
      }
      return { content: value };
    },
  };
}
```

**关键设计点:**

1. **store 是调用方注入的 `Map<string, string>`** — 与 M1"状态归调用方"原则一致。CLI 创建一个 `new Map()` 传进去,会话结束就没了。调用方想要持久化,自己 `JSON.stringify` 存盘
2. **memory_save 是同步 execute** — 写 Map 不需要 async
3. **memory_recall 的 isError=true** — 未命中时不是抛异常,而是返回 `isError: true` 的 `ToolExecuteResult`,让 LLM 能读到错误信息并继续
4. **value 限制为 string** — M1 不做嵌套 block,保持简单

### 5.5 read

```ts
// packages/tools-common/src/read.ts

import type { Tool } from "@my-agent-team/core";

export const readTool: Tool = {
  name: "read",
  description: "Read a file from the local filesystem and return its content as text",
  inputSchema: {
    type: "object",
    properties: {
      path: { type: "string", description: "Path to the file to read" },
    },
    required: ["path"],
  },
  async execute(input) {
    const { path } = input as { path: string };
    const file = Bun.file(path);
    if (!(await file.exists())) {
      return { content: `File not found: ${path}`, isError: true };
    }
    return { content: await file.text() };
  },
};
```

### 5.6 write

```ts
// packages/tools-common/src/write.ts

import type { Tool } from "@my-agent-team/core";

export const writeTool: Tool = {
  name: "write",
  description: "Write content to a file, overwriting if it already exists",
  inputSchema: {
    type: "object",
    properties: {
      path: { type: "string", description: "Path to the file to write" },
      content: { type: "string", description: "Content to write to the file" },
    },
    required: ["path", "content"],
  },
  async execute(input) {
    const { path, content } = input as { path: string; content: string };
    await Bun.write(path, content);
    return { content: `Wrote: ${path}` };
  },
};
```

**说明:**

- read 和 write 不做路径沙箱(M4 harness 加)
- 用 `Bun.file()` / `Bun.write()`,不引入 Node fs 模块
- `read` 返回文件内容(cat 语义),`write` 返回确认信息

### 5.7 不做的事

- ❌ 不做 `web_search` 多 provider 切换(只做 Tavily,M3 出现第二个 provider 时再抽)
- ❌ 不做 memory 持久化(调用方自己存 Map)
- ❌ 不做 memory_delete / memory_list(2 个够用,不够再加)
- ❌ 不做 `read` 的 offset/limit 参数(M4 需要时再加)
- ❌ 不做 `write` 的 append mode(M4 需要时再加)
- ❌ 不引入 zod / joi / valibot 校验 tool input

## 六、`apps/cli/`

### 6.1 定位

CLI 是 M2 的**手工验证入口**,不是正式产品。它是一个临时脚本,未来 M4 harness 完成后会被替换。

### 6.2 功能

```text
bun run apps/cli/src/main.ts \
  --model claude-opus-4-7 \
  --max-steps 32 \
  --system "You are a helpful assistant."
```

**启动后流程:**

1. 读 `ANTHROPIC_API_KEY` 和 `TAVILY_API_KEY` 环境变量,缺一报错退出
2. 创建 `AnthropicChatModel` + `Map` store + 6 个 tool
3. 组装 `messages` 数组(如果有 `--system` 参数,插入一条 system 消息)
4. 进入交互循环:
   - 打印 `> ` prompt
   - 读 stdin 一行
   - `messages.push({ role: "user", content: line })`
   - `for await (const msg of run(model, tools, messages, { maxSteps }))` — 流式打印
   - 回到 `> ` prompt
5. Ctrl+C 退出(SIGINT 捕获,不保存任何状态)

### 6.3 不做的事

- ❌ 不做 REPL 框架(用 `readline` 或 `process.stdin` 直接读,30 行足够)
- ❌ 不做颜色/高亮/markdown 渲染(纯文本输出)
- ❌ 不做配置文件(只用 env + CLI args)
- ❌ 不做 session 持久化/history 回放
- ❌ 不做 `--tools` 白名单/黑名单(全量加载 6 个 tool)
- ❌ 不写自动化测试(手工验证)

### 6.4 约 60-80 行,messages 数组由 CLI 持有和 fork
