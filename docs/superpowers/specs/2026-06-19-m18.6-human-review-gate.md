## 1. 背景

[Issue 协作工作流设计](../../architecture/foundations/issue-workflow.md) 把现状诊断为「建即跑、写死、单向、看不见」。到 M18.5 为止：「写死」已解（列可配置、绑真实 Agent、转移从 DB 读）、「建即跑」已解（草稿态 + 人工入列）、「单向」的**产出端**已解（交付物结构化 + 上下文累积，让一棒喂下一棒）。但**「单向」最后一块没解**：流程仍然只进不退——run 成功就自动推进到下一棒，一路冲到 `done`，**没有人来拍板「这算不算干完」，更没有「Review 不通过 → 退回开发返工」的回路**。

本棒落地设计的 ⑥：**人工验收闸门 + 返工**。这是整条 Issue 流里**唯一刻意保留的人类决策点**——其余推进都自动，只有 `in_review` 的去向（通过/打回）交给人。它带来两处结构性变化：

- **推进从「纯自动」长出一个「人工闸门」**：`onRunComplete` 当前对每个成功 run 都自动 `applyTransition` 到下一棒（§2.2）。本棒要让推进**在到达 `in_review` 时停住**，不再自动迈进 `done`；进 `done` 或退回 `in_progress` 都只能由一个显式的人类裁决事件触发。
- **转移表从「线性」长出「回退边」**：当前 `ORDER` 是严格线性序、`nextTransition` 假设每个 `from` 唯一 `to`（§2.1）。打回要加一条 `in_review → in_progress` 的**反向边**，转移表第一次从线性变成图。这要求 `LEGAL_TRANSITIONS`、`applyTransition` 校验、reactor 的「下一棒是谁」逻辑都正确处理「一个 from 有多条合法 to」。

为什么 ⑥ 自成一棒、不与 ⑤ 合并：⑤（M18.5）是「让数据在棒间结构化流动」的纯机制，不触碰状态机拓扑；⑥ 第一次改状态机拓扑（线性→图）并引入人类决策点，是另一个维度的演进。⑤的交付物上下文恰好是⑥的**载体**——打回反馈复用交付物机制承载（§3.4），所以⑥必须在⑤之后、且能直接吃到⑤的成果。

## 2. 当前代码事实

逐项已与 `c2905af` 代码核对：

1. **转移表是严格线性序，`nextTransition` 假设每个 `from` 唯一 `to`**：`orchestrator/transitions.ts:12` `ORDER = ["draft","planned","in_progress","in_review","done"]`；`nextTransition(table, from)`（`:64`）= `table.find(t => t.from === from)`——**只返回第一条匹配，隐含「每个 from 至多一条转移」**。`deriveLegalMap`（`:55`）按 `from → to[]` 建映射，结构上已支持一个 from 多个 to，但 `fixedTransitions()`（`:31`）只产相邻线性对，故当前 `LEGAL_TRANSITIONS` 每个 from 仍只有一个 to（终态 `done` 为空）。**本棒：`in_review` 要有两条合法去向（`done` 通过 / `in_progress` 打回），`nextTransition` 的「找一条自动推进的转移」语义需与「人工可选的合法去向」分开——前者只认前进边，后者认全部合法边。**

2. **`onRunComplete` 对每个成功 run 无条件自动推进到下一棒**：`reactor.ts:102-147`。成功后 `const t = nextTransition(table, issue.status); if (!t) return; advanced = issueSvc.applyTransition(issueId, t.to); await startStep(advanced)`——**只要 `nextTransition` 找得到下一棒就自动迈进并起下一棒**。`in_review` 当前若配了 ColumnConfig，reviewer 棒成功也会自动 `applyTransition(in_review → done)`（因为 `done` 是 `in_review` 的线性下一个）。**本棒：`in_review` 必须从「自动推进集合」里摘出——reviewer 棒成功只把 Issue 停在 `in_review` 等人裁决，`onRunComplete` 不得自动迈过这道闸门。**

3. **`run_origin.idempotency_key` 全局 UNIQUE，orchestrator 用 `issue:<id>:<status>:run` 做键，`INSERT OR IGNORE` 写入**：`events-db-migrations.ts:80` `CREATE UNIQUE INDEX idx_run_origin_idem ON run_origin(idempotency_key)`；`store.ts:67` `INSERT OR IGNORE INTO run_origin`；`reactor.ts:93` `idempotencyKey: issue:${issue.issueId}:${issue.status}:run`。**这是返工的致命约束（§3.3）**：返工第二次进 `in_progress` 时，`startStep` 又生成 `issue:<id>:in_progress:run` 这个**完全相同的键** → `INSERT OR IGNORE` 静默丢弃这一行 → 新 run **没有 run_origin** → run 跑完 `onRunComplete` 用 `getRunOrigin(runId)` 反查 `issueId` 得 null → **直接 return，返工棒永远不推进**。返工环在当前键设计下是死的。本棒必须把键改成 run 维度唯一（带 runId 或返工轮次），不再用 `(issueId, status)` 二元组。

4. **`onRunComplete` 用 `origin.idempotencyKey.split(":")[2]` 解 fromStatus 做幂等守卫**：`reactor.ts:127` `const fromStatus = origin.idempotencyKey?.split(":")[2]; if (fromStatus && issue.status !== fromStatus) return`。注意 M18.5 R3 已给 `run_origin` 加了**权威 `from_status` 列**（`events-db-migrations.ts:140`，`store.ts` `RUN_ORIGIN_COLS` 含 `from_status AS fromStatus`），但 reactor 这处幂等守卫**仍在解析字符串**、没切到 `origin.fromStatus`。**本棒改键（§3.3）后 `split(":")[2]` 会取错段——必须同时把这处守卫切到 `origin.fromStatus`（权威列），与 R3 对齐。**

5. **`applyTransition` 用 `LEGAL_TRANSITIONS` 校验合法性、CAS 防并发**：`issue/service.ts:56-66`。`const legal = LEGAL_TRANSITIONS[issue.status] ?? []; if (!legal.includes(to)) throw IllegalTransitionError`；随后 `port.setStatus(issueId, fromStatus, to, ts)` 做 compare-and-swap。**`LEGAL_TRANSITIONS` 来自 `transitions.ts` 的全局 `FIXED`（线性），不含 `in_review → in_progress`**——故当前即便手调 transition 端点想打回，也会被 `IllegalTransitionError` 挡住。**本棒：回退边 `in_review → in_progress` 必须进 `LEGAL_TRANSITIONS`（加进 `ORDER` 派生不行——`ORDER` 是线性序；要么单独维护一组回退边并入 `deriveLegalMap` 的输入）。**

6. **transition HTTP 端点是通用状态机驱动、forward 触发 orchestrator**：`issue/http.ts:69-95` `transition(req, issueId)`：解析 `{to}`，调 `svc.applyTransition`，**若 `ORDER.indexOf(to) > ORDER.indexOf(from) && to !== "done"` 则触发 `onIssueStarted`（起 orchestrator）**（`:80-88`）。这覆盖 `draft→planned`（启动）与人工前拖。**本棒新增的人工裁决（通过/打回）语义上不同于「拖卡片」——通过是 `in_review→done`（前进但 `to==="done"`，当前逻辑不触发 orchestrator，正确）；打回是 `in_review→in_progress`（`ORDER.indexOf` 是后退，当前逻辑不触发 orchestrator → 返工棒不会起）。所以打回不能走现有 transition 端点的触发逻辑，要么扩展它识别回退边、要么走专门的裁决端点（§3.5 选后者）。**

7. **交付物机制可承载打回反馈，且上下文累积已支持按 kind 取用**：`deliverable/domain.ts` `DeliverableRow{issueId, fromStatus, kind, fields, ref, runId, createdAt}`，追加式；`reactor.ts:47-54` `buildPromptVars` 按 kind 聚合最新、产 `{{deliverables.<kind>.fields.<key>}}` / `{{deliverables.<kind>.ref}}`（R4 双命名空间）。**打回反馈天然是一份「人产的交付物」**：`kind="rework_feedback"`、`fields={note: "<打回理由>"}`，落 `deliverable` 表后，返工的 developer 棒 prompt 写 `{{deliverables.rework_feedback.fields.note}}` 即可取用。**复用既有机制，不发明新表**——但人产的交付物没有 `runId`（不是某个 run 提交的），要确认 `runId` 可空（`domain.ts` 已是 `runId: string | null`，✓）且不撞 `(run_id, kind)` 幂等索引（`run_id IS NULL` 时部分唯一索引不约束，✓，见 `migrations.ts` 5012 的 `WHERE run_id IS NOT NULL`）。

8. **交付物提交服务 `submit` 接 `runId?` 可选，端点 `submitDeliverable` 走 R3 反查**：`issue/http.ts:100-127`。人产反馈不经 runner 工具回调，而是裁决端点内部直接调 `deliverableSvc.submit({issueId, fromStatus, kind:"rework_feedback", fields:{note}, ...})`，**不带 runId**——绕开 R3 的 run_origin 反查（那是给 Agent 工具回调用的），人产反馈的归属（issueId/fromStatus）由裁决端点已知上下文直接给。

9. **`buildPromptVars` 同 kind 取最新——返工反馈天然「最新一条生效」**：`reactor.ts:50-52` 后写覆盖。多次打回 → 多条 `rework_feedback`，返工棒只读最新那条（最近一次打回理由），符合直觉。历史打回全留在表里（追加式，§3.6 of M18.5），未来 Timeline（M18.7）可回放。

10. **Issue 状态机的 mermaid 设计图已含回退边、但 current 代码未实现**：`issue-workflow.md:52-61`（②的状态图）已画 `in_review --> in_progress: 人工打回（返工，见 ⑥）`；`:123-128`（⑥的 flowchart）画了完整闸门。**设计已锁，本棒是把这张图落进代码。** 不变量 6（`issue-workflow.md:173`）：「『是否完成』由人工闸门裁决；返工是转移表上的显式回退边，而非旁路」——本棒立这条契约。

11. **`startStep` 已能为任意 status 起对应转移棒**：`reactor.ts:58-99`。返工进 `in_progress` 后，`startStep` 用 `nextTransition(table, "in_progress")` 找到 developer 棒配置、构建上下文（此时上下文已含 `rework_feedback`）、起 run。**返工棒的起步完全复用 `startStep`，只要键不撞（§3.3）+ 上下文带上反馈（§7 复用）即可**——这是返工实现量小的关键：拓扑改造集中在「闸门停 + 回退边合法 + 键去撞」三点，起步逻辑零改动。

12. **前端看板有 `in_review` 列、拖拽即 transition**：（M18.4 落，`apps/web`）卡片拖入列 = `POST /api/issues/:id/transition {to}`。**本棒人工裁决的最小 UI：`in_review` 卡片上两个动作「通过」「打回（带反馈输入）」**——但 §5 把完整裁决 UI 列为非目标首版，本棒后端闸门 + 裁决端点为主，前端最小化（仅够触发裁决，复杂交互后置）。

## 3. 第一性原则

### 3.1 「干完」是人的判断，不是 run succeeded

reviewer 棒 run 成功，只说明「review 这个动作执行完了」，**不等于「这件活验收通过」**——review 的结论可能是「打回重做」。把 run 终态等同于验收结论，就是把人的判断权交给了「Agent 有没有跑完」。本棒立的契约（不变量 6）：**`in_review` 的去向由一个显式的人类裁决事件决定，run 终态只负责把 Issue 送到 `in_review` 这道闸门前停下。** 这是整条流唯一不自动的环节。

### 3.2 闸门是「停」，不是「新状态」

实现人工闸门不需要新状态——`in_review` 本身就是闸门所在。要做的是**把 `in_review` 从「自动推进集合」里摘出来**：`onRunComplete` 遇到「Issue 走到 `in_review`」时**不再自动迈进**，停在原地等裁决。这比新增一个 `awaiting_review` 状态更省——状态空间不膨胀，闸门语义落在「`in_review` 不自动出」这一条规则上。判定「哪些 from 自动推进、哪些停下等人」需要一个明确的边分类（§3.5 前进边 vs 闸门边）。

### 3.3 返工要求 run_origin 幂等键按 run 唯一，不按 (issueId, status)

这是本棒**最硬的一处机制约束**（§2.3）。当前 `idempotencyKey = issue:<id>:<status>:run` 把「同一 Issue 的同一 status 起的 run」视为同一个幂等单元——这在**只进不退**的线性流里成立（每个 status 一生只经过一次）。但返工**让同一个 status 被经过多次**（`in_progress` 第一次开发、打回后第二次开发……），相同的键 + `run_origin` 全局 UNIQUE + `INSERT OR IGNORE` → **第二次起的 run 的 origin 行被静默丢弃 → 反查不到 issueId → 返工棒跑完不推进**。返工环在当前键下根本跑不通。

修复：**幂等键改为 `runId` 本身**（runId 已 ulid 全局唯一）。前缀 `issue:<id>:<status>:` 不参与任何索引或查询逻辑——所有反查都走 `getRunOrigin(runId)`，前缀只是给人看的。M18.5 R3 已加权威 `from_status` 列，调试时该看的是结构化列。这样每次起 run（含返工重入同 status）都写得进 origin。但要核对一处下游：`onRunComplete` 当前从键里 `split(":")[2]` 解 fromStatus（§2.4）——用纯 `runId` 后键里彻底没有 status 可抠，必须**切到 R3 的权威 `origin.fromStatus` 列**（M18.5 已加，reactor 漏切）。改键 + 切权威列是同一处修复的两面。

> **为什么不保留旧键 + 加轮次后缀**：`issue:<id>:<status>:rework<n>:run` 也能去撞，但 `<n>` 从哪来要额外存返工计数，徒增状态。runId 本就唯一、本就在手，用它做幂等键最省——run_origin 的幂等本质是「同一个 run 别写两行」，按 runId 去重语义最直白。

### 3.4 打回反馈是一份「人产的交付物」，复用 ⑤ 的载体

打回时人写的反馈要喂进返工棒。它和 Agent 交付物**同构**——都是「某一棒产出、供下一棒结构化取用的内容」，区别只是生产者是人不是 Agent、没有 runId。所以**不发明新表**：打回反馈作为 `kind="rework_feedback"` 的交付物落 `deliverable` 表（`fields={note}`、`runId=null`），返工的 developer 棒 prompt 经 `{{deliverables.rework_feedback.fields.note}}` 取用（§7 复用 M18.5 全套机制）。这让「返工反馈如何进上下文」零新增机制——⑥ 直接站在 ⑤ 的肩上。

> 人产交付物的 `runId=null` 不撞 `(run_id, kind)` 幂等索引（部分唯一索引 `WHERE run_id IS NOT NULL`，§2.7）。多次打回产多条 `rework_feedback`，`buildPromptVars` 同 kind 取最新（§2.9）——返工棒读到的是最近一次打回理由，符合直觉，历史打回不丢。

### 3.5 裁决走专门端点，不复用 transition 的「拖卡片」语义

人工裁决（通过/打回）语义上不是「拖卡片改状态」：通过=`in_review→done` 且带「验收」语义；打回=`in_review→in_progress` 且**必须带反馈 + 必须触发返工棒起步**。现有 transition 端点（§2.6）的 orchestrator 触发逻辑按 `ORDER.indexOf` 前进/后退判定，**打回是后退、不会触发返工棒**。与其给通用 transition 端点塞进「识别回退边 + 落反馈 + 起返工棒」的特例，不如**新增一个语义明确的裁决端点** `POST /api/issues/:id/review-decision {decision: "approve"|"reject", note?}`：
- `approve` → `applyTransition(in_review → done)`，不起任何棒（done 是终态）。
- `reject` → 落 `rework_feedback` 交付物（`note` 必填）→ `applyTransition(in_review → in_progress)` → 触发 `onIssueStarted`（起返工 developer 棒，此时上下文已含反馈）。

这样 transition 端点保持「纯状态机 + 拖拽」语义不变，裁决的复合语义（落反馈 + 转移 + 起棒）收在一个意图明确的端点里。回退边 `in_review→in_progress` 仍要进 `LEGAL_TRANSITIONS`（裁决端点内部也调 `applyTransition`，要过合法性校验，§2.5）。

### 3.6 回退边进合法集，但不进「自动推进」集——两套边分开

转移表第一次有了两类边：**前进边**（`planned→in_progress` 等，reactor 自动推进走它）和**回退边**（`in_review→in_progress`，只能由人工裁决触发、reactor 绝不自动走）。两者都得是 `applyTransition` 眼里的「合法转移」（否则 CAS 前的 `LEGAL_TRANSITIONS` 校验会拦），但**只有前进边能进 `nextTransition` 的「找下一棒自动推进」**。所以：
- `LEGAL_TRANSITIONS`（合法性校验用）= 前进边 ∪ 回退边 ∪ 闸门边（`in_review→done`）。
- `nextTransition`（reactor 自动推进用）= 只认前进边，且**排除闸门边的 from**（`in_review` 不自动出，§3.2）。

把「合法吗」和「自动走吗」两个问题用两个数据结构回答，避免 reactor 误把回退边/闸门边当成自动推进路径。

## 4. 目标

1. **转移表长出回退边 + 闸门语义**（`transitions.ts`）：`in_review → in_progress`（回退边）、`in_review → done`（闸门边，人工触发）进 `LEGAL_TRANSITIONS`；新增「哪些 from 是人工闸门、不自动推进」的判定（`IN_REVIEW` 不自动出）。前进边集合（reactor 自动推进用）与合法边集合（校验用）分开。
2. **`onRunComplete` 在 `in_review` 停住**（`reactor.ts`）：reviewer 棒成功只把 Issue 推到 `in_review` 后**不再自动迈进 `done`**；`in_review` 的去向只由人工裁决决定。
3. **run_origin 幂等键改按 run 唯一**（`reactor.ts` + 必要时 store）：键从 `issue:<id>:<status>:run` 改为按 runId 唯一（如 `issue:<id>:<status>:<runId>` 或直接 runId），**根治返工重入同 status 撞键被静默丢弃的死环**（§3.3）。
4. **`onRunComplete` 幂等守卫切到权威 `from_status` 列**（`reactor.ts`）：`split(":")[2]` → `origin.fromStatus`（M18.5 R3 已加列，本棒补切），与改键解耦。
5. **新增人工裁决端点** `POST /api/issues/:id/review-decision`（`issue/http.ts` + `router.ts`）：`approve` → `in_review→done`；`reject` → 落 `rework_feedback` 交付物（`note` 必填）+ `in_review→in_progress` + 起返工棒。校验 Issue 必须处于 `in_review`（否则 409）。
6. **打回反馈复用交付物载体**（裁决端点内部）：`reject` 落 `kind="rework_feedback"`、`fields={note}`、`runId=null` 的交付物；返工 developer 棒 prompt 经 `{{deliverables.rework_feedback.fields.note}}` 取用（机制零新增，复用 M18.5）。
7. **前端最小裁决入口**（`apps/web`，最小化）：`in_review` 卡片上「通过 / 打回（带反馈输入框）」两个动作，调裁决端点。复杂裁决 UI 后置（§5）。
8. **测试收口**：transitions 回退边/前进边分离纯函数测试；reactor `in_review` 停 + 返工重入不撞键 + 推进 + 幂等守卫切列；裁决端点 approve/reject/非 in_review→409/reject 落反馈 + 起棒；端到端「reviewer 打回 → 返工棒 prompt 含反馈」。

## 5. 非目标

- **⑦ Issue Timeline**（按 issueId 聚合的追加式工作事件流：created/started/run.*/deliverable.submitted/status.advanced/**human.decided**）——M18.7。本棒人工裁决**不**顺带建 Timeline 事件；裁决只改 status + 落反馈交付物。`human.decided` 事件留给 ⑦。
- **⑧ Issue 级可观测**（`getRunOriginsByIssueId` 反查 + 详情抽屉 + 诊断聚合）——M18.7。
- **任意 DAG / 多级回退**：本棒只加**一条**回退边 `in_review → in_progress`（设计图 ⑥ 就这一条）。不支持 `done → in_review` 重开、不支持跳级回退（如 `in_review → planned`）。转移表是「线性序 + 一条回退边」的有限图，不是任意 DAG。
- **完整裁决 UI / 评论线索 / 多人会签**：本棒前端只做「通过 / 打回 + 一个反馈输入框」最小入口。富文本反馈、@人、多人审批、裁决历史展示——后置。
- **自动验收 / AI 验收**：闸门刻意是**人工**的（不变量 6）。不引入「reviewer 棒自己判通过就自动进 done」的旁路——那等于取消闸门。
- **强制 reviewer 棒先提交 review 交付物才能裁决**：裁决不强依赖 reviewer 是否提交了 `kind=review` 交付物（最佳努力，与 M18.5 §3.7 一致）。人可以在没有结构化 review 交付物时也裁决。
- **打回反馈的结构化 schema**：`rework_feedback` 的 `fields` 首版只约定一个 `note` 文本键，不定义更复杂的结构（分项打回、行级评论等）后置。
- **改 transition 端点的拖拽语义**：裁决走专门端点（§3.5），transition 端点保持纯状态机 + 拖拽不变。

## 6. 实施分组与方案

> 伪代码均对齐 §2 存量写法：transitions 纯函数 + reactor 单点控制流、issue service `applyTransition` + CAS、router if-链 + `withAuth`、真库测试 + 注入 idGen/now。转移表拓扑改造先行（P1），reactor 闸门 + 改键 + 切列其次（P2），裁决端点 + 反馈载体（P3），路由接线（P4），前端最小入口（P5），测试收口（P6）。

### Patch P1 — 转移表：回退边 + 前进边/合法边分离

**目标**：`in_review` 有两条合法去向（`done`/`in_progress`），reactor 自动推进只认前进边、且 `in_review` 不自动出。

```ts
// transitions.ts —— ORDER 仍是线性「列顺序」（看板列序不变），但新增回退边 + 闸门判定

export const ORDER: IssueStatus[] = ["draft", "planned", "in_progress", "in_review", "done"];

/** 回退边：人工打回触发，reactor 绝不自动走。本棒只此一条（设计 ⑥）。 */
export const BACKWARD_EDGES: ReadonlyArray<{ from: IssueStatus; to: IssueStatus }> = [
  { from: "in_review", to: "in_progress" },
];

/** 人工闸门：这些 from 的「出」必须由人裁决，reactor 不自动推进。 */
export const HUMAN_GATES: ReadonlySet<IssueStatus> = new Set<IssueStatus>(["in_review"]);

/** 合法转移集（applyTransition 校验用）= 线性前进边 ∪ 回退边。
 *  注意：in_review→done 本就是线性前进边（ORDER 相邻），已在内；in_review→in_progress 由回退边补入。 */
export const LEGAL_TRANSITIONS: Readonly<Record<IssueStatus, IssueStatus[]>> = (() => {
  const map = deriveLegalMap(FIXED);                  // 线性前进边（含 in_review→done）
  for (const e of BACKWARD_EDGES) (map[e.from] ??= []).push(e.to);
  return map;
})();

/** reactor 自动推进用：找「自动前进的下一棒」。
 *  闸门 from 直接返回 undefined——in_review 不自动出，等人裁决（§3.2/§3.6）。 */
export function nextTransition(
  table: ReadonlyArray<Transition>,
  from: IssueStatus,
): Transition | undefined {
  if (HUMAN_GATES.has(from)) return undefined;        // ★ 闸门：不自动推进
  return table.find((t) => t.from === from);
}
```

> `nextTransition` 加一行闸门短路即可——reviewer 棒成功后 `onRunComplete` 调 `nextTransition(table, "in_review")` 返回 undefined → 不自动推进（§2.2 的自动推进自然停在闸门）。`done` 仍因无后继转移而停（终态）。回退边只进 `LEGAL_TRANSITIONS`、不进 `table`（`transitionsForProject` 派生的前进表），故 reactor 永远看不到它，杜绝误自动返工。

**影响文件**：`orchestrator/transitions.ts`（改）。`column-config/service.ts` 的 `transitionsForProject`（派生前进表）**不改**——它只产前进边（回退边不需要 ColumnConfig，返工复用 `in_progress` 那一列的既有配置）。

### Patch P3 — 人工裁决端点 + 打回反馈载体

**P3-a 裁决端点**（`issue/http.ts` 新增 `reviewDecision(req, issueId)`，`issueRoutes` 已持有 `svc`/`opsStore`/`deliverableSvc`，§2.8）：

```ts
const reviewDecisionSchema = z
  .object({
    decision: z.enum(["approve", "reject"]),
    note: z.string().trim().optional(),
  })
  .refine((d) => d.decision === "approve" || (d.note && d.note.length > 0), {
    message: "note is required when rejecting",
    path: ["note"],
  });

/** POST /api/issues/:id/review-decision → 200 { issue } | 400 | 404 | 409 */
async reviewDecision(req: Request, issueId: string): Promise<Response> {
  const parsed = reviewDecisionSchema.safeParse(await req.json().catch(() => ({})));
  if (!parsed.success)
    return json({ error: "Validation failed", details: parsed.error.issues }, 400);

  const issue = svc.port.getIssue(issueId);
  if (!issue) return json({ error: "Not found" }, 404);
  // 裁决只在闸门处有意义：Issue 必须正处于 in_review
  if (issue.status !== "in_review")
    return json({ error: `issue not awaiting review (status=${issue.status})` }, 409);

  try {
    if (parsed.data.decision === "approve") {
      const updated = svc.applyTransition(issueId, "done");   // 终态，不起棒
      return json({ issue: updated });
    }
    // reject：先落打回反馈交付物（人产、runId=null），再退回 in_progress 起返工棒
    deliverableSvc.submit({
      issueId,
      fromStatus: "in_review",
      kind: "rework_feedback",
      fields: { note: parsed.data.note! },
      // 无 ref、无 runId（人产，§3.4）
    });
    const updated = svc.applyTransition(issueId, "in_progress");  // 回退边（P1 已纳入 LEGAL_TRANSITIONS）
    // 起返工棒：此时上下文已含 rework_feedback，developer 棒 prompt 取 {{deliverables.rework_feedback.fields.note}}
    void onReviewRejected?.(updated).catch((e) =>
      console.error(`[orchestrator] rework startStep failed for ${issueId}: ${String(e)}`),
    );
    return json({ issue: updated });
  } catch (err) {
    if (err instanceof IssueNotFoundError) return json({ error: err.message }, 404);
    if (err instanceof IllegalTransitionError) return json({ error: err.message }, 409);
    throw err;
  }
}
```

> `onReviewRejected` 是 `issueRoutes` 新增的可选回调（同现有 `onIssueStarted` 范式，§2.6/main.ts:347），main.ts 接 `orchestrator.startStep`。返工棒起步**完全复用 `startStep`**（§2.11）——`startStep(in_progress 的 issue)` 走 `nextTransition(table, "in_progress")` 找 developer 棒、`buildPromptVars` 此时已能读到刚落的 `rework_feedback`、改键后 origin 写得进、run 跑完正常推进到 `in_review` 再次等裁决。返工闭环成立。

> **顺序要点**：必须**先 `deliverableSvc.submit` 落反馈、再 `applyTransition` + 起棒**。因为 `startStep` 内 `buildPromptVars` 读 `listByIssue` 取上下文——反馈必须先落库，返工棒才取得到（同 M18.5 §3.7「提交先于读取」时序）。

**P3-b 交付物提交无需扩展**：`deliverableSvc.submit` 签名已接 `{issueId, fromStatus, kind, fields, ref?, runId?}`（M18.5），`runId` 省略即人产。`rework_feedback` 的 `runId=null` 不撞 `(run_id, kind)` 部分唯一索引（§2.7）。**无新迁移、无服务改动。**

**影响文件**：`features/issue/http.ts`（改 `issueRoutes` opts 加 `onReviewRejected` + 加 handler）。

## 7. 验收标准

1. 配好各列 ColumnConfig 的 Project 跑一个 Issue：`planned→in_progress→in_review` 三棒自动推进（同今），但走到 `in_review` 后 **reviewer 棒成功不再自动进 `done`**——Issue 停在 `in_review` 等人。
2. 对 `in_review` 的 Issue 调 `review-decision {decision:"approve"}` → Issue 进 `done`，不起任何棒。
3. 对 `in_review` 的 Issue 调 `review-decision {decision:"reject", note:"X"}` → ①落一份 `kind="rework_feedback"`、`fields.note="X"`、`runId=null` 的交付物；②Issue 回 `in_progress`；③自动起返工 developer 棒。
4. **返工棒的 prompt（developer 列模板含 `{{deliverables.rework_feedback.fields.note}}`）渲染出打回理由 "X"**——返工反馈经 ⑤ 载体喂进返工棒。
5. **返工死环根治**：返工棒（第二次进 `in_progress` 的 run）的 `run_origin` 写得进（幂等键含 runId，不撞第一次开发棒的键）；run 成功后 `onRunComplete` 反查得到 issueId、正常推进回 `in_review` 再次等裁决。可反复打回多轮。
6. 多次打回产多条 `rework_feedback`，返工棒只读最新一条（最近一次打回理由），历史打回不丢（追加式）。
7. 回退边只人工触发：reactor 永不自动走 `in_review→in_progress`；`nextTransition(table, "in_review")` 恒返回 undefined。
8. 裁决端点的状态守卫：对非 `in_review` 的 Issue 调 `review-decision` → 409；`reject` 不带 `note` → 400。
9. `applyTransition` 合法性：`in_review→in_progress`（回退）与 `in_review→done`（通过）均合法通过校验；非法转移（如 `in_review→planned`）仍 `IllegalTransitionError`。
10. M18.1~M18.5 不回归：Issue CRUD、Project、ColumnConfig、草稿态拖拽启动、SSE 实时、交付物结构化提交、`{{deliverables.x.y}}` 上下文取用、`start_new_conversation`/`submit_deliverable` 工具注入全部照旧；transition 端点拖拽语义不变。
11. `bun test` 全绿；`turbo run build` 后 `turbo run lint typecheck` 通过（注意 runner-protocol dist 需先 build，避免陈旧 d.ts 假阳性）。

## 8. 风险与权衡

- **改 `run_origin` 幂等键的回溯兼容**：改键只影响**新写入**的 origin 行（旧行保留旧键格式）。`onRunComplete` 切到 `origin.fromStatus`（权威列）后不再依赖键格式解析，故新旧行都能正确反查 fromStatus（旧行 R3 迁移给了 `from_status DEFAULT ''`——历史行 fromStatus 为空，幂等守卫 `if (fromStatus && ...)` 对空串短路跳过，不误拦，与改键前行为一致）。风险可控。
- **闸门停在 `in_review` 但无人裁决 → Issue 永久卡住**：这是**设计本意**（人工闸门就是等人）。但缺乏「待裁决」的显式信号，人可能不知道有 Issue 在等。缓解：本棒靠看板 `in_review` 列可见即提示；显式待办/通知留 ⑦/⑧（Timeline + 可观测）。接受「卡住直到人来」为正确语义，不引入超时自动通过（那等于取消闸门）。
- **`reject` 落反馈与 `applyTransition` 非原子**：裁决端点先 `submit` 反馈再 `applyTransition`，两步非事务。若 `applyTransition` 失败（如并发把 Issue 已挪走，CAS 丢）→ 反馈已落库但状态没退回 → 多出一条孤儿 `rework_feedback`。缓解：孤儿反馈无害（下次真返工时仍是最新可用上下文，或被更新的反馈覆盖）；CAS 失败本就罕见（裁决是人工低频操作）。权衡：不为低频人工操作引入跨表事务，接受幂等/最终一致。
- **返工棒复用 `in_progress` 列的同一 ColumnConfig**：返工和首次开发跑的是同一个 developer 配置（同 agent、同 promptTemplate）。区别只在上下文多了 `rework_feedback`。风险：若模板没写 `{{deliverables.rework_feedback...}}`，返工棒读不到反馈、退化成「重跑一遍开发」。缓解：developer 列模板应显式纳入返工反馈占位符（文档/示例引导，§9）；缺失即降级不崩（M18.5 §3.7 缺失回退空串）。
- **回退边引入「图」后，未来多回退边的扩展性**：本棒硬编码单条 `BACKWARD_EDGES`。未来若要 `done→in_review` 重开或多级回退，`BACKWARD_EDGES`/`HUMAN_GATES` 要扩、reactor 的「自动 vs 人工」分类要重审。本棒只锁一条边（§5），把图的复杂度压到最小，未来按需扩。
- **裁决端点信任调用方已认证**：`review-decision` 走 `withAuth`（同其他 issue 端点），但不校验「裁决人是否有验收权限」（无角色系统）。本棒接受「认证即可裁决」，细粒度权限留未来。
