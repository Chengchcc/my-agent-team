# M16.1 — Observability Console：业务动线、诊断页、控制面

> 根因：M16 已经把 runtime ops、runner health、surface diagnostics、trace context 接到 backend 和 Web，但 Web 观测面还停留在"把表和卡片摆出来"。这不够。用户进入观测面时，不是在找某张表，而是在回答一个问题：**现在哪里坏了，我该做什么**。
>
> M16.1 的目标是把 M16 的观测数据产品化：导航进入同一个 AppShell，首页先给健康判断，Run Detail 给诊断路径和控制动作，Trace View 给本地 waterfall，Surface View 解释 Lark/Web 投影链路。它不改变 M16 的事实源，只把已有 facts 组织成可操作的业务动线。
>
> 关联：[M16 Runtime Observability](./2026-06-14-m16-runtime-observability-design.md) · [M15.1 Lark Bot Polish](../retros/2026-06-14-m15.1-retro.md) · [M14.7 Resident Runner](../retros/2026-06-12-m14.7-retro.md) · [14-event-log](../../architecture/14-event-log.md) · [16-resident-runner](../../architecture/16-resident-runner.md)。
>
> 基线 commit：`d71a0e8`（`ux: Dashboard → Observability`，已包含 M16 runtime-ops API、Web `api.ts` BFF client、`/ops` 页面、run detail、trace degraded page、NavRail Observability 入口）。

## 一、问题：观测面还没有业务动线

### 1.1 导航断层

当前 Observability 入口被加到了 `NavRail`，但 `/ops` 页面不在 `(main)` route group 下面。用户从主产品导航点击 Observability 后，会进入一个没有 AppShell 的页面。结果是：入口存在，但产品空间断开。

M16.1 必须让 Observability 成为主产品里的一个一级业务域，而不是一个孤立调试页。

### 1.2 首页没有回答"哪里不健康"

当前 `/ops` 首页展示：

```text
Observability
Agent Runtimes
Recent Runs
```

这只是数据陈列。用户仍需要自己从 runtime card、run table、heartbeat age、last event 里推断问题。真正的首页应该先回答：

- 当前有多少 running run？
- 哪些 run heartbeat stale？
- 哪些 runner detached / noop？
- 哪些 surface degraded？
- 哪些 agent checkpointer / workspace 不健康？
- 哪几个对象需要马上处理？

### 1.3 Run Detail 是 DB viewer，不是诊断页

当前 Run Detail 按实体展示：

```text
Run Info
Attempts
Event Log
Ops Events
```

这对开发者调试有用，但对处理事故不够。用户需要的是：

1. 这个 run 是否还在推进；
2. 卡在哪个责任域；
3. 有什么证据；
4. 现在可以 cancel、recover、等待，还是需要看 surface。

### 1.4 Trace View 承诺大于内容

当前 Trace list 有 "Trace View" 链接，但详情页只说明 OTLP waterfall 不可用。即使没有外部 collector，M16 本地也已有 `run_origin`、`run_ops_event`、attempt、EventLog summary，可以组成一个 degraded local waterfall。Trace 页面不应空转。

### 1.5 Surface Diagnostics 没有成为一等页面

M16 已经让 Lark bot 上报脱敏 surface heartbeat，但 Web 里只是把 surfaces 塞在 AgentRuntimeCard 里。用户问"为什么 Lark 卡片没更新"时，需要一条从 run 到 surface projection 的路径，而不是去 agent card 里猜。

### 1.6 诊断要用的字段，后端还没真实暴露

这是 M16.1 最容易踩空的地方。M16 已经把诊断需要的部分类型声明出来了，但实现还没落地：

1. **transport kind 退化成二值**。`RunOpsListItem.runnerTransport` 与 attempt `transport` 类型都是 `"attached" | "noop" | "detached"`，但 `RuntimeOpsService.listRuns` 和 `getRunDetail` 的实现只产出 `session ? "attached" : "detached"`——`"noop"` 永远不会出现。也就是说：backend restart 后挂在 `NOOP_TRANSPORT` 上的 run，会被报成 `attached`，UI 无法据此判断"控制面已 detached"。
2. **`recover` 的 `reattached` 状态是死状态**。`RecoverRunResult` 类型里有 `{ state: "reattached" }`，但 `recover()` 实现从不返回它，只会返回 `already_terminal / marked_interrupted / waiting`；且 heartbeat stale 时直接 `UPDATE run/attempt`，不走 EventLog、不触发 `onRunComplete`。
3. **`traceId` 依赖 `run_origin`**。`listRuns` 通过 `opsStore.getRunOrigin(runId)` 取 traceId，所以 trace 链路是否完整，取决于 M16 review 里 `run_origin` 幂等键是否已修。

结论：M16.1 的诊断与控制动线，必须先有一个 backend 前置改造，把这三件事补成真实信号；否则前端诊断函数会建立在不存在的字段上。这个前置工作收进 Part 0。

## 二、目标形态：从发现问题到执行动作

M16.1 的产品动线是：

```text
发现问题 → 定位对象 → 判断责任域 → 执行控制动作 → 查看恢复结果
```

对应页面：

```text
/ops                         Overview：系统健康与待处理事项
/ops/runs                    Runs：可筛选 run 列表
/ops/runs/:runId             Run Diagnosis：诊断、证据、控制动作
/ops/agents                  Agent Readiness：agent runtime 健康矩阵
/ops/agents/:agentId         Agent Runtime Detail：runner / checkpointer / workspace / surfaces
/ops/traces                  Trace Paths：trace 列表
/ops/traces/:traceId         Local Trace Waterfall：本地 waterfall + related runs
/ops/surfaces                Surface Diagnostics：Lark/Web 投影健康
```

```mermaid
flowchart LR
  NAV[NavRail Operations] --> OVERVIEW[/ops Overview]
  OVERVIEW --> RUNS[/ops/runs]
  OVERVIEW --> AGENTS[/ops/agents]
  OVERVIEW --> SURFACES[/ops/surfaces]
  RUNS --> RUN[/ops/runs/:runId]
  RUN --> TRACE[/ops/traces/:traceId]
  RUN --> AGENT[/ops/agents/:agentId]
  TRACE --> RUN
  SURFACES --> RUN
  AGENTS --> AGENT
```

## 三、不变量

1. **M16 facts 不搬家**：EventLog 仍是 run 事件事实源；conversation ledger 仍是用户可见事实源；RuntimeOps 只保存诊断事实。
2. **UI 不创造状态**：running、succeeded、error、aborted、interrupted 仍来自 backend；UI 只派生 diagnosis，不写 run status。
3. **控制动作走 ops API**：cancel / recover 不直接改 DB；UI 只调用 backend 已有幂等 API。
4. **trace 可降级**：没有 OTLP collector 时，Trace View 用本地 `run_origin + run_ops_event + attempts` 组成 local waterfall。
5. **surface 与 run 分离**：run succeeded 不代表 Lark/Web projection succeeded；surface health 独立展示。
6. **观测默认脱敏**：UI 不展示 message text、tool input 原文、Lark chat_id/open_id、profile secret。
7. **主产品导航不断层**：Observability 必须在 AppShell 内，不能进入后丢失 NavRail。
8. **错误态可解释**：loading、not found、unauthorized、backend unavailable 必须分开显示。
9. **诊断只用真实字段**：UI 派生 diagnosis 只能基于 backend 真实产出的字段；类型里声明但实现未产出的状态（如当前的 `transport="noop"`、`recover="reattached"`）必须先在 backend 落地，再被前端依赖。
10. **鉴权走既有 BFF**：所有 ops 请求继续走 `apiFetch → /api/bff/[...path]`，由 BFF 注入 `x-auth-token` / `x-user-id`；M16.1 不新建鉴权链路。

## 四、信息架构

### 4.1 NavRail

把 Observability 从单个链接升级为 Operations 域。第一版先只显示一个一级入口 `Operations`，进入 `/ops` 后用页面内 tabs 承载二级导航：

```text
Overview | Runs | Agents | Traces | Surfaces
```

路由必须迁入 `(main)`：

```text
apps/web/src/app/(main)/ops/page.tsx
apps/web/src/app/(main)/ops/runs/page.tsx
apps/web/src/app/(main)/ops/runs/[runId]/page.tsx
apps/web/src/app/(main)/ops/traces/page.tsx
apps/web/src/app/(main)/ops/traces/[traceId]/page.tsx
apps/web/src/app/(main)/ops/surfaces/page.tsx
apps/web/src/app/(main)/ops/agents/[agentId]/page.tsx
```

### 4.2 Overview 页面

Overview 不平铺前 10 个 agent card，而是先给系统判断。

```text
Operations Overview

Health summary
- Running runs
- Stale runs
- Detached/noop runners
- Degraded agents
- Surface errors

Needs attention
- Run A: heartbeat stale, recover available
- Agent B: runner offline
- Lark surface: card_update_failed increased

Recent activity
- recent abnormal ops events
- recent completed runs
```

### 4.3 Runs 页面

Runs 页面是工作队列。筛选：status / transport / agent / trace present / heartbeat。Diagnosis 列为 UI 派生字段。

### 4.4 Run Diagnosis 页面

Run Detail 改成诊断页。首屏显示 diagnosis、owner、execution path、control action、evidence、diagnostic timeline。

### 4.5 Trace 页面

必须先有 local mode。用 `run_origin + run_ops_event` 组装本地 waterfall。Backend `GET /api/ops/traces/:traceId`。

### 4.6 Surface Diagnostics 页面

回答"run 成功后用户是否真的看到了结果"。Lark projection path 展示 counters。Backend `GET /api/ops/surfaces`。
