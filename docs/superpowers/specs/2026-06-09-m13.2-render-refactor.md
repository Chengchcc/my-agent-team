# M13.2 — Frontend Render Refactor: Single Reducer + Draft Model

> M13.2 把会话页"权威消息（/events）+ 实时增量（/stream）"的双源渲染重写为单一有序消息列表 + 至多一条 draft，对齐逻辑下沉到纯 reducer。

## 背景

`9a05f3b`（M13.1 后端 done-on-complete）把 run 结束时刻变确定后，暴露了两个前端潜伏缺陷：

- **B1 文本错位/重复**：`finalizeBlocks` 做 delta 段落块（N 个）对 authoritative 单 text 块（1 个）的 1 对 1 覆盖，`blocks[1..N]` 成孤儿残留。
- **B2 stream 与 timeline 最后一条重复**：Timeline 的"隐藏 live 项"与"渲染 live tail"分别由 `connection` 与 `isStreamingDone`/`lastLiveSeq` 两个不同步状态驱动，存在两者皆真的中间帧。

共同根因：**双源各自渲染同一段文本，靠 `connection`/`liveStatus`/`runStatus`/`isStreamingDone`/`lastLiveSeq` 互相推导对齐**。

## 设计

- 新增 `lib/conversation-reducer.ts` — 纯函数状态机，承载全部对齐/去重/互斥逻辑
- 新增 `hooks/useConversation.ts` — 唯一 hook，聚合 history + /events + /stream + currentRun poll
- 改 `ConversationCanvas` — 只消费 `useConversation`，渲染降到 messages.map + 至多一条 draft
- 改 `Timeline` — 删除隐藏判断与并存 tail，单纯渲染传入列表
- 删 `lib/stream-ast.ts`、`useDeltaStream`、`useTimeline`、`useLiveEvents`（被 `useConversation` 取代）
- 删 `StreamingBlocks.tsx`、`StreamingMessage.tsx`（draft 改用 `DraftMessage`）

**核心不变量**：任一渲染帧里，一条 assistant 文本要么以 draft 呈现，要么以权威 message 呈现，绝不同时。该互斥由 reducer 在同一次 dispatch 内保证（`events/message(assistant)` → 原子清 draft）。

## 技术契约

- 客户端不分块 markdown — 仅累积字符串，结构解析在 `<Markdown>`
- ephemeral 事件（text_delta/tool_start/tool_end）永不进 messages，只进 draft
- 权威 content（tool_use/tool_result）只来自 /events，不来自 /stream
- busy/label 只读 `run.phase` 单一来源
- 后端协议、SSE 通道、checkpointer 不变

## 数据流

```
/events (durable) ──→ dispatch("events/message") ──→ messages[] ←── Timeline 渲染
/stream (ephemeral) → dispatch("stream/delta")  ──→ draft.text  ←── DraftMessage 渲染
                                        工具事件 → draft.tools
currentRun poll ────→ dispatch("run/completed") ──→ run.phase (兜底)
```

Reducer 在一次 dispatch 内完成：
- 乐观 user → 权威 user echo 替换（不重复）
- 权威 assistant 到达 → draft 清空（互斥）
- done/completed → phase 迁移（幂等，interrupted/error 不覆盖）
