## 二、Code Review 发现的 Bug

### B1: `pendingEvents` 在 `fireBeforeStop` 后未 drain（已修复）

**文件**: `packages/framework/src/create-agent.ts:458`
**根因**: `beforeStop` 的 fallback plan generation 调用 `ctx.emit()` 推 `todo_update`，但 `runLoop` 的 stop-gate 分支（`toolUses===0`）不走 `executeOne`，`pendingEvents` 永不排空。
**影响**: resume 轮或门关时若触发 beforeStop 现场生成 plan，`todo_update` 事件丢失，web UI 看不到 plan。
**修复**: 在 `fireBeforeStop` 后、`if (verdict?.continue)` 前 drain `pendingEvents`。

### B2: `fireBeforeRun` 返回值引用比较总是成立（未修，低优先级）

**文件**: `packages/framework/src/create-agent.ts` run() 方法
**根因**: `fireBeforeRun` 总是返回新数组（spread），`seeded !== thread.messages` 永远为 true，导致每次 run 都多一次 `save()` 调用。
**影响**: 额外一次 checkpoint save（~1ms），不影响正确性。
**修不修**: 低优先级。save 是幂等的，性能影响可忽略。

### B3: `injectIntoSystem` 假设 system content 是 string

**文件**: `packages/plugin-task-guard/src/task-guard.ts:55`
**根因**: `${sys.content}\n\n${block}` 在 system content 为 `ContentBlock[]` 时会产出 `[object Object]\n\n<todo>...`。
**影响**: system prompt 使用 `ContentBlock[]` 的 agent 会看到损坏的 system 内容。当前 harness 的 systemPrompt 总是 string，不受影响。
**修不修**: 应在注入前做 `typeof sys.content === 'string'` 检查，非 string 时跳过注入。

## 三、设计张力（未解决，已标注）

### T1: `onUpdate` 在 `beforeModel` 每轮重绑

`beforeModel` 每 step 触发一次，每次都重新创建箭头函数赋值给 `onUpdate`。这是因为 `ctx` 引用可能变化（虽然实际上不变）。无功能影响，但暴露了 "closure 桥接 ctx" 模式的内在张力——工具拿不到 ctx，钩子必须反复搭桥。

### T2: 冷审轮 verdict 解析脆弱

`entry.ts` 的 JSON 提取用 `/\{[\s\S]*\}/`，匹配最后一个 JSON 对象。如果模型在回复中嵌入了示例 JSON（"reply with {\"complete\": false, ...}"），可能匹配到错误的对象。不如用更精确的锚定或要求模型只输出 JSON（不加 markdown 包裹）。

### T3: `maxForceContinues` 是 run 选项而非插件选项

终止门的行为（续跑几次）受 framework 参数控制，但判据（何时该续跑）在 plugin。两者分层正确——机制在 framework，策略在 plugin——但调用方要同时理解两个概念才能正确配置。

## 四、做得好的

1. **fail-open 纪律一致**: 插件三处 + 冷审轮全部 fail-open，与现有 `fireBeforeModel`/`fireAfterModel` 错误隔离同模式
2. **复用现有机制**: `pendingEvents` 模式与 `checkpointer.appendEvent` 同款异步收集；`routeEvents` 是原 for-await 循环的 1:1 提取
3. **零后端改动**: `todo_update` 作为 durable AgentEvent，supervisor/service 的所有路由逻辑零改动——这是对 EventLog 设计的前瞻验证
4. **测试覆盖**: 框架 79 + 插件 14 + runner 32 测试，覆盖 beforeRun/beforeStop/todo_write/冷审/fail-open/零回归
5. **closure 绑 threadId**: 对"Tool.execute 无 ctx"问题的标准化解法——钩子搭桥，工具取用

## 五、如果重来一次

1. **先写 framework 测试再写 plugin**: plugin 测试依赖 framework 的 `todo_update` 事件+`pendingEvents` drain 机制。如果 framework 测试先落地（scriptedModel + 钩子桩），plugin 开发会更快。
2. **冷审 verdict 用结构化输出**: 与其正则解析 JSON，不如让冷审 fork agent 用 `tool_use` 返回结构化裁决——更可靠，且与现有 tool 基础设施一致。
3. **Runner 重构先行**: `routeEvents` 提取是正确方向，但应该作为独立 PR 在 M14.6 之前落地，减少 diff 体积。
