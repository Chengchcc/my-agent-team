# M18.2 Spec — Orchestrator Reactor：转移表驱动的 Issue 自动推进

> **Status:** Spec → Ready for implementation.
> **Baseline commit:** `origin/master` @ `b08d515`。
> **关联:** `docs/architecture/backend/orchestrator.md` · `docs/architecture/flows/e2e-issue-lifecycle.md` · `docs/architecture/foundations/issue.md` · `docs/architecture/design-philosophy.md` · `apps/backend/src/features/orchestrator/`（新增）· `apps/backend/src/features/issue/`（M18.1 已落）· `apps/backend/src/features/run/supervisor.ts` · `apps/backend/src/features/run/events-db-migrations.ts` · `apps/backend/src/features/runtime-ops/`（run_origin）· `apps/backend/src/features/conversation/projection.ts`（onRunComplete）· `apps/backend/src/main.ts`。

M18.1 把 [Issue](../../architecture/foundations/issue.md) 立成领域本体，给了它唯一的状态写入命令 `applyTransition(issueId, to)`——但谁来调它，仍是空的：M18.1 的看板只读，状态推进只能靠人手工 POST。M18.2 落地 [Orchestrator](../../architecture/backend/orchestrator.md)，把"谁来调 `applyTransition`、调完跑下一棒"这件事从人手里接过来，固化成显式机制：**一张转移表 + 两个纯函数 + 一个回填监听器**。

落地后，`backend/orchestrator.md` 与 `flows/e2e-issue-lifecycle.md` 从 `status: design` 翻成 `status: current`，并补 `last_verified_against_code`。

本子里程碑**刻意不做**：拖拽交互与乐观更新、SSE 实时增量（M18.3）；Project git 子目录克隆与绑定（M18.4）；可配置 DAG 工作流引擎（未来工作）。M18.2 只保证"一个 Issue 被创建后，能在转移表的驱动下，由对应 agent 一棒接一棒自动跑到 done，全程状态显式落在 Issue 上"。

## 1. 背景

[Orchestrator](../../architecture/backend/orchestrator.md) 的设计已锁定：它**取代** @提及自动触发作为 Issue 推进的驱动方式。现状的多 Agent 协作靠 `onRunComplete` 扫 assistant 文本里的 `@`、命中就 `triggerMentionedAgents` 起下一棒（`conversation/projection.ts:250`）——这条链路隐式、不可观测、难约束。Orchestrator 把"下一步是什么"从模型自由文本里拿出来，固化成一张写死的线性转移表。

M18.1 已经为这一步铺好了地基：状态空间收敛到单一转移表、`applyTransition` 是 status 的唯一物理写入入口、CAS 保证并发下"一次只推一格"。M18.2 要做的是给这个入口接上**自动触发源**——run 终态。整条链路的执行层（起 run、续跑、收尾）完全复用 [RunSupervisor](../../architecture/backend/run-supervisor.md)，Orchestrator 不发明任何新执行机制（[issue.md](../../architecture/foundations/issue.md) 不变量③）。

设计确立了三条关键事实：M18.1 只立住了"同一个状态命令"；M18.2 兑现剩下两条的前半——"状态推进带副作用"中的核心副作用即"起下一棒"，而"失败要回滚"在本里程碑收敛为"run 没到 succeeded 终态就不推进、Issue 停在原状态"。

## 2. 当前代码事实

基于 M18.1 完成态，逐项已与代码核对：

1. **`applyTransition` 是 status 唯一写入口**：`issue/service.ts` 的 `createIssueService(deps)` 工厂返回 `applyTransition(issueId, to)`——先 `port.getIssue` 取当前 status，查 `LEGAL_TRANSITIONS` 校验合法性，再经 `port.setStatus`（CAS）落库，返回更新后的 `IssueRow`。非法转移抛 `IllegalTransitionError`，不存在抛 `IssueNotFoundError`。M18.2 的回填器**复用这个方法**，不旁路写 status。
2. **转移表当前在 `issue/entities.ts`，只有 `{from,to}`**：M18.1 的 `TRANSITIONS` 是 `as const satisfies ReadonlyArray<{from,to}>`，`ISSUE_STATUSES` / `LEGAL_TRANSITIONS` 从它派生。M18.2 要给每条转移加 `agentId` 与 `promptTemplate`，并把整张表**迁出 issue 模块、移交 orchestrator 模块**（对齐 orchestrator.md「M18.2 起整张表移交 orchestrator」）。
3. **run 完成钩子签名固定**：`supervisor.onRunComplete(fn)`，`fn: (threadId, runId, status, kind) => void | Promise<void>`（`run/supervisor.ts:269`）。supervisor **await** 这些监听器（critical sink）。`status` 是运行态字符串，终态取值 `"succeeded" | "error" | "aborted"`；`kind` 取值 `"main" | "reflect"`。当前唯一注册者是 `conversation/projection.ts` 的 `onRunComplete`（`main.ts:107`）。M18.2 **新增一个监听器**，与现有的并存（`#onRunComplete` 是数组，支持多监听器）。
4. **起 run 的入口是 `supervisor.startMainRun`**：签名 `startMainRun(runId, threadId, spec, opts?)`（`run/supervisor.ts:341`），`spec` 是 `Record<string, unknown>`，其中 `spec.agentId` 决定跑哪个 agent（`run/supervisor.ts:350`）。`opts` 可带 `preloadedMessages` / `surfaceContext` / `trace`。
5. **spec 由 `buildAgentSpecV2` 构造**：`conversation/conv-svc-factory.ts:120`，签名 `buildAgentSpecV2(db, agentSvc, threadId, input, overrides?)`。它**当前从 threadId 解析 memberId、再查 member 的 agent_id**（`conv-svc-factory.ts:134-138`）——这是 conversation 路径的拼装约定。Orchestrator **不走这条 member 推导**：它直接知道转移要哪个 agentId，自己构造 spec（见 §3.4）。
6. **`threadId` 是拼装标识，非持久化本体**：`deriveThreadId(convId, memberId)` = `` `${convId}:${memberId}` ``（`conversation/service.ts:28`），`parseThreadId` 是其逆（`service.ts:35`）。它没有自己的表和生命周期，只是运行时拼出来的执行线程 key。Issue 自带的 `threadId`（M18.1 存在 `issue.thread_id` 列）同理——它是这个 Issue 的执行脉络标识，**"这一棒由哪个 agent 干"与 threadId 怎么拼解耦**，由 Orchestrator 的转移表决定。
7. **agent 是文件驱动、模板自带角色**：`AgentRow` 有 `template`（`agent/domain.ts:4`）与 `workspacePath`，建 agent 时 `materializeWorkspace(agentId, template)`（`agent/agent-svc-factory.ts:27`）把模板物化成工作区。所谓 planner / developer / reviewer **就是不同 template 物化出来的不同 agent**——角色性长在 agent 文件里，系统里**没有、也不需要**一个独立的 `role` 概念。转移表只引用 `agentId`。
8. **agent 存在性查询**：`agentSvc.getById(agentId)`（`conv-svc-factory.ts:139` 已在用）返回 `AgentRow`，archived 的 agent 有 `archivedAt` 字段（`domain.ts:13`）。Orchestrator 起 run 前用它做存在性校验。
9. **`run_origin` 在 events.db**：迁移 `events_v8_run_origin`（`run/events-db-migrations.ts:67`），`RunOriginRow` 字段见 `runtime-ops/types.ts:28`（runId / conversationId / sourceLedgerSeq / agentMemberId / surface / traceId / traceparent / idempotencyKey / createdAt）。写入经 `opsStore.insertRunOrigin`（`conv-svc-factory.ts:87`），`RuntimeOpsStore` 持 `eventsDb`（`main.ts:59`）。**这是 run 溯源的机制边界**，M18.2 在它上面挂 `issueId` 关联。
10. **两个 db handle 在 main.ts 共存**：`db = openDb(backend.db)`、`eventsDb`（`main.ts`）。Issue 表在 backend.db，run_origin 在 events.db。回填器跨两库读取是现状常态（ops service 已如此），不引入新模式。
11. **feature 骨架**：`features/<feature>/` 统一 `entities.ts` / `ports.ts` / `adapter-sqlite.ts` / `service.ts` / `http.ts` / `index.ts`；service 是工厂函数（非 class），仅 Error 子类是 class；DDL 进集中迁移，adapter 只做 CRUD。Orchestrator 模块按此骨架落（它无独立持久化本体，骨架按需裁剪，见 §3.6）。
12. **migrations id 分段**：backend.db 的 `BACKEND_MIGRATIONS` 当前最大 `id: 5009`（M18.1 的 `backend_v23_issue`）；events.db 的迁移在 `events-db-migrations.ts`，当前最大是 `events_v8_run_origin`。run_origin 加列走 events.db 迁移（见 §3 Patch）。

## 3. 第一性原则

### 3.1 "下一步是什么"是显式机制，不从模型文本推断

按 [orchestrator.md](../../architecture/backend/orchestrator.md) 不变量①：「下一步是什么」由显式转移表决定，不从 Agent 自由文本里推断。现状 @提及的脆点是它把"下一棒该谁干"藏在上一个 agent 的产出文本里——模型少打一个 `@`、换个说法，链路就断。Orchestrator 把这件事固化成转移表的一条记录：`from` 状态跑完 → 查表得 `to` 和该转移的 `agentId` → 起下一棒。链路不再依赖模型的措辞。

### 3.2 status 推进仍是单一写者，回填器复用 `applyTransition`

[orchestrator.md](../../architecture/backend/orchestrator.md) 不变量②：Issue 的状态推进只经 status 回填监听器写入，单一写者。落到代码上，这**不是**让回填器自己写 status——而是回填器调用 M18.1 已经立好的唯一入口 `applyTransition`。这正是设计哲学 §3.7"一条事实一个权威物理入口"的延续：M18.1 立了入口，M18.2 给入口接了自动触发源，而**没有**新开一个写法。"人工调 `applyTransition`（M18.1）"和"回填器调 `applyTransition`（M18.2）"是同一个入口的两个调用方，不是两套写者。

### 3.3 run → Issue 关联落在机制边界 run_origin，不污染 Issue 本体

回填器拿到的是"一个 run 终态（runId）"，它要反查"这是哪个 Issue 的哪一棒"。这个映射放哪？按设计哲学 §3「实现机制不能上浮成业务心智」+ §4「概念要少」：

`run_origin` 本就是"一次 run 从哪来"的溯源机制边界（surface/ops 用的旁路读模型），给它加一列 `issue_id` 是在**已有机制边界上挂一条溯源事实**，不引入新名词。反过来若在 `Issue` 表加 `currentRunId`，就是给领域本体塞进"当前在跑哪个 run"的可变机制状态——违反 §2「同一语义对象只能有一个本体」的精神，且并发下要管这个字段的写清时序，正是哲学红旗信号「一个可见/terminal 状态靠旁路可变字段推断」。

> Issue 本体保持只有 `status` 一处可变；"run 属于哪个 Issue"放在 run_origin。这是"暴露业务、隐藏机制"。

### 3.4 Orchestrator 直接按转移构造 spec，不借道 conversation 的 member 推导

`buildAgentSpecV2` 当前从 threadId 解析 memberId、查 member 拿 agentId（§2.5）——这是 conversation 路径的拼装约定，**不是** Orchestrator 该继承的约束。`threadId` 是拼装标识、非本体（§2.6），"这一棒谁干"由转移表的 `agentId` 直接给定。所以 Orchestrator 起 run 时：用转移的 `agentId` + Issue 的 `threadId`，自己拼出 spec（agentId / model / threadId / input），经 `startMainRun` 起 run。**Issue 仍是一条执行脉络（一个 threadId），三个角色在它上面接力**，符合 [issue.md](../../architecture/foundations/issue.md) 不变量③"复用现有 supervisor/checkpointer，不发明新执行机制"。

这要求把"从 agentId 构造 spec"的能力，从 conversation 的 member 推导里解耦出来。最小做法：Orchestrator 调 `agentSvc.getById(agentId)` 拿 agent 配置，自己拼 spec 字段（复用 `buildAgentSpecV2` 内部已有的 model/permission/maxSteps 拼装逻辑，但跳过 member 解析那一步）——见 Patch P3。

### 3.5 角色由 agent 文件承载，转移表只引用 agentId，缺失即报错不推进

系统里没有 `role` 概念，agent 经 template 物化角色（§2.7）。所以转移表里每条转移直接写 `agentId`（指向一个已经配置好角色模板的 agent）。配置能力即"建好 agent（选 template）+ 在转移表引用其 agentId"，**不新增任何角色注册表**（§4「概念要少」）。

但引用就可能悬空。延续 M18.1 review 暴露的"引用完整性"教训：回填器起每一棒前 `agentSvc.getById(agentId)` 校验该 agent 存在且未 archived。**缺失 → 不推进 status、不起下一棒、记一条可观测错误**，Issue 停在当前状态等人修配置——而不是静默卡死或推进到一个没人能跑的状态。这是 [e2e-issue-lifecycle.md](../../architecture/flows/e2e-issue-lifecycle.md) 失败模式"卡在某状态不推进"的一种受控形态：可观测、可定位、可恢复。

### 3.6 Orchestrator 是机制，骨架按需裁剪

Orchestrator 没有独立的领域本体——它不持久化新事实（关联落在 run_origin，状态落在 Issue）。所以它的模块骨架按需裁剪：有 `transitions.ts`（转移表 + 两个纯函数）、`reactor.ts`（回填监听器工厂）、`index.ts`，**没有** `adapter-sqlite.ts`（无自己的表）、**没有**独立 `entities.ts`（转移表即其全部"数据"）。这与现状非均匀骨架一致（run 有 entities 无 ports、runtime-ops 用 types.ts），不强凑六文件。

### 3.7 与 @提及自动触发的隔离：用 run_origin.issueId 判别，不打散标记

issue-driven 的 run 和 conversation-driven 的 run 都会触发 `onRunComplete`。若两条驱动都对同一个 run 终态起下一棒，控制环就碎（设计哲学 §7）。隔离判据**复用 §3.3 的同一条事实**：`run_origin.issueId` 非空 ⇔ 这是 issue-driven run ⇔ 走 Orchestrator 回填、`onRunComplete` 里的 @mention 触发跳过它。不需要在 run 上再打一个 `isIssueDriven` 散标记——一个事实一个权威入口。

## 4. 目标

1. 新增 `apps/backend/src/features/orchestrator/`：转移表（带 `agentId` + `promptTemplate`）、两个纯函数（`renderPrompt` + 回填推进）、回填监听器工厂。
2. 转移表从 `issue/entities.ts` 迁入 orchestrator，并扩字段；`issue/entities.ts` 改为从 orchestrator 导入状态派生所需的最小信息（或保留 `IssueStatus` 类型、转移表移走），保证 M18.1 的 `ISSUE_STATUSES` / `LEGAL_TRANSITIONS` 派生不破。
3. `run_origin` 加 `issue_id` 列（events.db 迁移）；起 issue-driven run 时写入。
4. 回填监听器挂 `supervisor.onRunComplete`：run `succeeded` + 是 issue-driven → 查转移表 → `issueSvc.applyTransition` → 有下一条则起下一棒；幂等"一次终态推进一格"。
5. agent 存在性校验：起每一棒前 `getById`，缺失则不推进、不起 run、记可观测错误。
6. `onRunComplete`（conversation 路径）跳过 issue-driven run 的 @mention 触发。
7. Issue 创建后的"起第一棒"接线：建 Issue（status=planned）后，由 Orchestrator 起 planned 对应的第一棒（触发点见 §6 Patch P5）。
8. `main.ts` 装配：构造 orchestrator reactor，注册到 supervisor。

## 5. 非目标

- **拖拽交互、乐观更新、回弹、SSE 实时增量**：M18.3。M18.2 看板仍是 M18.1 的只读刷新态；自动推进的结果靠看板的 `refetchInterval` 拉取可见。
- **Project git 子目录克隆与绑定**：M18.4。`projectId` 仍只是字符串外键。
- **可配置 DAG 工作流引擎**（分支、并行、条件跳转、可视化编辑）：未来工作。M18.2 只做固定线性表。
- **转移表从 DB/配置动态读取**：M19。M18.2 转移表是模块内常量；演进位已在 orchestrator.md 留好，届时只换"转移表数据从哪来"，纯函数与回填器不动。
- **副作用动作的真实集成**（MR、Coding Agent、外部信号驱动终态）：转移可声明 `promptTemplate` 让 agent 去干活，但"起下一棒"之外的副作用（如自动建 MR）不在本里程碑硬做。
- **renderPrompt 引入模板 DSL**：刻意不做（orchestrator.md 不变量③），仅 `{{}}` 字符串插值。
- **新增 per-resource BFF / 前端改动**：M18.2 是纯后端机制里程碑，前端零改动（看板已在 M18.1 交付）。

## 6. 实施分组与方案

> 伪代码均已对齐 §2 的存量写法：service/工厂用函数、Error 用 class、DDL 进迁移、监听器经 `supervisor.onRunComplete` 注册、起 run 经 `startMainRun`。

### Patch P1 — 转移表迁入 orchestrator 并扩字段

**目标**：把转移表从 issue 模块迁到 orchestrator，每条转移补 `agentId`（这一棒谁干）与 `promptTemplate`（给它什么 prompt）。状态派生仍是转移表的纯函数投影——M18.1 的"单一事实来源"不破，只是来源换了归属模块。

`apps/backend/src/features/orchestrator/transitions.ts`：

```ts
import type { IssueStatus } from "../issue/entities.js";

/** ── 单一事实来源（M18.2 起归属 orchestrator）──────────────
 *  状态空间 + "每一步谁干、给什么 prompt" 都由这张表唯一定义。
 *  M18.2 它是固定线性常量；M19 升级为"从配置/DB 读"——届时下面的纯函数与
 *  reactor 完全不用改，扩展点收敛到"转移表数据从哪来"一处。
 *  **不变量：必须按生命周期拓扑顺序书写**（看板列序 = 书写序，见 deriveStatuses）。 */
export interface Transition {
  from: IssueStatus;
  to: IssueStatus;
  /** 这一棒由哪个 agent 干。指向一个已配置（template 物化角色）的 agent。
   *  系统无独立 role 概念——角色性长在 agent 文件里。 */
  agentId: string;
  /** 起 run 时的 prompt 模板，仅 {{var}} 字符串插值，无 DSL。 */
  promptTemplate: string;
}

export const TRANSITIONS = [
  {
    from: "planned",
    to: "in_progress",
    agentId: "planner",
    promptTemplate: "为 Issue「{{title}}」制定开发计划并开始实现。",
  },
  {
    from: "in_progress",
    to: "in_review",
    agentId: "developer",
    promptTemplate: "完成 Issue「{{title}}」的实现，提交待 Review。",
  },
  {
    from: "in_review",
    to: "done",
    agentId: "reviewer",
    promptTemplate: "Review Issue「{{title}}」的实现，通过则标记完成。",
  },
] as const satisfies ReadonlyArray<Transition>;

/** 从转移表派生有序状态集合（= 看板列顺序）。取首次出现顺序（Set 插入序）。 */
export function deriveStatuses(table: ReadonlyArray<Transition>): IssueStatus[] {
  const seen = new Set<IssueStatus>();
  for (const t of table) {
    seen.add(t.from);
    seen.add(t.to);
  }
  return [...seen];
}

/** 从转移表派生 from → 合法 to 列表的映射。 */
export function deriveLegalMap(
  table: ReadonlyArray<Transition>,
): Record<IssueStatus, IssueStatus[]> {
  const map = Object.fromEntries(deriveStatuses(table).map((s) => [s, [] as IssueStatus[]]));
  for (const t of table) map[t.from]!.push(t.to);
  return map as Record<IssueStatus, IssueStatus[]>;
}

/** 查 from 状态对应的那条转移（线性表里 from 唯一）。无则返回 undefined（如终态 done）。 */
export function nextTransition(
  table: ReadonlyArray<Transition>,
  from: IssueStatus,
): Transition | undefined {
  return table.find((t) => t.from === from);
}

export const ISSUE_STATUSES: readonly IssueStatus[] = deriveStatuses(TRANSITIONS);
export const LEGAL_TRANSITIONS: Readonly<Record<IssueStatus, IssueStatus[]>> =
  deriveLegalMap(TRANSITIONS);
```

### Patch P3 — Orchestrator 起 run：按转移构造 spec + agent 校验

**目标**：把"按 agentId + threadId 起一棒 issue-driven run"封装成 orchestrator 的一个能力，含 §3.5 的 agent 存在性校验，并写 run_origin.issueId。

`apps/backend/src/features/orchestrator/reactor.ts`（起 run 部分）：

```ts
import { renderPrompt } from "./render.js";
import { nextTransition, TRANSITIONS } from "./transitions.js";

export class OrchestratorAgentMissingError extends Error {
  constructor(agentId: string, issueId: string) {
    super(`Orchestrator: agent not found or archived: ${agentId} (issue ${issueId})`);
    this.name = "OrchestratorAgentMissingError";
  }
}

export interface OrchestratorDeps {
  issueSvc: IssueService;
  agentSvc: AgentService;
  supervisor: RunSupervisor;
  opsStore: RuntimeOpsStore;
  buildSpec: (agentId: string, threadId: string, input: string) => Promise<Record<string, unknown>>;
  idGen: () => string;
  now?: () => number;
}

export function createOrchestrator(deps: OrchestratorDeps) {
  const { issueSvc, agentSvc, supervisor, opsStore, buildSpec, idGen } = deps;

  /** 为某个 Issue 的当前 status 起对应转移的那一棒。
   *  缺转移（终态）→ 静默停止；缺 agent → 抛错、不起 run、Issue 不推进。 */
  async function startStep(issue: IssueRow): Promise<{ runId: string } | null> {
    const t = nextTransition(TRANSITIONS, issue.status);
    if (!t) return null; // 终态（done），无下一棒

    // §3.5 引用完整性：agent 必须存在且未 archived
    const agent = await agentSvc.getById(t.agentId).catch(() => null);
    if (!agent || agent.archivedAt != null) {
      throw new OrchestratorAgentMissingError(t.agentId, issue.issueId);
    }

    const runId = idGen();
    const prompt = renderPrompt(t.promptTemplate, { title: issue.title, issueId: issue.issueId });
    const spec = await buildSpec(t.agentId, issue.threadId, prompt); // §3.4 直接按 agentId 拼，不走 member 推导

    await supervisor.startMainRun(runId, issue.threadId, spec);

    // §3.3 + §3.7：写 issue_id 关联（也是 @mention 隔离判据）
    opsStore.insertRunOrigin({
      runId,
      issueId: issue.issueId,
      conversationId: "", // issue-driven run 不归属 conversation
      sourceLedgerSeq: 0,
      agentMemberId: t.agentId,
      surface: "orchestrator",
      traceId: "",
      traceparent: "",
      idempotencyKey: `issue:${issue.issueId}:${issue.status}:run`, // 同状态只起一棒（幂等）
      createdAt: (deps.now ?? Date.now)(),
    });

    return { runId };
  }

  return { startStep, /* onRunSucceeded 见 P4 */ };
}
```

> `buildSpec` 是注入的"按 agentId 构造 spec"闭包，在 main.ts 用 `agentSvc.getById` + 现有 model/permission 拼装实现（§3.4），与 conversation 的 `buildAgentSpecV2` 解耦。`idempotencyKey` 用 `issue:<id>:<status>:run` 保证"一个 Issue 在某状态下只起一棒"——run_origin 已有 `idx_run_origin_idem` 唯一索引（§2.9），重复起会被它挡住，天然幂等。

**影响文件**：`orchestrator/reactor.ts`（新增）、`orchestrator/render.ts`（P5）。

### Patch P4 — 回填监听器：run 终态 → 推进 → 起下一棒

**目标**：把 run `succeeded` 终态接到 `applyTransition`，推进一格后起下一棒，直到终态停止。幂等"一次终态推进一格"。

`apps/backend/src/features/orchestrator/reactor.ts`（回填部分，续 P3 的工厂）：

```ts
  /** 注册为 supervisor.onRunComplete 监听器。只处理 issue-driven 的成功终态。 */
  async function onRunComplete(threadId: string, runId: string, status: string, kind: string) {
    if (kind === "reflect") return;

    // §3.7：是不是 issue-driven？查 run_origin.issueId
    const origin = opsStore.getRunOrigin(runId);
    const issueId = origin?.issueId;
    if (!issueId) return; // conversation-driven，交给 conversation 的 onRunComplete 处理 @mention

    if (status !== "succeeded") {
      // §3.5 / e2e 失败模式：run 没成功 → 不推进，Issue 停在原状态（可观测：查这次 run）
      console.warn(`[orchestrator] run ${runId} for issue ${issueId} ended ${status}; not advancing`);
      return;
    }

    const issue = issueSvc.port.getIssue(issueId);
    if (!issue) return; // Issue 已删，无可推进

    const t = nextTransition(TRANSITIONS, issue.status);
    if (!t) return; // 当前已是终态（done）

    // §3.2 单一写者：推进经 applyTransition（含 CAS，"一次终态推进一格"）
    let advanced: IssueRow;
    try {
      advanced = issueSvc.applyTransition(issueId, t.to);
    } catch (err) {
      // 非法转移 / 丢 CAS（终态被重复消费）→ 幂等吞掉，记日志
      console.warn(`[orchestrator] applyTransition skipped for ${issueId}: ${String(err)}`);
      return;
    }

    // 起下一棒（若 advanced 仍有后继转移）
    try {
      await startStep(advanced);
    } catch (err) {
      // agent 缺失等：Issue 已推进到 advanced.status 但起不了下一棒 → 停在该状态，可观测
      console.error(`[orchestrator] failed to start next step for ${issueId}: ${String(err)}`);
    }
  }

  return { startStep, onRunComplete };
```

**幂等说明**：CAS 保证即使同一 run 终态被重复投递（reaper 重入、重启），`applyTransition` 第二次会因 `expectFrom` 不匹配丢 CAS 抛错被吞掉，状态不会被推两格。配合 P3 的 `idempotencyKey`（同状态只起一棒），满足 e2e-issue-lifecycle 失败模式"同一状态被起两次"的防护。

**注意终态推进与起下一棒的次序**：先 `applyTransition`（事实落库）再 `startStep`（起下一棒）。若 `startStep` 失败，Issue 已停在 `advanced.status`（如 in_review），人可介入修 agent 配置后手工或重试推进——状态不丢、可恢复。

**影响文件**：`orchestrator/reactor.ts`。

## 7. 验收标准

1. 建一个 Issue（planned）→ 自动起 planner run → run succeeded → Issue 自动进 in_progress 并起 developer run → … → done 停止，全程无人工干预。
2. 任一棒 run 失败（error/aborted）→ Issue 停在当前状态，不前进；日志可见原因。
3. 转移表引用一个不存在/已 archived 的 agentId → 起该棒时抛 `OrchestratorAgentMissingError`，Issue 停在上一个状态，不静默卡死。
4. 同一 run 终态被重复投递 → Issue 状态只推进一格（CAS 幂等）。
5. conversation-driven 的普通对话 run 终态 → @mention 自动触发照常工作，orchestrator 不介入；issue-driven run 终态 → 只走 orchestrator，不触发 @mention。
6. M18.1 的 `ISSUE_STATUSES` / `LEGAL_TRANSITIONS` / `applyTransition` / 看板 / `/api/issue-meta` 行为不回归（转移表迁移无副作用）。
7. `bun test` 全绿；`turbo run lint typecheck` 通过。

## 8. 风险与权衡

- **转移表迁移的依赖方向**：issue↔orchestrator 双向引用风险，靠"`IssueStatus` 纯类型留在 issue、转移表与运行时逻辑在 orchestrator、issue 经函数注入回调用 orchestrator"消解，无运行时环（§6 P1/P5）。
- **跨库读取**（Issue 在 backend.db、run_origin 在 events.db）：回填器一次终态读两库，是现状常态（ops service 已如此），无新模式；性能上每次终态多一次 run_origin 点查，可忽略。
- **best-effort 起第一棒**：创建成功但起第一棒失败时，Issue 停在 planned，需人工/重试推进——可观测（日志）、可恢复（重新触发），不破坏数据。
- **idempotencyKey 复用 run_origin 唯一索引**：`issue:<id>:<status>:run` 保证同状态只起一棒；但若一个 Issue 合法地回到同一状态（M18 线性表不会，未来可配置 DAG 可能），该 key 会误挡——属未来工作的已知边界，M18.2 线性表下安全。
- **conversation_id 空串**：issue-driven run_origin 的 `conversation_id` 写空串（该 run 不归属 conversation）。需确认 ops 查询/展示对空 conversation_id 容忍（M18.2 测试覆盖）。
