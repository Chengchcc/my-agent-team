## Actual vs Spec

### Matched

- 4 个新 tool 全部常量导出，无 `cwd` 工厂参数（Q3=D 决议落地）
- `editTool`: 0 匹配 → isError，多匹配 → isError（含匹配次数），文件不存在 → isError，不抛
- `bashTool`: `Bun.spawn(["bash", "-c", command])`，默认 30s，上限 600s，非 0 退出码 → isError，stdout+stderr 同体
- `grepTool`: 系统 `rg -n --color=never`，rg 退出码 0=匹配/1=无匹配/2+=错误，glob 过滤
- `globTool`: `Bun.Glob`，`onlyFiles: true`，上限 500 条，无匹配 `(no matches)`
- harness 包名 `@my-agent-team/harness`（Q2=A），路径吃掉现存空心 `packages/harness/`
- 6 个默认 tool（read/write/edit/bash/grep/glob，Q9=A），web_fetch/web_search 不默认装（Q10=A）
- 2 个默认 plugin：fs-memory `dir=workspace`（Q11=A）+ progressive-skill `dir=workspace/skills`（Q12=A）
- bootstrap 并行读 6 文件 `Promise.all`（Q14=A）
- systemPrompt 5 段 XML 短名 `<soul>/<user>/<tools>/<agents>/<recent-work>`（Q8=A），空段保留外壳
- workspace 全空 → 硬编码兜底 prompt，含 workspace 路径（Q7=A）
- daily-log 读 today + yesterday，`isoDate` 同 `YYYY-MM-DD`（Q6）
- workspace 不存在不 auto-mkdir
- 不读 `process.env`，不假设 `process.cwd()`
- 不解析 AGENTS.md 为结构化配置
- tool 错误全部 `isError: true`（不抛），与 M1 纪律一致
- `bash` 不做命令黑名单
- `edit` 0/多匹配返回 isError

### Differences

1. **`permissionMode` 未传入 framework**。Spec 说 "透传 framework，由 framework 既有 permissionMode 机制兜底"，但实际上 framework `AgentConfig` 不含 `permissionMode` 字段（`grep -r permissionMode packages/` 零匹配）。实现中 `createGenericAgent` 接受该参数，取值 `_permissionMode`，不作任何传递——permission 机制完整落地需 M8。**这是 spec 架构假设与代码真实状态之间的 gap，不是实现偏差。**

2. **config-tool 间重名检测补充**。Spec 说 "复用 M5 validatePlugins" 处理重名 fail-fast。实测 `validatePlugins` 只检查 config-tools vs plugin-tools 的冲突，不检查 config-tools 内部的重复。补充了 `checkDuplicateNames` 函数：先用 `Set` 收集默认 tool/plugin 名，再逐个检查 extra 是否碰撞。**这是 M5 `validatePlugins` 的覆盖盲区，不是 spec 错误——spec 对 `validatePlugins` 的覆盖范围有误判。**

3. **`grepTool` ENOENT 兜底**。`Bun.spawn` 在可执行文件不存在时抛 `ENOENT` 异常而非返回 exit code，spec 描述的 "rg 退出码 2+ = isError" 路径不会被执行到。实现中 `let proc = Bun.spawn(...)` 外包了 try/catch，捕获 spawn 失败并返回安装指引。

4. **`globTool` 截断消息微调**。Spec 写 `(truncated at 500)`，实现为 `... (truncated at 500)`。前导 `...` 暗示前面有内容被切断——与 Claude Code 原生 tool 的截断风格一致。

5. **CLI 泛用 harness 循环缺少 try/catch**。旧 REPL `runLegacyRepl` 有 `try { for await (msg) {} } catch (err) { console.error }` 包裹每次交互，让用户在工具异常后继续对话。新 `runGenericHarnessLoop` **没有等效保护**——`agent.run()` 如果抛异常，整个进程 crash。这是代码 review 发现的回归 bug。

6. **`createGenericAgent` 测试：弱断言**。6 个测试中有 3 个只用 `expect(agent.thread.messages).toBeDefined()` 验证 agent 创建成功，但没有验证实际装配的工具名、plugin 名、systemPrompt 内容。Spec 要求 "默认装 6 tools" 和 "默认装 2 plugins" 的验收，实际上靠的是代码常量（inline 的 `defaultTools` 数组）而非测试证据。

## 深入 Review 发现的问题

### P1: CLI 泛用 harness 循环无异常保护

```ts
// runGenericHarnessLoop — M6 新增（无 try/catch）
while (true) {
  const line = await rl.question("> ");
  if (line === "") continue;
  for await (const event of agent.run(line)) {  // ← 抛异常 = 进程 crash
    renderEvent(event);
  }
}

// runLegacyRepl — M2 既存（有 try/catch）
while (true) {
  const line = await rl.question("> ");
  try {
    for await (const msg of run(model, tools, messages, ...)) { ... }
  } catch (err) { console.error(...) }  // ← 优雅降级，对话继续
}
```

**影响**：tool 抛异常（如 `Bun.spawn` spawn 失败）时，旧 REPL 打印错误后继续等待下一条输入；新 harness 模式 crash 整个进程。**这是一个行为回归。**

**修复建议**：在 `for await` 外包 try/catch，与旧 REPL 一致。

### P2: `bashTool` killed 进程显示 `exit=null`

```ts
proc.kill()  // sends SIGTERM → proc.exited resolves to null in Bun
// body = `exit=null\n--- stdout ---\n...`
```

Bun 的 `Subprocess.exited` 在进程被信号杀死时返回 `null`（不是信号编号）。模板字符串渲染 `exit=null`，对 LLM 来说这个信息不如 `exit=signal` 或 `exit=killed` 有用。

**修复建议**：`const code = exitCode === null ? 'killed' : exitCode`

### P3: `createGenericAgent` 测试弱断言（4/6）

已在上节详细论述。这 4 个弱测试是 TDD 边界上的灰色地带——测试确实先于实现写了，但断言强度不足以验证 spec 语义（"默认装 6 tool"、"默认装 2 plugin"）。

**根本原因**：framework 的 `Agent` 接口不暴露 `tools` / `plugins` / `systemPrompt` 字段（只有 `thread: Thread`），agent 创建后无法从外部检查其内部配置。测试只能通过行为（跑一轮对话看结果）间接验证，或者 mock `createAgent`。

**修复建议**：M7 考虑 `AgentConfig` 加 `readonly config: Readonly<AgentConfig>` 暴露原配置，使测试可以直接断言 tool/plugin 列表。

### P4: `permissionMode` 被无声忽略

```ts
const { permissionMode: _permissionMode = "ask", ... } = opts;
// _permissionMode 在后续代码中从未被引用
```

Spec 要求 "透传 framework"，但 framework 无此字段。实现选择接收但忽略，不加任何日志或存储。这意味着调用方传 `permissionMode: "deny"` 得到的行为与 `"auto"` 完全相同——不会报错、不会警告、不会生效。**静默失效是最糟糕的 API 行为。**

**修复建议**：至少 `logger.warn("permissionMode not yet enforced; will be supported in M8")` 告知调用方此参数当前无效。或者在 M8 落地前暂时不暴露此参数。

### P5: `bootstrap.test.ts` 复制了 `isoDate` 逻辑

测试文件定义了：
```ts
function isoStr(d: Date): string {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, "0");
  const day = String(d.getDate()).padStart(2, "0");
  return `${y}-${m}-${day}`;
}
```

这与 `daily-log.ts` 的 `isoDate` 完全重复。应该直接从模块导入而非复制。**违反了"单一真相源"原则，未来改日期格式会漏改测试。**

### P6: harness 包无 README

Spec 要求 "包 README 写明 `rg` 是系统依赖"。当前 harness 包没有 README.md。同时 `grepTool` 的 description 写了依赖提示，但开发者安装 `@my-agent-team/harness` 时不会看到 tool description——只有 LLM 调用 tool 时才看到。**人类开发者缺少系统依赖文档。**

### P7: `globTool` 不处理 Bun.Glob 构造异常

`new Bun.Glob(pattern)` 在非法 pattern 时抛异常（如 unmatched `[`）。当前代码不捕获，依赖 framework 的 catch-all 转 `isError: true` 的 tool_result。行为正确但错误信息不友好（LLM 看到的是原始异常 message，不如 tool 自己格式化）。

这不紧急——LLM 通常传合法 glob pattern——但值得在 tool description 里加一个合法 pattern 的例子。

## Lessons

1. **"Framework 既有 X" 必须 grep 确认，不能信 spec**。`permissionMode` 在 spec 出现 5+ 次，全部假设 framework 已支持。`grep -r permissionMode packages/` 返回零匹配只用 2 秒，不做就等于戴着 spec 的滤镜写代码。教训：**spec 里的架构假设是待验证的，不是已证实的。**

2. **`validatePlugins` 命名暗示全量校验，实际只覆盖跨层冲突**。M5 的 validatePlugins 不检查 config-tools 内部重复，因为当时没有两个来源同时往 config.tools 塞 tool 的场景。M6 引入了 "defaults + extras" 两个 config-tool 来源，才暴露这个盲区。教训：**新使用模式会暴露既有 API 的覆盖盲区——即使 API 在原场景下是正确的。**

3. **CLI 的新旧两条路径让 regression risk 翻倍**。M2 REPL 的 try/catch 模式在 M6 新增路径中没有被保留。这不是疏忽——是两条代码路径没有抽象出共享的异常保护层。教训：**加 flag 分支时，先抽公共逻辑（error handling loop），再在两个分支里复用，而不是复制粘贴旧逻辑。**

4. **TDD 纪律有进步，但断言强度是新的弱点**。M5 两次松动是"先写实现后补测试"（过程级违规）。M6 严格遵守了 RED→GREEN 顺序，但遇到了新问题：**GREEN 只能验证"不抛"，不能验证"装配正确"**。这是 framework `Agent` 接口不暴露内部配置导致的测试可观测性瓶颈。教训：**TDD 的成功不只是"测试先写"，还取决于被测接口的可观测性——如果接口不暴露关键信息，测试就只能是烟雾测试。**

5. **`Bun.spawn` 的类型间隙值得上游修复**。`proc.stdout` 的类型是 `number | ReadableStream<Uint8Array> | undefined`，但当传入 `stdout: "pipe"` 时，运行时永远是 `ReadableStream<Uint8Array>`。TypeScript 无法根据字面量参数收窄返回类型，导致到处都是 `as ReadableStream<Uint8Array>`。这不是设计问题，是 bun-types 的类型签名精度不够。

6. **14 条 grill 决议 = 高质量的 spec**。M6 的 spec 比 M4/M5 更成熟——14 条 grill 决议覆盖了包命名（Q2）、工具工厂 vs 常量（Q3）、runtime 选择（Q4）、系统依赖策略（Q5）、归档天数（Q6）、兜底行为（Q7）、标签命名（Q8）、默认装配列表（Q9–Q13）、并行策略（Q14）。**决议密度高意味着实现时的决策成本低**——每个 Q 都是"已经有人替你想过了"的信号。只有 Q3 经历了撤销（原草案讨论过工厂方案，最终决议回归常量），说明 grill 本身就包含了"否决不合理方案"的价值。

7. **Harness 是 thin assembly layer——代码量证实了架构判断**。架构文档把 harness 定义为"有观点的产品层：内置 tools + system prompt + 权限策略"。M6 实现下来生产代码 ~360 LOC，远低于 M4 framework（986 LOC）和 M5 plugin（~650 LOC）。harness 不做新协议、不做新算法、不引入外部依赖——它只是把 framework + plugin + tool 按约定路径装配起来。**这个代码量恰好验证了"assembly layer"的定位是准确的。**
