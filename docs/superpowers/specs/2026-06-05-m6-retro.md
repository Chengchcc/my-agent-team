## Actual vs Spec

### Matched

- 4 个新 tool 全部常量导出，无 `cwd` 工厂参数（Q3=D 决议落地）
- `editTool`: 0 匹配 → isError，多匹配 → isError（含匹配次数），文件不存在 → isError，不抛
- `bashTool`: `Bun.spawn(["setsid", "bash", "-c", command], { cwd })`，默认 30s，上限 600s，非 0 退出码 → isError，stdout+stderr 同体，exit=null → "killed"
- `grepTool`: 系统 `rg -n --color=never`，rg 退出码 0=匹配/1=无匹配/2+=错误（含 exit code），glob 过滤
- `globTool`: `Bun.Glob`，`onlyFiles: true`，上限 500 条，显式 `truncated` flag（非 count-based 判据），无匹配 `(no matches)`
- harness 包名 `@my-agent-team/harness`（Q2=A），路径吃掉现存空心 `packages/harness/`
- 6 个默认 tool（read/write/edit/bash/grep/glob，Q9=A），web_fetch/web_search 不默认装（Q10=A）
- 2 个默认 plugin：fs-memory `dir=workspace`（Q11=A）+ progressive-skill `dir=workspace/skills`（Q12=A）
- bootstrap 并行读 6 文件 `Promise.all`（Q14=A）
- systemPrompt 6 段 XML 标签（Q8=A + N13），空段保留外壳
- workspace 全空（或纯 whitespace）→ 硬编码兜底 prompt，含 workspace 路径（Q7=A + N8）
- daily-log 读 today + yesterday，`isoDate` 同 `YYYY-MM-DD`（Q6），单次 `Date.now()` 防午夜 race（N7）
- workspace 不存在不 auto-mkdir
- 不读 `process.env`，不假设 `process.cwd()`
- 不解析 AGENTS.md 为结构化配置
- tool 错误全部 `isError: true`（不抛），与 M1 纪律一致
- `bash` 不做命令黑名单
- `edit` 0/多匹配返回 isError；大文件安全版 `indexOf` 替代 `split`（N4）

### Differences（含 spec 偏差 + 实现后修正）

1. **`permissionMode` 未传入 framework**。Spec 假设 framework "既有 permissionMode 机制"，但 `AgentConfig` 无此字段。实现接受该参数但暂不传递——permission 完整落地需 M8。（P4，未修——需 framework 侧配合）

2. **config-tool 间重名检测补充**。Spec 说"复用 M5 validatePlugins"，但 `validatePlugins` 只检查 config vs plugin 冲突。补充了 `checkDuplicateNames` 函数处理 config-tool 内部 + plugin 同名重复。

3. **`grepTool` ENOENT 兜底**。`Bun.spawn` 在 rg 不存在时抛 `ENOENT` 而非返回 exit code。加了 try/catch 兜底；exit≥2 的错误消息也改为含 exit code 的格式（N3）。

4. **`<workspace>` 段新增（N13）**。spec 的 systemPrompt 只有 5 段（soul/user/tools/agents/recent-work），实现中新增了第 0 段 `<workspace>`——告诉 LLM 根目录路径、文件路径解析基准、bash 默认 cwd、当天日期。架构文档 §三 同步更新为 6 段。

5. **bashTool `cwd` 字段（N14）**。架构文档 §四 "bash cwd = workspace" 最初实现遗漏——bashTool 不设 cwd，也不暴露 cwd 参数。修正：bashTool inputSchema 新增可选 `cwd` 字段，systemPrompt `<workspace>` 指引 LLM 默认传 `cwd='${workspace}'`。**不做 harness 层 wrap**——LLM 可以按用户要求覆盖 cwd（如用户说"去 /tmp 跑"）。

6. **CLI 泛用 harness 循环缺 try/catch（P1）**。已修复——`runGenericHarnessLoop` 现在与 `runLegacyRepl` 一致的 try/catch 包裹。

7. **`createGenericAgent` 测试弱断言（P3/N9）**。原 "default-assembles 6 file tools" 测试注释声称 10 tools 但断言只有 `toBeDefined()`。已重写为验证 systemPrompt 注入的实质性断言。

## Lessons（增补 N13/N14 后的洞察）

1–7 同前。新增：

8. **架构契约需要实现侧主动 verify，不能靠"spec 写了就等于做了"**。N14（bash cwd=workspace）是架构文档 §四 白纸黑字的契约，但实现完全遗漏——bashTool 不设 cwd、不暴露 cwd 参数、harness 不注入。这个 gap 在自 review 和 retro 中都没被发现，直到独立 review 逐条对照架构文档才暴露。教训：**实现完成后逐条对着架构文档跑一遍 checklist，比再跑一遍测试更有价值。**

9. **Harness 层的 "wrap vs prompt" 决策边界**。N14 修复的第一版用了 harness 层 wrap（强制注入 cwd），被指正后改为 LLM 自主传递。这个演变揭示了一条设计原则：**harness 通过 systemPrompt 引导默认行为，而非通过代码强制执行**。强制执行锁死了用户覆盖的路径（如去 `/tmp` 跑命令）；prompt 引导保留了灵活性，信任 LLM + permissionMode 做正确的事。这条原则应该写进 harness 架构文档。

10. **N13 和 N14 是同一个根因的两个症状**。根因：LLM 不知道自己在哪个目录工作。N13（不知道 workspace 路径）和 N14（bash 不在 workspace 执行）都是这个信息断层的表现。修复也是互补的：`<workspace>` 段告诉 LLM "你的根目录是 X"；bashTool 的 `cwd` 字段给了 LLM 把这条信息转化为正确行为的工具。
