# M18.1 Spec — Issue Entity & Storage：Issue 本体、单一状态命令与看板基础渲染

> **Status:** ✅ Complete (final commit `b08d515`)。
> **Baseline commit:** `origin/master` @ `b08d515`。
> **关联:** `docs/architecture/foundations/issue.md` · `docs/architecture/backend/orchestrator.md` · `docs/architecture/flows/e2e-issue-lifecycle.md` · `docs/architecture/backend/data-model.md` · `docs/architecture/design-philosophy.md` · `apps/backend/src/features/issue/`（新增）· `apps/backend/src/infra/sqlite/migrations.ts` · `apps/backend/src/http/router.ts` · `apps/backend/src/main.ts` · `apps/web/src/app/(main)/issues/`（新增）· `apps/web/src/lib/api.ts` · `apps/web/src/components/NavRail.tsx`。

M18 要把 [Issue](../../architecture/foundations/issue.md) + [Orchestrator](../../architecture/backend/orchestrator.md) 从 `status: design` 落地为后端真实能力 + 前端可操作的看板。M18.1 是这条链路的**地基**：先把 Issue 这个新领域对象、它唯一的状态写入命令、以及一个只读看板渲染出来——但**不引入任何自动推进**（那是 M18.2 Orchestrator 的事）。

落地后，`foundations/issue.md` 中"Issue 数据模型""Kanban 是视图"两节从 `status: design` 翻成 `status: current`，并补 `last_verified_against_code`。

本子里程碑**刻意不做**：状态自动推进（M18.2）、拖拽交互与乐观更新（M18.3）、Project git 子目录绑定（M18.4）。M18.1 只保证"Issue 能被建出来、停在某个状态、在看板上按状态分列显示"。

## 1. 背景

现状里，多 Agent 协作靠 **@提及自动触发**：一个 Agent 的产出文本被扫描，命中 `@某人` 就起下一棒（见 [conversation-and-members](../../architecture/conversation/conversation-and-members.md)）。这条链路隐式、不可观测、难约束——"一件活推进到第几步"没有任何显式载体。

[foundations/issue.md](../../architecture/foundations/issue.md) 已经论证：Conversation / Run / Message 都无法表达"一件可跨多次运行、按状态推进的活"，所以 **Issue 是 M18 唯一新增的领域本体**。Kanban 与 Project 都不是新本体——Kanban 是 Issue 按 status 分组的视图，Project 是 Issue 的归属（git 子目录 + 元数据）。

本设计采用线性状态机：`planned → in_progress → in_review → done`，确立了三条会贯穿整个 M18 的关键事实——拖拽与按钮触发同一个状态命令、状态推进带副作用且失败要回滚、终态由外部信号驱动。M18.1 先把"同一个状态命令"这个抽象立住，副作用与回滚留给 M18.2。

## 2. 当前代码事实

基于 M17.5 完成态，逐项已与代码核对：

1. **feature 按需选用文件，非统一骨架**：各 feature 根据存储需求选用不同模式——
   - `conversation`：Row 类型放在 **`ports.ts`**（无 `entities.ts`），有 `adapter-sqlite.ts` 做持久化。
   - `run`：有 `entities.ts`，但**无 `ports.ts`/`adapter-sqlite.ts`**——运行事实由 `RunSupervisor` 写入 `events.db`，不走 Port/adapter。
   - `runtime-ops`：类型在 **`types.ts`**，存储在 `store.ts`。
   - **Issue 需要自己的关系型存储**（建/读/列 + CAS 改状态），因此采用 **conversation 的 adapter 模式**：`entities.ts`(Row) + `ports.ts`(Port) + `adapter-sqlite.ts`(实现) + `service.ts` + `http.ts` + `index.ts`。这是"按存储需求选模式"，不是"套用全仓统一骨架"。
2. **service 是工厂函数，不是 class**：`export function createRunService(deps)`（`run/service.ts:52`）、`export function createConversationService(deps)`（`conversation/service.ts:66`）。deps 里注入 `idGen: () => string`（运行期传 `ulid`）。**只有 Error 子类是 `class`**（`ThreadBusyError extends Error`）。service 末尾 `export type XxxService = ReturnType<typeof createXxxService>`。**conversation service 在返回对象里暴露 `port` 给 HTTP 层**（thin-adapter，service.ts:198）；run service **不暴露 port**（其存储不走 Port）。Issue 因走 adapter 模式，**照 conversation 暴露 `port`**。
3. **建表集中在迁移文件，adapter 不建表**：所有 DDL 写在 `apps/backend/src/infra/sqlite/migrations.ts` 的 `BACKEND_MIGRATIONS` 数组，由 `openDb()`（`infra/sqlite/db.ts`）按 `id` 升序执行并落 `_migrations` 去重表。命名规约 `backend_v<seq>_<slug>`，`id` 严格递增分段（1–999 core，4000–4999 conversation/member/ledger，5000–5999 alteration）。当前最大条目是 `id: 5008` / `backend_v22_projection_messages`。`adapter-sqlite.ts` 只做 CRUD（`sqliteConversationAdapter(db): ConversationPort` 内全是 `db.run/db.query`，**无任何 `CREATE TABLE`**）。
4. **adapter 测试用真库**：`conversation/adapter-sqlite.test.ts` 直接 `openDb('/tmp/test-...db')` 拿到一个已跑完迁移的 Database，再 `sqliteConversationAdapter(db)`。所以**新表必须先在迁移里注册，测试才建得出来**。
5. **后端存储分库**：`backend.db` 持有 agents / conversation / member / conversation_ledger / projection_messages；`events.db` 持有 run / attempt / event_log 等运行事实（见 [data-model](../../architecture/backend/data-model.md)）。`main.ts` 里 `const db = openDb(`${config.dataDir}/backend.db`)`。
6. **HTTP 汇聚在 `http/router.ts`**：`createRouter(token, features)`，`features` 是 `FeatureSet` interface（`router.ts:10`，字段如 `agents/runs/threadProjections/conversations?/ops?`）。路由用正则匹配 path + `method` 判断，每条经 `withAuth(handler, token)` 包裹；顶层 `try/catch` 把 `HttpError` 映射状态码。feature `http.ts` 内部用 `z.object` 校验 body、`json(body, status)` 构造响应、`catch (err) { if (err instanceof XxxError) return json(..., 4xx) }` 映射自定义错误（`run/http.ts:36` `ThreadBusyError → 409`、`RunNotFoundError → 404`、`TooManyRunsError → 429`）。
7. **feature 在 `main.ts` 装配**：`adapter → createXxxService(deps) → xxxRoutes(svc, ...)`，再塞进 `createRouter(config.authToken, { agents, runs, threadProjections, conversations, ops })`。共享 ID 生成器 `ulid`（`infra/ids.ts`）。共享响应助手 `json / parseJsonBody / sseResponse`（`http/response.ts`）。
8. **Issue 绑定 `threadId`**：thread 是已存在的运行载体（`run` feature 以 threadId 为运行单元，`deriveThreadId(convId, memberId)` / `parseThreadId`），Issue 复用它即可，无需新发明执行机制。
9. **前端 BFF 是单一兜底代理**：`apps/web/src/app/api/bff/[...path]/route.ts` 一个 catch-all，导出 `GET/POST/PATCH/DELETE = handler`，经 `proxyRequest(req, path, userId)`（`lib/bff.ts`）把 `/api/bff/<path>` 透传到 `${BACKEND_URL}/api/<path>`。**没有任何 per-resource 的 BFF 路由文件**，新增后端端点前端无需加 BFF 文件。
10. **前端走 typed api 层**：`apps/web/src/lib/api.ts` 导出 `api` 对象（`listAgents/createAgent/listOpsRuns/...`），底层 `apiFetch(path)` 自动加 `/api/bff/` 前缀、带 cookie、401 跳登录、抛 `ApiError(status, msg)`。页面用 `@tanstack/react-query` 的 `useQuery({ queryKey, queryFn: () => api.xxx() })`（参考 `(main)/ops/runs/page.tsx`，`"use client"` + `export const dynamic = "force-dynamic"`）。
11. **`(main)` 路由组**：现有 `agents` / `conversations` / `ops` 三个，加 `issues` 与之并列。
12. **导航是硬编码**：`components/NavRail.tsx` 里 Workspace / Conversations / Operations 三段写死，Operations 用常量 `OPS_LINKS`。新加看板入口要改这里（折叠态、展开态各一处）。
13. **TS target = ES2023**：`tsconfig.base.json:20` 与 `apps/web/tsconfig.json:3` 都是 `"target": "ES2023"`。**`Object.groupBy` 是 ES2024，类型层不可用**——分组必须手写 `reduce` / `for` 循环 / `Map`。
14. **状态徽标**：`components/RunStatusBadge.tsx` 用 `Badge`（`ui/badge.tsx`，variant `default|secondary|destructive|outline`）做一个 `Record<status, {label, variant}>` 映射，Issue 卡片照此风格做一个 `IssueStatusBadge`。

## 3. 第一性原则

### 3.1 Issue 是领域本体，必须有独立存储与生命周期

按[设计哲学](../../architecture/design-philosophy.md)"每个新概念必须说明为什么已有概念不能表达"：Conversation 是"谁和谁在对话"、Run 是"一次执行"、Message 是"一条消息"，三者都无法表达"一件活的多运行状态推进"。Issue 通过了这道闸——它有自己的身份、状态机生命周期、单一 terminal（done）。因此它配有独立的表、独立的 service、独立的 status。

### 3.2 status 单一写者：所有状态变更经唯一命令 `applyTransition`

拖拽卡片和点按钮触发"效果一致"——它们不是各写各的状态，而是触发同一个状态变更。落到我们的架构上，这意味着 **status 只有一个物理写入入口 `applyTransition(issueId, to)`**：

- M18.1：人工调用（建 Issue 后的显式状态变更、未来的拖拽）走它。
- M18.2：Orchestrator 的回填监听器也走它（run 终态 → `applyTransition`）。

这正是设计哲学 §3.7"一条事实一个权威物理入口"在 Issue 上的应用，也避免了"机器一个写法、人一个写法"的双写者漂移。Port 上**不暴露**通用 `updateIssue`——CRUD 只允许建/读/列，status 只能经 `setStatus`，而 `setStatus` 只被 `service.applyTransition` 调用，从存储层就堵死旁路。

### 3.3 状态空间是单一事实来源，看板列从它派生

看板有哪几列，本质是 Issue 状态空间（`planned/in_progress/in_review/done`）的投影。状态空间的唯一来源是状态机定义本身（M18.2 起由 Orchestrator 的转移表锚定；M18.1 阶段先由 Issue 模块导出一份状态元数据）。

**前端绝不硬编码列集合**——否则 M19 转移表可配置后，前端要跟着改，这就是设计哲学警惕的"模块边界冒充领域边界"。前端拿到后端声明的有序状态列表，渲染出对应的列。

### 3.4 看板是视图，BFF 是透传，分组是 trivial 渲染

按设计哲学 §6"投影是手段，不是主线"：看板 = `listIssues()` 拉平结果按 `status` 分组。这个分组是纯客户端渲染（**ES2023 下手写 reduce/Map，不能用 `Object.groupBy`**），**既不在后端新增分组端点，也不在 BFF 里做业务派生**——事实来源在后端一处，surface 只渲染。前端 BFF 已经是单一兜底代理（§2.9），Issue 接口零新增 BFF 文件即被透传。

### 3.5 转移合法性由状态机校验，非法变更拒绝

M18 转移表是固定线性的。`applyTransition(issueId, to)` 必须校验 `to` 相对当前 status 是否"可达"——M18.1 阶段先用一份与转移表同源的"合法 from→to 集合"校验，非法转移返回 409 并不改库。这保证看板拖拽（M18.3）天然只能在合法相邻状态间移动，状态空间不被旁路撑破。

1. 新增 `apps/backend/src/features/issue/`，按现有 feature 骨架（entities/ports/adapter/service/http/index）落 Issue。

- **拖拽交互、乐观更新、回弹、SSE 实时增量**：M18.3。注意 M18.1 **仍交付一个看板前端**（见 §4 目标⑥、Patch P5）——按状态分列、卡片渲染、建 Issue 表单都做；只是这块看板是**只读 + 刷新拉取**的，不含拖拽改状态与实时增量推送。换言之，"不做前端"是误读：M18.1 不做的是**拖拽/SSE 这层交互**，不是整个看板页面。

### Patch P1 — Issue 实体与状态空间（转移表为唯一源）

**目标**：定义 Issue 领域对象，并把状态空间收敛到**单一事实来源**——一张转移表，状态集合与合法转移都从它派生，杜绝"两份常量手维护一致"的漂移与扩展隐患。

`apps/backend/src/features/issue/entities.ts`：

```ts
/** Issue — a work unit with its own state-machine lifecycle, spanning
 *  multiple runs on a single bound thread. The only new domain ontology in M18. */
export interface IssueRow {
  issueId: string;
  projectId: string;
  title: string;
  status: IssueStatus;
  /** The thread this Issue runs on. Reuses the existing run载体, no new exec mechanism. */
  threadId: string;
  createdAt: number;
  updatedAt: number;
}

export type IssueStatus = "planned" | "in_progress" | "in_review" | "done";

/** ── 单一事实来源 ──────────────────────────────────────────
 *  状态空间由这张转移表唯一定义。M18.1 它是固定常量（对应 Orchestrator
 *  「固定线性转移表」设计）；M18.2 起整张表移交 orchestrator 模块，M19 它从
 *  `const` 长成「从配置/DB 读」——届时下面两个派生函数完全不用改。 */
const TRANSITIONS = [
  { from: "planned", to: "in_progress" },
  { from: "in_progress", to: "in_review" },
  { from: "in_review", to: "done" },
] as const satisfies ReadonlyArray<{ from: IssueStatus; to: IssueStatus }>;

/** 从转移表派生有序状态集合（= 看板列顺序）。不再手写第二份常量。
 *  **不变量：`TRANSITIONS` 必须按生命周期拓扑顺序书写。**
 *  本函数取"首次出现顺序"(Set 插入序)，所以看板列顺序 = 转移表书写顺序。
 *  新增转移（如 `planned→cancelled`）要插在保证拓扑序的位置，否则列序错乱。
 *  M19 若改为从配置/DB 读，需在数据源侧保证同一顺序约束。 */
function deriveStatuses(table: typeof TRANSITIONS): IssueStatus[] {
  const seen = new Set<IssueStatus>();
  for (const t of table) {
    seen.add(t.from);
    seen.add(t.to);
  }
  return [...seen];
}

/** 从转移表派生 from → 合法 to 列表的映射。 */
function deriveLegalMap(table: typeof TRANSITIONS): Record<IssueStatus, IssueStatus[]> {
  const map = Object.fromEntries(deriveStatuses(table).map((s) => [s, [] as IssueStatus[]]));
  for (const t of table) map[t.from]!.push(t.to);
  return map as Record<IssueStatus, IssueStatus[]>;
}

export const ISSUE_STATUSES: readonly IssueStatus[] = deriveStatuses(TRANSITIONS);
export const LEGAL_TRANSITIONS: Readonly<Record<IssueStatus, IssueStatus[]>> =
  deriveLegalMap(TRANSITIONS);
```

**修复思路**：唯一要人维护的是 `TRANSITIONS`；`ISSUE_STATUSES`（前端列）与 `LEGAL_TRANSITIONS`（转移校验）都是它的纯函数投影。加一个状态 = 在转移表加一条，集合与校验自动跟上，不会漏改。M19 转移表可配置后，**后端派生逻辑**(`deriveStatuses`/`deriveLegalMap`/`applyTransition`)零改动；但 `IssueStatus` union 类型与前端 `COLUMN_LABEL` 映射在**新增状态值**时仍需扩一处（label 缺失时 `?? s` 兜底为原始字符串，不崩但不美观）。扩展点收敛但非完全零改。

**影响文件**：`apps/backend/src/features/issue/entities.ts`（新增）。

export interface CreateIssueInput {
  issueId: string;
  projectId: string;
  title: string;
  threadId: string;
  createdAt: number;
}

export interface IssuePort {
  createIssue(input: CreateIssueInput): IssueRow; // 初始 status = "planned"

**P2-c adapter**（`apps/backend/src/features/issue/adapter-sqlite.ts`，**无 `CREATE TABLE`**，仿 `sqliteConversationAdapter` 的逐字段映射写法）：

```ts
import type { Database } from "bun:sqlite";
import type { IssueRow, IssueStatus } from "./entities.js";
import type { CreateIssueInput, IssuePort } from "./ports.js";

type Raw = {
  issue_id: string; project_id: string; title: string;
  status: IssueStatus; thread_id: string; created_at: number; updated_at: number;
};
const toRow = (r: Raw): IssueRow => ({
  issueId: r.issue_id, projectId: r.project_id, title: r.title,
  status: r.status, threadId: r.thread_id, createdAt: r.created_at, updatedAt: r.updated_at,
});

export function sqliteIssueAdapter(db: Database): IssuePort {
  return {
    createIssue(input: CreateIssueInput): IssueRow {
      db.run(
        `INSERT INTO issue (issue_id, project_id, title, status, thread_id, created_at, updated_at)
         VALUES (?, ?, ?, 'planned', ?, ?, ?)`,
        [input.issueId, input.projectId, input.title, input.threadId, input.createdAt, input.createdAt],
      );
      return { ...input, status: "planned", updatedAt: input.createdAt };
    },
    getIssue(issueId: string): IssueRow | null {
      const r = db.query("SELECT * FROM issue WHERE issue_id = ?").get(issueId) as Raw | undefined;
      return r ? toRow(r) : null;
    },
    listIssues(opts?: { projectId?: string }): IssueRow[] {
      const rows = (opts?.projectId
        ? db.query("SELECT * FROM issue WHERE project_id = ? ORDER BY created_at DESC").all(opts.projectId)
        : db.query("SELECT * FROM issue ORDER BY created_at DESC").all()) as Raw[];
      return rows.map(toRow);
    },
    setStatus(issueId: string, expectFrom: IssueStatus, to: IssueStatus, updatedAt: number): boolean {
      // CAS：WHERE 同时锁 issue_id 与旧 status，并发只有一个能命中
      db.run("UPDATE issue SET status = ?, updated_at = ? WHERE issue_id = ? AND status = ?",
        [to, updatedAt, issueId, expectFrom]);
      const { n } = db.query("SELECT changes() AS n").get() as { n: number };
      return n > 0;
    },
  };
}
```

> CAS 命中判定用 `SELECT changes()` 读受影响行数（bun:sqlite 写后即时有效）；落地时与 conversation adapter 写法对齐。要点是 `UPDATE ... WHERE issue_id=? AND status=?` 这条语句本身。

**修复思路**：Port 上**没有**通用 `updateIssue`——status 只能经 `setStatus`，从存储层保证单一写者（§3.2）。建 Issue 时 status 由 SQL 字面量固定为 `planned`，不接受调用方传入初始状态。`setStatus` 带 `expectFrom` 做 CAS，把"并发推进"在最底层一并解决（§8）。

**影响文件**：`apps/backend/src/infra/sqlite/migrations.ts`（改：追加迁移）、`apps/backend/src/features/issue/ports.ts`、`adapter-sqlite.ts`、`adapter-sqlite.test.ts`（新增；测试用 `openDb('/tmp/test-issue-...db')` 拿已迁移库，仿 `conversation/adapter-sqlite.test.ts`）。

### Patch P3 — Service 工厂：建 Issue 与 applyTransition 单一命令

**目标**：实现唯一状态命令 `applyTransition`，含合法性校验与错误语义。**用工厂函数 + 注入 `idGen`，与 `createRunService`/`createConversationService` 一致**（§2.2）。

`apps/backend/src/features/issue/service.ts`：

```ts
import { LEGAL_TRANSITIONS, type IssueRow, type IssueStatus } from "./entities.js";
import type { IssuePort } from "./ports.js";

export class IssueNotFoundError extends Error {
  constructor(id: string) { super(`Issue not found: ${id}`); this.name = "IssueNotFoundError"; }
}
export class IllegalTransitionError extends Error {
  constructor(msg: string) { super(`Illegal transition: ${msg}`); this.name = "IllegalTransitionError"; }
}

export interface IssueServiceDeps {
  port: IssuePort;
  idGen: () => string;      // 运行期传 ulid（infra/ids.ts）
  now?: () => number;       // 默认 Date.now（与 conversation/run 一致用 Date.now()）
}

export function createIssueService(deps: IssueServiceDeps) {
  const { port, idGen } = deps;
  const now = deps.now ?? Date.now;

  return {
    port, // 暴露 port 给 HTTP 层（thin adapter，对齐 conversation service:198）

    createIssue(input: { projectId: string; title: string; threadId: string }): IssueRow {
      return port.createIssue({
        issueId: idGen(),
        projectId: input.projectId,
        title: input.title,
        threadId: input.threadId,
        createdAt: now(),
      });
    },

    /** The single status writer. 人工(拖拽/按钮) 与 机器(M18.2 回填) 共用此入口。 */
    applyTransition(issueId: string, to: IssueStatus): IssueRow {
      const issue = port.getIssue(issueId);
      if (!issue) throw new IssueNotFoundError(issueId);
      const legal = LEGAL_TRANSITIONS[issue.status] ?? [];
      if (!legal.includes(to)) throw new IllegalTransitionError(`${issue.status} → ${to}`);
      const ts = now();
      const ok = port.setStatus(issueId, issue.status, to, ts); // CAS
      if (!ok) throw new IllegalTransitionError(`${issue.status} → ${to} (lost CAS)`);
      return { ...issue, status: to, updatedAt: ts };
    },
  };
}

export type IssueService = ReturnType<typeof createIssueService>;
```

**修复思路**：`applyTransition` 是 §3.2 的物理实现——人工与机器（M18.2）共用同一入口。M18.1 它只改 status；M18.2 会在此之上挂"进入某状态触发副作用、副作用失败回滚"的事务语义，但**入口签名不变**。校验失败抛 `IllegalTransitionError`，HTTP 层映射 409；CAS 失败（并发）同样落 409。

**影响文件**：`apps/backend/src/features/issue/service.ts`、`service.test.ts`（新增）。

### Patch P4 — HTTP 路由工厂、状态元数据端点、挂载与装配

**目标**：暴露建/查/列 Issue、applyTransition、状态空间元数据，并把路由真正挂上去。挂载要动 `router.ts`（FeatureSet + 路由块）**和** `main.ts`（装配）——这两处不能漏（§2.6、§2.7）。

**P4-a 路由工厂**（`apps/backend/src/features/issue/http.ts`，仿 `conversation/http.ts`。注意 `conversationRoutes(svc, idGen)` 带第二参 `idGen`，而 `issueRoutes(svc)` **不需要**——Issue 的 `idGen` 已在 `createIssueService` deps 注入，路由层不碰 ID 生成）：

```ts
import { z } from "zod";
import { json } from "../../http/response.js";
import { ISSUE_STATUSES, type IssueStatus } from "./entities.js";
import { IllegalTransitionError, IssueNotFoundError, type IssueService } from "./service.js";

const createSchema = z.object({
  projectId: z.string().min(1),
  title: z.string().min(1),
  threadId: z.string().min(1),
});
const transitionSchema = z.object({ to: z.enum(ISSUE_STATUSES as readonly [string, ...string[]]) });

export function issueRoutes(svc: IssueService) {
  return {
    /** POST /api/issues → 201 { issue } */
    async create(req: Request): Promise<Response> {
      const parsed = createSchema.safeParse(await req.json().catch(() => ({})));
      if (!parsed.success) return json({ error: "Validation failed", details: parsed.error.issues }, 400);
      return json({ issue: svc.createIssue(parsed.data) }, 201);
    },
    /** GET /api/issues?projectId= → 200 { issues } */
    list(req: Request): Response {
      const projectId = new URL(req.url).searchParams.get("projectId") ?? undefined;
      return json({ issues: svc.port.listIssues(projectId ? { projectId } : undefined) });
    },
    /** GET /api/issues/:id → 200 { issue } | 404 */
    get(_req: Request, issueId: string): Response {
      const issue = svc.port.getIssue(issueId);
      if (!issue) return json({ error: "Not found" }, 404);
      return json({ issue });
    },
    /** POST /api/issues/:id/transition { to } → 200 { issue } | 404 | 409 */
    async transition(req: Request, issueId: string): Promise<Response> {
      const parsed = transitionSchema.safeParse(await req.json().catch(() => ({})));
      if (!parsed.success) return json({ error: "Validation failed", details: parsed.error.issues }, 400);
      try {
        return json({ issue: svc.applyTransition(issueId, parsed.data.to as IssueStatus) });
      } catch (err) {
        if (err instanceof IssueNotFoundError) return json({ error: err.message }, 404);
        if (err instanceof IllegalTransitionError) return json({ error: err.message }, 409);
        throw err;
      }
    },
    /** GET /api/issue-meta → 200 { statuses } */
    meta(): Response {
      return json({ statuses: ISSUE_STATUSES });
    },
  };
}
```

## 7. 不变量

1. **status 单一写者**：所有 status 变更只经 `IssueService.applyTransition` → `IssuePort.setStatus`；CRUD 不含直接改 status 的路径，Port 上无通用 `updateIssue`。
2. **状态空间单一来源**：列集合、转移校验都从 `entities.ts` 的 `TRANSITIONS` 派生（`ISSUE_STATUSES` / `LEGAL_TRANSITIONS`）；前端列从 `/api/issue-meta` 来，不硬编码。
3. **转移合法性**：`applyTransition` 拒绝非 `LEGAL_TRANSITIONS` 允许的变更（409），不改库；并发竞争经 CAS 落 409。
4. **建表集中**：Issue 表 DDL 只在 `infra/sqlite/migrations.ts`，adapter 不含 `CREATE TABLE`。
5. **BFF 无业务逻辑且零新增文件**：Issue 经通用 `[...path]` 透传，分组等渲染逻辑在前端组件层。
6. **Issue 复用执行载体**：Issue 通过 `threadId` 绑定既有 thread，不发明新的运行机制。
7. **约定一致**：service 是 `createIssueService(deps)` 工厂 + 注入 `ulid`；Error 是 class；前端走 `lib/api.ts` typed 方法，不裸 fetch；分组手写（ES2023 无 `Object.groupBy`）。

## 8. 失败模式

- 建 Issue 时 `threadId` 不存在：M18.1 不校验 thread 存在性（解耦，thread 可后绑），仅存字符串；M18.2 起运行时若 thread 缺失由 RunSupervisor 报错。
- createIssue 遇 DB 约束冲突（ULID 碰撞极罕见、thread_id 约束等）→不捕获→router 兜底 500，与 conversation 一致。
- `applyTransition` 并发：两个请求同时推进同一 Issue，`setStatus` 用单条 `UPDATE ... WHERE issue_id=? AND status=?`（CAS）保证只有一个成功，另一个 CAS 不命中（`changes()=0`）→ service 抛 `IllegalTransitionError` → 409。幂等重复 transition（已在目标态）同样走 CAS 失手 → 409。
- 校验分两层：Zod 校 `to` 是否合法枚举值（非法 → 400），`LEGAL_TRANSITIONS` 校转移是否可达（不可达如 `planned→done` → 409）。两层防御，语义独立。
- 前端 `/api/issue-meta` 拉取失败：`IssueBoard` 收到空 `statuses` → 降级为平铺列表（不分列），避免把有效 issue 数据一起藏掉。
- 迁移 `id` / `name` 冲突：新迁移 `id` 必须严格大于现有最大值（当前 5008），选 5009 落 5000 段；`name` 必须全新（`_migrations` 用 `name` 去重，复用旧名会被跳过导致表建不出）。

## 9. 验收

1. `POST /api/issues`（经 `/api/bff/issues`）建出 Issue，停在 `planned`；`GET /api/issues` 看到它。
2. `POST /api/issues/:id/transition {to:"in_progress"}` 成功；`{to:"done"}`（跨列）返回 409 且不改库。
3. 前端 `(main)/issues` 看板四列，新建的 Issue 出现在"计划中"列；列顺序来自 `/api/issue-meta`；NavRail 有入口。
4. `bun run typecheck`（含确认无 `Object.groupBy` 报错）/ `bun run test`（含 `issue/adapter-sqlite.test.ts`、`service.test.ts`）/ `bun run lint` 全绿。
