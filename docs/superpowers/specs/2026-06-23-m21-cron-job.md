# M21 Spec — CronJob：按时间表反复触发 Agent 运行的定时规则（前后端）

> **Status:** 📝 Design → Implementation
> **Baseline:** M20（drizzle-orm 重构）完成态。
> **关联:** `docs/architecture/foundations/cron-job.md`（设计基线）· `docs/architecture/foundations/issue.md`（同级触发型实体）· `docs/architecture/backend/run-supervisor.md` · `docs/architecture/backend/conversation-projection.md` · `docs/architecture/design-philosophy.md`

**Goal:** 落地 CronJob——「一条按时间表反复触发 Agent 运行的定时规则」——后端真实能力 + 前端可 CRUD 的配置界面。

**Non-goals:**
- 不发明新执行机制（复用 dispatcher/supervisor/checkpointer）
- 不写 OS crontab（只用 `Bun.cron(schedule, handler)` 进程内形态）
- 不引入分布式调度 / 多进程协调
- 不做 cron 表达式可视化编辑器 / 秒级精度 / 时区选择 UI
- 不让 cron run 参与 @提及级联
- 不做 per-resource BFF 路由文件

---

## 1. 背景

系统现在的运行都是被动触发：@提及、Issue 编排、手动。缺的是主动、周期性的起点——「每天早上九点让运维 Agent 巡检一遍」。触发源是时钟。

CronJob 是 M21 唯一新增的领域本体。它和 Issue 是兄弟：都是触发型实体、都自带 `threadId`、都复用现有执行层，区别只在触发源——Issue 由人/编排器在状态间推动，CronJob 由时钟周期性点火。

## 2. 当前代码事实

1. **feature 骨架**：`features/project/` 的 adapter 模式——entities.ts + ports.ts + adapter-sqlite.ts + service.ts + http.ts + index.ts。service 是工厂函数，只有 Error 子类是 class。
2. **建表走 drizzle schema（M20 起）**：DDL 在 `infra/db/schema.ts`，迁移由 `scripts/gen-drizzle.sh` 生成，不再手写 migration。
3. **统一 run-start 机制**：`features/run/dispatcher.ts` 的 `createRunDispatcher` 是所有起运行路径的唯一入口。`DispatchCause.kind: RunOriginKind`。
4. **RunOriginKind 是封闭联合**：`"orchestrator" | "mention" | "manual"`。`RunOriginRow` 有可空 `issueId?`。
5. **会话投影按 originKind 隔离**：`projection.ts` 对 `"orchestrator"` 直接 `clearAccumulator + return`。
6. **实体自带会话套路**：Issue 在 create 时 best-effort 建 `conversationId = issueId` + `addMember("owner")`，`threadId = "<issueId>:owner"`。
7. **supervisor**：有 `startMainRun`、`cancel(runId)`、`onRunComplete(listener)`、`dispose()`、reaper。
8. **retry ops 事件已存在**：`retry_requested` / `retry_started`。
9. **HTTP 汇聚**：`http/router.ts` 的 `createRouter(token, features)`，`FeatureSet` 字段。
10. **前端 BFF 单一兜底代理**：`/api/bff/[...path]` 透传所有 `/api/bff/*`。
11. **前端 typed api + react-query**：`lib/api.ts` + `useQuery/useMutation`。
12. **表单模板**：`ProjectForm.tsx` — Dialog + react-hook-form + zod + sonner toast。
13. **导航硬编码**：`NavRail.tsx` 的 "Navigate" 段。
14. **`Bun.cron`**：Bun 内建 API，需 Bun ≥ 1.3.11，UTC 解释，不重叠保证。

## 3. 第一性原则

1. **CronJob 是领域本体**：必须独立存储与触发规则。
2. **事实在表里，定时器是投影**：`cron_job` 表是事实，`Bun.cron` 句柄是投影，重启按表重建。绝不用 OS crontab 形态。
3. **不发明执行机制**：复用 dispatcher / 自带会话 / supervisor。
4. **cron run 必须隔离出 @提及级联**：与 orchestrator 同款隔离。
5. **超时分两层**：per-job 看门狗 + 全局 reaper 兜底，各司其职。
6. **重试是同一棒内的重起**：与 Bun.cron 不重叠保证正交。

## 4. 验收标准

1. `cron_job` 表在 `backend.db`；`run_origin` 含可空 `cron_job_id`。
2. CRUD + enable API 全部生效，返回正确状态码。
3. enabled job 启动/create 时注册 Bun.cron；disable/delete 正确 stop；shutdown 全停。
4. 到点触发经 `dispatcher.dispatch({kind:"cron"})` 起 run，`threadId = "<cronJobId>:owner"`，输出累积进 `/conversations/<cronJobId>`，前端点卡片可达。
5. cron run 在 projection 被隔离。
6. timeoutMs > 0 到点 cancel；非成功收尾在 maxRetries 内重起并写 ops 事件。
7. 绝无 OS crontab 形态调用。
8. 全 workspace typecheck / lint / test 通过。

## 6. 风险

1. **Bun 版本**：`Bun.cron` 需 ≥ 1.3.11，`packageManager` 写 `bun@1.1.0`。P0 先核实。
2. **命名冲突**：Bun 的 `CronJob` 类型 vs domain `CronJob`，import 用别名 `BunCron`。
3. **idempotencyKey 唯一约束**：重试必须带 `:run:<attempt>` 后缀。
4. **不重叠 vs 重试**：handler 必须 await 重试链落定后才返回。
5. **进程内定时器不跨重启**：at-most-once 语义，不做 misfire 补偿。
6. **会话 bootstrap best-effort**：建会话失败只 log 不抛。
7. **scheduler 与 service 职责边界**：service 写表，scheduler 管句柄。http 层先 service 后 scheduler。
8. **onRunComplete 监听器叠加**：retry 监听器的退避不能阻塞 supervisor 的 onRunComplete 链。实施时务必核对 supervisor 调用语义。
