# M16.3 — Run Insights: Structured LLM/Tool Call Timeline from EventLog

> **Status:** Spec (approved) → Implementation in progress.
> **Baseline commit:** `5f816c7` (M16.2 complete).
> **关联:** [14-event-log](../../architecture/14-event-log.md) · [M16.2 Retro](./2026-06-14-m16.2-observability-polish.md)

## 一句话定位

Run Insights 是 EventLog 的只读派生投影——把一次 run 还原成「调了几次模型、烧了多少钱、用了哪些工具」的结构化时间线。EventLog 是唯一 trace 事实源。

---

## 一、动机

M16.1/16.2 把观测面做成了运维事故台（卡住/断连/cancel/recover）。开发者/产品需要另一类答案：这次 run 调了几次 LLM、烧了多少 token、花了多少钱、每步多久、最慢的是哪步。这是 LangSmith / Fornax Trace 视角。

当前现状：token usage 在适配器拿得到但只写进废弃的 checkpointer 审计层，从没进 EventLog；model name、cost、per-call latency、TTFT 全不存在。

---

## 二、目标形态（三层）

```
采集层(Part A)  framework run loop 把 llm_call/tool_call 指标写进 EventLog
查询层(Part B)  backend 从 EventLog 聚合 RunInsights（根汇总 + per-call + 归因）
展示层(Part C)  Web Run 详情加 Insights 区块，Overview 加聚合图表
```

---

## 三、不变量

1. **EventLog 是唯一 trace 事实源** — llm_call/tool_call 指标 append 进 EventLog，不另起表
2. **采集只读不改控制流** — 埋点只读已有数据，不改 run status / checkpointer 恢复
3. **指标可丢可降级** — 未采集显示「未采集」，不崩溃不伪造 0
4. **cost 是派生展示值** — cost = token × price table，token 才是事实
5. **脱敏延续** — 工具 input/output 默认摘要，raw 走 disclosure
6. **废弃层不复活** — model_end CheckpointEvent 保留但 UX 不再依赖它

---

## 四、Part A — 采集层

### 4.1 节点模型

| 节点 | 是否存在 | 处理 |
|------|---------|------|
| root (run) | ✅ | run 级根汇总，从 run 表 + 聚合派生 |
| llm_call | ✅ | 新增持久化事件 |
| tool_call | ✅ | 新增持久化事件 |
| interrupt | ✅ 已有 | 复用，标「⏸ 暂停点」 |
| retrieval / RAG | ❌ | 不做 |
| planner / intent | ❌ | 不做 |

### 4.2 新增 AgentEvent types

```ts
| { type: "llm_call"; payload: {
    step: number; model: string;
    usage: { input: number; output: number; cacheCreate?: number; cacheRead?: number };
    latencyMs: number; ttftMs?: number; stopReason?: string;
  }}
| { type: "tool_call"; payload: {
    step: number; id: string; name: string;
    latencyMs: number; isError: boolean;
  }}
```

### 4.3 埋点位置

- **runLoop**: model_start 前记 llmStart，stream 首个 text delta 记 TTFT，stopReason 捕获，model_end 后 yield llm_call
- **executeOne**: 工具执行完成后 yield tool_call（需 step 作为新参数传入）
- **runner-daemon**: 新 type 不在 delta 白名单，自动走 else 分支落库，无需改代码

### 4.4 适配器改动

- `AIMessageChunk.usage` 加 `cacheCreate?` / `cacheRead?`
- `AnthropicChatModel.stream` 补取 `cache_creation_input_tokens` / `cache_read_input_tokens`
- `ChatModel.id` 可选，loop 读取时 `rt.model.id ?? "unknown"` 兜底

## 五、Part B — 查询层

### 5.1 Price table

代码常量起步，USD per 1M tokens，按 model id 前缀匹配。未知 model 返回 null。

### 5.2 RunInsights 聚合

- 根汇总：status / totalLatency / totalCost / llmCalls / toolCalls / token 合计
- 自动归因：slowestCall / failedCall / interruptedAt
- per-call 时间线（按 step 嵌套）
- toolBreakdown

### 5.3 端点

```
GET /api/ops/runs/:runId/insights        → RunInsights
GET /api/ops/insights/summary?from=&to=  → InsightsSummary
```

---

## 六、Part C — 展示层

### 6.1 图表库

shadcn/ui Charts（Recharts），copy-in chart.tsx，package.json 加 recharts。

### 6.2 Run 详情 Insights 区块

根汇总条（最上）+ call 时间线（step 嵌套，LLM/工具/interrupt），可展开 disclosure。

### 6.3 Overview 图表

Token 趋势折线 + Cost by agent/model 柱状 + Top tools 横向柱。

### 6.4 未采集态

无 llm_call 事件显示「本次 run 未采集运行指标」。cost null 标「未知价」。

## 八、测试计划

- 适配器：mock Anthropic finalMessage 带 cache token，断言透出
- run loop：mock run 断言 EventLog 收到 llm_call + tool_call
- pricing：已知/未知 model，cache token 计入
- insights：多 step run 聚合 + 归因正确；老 run 不崩溃
- 前端：未采集态 / 正常态 / cost null / TTFT 缺省

---

## 九、验收标准

1. 真实 run 后 EventLog 含 llm_call (model/token/latency/TTFT) 和 tool_call (latency/isError)
2. cache token 被采集不再丢弃
3. GET /ops/runs/:id/insights 返回根汇总 + per-call + 归因
4. Run 详情展示根汇总条 + call 时间线
5. Overview 展示 token 趋势 / cost 分布 / top tools 图表
6. cost 标注「估算」，未知 model 不伪造 0
7. 未采集老 run 显示「未采集」不崩溃
8. 埋点不改控制流 / status / checkpointer 恢复
9. 新增函数有单测

---

## 十、边界

1. 不接入外部 OTLP collector，不建 otel_span 表
2. 不做精确计费/对账（估算是按价表）
3. 不改 run status 枚举、不改诊断判定逻辑
4. 不为非 Anthropic 适配器补齐 usage 采集
5. 不做检索/RAG/planner 节点
