# M17.4 Mechanism Boundary Consolidation — 收口机制层泄漏、删死代码、修真 bug

> **Status:** ✅ Complete
> **Baseline:** `6b4f2e3`
> **关联:** `docs/architecture/design-philosophy.md` · `docs/architecture/backend/conversation-projection.md` · `packages/message/src/helpers.ts` · `packages/event-log/src/index.ts` · `packages/agent-spec/src/index.ts` · `packages/agent-fs/src/types.ts` · `apps/backend/src/features/conversation/*.ts` · `apps/backend/src/main.ts` · `apps/lark-bot/src/render.ts`

## Summary

M17.1–M17.3 把消息语义下压成唯一 `Message` 本体、把 revision 流和 wire codec 收口到单一所有者。M17.4 处理这条主线收口后**残留在机制层的脏点**：仍在主链路上浮的实现机制、声明了从未消费的死类型、以及两处会在真实运行中导致数据丢失/消息永不关闭的功能 bug。

四组 patch：

- **Patch C（真 bug，最高优先级）** — 丢消息、Message 永不 terminal
- **Patch B（死代码）** — 删除声明却从未被生产路径消费的类型
- **Patch D（结构收敛）** — 把上浮到主链路的机制压回机制层
- **Patch A（纯文档漂移）** — 修正架构文档的类型名、位置、签名

## Findings

### 真 bug

1. **`assistantMessageId(runId)` 硬编码序号 `0`** — 函数签名的 ordinal 被焊死，无法表达"同一 run 多条 assistant message"的语义（即使当前产品语义是单消息，接口不应预判）
2. **`RunAccumulator` 是模块级内存态** — 进程重启后 terminal revision 写不出去，Message 永远停在 open 态

### 死代码

3. **`EVENT_LOG_MIGRATIONS`** — 死迁移，只被自身测试引用
4. **`AgentSpecV1` 做 canonical `AgentSpec`** — 生产路径实际解析 V2
5. **`AgentFsDomain.runner_state`** — 声明但从未消费

### 文档漂移

6. **编造类型名 `ConversationMessageRevision`** — 真实类型是 `MessageRevision`
7. **实现位置和签名写错** — 声称在 main.ts，实际在 projection.ts；签名 4 参 vs 真实 6 参

### 结构泄漏

8. **`event-log` 是唯一顶层独立包机制边界** — 唯一消费者是 backend
9. **`checkpoint_messages` 借 checkpointer 的存储名** — projection 借用别的机制的 DDL
10. **`threadId` 在不同文件用不同规则解析** — 首冒号 vs 末段切分
11. **`reflect:` 魔法字符串前缀做语义分发** — run 表已有 `kind` 列
12. **Lark 双解析读路径** — `render(content)` 4-shape sniff + `renderRevision`

## Implementation

### Patch C — 真 bug

**C1: `assistantMessageId` ordinal 参数化，当前语义下恒为 0**
- `packages/message/src/helpers.ts`: `assistantMessageId(runId, ordinal)` — ordinal 由调用方传入，不再硬编码在函数体内
- `packages/framework/src/run-loop.ts`: `buildAssistantRevision` 接受 ordinal 形参；当前产品语义是一条 run = 一个增长气泡（M17.2 full-run visibility），ordinal 为 `const 0`，blocks/tools 跨 step 累积不复位。保留 ordinal 接口为未来多消息语义预留
- `apps/backend/src/features/conversation/projection.ts`: terminal fallback 传 0
- 新增 K-turn 回归测试：断言同 run 所有 revision 共享单一 messageId，末态为 `done`，无 open 段残留

> **设计决策**：ordinal 形参存在但当前恒为 0。这不是"没做完"——当前语义确实是一个 run 产出单条气泡，强行分段（v2 尝试过）会导致中间段永远没有 terminal 写入、引入比原 bug 更严重的 open-message-leak。ordinal 接口是给未来如果真的需要"一个 run 开多条独立 message"时预留的钩子。

**C2: terminal 写入去内存化**
- `projection.ts`: `onRunComplete` 不再依赖 `activeConversations` Set（重启/reaper 路径下恒空），改为从 `threadId` 通过 `parseThreadId` 直接解析 `conversationId`
- accumulator 为空时 fallback 到 `findLatestAssistantRevision(port, cid, runId)` 扫描 ledger
- 新增 `ledgerHasTerminalForMessage` 按 `(runId, messageId, state)` 判重，避免 `updatedAt` 变化导致字节去重失效、重启/reaper 重复触发时写入重复 terminal
- 删除 `buildPreloadedMessages` 里 `legacyDropped` 计数
- `onRunComplete` 签名删除 `activeConversations` 参数（不再需要）

### Patch B — 死代码

- **B1**: 删除 `EVENT_LOG_MIGRATIONS` 导出及其测试
- **B2**: 删除 `AgentSpecV1` schema、`AGENT_SPEC_V1_VERSION`、`CURRENT_SCHEMA_VERSION`；`AgentSpec` → `z.infer<typeof AgentSpecV2>`
- **B3**: `AgentFsDomain` 删除 `runner_state`

### Patch D — 结构收敛

- **D1+D2**: `packages/event-log/` → `apps/backend/src/features/event-log/`；event_log DDL 加入 `EVENTS_DB_MIGRATIONS`（id 3010）；删除 `@my-agent-team/event-log` workspace 依赖
- **D3**: `checkpoint_messages` → `projection_messages`（migration `backend_v22_projection_messages` id 5008）；DDL 所有者改为 backend
- **D4**: `parseThreadId()` 导出（与 `deriveThreadId` 互逆）；全仓 5 文件裸 `split(":")` 替换
- **D5**: `onRunComplete`/`onRunEvent` 回调新增 `kind` 参数；projection 用 `kind === "reflect"` 替代 `threadId.startsWith("reflect:")`
- **D6**: 删除 Lark `render(content)` 4-shape sniff；测试迁移到 `renderRevision`

### Patch A — 文档

- `conversation-projection.md`: 全文 `ConversationMessageRevision` → `MessageRevision`；实现位置 → `projection.ts`；签名 → 6 参 + `kind`；补齐 `tools`/`visibility`/`updatedAt`/`error` 字段
