# M1 Spec — Core 协议与最小运行内核

> 基于 `next` 分支当前 monorepo 骨架(Bun + Turborepo)。
> 目标:用 ~300 行代码实现一个能跑起来的最小 agent 内核,**不引入任何超出必要的抽象**。

---

## 一、设计原则

### 1. 第一性原理 (First Principles)

不从模式 / 框架 / 习惯出发,只从**不可否认的事实**推出最小结构。

**事实清单:**

1. 用户给一段 prompt(自然语言意图)
2. 系统调 LLM,LLM 可能想调 tool
3. tool 执行,结果回喂 LLM
4. LLM 决定继续调 tool 还是结束
5. 整个过程产生流式输出,用户实时看

**所有其他东西**(session / plugin / capability / event bus / middleware / checkpoint / subagent)**都是派生需求**,**不是本质**,在 M1 中**一律不做**。

### 2. 奥卡姆剃刀 (Occam's Razor)

能用 code review 拦的,不写检查脚本。
能用 TS 类型表达的,不写运行时校验。
能用 package.json deps 声明的,不引专门工具。
能用 JS 原生函数能力做的,不发明协议字段。

### 3. 状态归调用方 (State Belongs to Caller)

参考 Claude Code SDK 的做法:**框架不持有会话状态**,`messages` 数组在调用方手里。

- 多轮 = 调用方传同一个数组
- Fork / 压缩 / 持久化 = 调用方自由处理
- 框架只负责 "给我消息数组,我给你新消息流"

### 4. AsyncIterable 是 first-class 输出

不发明 EventBus / Observer。流式输出就是 `AsyncIterable<Message>`,用户 `for await` 消费即可。

### 5. 抽象时机 = Rule of Three

不为想象需求建抽象,**只为已发生 3+ 次的痛点建抽象**。M1 之后任何新协议字段都要列出 3 个真实使用场景才能加。

## 四、L1 协议设计

### 4.1 Message 与 ContentBlock

对齐 Anthropic Messages API 字段名 `tool_use_id`(snake_case),便于 adapter 零翻译。

```ts
// packages/core/src/message.ts

export type Role = 'user' | 'assistant' | 'system';

export interface TextBlock {
  type: 'text';
  text: string;
}

export interface ToolUseBlock {
  type: 'tool_use';
  id: string;          // LLM 生成的 call id
  name: string;        // tool 名
  input: unknown;      // tool 参数,JSON 兼容
}

export interface ToolResultBlock {
  type: 'tool_result';
  tool_use_id: string;
  content: string;     // 简化:M1 只支持纯文本结果
  is_error?: boolean;
}

export type ContentBlock = TextBlock | ToolUseBlock | ToolResultBlock;

export interface Message {
  role: Role;
  content: string | ContentBlock[];
}
```

**说明:**

- `content` 允许 `string` 或 `ContentBlock[]`,兼容简单文本场景与多模态/工具场景
- `ToolResultBlock.content` 简化为 `string`,M1 不支持嵌套 block;有 image 需求再升 union
- 保留 `system` role,让调用方用同一个 `messages` 数组表达 system prompt;M1 不额外引入 `systemPrompt` 参数,减少抽象和心智负担
- 不引入 brand type(AgentId / SessionId),因为 M1 没有 session 概念

### 4.2 ChatModel

```ts
// packages/core/src/chat-model.ts

import type { Message } from './message';

export interface AIMessageChunk {
  /**
   * 增量块:本 chunk 新增的内容。
   * - text:增量文本
   * - tool_use:首次见到 tool call(id + name 已定)
   * - input_json_delta:tool input JSON 字符串增量,需按 id 累加后 JSON.parse
   */
  delta?:
    | { type: 'text'; text: string }
    | { type: 'tool_use'; id: string; name: string }
    | { type: 'input_json_delta'; id: string; partial_json: string };

  /** 完成标志 */
  done?: boolean;

  /** 对齐 Anthropic stop_reason 命名与取值;done chunk 应尽量携带,但 runtime 不依赖它 */
  stopReason?: 'end_turn' | 'tool_use' | 'max_tokens' | 'stop_sequence' | 'pause_turn' | 'refusal';

  /** token 用量(仅 done=true 时有效) */
  usage?: { input: number; output: number };
}

export interface ChatModelOptions {
  signal?: AbortSignal;
}

export interface ChatModel {
  readonly id?: string;
  stream(messages: readonly Message[], options?: ChatModelOptions): AsyncIterable<AIMessageChunk>;
}
```

**关键设计点:**

1. **只有 `stream`,没有 `invoke`** —— invoke = `collectStream(stream)`,核心包提供 helper,接口本身只暴露一个方法。最小原语。
2. **`input_json_delta.partial_json` 是 string** —— 对齐 Anthropic SSE 的 tool input 增量形态;mock model 一次发完即可
3. **不提供 `blocks` 快照字段** —— runtime 自己从 delta 累积完整 blocks,避免 snapshot + delta 双语义
4. **`stopReason` 对齐 Anthropic `stop_reason`** —— adapter 只做命名转换,不做语义翻译
5. **`signal` 在 options 里** —— Web 标准
6. **`messages` 是 readonly 输入** —— model 只读上下文,不修改调用方状态;状态推进只由 `run()` 负责

### 4.3 Tool

```ts
// packages/core/src/tool.ts

export interface ToolExecuteResult {
  /** 返回给 LLM 的文本 */
  content: string;
  /** 是否标记为错误(影响 tool_result.is_error) */
  isError?: boolean;
}

export interface Tool {
  readonly name: string;
  readonly description: string;
  /** JSON Schema 形态的 input 描述,给 LLM 看 */
  readonly inputSchema: Record<string, unknown>;

  /**
   * 执行 tool。
   * @param input  LLM 给的参数(unknown,调用方负责 validate / cast)
   * @param signal 上游 abort signal
   */
  execute(input: unknown, signal?: AbortSignal): ToolExecuteResult | Promise<ToolExecuteResult>;
}
```

**说明:**

- `inputSchema` 是给 LLM 的 hint,M1 不强制 runtime 用它做 validate(用户在 `execute` 内自己 validate)
- `name` 必须唯一;M1 不做运行时校验,重复时 `Map` 构造会以后者覆盖前者
- 不引入 Zod / Joi 等第三方依赖
- 不引入 "Tool.parse" 阶段;校验是 execute 的内部职责

### 4.4 Errors

M1 不定义自有错误类。model stream 抛出的错误自然向外透传;tool 执行错误由 `run()` 转成 `tool_result` 的 `is_error=true`;tool not found 同样转成 `tool_result` 错误。

## 五、L2 运行内核

### 5.1 公共 API

```ts
// packages/core/src/run.ts

import type { ChatModel } from './chat-model';
import type { Tool } from './tool';
import type { Message, ContentBlock } from './message';

export interface RunOptions {
  signal?: AbortSignal;
  /** 最大循环步数,防止失控。默认 32。达到后静默结束;提示用户继续属于 harness 职责。 */
  maxSteps?: number;
}

/**
 * 运行 agent 直到 LLM 不再调 tool 或达到 maxSteps。
 *
 * 状态归属:调用方持有 messages 数组(本函数会原地 push)。
 * 流式输出:每次 LLM 增量产出和每条 tool 结果都会被 yield。
 */
export async function* run(
  model: ChatModel,
  tools: readonly Tool[],
  messages: Message[],
  options?: RunOptions,
): AsyncIterable<Message> { /* impl */ }
```

### 5.2 实现要点(伪代码)

```ts
export async function* run(model, tools, messages, options = {}) {
  const { signal, maxSteps = 32 } = options;
  const toolMap = new Map(tools.map(t => [t.name, t]));

  for (let step = 0; step < maxSteps; step++) {
    if (signal?.aborted) return;

    // 1. 调 LLM,累积 chunks 成 ContentBlock[]
    const blocks: ContentBlock[] = [];
    const partialJson = new Map<string, string>();    // tool_use_id → 累积的 partial_json 字符串

    for await (const chunk of model.stream(messages, { signal })) {
      if (signal?.aborted) return;
      mergeChunkIntoBlocks(blocks, partialJson, chunk);
      // 增量 yield 给用户(同一条 assistant message,blocks 不断扩充)
      yield { role: 'assistant', content: structuredClone(blocks) };
      if (chunk.done) break;
    }

    if (signal?.aborted) return;
    if (blocks.length === 0) return;

    // 2. finalize:把 partial JSON 解析成 input;解析失败则保留原始字符串
    finalizeToolUseInputs(blocks, partialJson);

    const assistantMsg: Message = { role: 'assistant', content: blocks };
    messages.push(assistantMsg);

    // 3. 收集 tool_use,无则结束;M1 不按 stopReason 控制流程
    const toolUses = blocks.filter((b): b is ToolUseBlock => b.type === 'tool_use');
    if (toolUses.length === 0) return;

    // 4. M1 串行执行 tools;未来有真实需求再改并行
    const results: ContentBlock[] = [];
    for (const tu of toolUses) {
      if (signal?.aborted) return;
      const tool = toolMap.get(tu.name);
      if (!tool) {
        results.push({
          type: 'tool_result',
          tool_use_id: tu.id,
          content: `Tool not found: ${tu.name}`,
          is_error: true,
        });
        continue;
      }
      try {
        const out = await tool.execute(tu.input, signal);
        results.push({
          type: 'tool_result',
          tool_use_id: tu.id,
          content: out.content,
          is_error: out.isError,
        });
      } catch (err) {
        results.push({
          type: 'tool_result',
          tool_use_id: tu.id,
          content: err instanceof Error ? err.message : String(err),
          is_error: true,
        });
      }
    }

    if (signal?.aborted) return;

    const userMsg: Message = { role: 'user', content: results };
    messages.push(userMsg);
    yield userMsg;
  }
}
```

**约 80 行实际代码 + 30 行内部 helper(`mergeChunkIntoBlocks` / `finalizeToolUseInputs`)**。`finalizeToolUseInputs` 对累积的 partial JSON 调 `JSON.parse`;解析失败时把原始字符串作为 `input`。abort 后不把部分 assistant blocks 或部分 tool results push 回 `messages`,但已 yield 的实时 snapshot 保持已发送。这两个 helper 不公开导出;按奥卡姆剃刀,只暴露 M1 调用方真实需要的 `collectStream`。

### 5.3 Helper API(公开导出)

```ts
// packages/core/src/stream-utils.ts

/** 把流式 chunks 收集成完整 Message */
export async function collectStream(
  stream: AsyncIterable<AIMessageChunk>,
): Promise<{ blocks: ContentBlock[]; stopReason?: AIMessageChunk["stopReason"]; usage?: AIMessageChunk["usage"] }>;
```

供"我只想要最终结果不要流"的场景使用。

### 5.4 不做的事(M1 明确范围外)

- ❌ M1 不实现并发 tool 执行(串行即可;未来出现真实需求再改并行)
- ❌ 不实现 retry / cache / fallback(用户自己包装 model)
- ❌ 不实现 budget / 超时(`signal` + `maxSteps` 够用)
- ❌ 不实现权限 gate(无需要,L4 出现时再说)
- ❌ 不实现 session store / 持久化(状态归调用方)
- ❌ 不实现 EventBus(AsyncIterable 即事件流)
- ❌ 不实现 Plugin 协议(任何扩展都是包装 model/tool 函数)

## 六、test-helpers 包

仅一个 mock model,用于 M1 自测和未来 milestone 复用。

```ts
// packages/test-helpers/src/echo-model.ts

import type { ChatModel, AIMessageChunk, Message } from '@my-agent-team/core';

export interface EchoScript {
  /** 第 N 轮(从 0 起)的响应。超出长度则返回最后一条。 */
  turns: Array<
    | { type: 'text'; text: string }
    | { type: 'tool_call'; id: string; name: string; input: unknown }
  >;
}

/**
 * 可脚本化的测试用 ChatModel。
 * 按 messages 中 assistant 的数量决定当前是第几轮,返回对应脚本。
 */
export function echoModel(script: EchoScript): ChatModel {
  return {
    id: 'echo',
    async *stream(messages: readonly Message[]): AsyncIterable<AIMessageChunk> {
      const turn = countAssistantMessages(messages);
      const item = script.turns[Math.min(turn, script.turns.length - 1)];
      // 模拟流式:一次性发完即可
      if (item.type === 'text') {
        yield { delta: { type: 'text', text: item.text } };
        yield { done: true, stopReason: 'end_turn' };
      } else {
        yield { delta: { type: 'tool_use', id: item.id, name: item.name } };
        yield { delta: { type: 'input_json_delta', id: item.id, partial_json: JSON.stringify(item.input) } };
        yield { done: true, stopReason: 'tool_use' };
      }
    },
  };
}
```

不暴露 `mockTool`,用户在测试里直接写一个 5 行的 Tool 实现。

## 八、测试要求

### 8.1 单元测试(`packages/core/src/*.test.ts`)

测试文件贴近被测源码,便于开发时马上找到对应测试。

| 文件 | 覆盖场景 |
|---|---|
| `src/stream-utils.test.ts` | collectStream 公开行为:从 delta 流得到完整 blocks,并保留 stopReason/usage |
| `src/run.test.ts` | 见下方场景 |

### 8.2 `run.test.ts` 场景清单(P0)

1. **single-turn text** —— EchoModel 返回纯文本,run 产出一条 assistant 消息,messages 数组长度 +1,迭代结束
2. **multi-turn with tool** —— 第一轮 LLM 调 tool,第二轮返回文本;验证 messages 数组最终包含 user(初始)+ assistant(tool_use)+ user(tool_result)+ assistant(text),共 4 条
3. **tool not found** —— LLM 调一个不存在的 tool,验证 tool_result 标记 is_error=true 且 LLM 能进入下一轮
4. **tool execute throw** —— tool 抛错被捕获成 is_error 的 tool_result,不中断 run
5. **abort mid-stream** —— signal abort 时 run 中止,不把部分 assistant blocks push 回 messages
6. **abort during tools** —— tool 执行阶段 abort 时 run 中止,不把部分 tool_result push 回 messages
7. **maxSteps 防失控** —— EchoModel 永远调 tool,maxSteps=3 时正好跑 3 步退出
8. **多个 tool_use 串行** —— 一轮 LLM 返回两个 tool_use,两个都执行,results 按顺序 push
9. **empty stream** —— model 只返回 done 或空 stream 时,run 不 push 空 assistant message

### 8.3 测试工具

- Bun test runner(`bun test`)
- 不引入 vitest / jest
- `tsconfig.json` 用于 build,排除 `src/**/*.test.ts`
- `tsconfig.test.json` 继承 `tsconfig.json`,覆盖 `include` 并设置 `exclude: []` 以纳入源码和测试文件;`typecheck` 只运行 `tsc -p tsconfig.test.json --noEmit`,不重复跑主 tsconfig
- `_template/package/` 也采用同一约定,后续新包默认生成 `tsconfig.test.json`

- 真 LLM(M2)
- Adapter 翻译(M2)
- 并发 tool(暂未实现)
- 持久化 / session(无)

## 附录 A:与 LangChain / Claude Code 的对标取舍

| 维度 | LangChain | Claude Code | M1 选择 |
|---|---|---|---|
| Message 类型 | LangChain Core (`HumanMessage`/`AIMessage` 类) | Anthropic SDK 原生 union | 后者(plain object,字段对齐 Anthropic API) |
| ChatModel 接口 | `invoke` + `stream` 双方法 | `query()` AsyncIterable | 只保留 `stream`,`invoke` = helper |
| 状态管理 | StateSchema + reducer + Channel | 调用方持有 messages | 后者 |
| 扩展机制 | Middleware (wrap hook) | 装饰 tool + permissionMode | 都不做,用户自己包函数 |
| Subagent | createAgent 嵌套 + handoff | Task tool 递归 query | 都不做(M1 范围外) |
| Streaming 输出 | AsyncIterable | AsyncIterable | 抄 |
| Graph runtime | LangGraph | 无 | 不做,简单 while 循环 |
| Checkpoint | langgraph-checkpoint | 无 | 不做 |
| 多 provider | initChatModel | 仅 Anthropic | M2 起做 Anthropic,后续按需 |

## 附录 B:设计纪律(永久生效)

写进 `docs/architecture/00-overview.md` 的 README 章节:

1. **No protocol without proven need** — 协议字段只为已发生的痛点加,不为想象需求加
2. **No deep imports** — 跨包只能从 index.ts 进
3. **Composition over hooks** — 用户能用 JS 函数嵌套解决的,不加协议字段
4. **State belongs to caller by default** — 框架持有状态需要充分理由
5. **Layer downward dependency** — L4→L3→L2→L1,反向 = bug
6. **AsyncIterable is the event stream** — 不重新发明 EventBus / Observer
7. **One concept, one name** — 同一事物全项目同一称呼
8. **Rule of three** — 第三次重复出现才提取抽象
