## Grill 决议落地

12 条 grill 决议全部落地。Q12（gray-matter `^4.1.0`）因 npm 无此版本改为 `^4.0.3`——属 spec 假设错误而非实现偏差。

## 偏离 spec 的点

1. **gray-matter 版本**（§4.3 Q12）：spec 假定 `^4.1.0`，npm 实际最高 `4.0.3`。无功能差异。
2. **测试文件组织**：spec 列了 `skill-dir-replacement.test.ts` 独立文件，实际合并到 `progressive-skill.test.ts` 里——一个文件少但覆盖完整（review 补了缺失的"其他占位符保持原文" case）。
3. **lifecycle 测试方法**：spec 建议的 `Bun.file(dir).exists()` 对目录不生效，改用 `stat()` 验证。
4. **collision 测试简化**：spec 的 `-2.md` 精确后缀测试因时间戳精度不可靠，改为"同一内容多次写入 path 唯一"（`Set.size === N`）。

## Lessons Learned

1. **npm 版本不可假设** — spec 写 `^4.1.0` 时应先 `npm view gray-matter versions` 确认。Grill 阶段拍的是"对版本的偏好"而非"对版本的验证"。
2. **模块级 cache 在多实例场景有理论碰撞风险** — cache key 是 `mtimeMs`，两个不同目录的 mtime 可能相同。v1 单实例场景无影响，但在 code review 中应明确文档化这个限制。
3. **TDD 纪律有两次松动** — `cache.ts` 和 `progressive-skill.ts` 先写实现后补测试，后续 review 发现 `let skills` 隐式 any 正是这个顺序导致的——如果测试先行，类型错误在 RED 阶段就被 TS 编译器拦住。
4. **`Bun.file().exists()` 对目录返回 false** — bun 的 `Bun.file()` 只用于文件，目录存在性检查必须用 `fs.stat`。
