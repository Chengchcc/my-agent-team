# M17.5 Pipeline Tiering & Package Boundaries — 链路一致性分层 + 包边界纠偏

> **Status:** Spec → Ready for implementation.
> **Baseline commit:** `c3a3146` (M17.4 Patch C v3 完成,Patch C 真 bug 全部收口)。
> **关联:** `docs/architecture/foundations/facts-and-projections.md` · `docs/architecture/backend/run-supervisor.md` · `docs/architecture/backend/conversation-projection.md` · `docs/architecture/conversation/ledger.md` · `docs/architecture/backend/data-model.md` · `docs/architecture/runner/agent-file-system.md` · `apps/backend/src/features/run/supervisor.ts` · `apps/backend/src/features/run/service.ts` · `apps/backend/src/features/conversation/projection.ts` · `apps/backend/src/features/conversation/service.ts` · `apps/backend/src/features/conversation/ports.ts` · `apps/backend/src/main.ts` · `apps/lark-bot/src/sse-watcher.ts` · `packages/runner-daemon/src` · `packages/agent-spec/src` · `packages/agent-fs/src` · `packages/conversation/src` · `packages/message/src` · `packages/tools-common/src/agent-fs-like.ts`。

M17.5 处理四类已经攒到必须单独承载的债,它们彼此独立,但同属"边界没画对":

1. **链路一致性分层缺失**:长 e2e 链路把强一致的控制/业务信号,和弱一致的可观测/副作用,焊死在同一条 `await` 链或同一个 `try/catch` 里,失败语义互相污染。这不是 M17.4 收口的"机制层泄漏",而是更上层的**职责定级**问题——每个 sink 没有声明自己的一致性等级。
2. **包边界不合理**:有的包薄到不该独立存在(`agent-spec`),有的契约被复刻成多份同形定义(`AgentFsLike` 与 `agent-fs` 的 backend 接口),增加认知负担和维护漂移面。
3. **对话事实双来源**:`conversation_ledger` 与 `event_log` 被同时标为对话消息的事实来源,但 ledger 里的 assistant 消息其实是从 event_log 派生的——一条业务事实有两个权威载体,"以谁为准/坏了重建谁"说不清。
4. **领域没拆干净**:同一领域对象(`LedgerEntry`、`Member`、`MessageId`、run 终态、并发锁)在 backend/web/lark/runner 各复制一份同形定义或各写一套规则,字段与枚举已开始漂移——"模块边界冒充了领域边界"(Conway)。它与第 3 条同根(都违反"一个本体/一个权威载体"),只是表现为复制而非派生。

这不是一次兼容性改造。旧的 fire-and-forget 顺序、旧的混职责回调、薄包的独立发布单元、复刻的同形接口、派生出来的第二份"事实"、各模块各抄一份的领域模型,都是本次要拆除的技术债,不作为兼容目标保留。

## 1. 背景

M17.1–M17.4 已经把消息本体下压、把机制层(EventLog 折叠、checkpoint 改名、`buildAssistantRevision` 归位)收口、把 Conversation Projection 立为正式概念,并修掉了 `assistantMessageId` ordinal 与 terminal 去重两个真 bug。

但这些工作都停在"单点正确"。当我们沿着 `日志是运维可观测的,会话才是业务功能` 这把尺子复查长链路时,发现 M17.4 确认的"账本对事件日志主次颠倒"不是孤例,而是一个在 supervisor → projection → service → daemon → sse-watcher 多处重复出现的**反模式**。同时,在梳理包清单时发现两处包边界问题:`agent-spec` 只有 44 行、单一消费者;`AgentFsLike` 这个文件系统契约在 `tools-common` 定义,而 `agent-fs` 又用 `ReadableBackend/WritableBackend` 复刻了同一组方法签名,`AgentFS` 类也是同形。

M17.5 要处理的就是这两条:**给每条 e2e 链路的每个 sink 定级,让失败语义不再互相绑架;把不该独立的包并回去,把复刻的同形契约归一。**

## 2. 当前代码事实

基于 `c3a3146`:

### 2.1 链路一致性

1. `supervisor.ts` `run_done` case(L495–525):先 `await Promise.all(this.#onRunComplete.map(fn => fn(...)))`,**之后**才 `if (transport) transport.send({ type: "run_finalized", runId })`。注释自认"Only send run_finalized after all listeners complete"。
2. daemon 的 reflect **唯一**靠 `run_finalized` 触发:`#drive` 发 `run_done` → 设 `#finalized` → 等 backend 回 `run_finalized` → `#fireReflect`。
3. `supervisor.ts` `event` case(L453–487):`eventLog.append` 失败 `throw`(强一致,阻止 `run_done` 带着残缺日志成功);紧接着 `onRunEvent` listener 是 `void Promise.resolve(fn(...)).catch(...)`(fire-and-forget,失败仅 `console.error`)。
4. `main.ts` `onRunEvent`(L101–135):一个回调里同时做 todo 累积 + @mention 正则扫描 + 投影编排;@mention 正则在每条 streaming revision 上重复跑。
5. `projection.ts` `onRunComplete`(L221–334):一个函数顺序做 `await projectionChain` → 写终态 revision + `broadcastMessage` → `clearAccumulator` → `appendTodo` → `triggerMentionedAgents`(fire-and-forget) → `completeRun`。其中**终态写 + broadcast 段没有任何 `try/catch` 包裹**,而 `completeRun(cid, threadId, runId)`(释放 M10 锁)在所有 try 块**之外**的最后一行。
6. `service.ts` `forkAgentRuns`(L163–199):`activeConversations.add` + `pendingRuns.set` 与 fork 失败的 catch 减一、`finally` 删锁混在同一个 try/finally。`completeRun`(L430–438)和 `triggerMentionedAgents`(L443–478)各自又读写 `pendingRuns`/`activeConversations`,并发控制状态分散在三处。
7. daemon `#drive` 的 finally:`#runs.delete` → `await send(run_done)` → `if (wantsReflect) #finalized.set`。`send` 抛错则 `#finalized.set` 不执行,run 已从 `#runs` 删除 → 既不 active 也不 finalized,reflect 丢失,只能靠 reaper 兜底。
8. `sse-watcher.ts` `processEntry`:`onSend`(投递)与 `upsertMessageDelivery`(记账)非原子;投递成功、写 delivery 前崩溃,重连因 `pushedSeq` 未推进会重发同一条。
9. reaper 路径释放锁是 fire-and-forget(未 await),`run_done` 路径是 `await`——同一业务回调两条调用约定不一致。

### 2.2 包边界

10. `packages/agent-spec/src/index.ts`:仅 44 行,导出 `AgentSpecV2`(zod discriminated union)及其类型别名。**唯一消费者**是 `packages/runner-daemon/src/runner-daemon.ts`(`AgentSpecV2.safeParse`,L141/L214)。`runner-protocol` 的 `HostToRunner.start` 把 spec 定义成 `z.record(z.unknown())`(不透明),由 daemon 用 `AgentSpecV2` 二次校验。

### 2.3 对话账本的双事实来源

14. `docs/architecture/foundations/facts-and-projections.md` 把 `conversation_ledger` 与 `event_log` **同时**标为"事实来源=是、可重建=否"。但二者对"assistant 消息"的关系并不对等:assistant 消息的**内容**是先落 `event_log`,再经投影桥(`main.ts onRunEvent` → `projection.ts`)派生进 ledger 的——ledger 里的 assistant 消息是 event_log 的**派生物**,却也被当成事实来源。
15. 物理入口不对称:人类/系统消息由 Conversation Service 的 `postMessage` → `appendAndBroadcast` → `appendLedgerEntry` **直接写入 ledger**(role=user/system,state=done);assistant 消息没有直接入口,只能以 run 事件存在 → 过 `event_log` → 投影 → 才进 ledger。于是 ledger **半截自产、半截下游于另一机制**,不自洽。
16. 注意边界事实:`projection.ts` 与 `packages/conversation` **都不 import event-log**——耦合是数据流上的(经 `main.ts` 把 `supervisor.onRunEvent` 回调拼起来),不是包依赖。所以这不是"包依赖反向",而是"**事实定位过重**":一条业务事实(对话消息)被安排了两个权威载体。同时,`facts-and-projections.md` 已立的硬边界——"只有会话投影这一处能把运行事件变成对话可见消息""成员加入通知根本不属于任何一次运行"——说明 event_log **不该**承载对话事件,这排除了"把 ledger 整体降为 log 的纯投影"那条路。

### 2.4 领域归一(D7 同形:领域没拆干净)

D7 不是孤例。沿同一把尺子("一条业务事实只有一个权威载体;同一领域对象只有一个本体")复查全仓,发现以下同形债。按"病型"归三类。

**同模型复制型(Conway:模块边界冒充领域边界):**

**机制上浮型(同一规则散落多层,无单一所有者):**

## 3. 第一性原则

### 3.1 每个 sink 必须声明一致性等级

链路上每个写入点(sink)要么是 **critical**(业务事实,失败必须传播、保序、可观测),要么是 **best-effort**(可观测/副作用,失败只记录、不影响主流程)。把两者焊在同一条 `await` 链或同一个 `try/catch` 里,会让一方的失败语义污染另一方:critical 失败被吞 → 业务静默丢数据;best-effort 失败上抛 → 控制流被无关副作用卡死。

| 等级 | 例子 | 失败语义 | 调用约定 |
|------|------|----------|----------|
| critical | 账本终态写、对话锁释放、并发计数 | 传播(throw / reject) | `await`,保序 |
| best-effort | EventLog、ops、SSE 投递、@mention 触发 | 仅 `console.error` | fire-and-forget,不阻塞 |

### 3.2 控制信号优先于业务副作用

`run_finalized` 这类**控制信号**驱动的是运行时状态机(reflect 触发、锁、队列),它的及时性是正确性的一部分。控制信号必须在业务写入**之前或并行**发出,绝不能排在 `Promise.all(业务副作用)` 之后。否则任一业务 listener 卡住/抛错,控制信号永不发出,运行时被业务副作用绑架。

### 3.3 并发控制是独立状态机,不依附 try/catch

锁与计数(`activeConversations`/`pendingRuns`)是显式的并发控制状态,它的迁移规则不应散落在业务回调的 `catch`/`finally` 分支里。状态机应有单一所有者和明确的迁移入口(acquire/release/decrement),与任何业务 `try/catch` 无关。

### 3.4 IO 与记账要么同事务,要么幂等

"对外投递"和"记下已投递"必须原子:要么同事务提交,要么用幂等键让重发覆盖(update 态)而非追加。两步之间崩溃不得产生重复对外可见效果。

### 3.5 包是依赖边界,不是文件抽屉

一个包的存在理由是它**划定了一条依赖边界**——被多个独立单元共享,或需要独立的版本/可见性约束。仅仅因为"这是一类东西"就建包,只会增加 workspace 协议、构建产物、import 路径的固定成本。单一消费者的薄包应并回其消费者;被多端共享的薄包应保留。

### 3.6 同一契约只能有一处定义

一个语义契约(如"Agent 能读写的文件系统长什么样")在整个仓库只能有一处**规范定义**;其它地方引用它,而不是复刻一份同形声明。多份同形定义会随时间漂移(一处加了方法另一处没加),并让"谁是真相"变得模糊。

### 3.7 一条业务事实只有一个权威物理入口

一类业务事实(如"对话里的一条消息")必须有**唯一的权威写入入口**。如果同一类事实有两个被同时标记为"事实来源"的载体,而其中一个的内容又是从另一个**派生**出来的,那它就不是真正的事实来源——它是投影。两个都自称真相,会让"以谁为准""坏了重建谁"永远说不清。

判据很简单:**一个事实来源的内容,不应该需要从另一个事实来源推导出来。** 若需要推导,被推导方就该降级为投影(只读、可重建),把"事实来源"这顶帽子还给真正的源头。

### 3.8 同一领域对象只有一个本体,模块边界不是领域边界

一个领域对象(Message、Member、LedgerEntry、Run、MessageId)在整个仓库只能有**一个规范本体**:一处类型/schema 定义、一处 id 生成规则、一处 codec(序列化/反序列化)、一处 terminal 判定谓词。其它模块**引用**这个本体,而不是"因为我在另一个模块/存储/surface 里,所以复制一份同形声明"。

判据(对照 `design-philosophy.md` 红旗):它们是否描述同一个"谁在什么时候说了什么"、共享同一个完成/失败状态、在 UI 里 collapse 成同一条?是,就该共用本体。一旦同一对象出现 backend 版 / web 版 / lark 版 / runner 版,且字段或枚举开始不一致,就是"模块边界冒充了领域边界"(Conway)——这不是抽象升级,是概念债,迟早漂移成 bug。

机制(id 怎么生成、content 怎么编码、terminal 怎么判定)是领域本体的一部分,应**下沉到本体所在的包**统一导出,而不是让每个调用方各写一份字面量。

## 4. 目标

1. 给 supervisor → projection → service → daemon → sse-watcher 链路上的每个 sink 定级,落实"控制信号优先"与"sink 分级",修掉 H1/H2/H3(高)。
2. 把混职责的回调(`projection.ts onRunComplete`、`main.ts onRunEvent`、daemon `#drive`、`sse-watcher processEntry`)按一致性等级拆开,修掉 M4/M5/M6(中)与 L7/L8/L9(低)。
3. 把对话锁/计数收敛成单一所有者的显式状态机,消除分散在三处的迁移逻辑。
4. 把 `agent-spec` 并回 `runner-daemon`,消除单一消费者薄包。
5. 把 `AgentFsLike` 下沉 `core` 立为文件系统契约的唯一规范定义,让 `agent-fs`/`tools-common` 引用它而非复刻同形接口。
6. 保留 `conversation` 包(多端共享的领域契约),并在文档里记录"为什么它薄但该留",作为 3.5 的判例。
7. 把 assistant 消息的权威入口收回 ledger,消除"账本/事件日志双事实来源",`event_log` 回归纯执行细节,投影桥降级为可观测扇出(治本 H3)。
8. `LedgerEntry`/`Member` 立单一本体,删除 backend/lark/web 的复刻同形定义,消除字段与枚举漂移。
9. messageId 生成、content codec、terminal 谓词三类机制下沉本体所在包,调用方一律引用,消除散落字面量与裸字符串判定。
10. run 的"存在"(reflect run)与"完成"(终态)各归一个权威入口,消除 daemon 内存先于 DB、reaper 与 run_done 两套语义。
11. 会话/线程并发锁合并成单一闸门,两条入口路径不再各算一套 busy 状态。

## 5. 非目标

- 不改 EventLog 的强一致语义本身(它已经是 critical sink,`append` 失败上抛是对的;本次只是不让它和 best-effort 投影焊在一起,并在 P7 把它的**职责范围**收窄到纯执行细节——语义不变,承载内容变窄)。
- 不改 reflect 的业务语义(只保证它的触发信号不被业务副作用阻塞)。
- 不引入消息队列/外部 broker——sink 分级靠进程内的调用约定实现,不引入新基础设施。
- 不动 `conversation`、`message`、`core` 这类有真实跨端消费者的包边界(但 P6 会把 `AgentFsLike` **加入** `core` 的导出——这是扩充 core 的契约面,不是改动既有边界)。
- 不在本里程碑重做 sse-watcher 的飞书投递协议;M6/幂等键属于结构修复,不扩展为投递层重写。
- P7 不采用方案 A(把 ledger 整体降为 event_log 的纯投影/CQRS)——它会迫使 event_log 承载不属于任一次运行的对话事件,破坏既有硬边界。
- P8 不合并 `MessageAuthor`(消息作者角色:system/user/agent/tool)与会话 `Member`(参与者身份:human/agent)——两者不同层,本次只厘清注释,不强行归一。
- 不动 `thread-projection`(`projection_messages`):它服务模型 fork/resume,可辩护为 checkpoint 类机制投影;列入存疑待评估,本里程碑不重构。
- P9 不重做 content 的 wire 格式(blocks schema 本身不变);只把"编解码 + 抽 text"的实现收敛到单一 codec,不改数据形状。

## 6. 实施分组与方案

### Patch P1 — 控制信号优先(H1, M5)

**目标:** `run_finalized` 不再排在业务 listener 之后;daemon 本地状态机更新不依赖 IO 成功。

`supervisor.ts` `run_done` case 调整顺序与失败隔离:

```text
case "run_done":
  更新 attempt/run 状态(critical,本地 DB,保留)
  this.#active.delete(runId)
  // 控制信号优先:先发 run_finalized,再跑业务 listener
  if (transport) transport.send({ type: "run_finalized", runId })
  appendRunEvent({ kind: "run_finalized_sent" })   // 观测
  // 业务 listener 降为不阻塞控制信号;各自 catch,互不传染
  for (const fn of this.#onRunComplete) {
    void Promise.resolve(fn(threadId, runId, status, kind)).catch(logListenerError)
  }
```

> 注意:`run_finalized` 与 onRunComplete 之间没有真实数据依赖——reflect 只需要信号,锁释放/投影是 listener 各自的事。把信号提前是安全的。

daemon `#drive` 的 finally 解耦本地状态与投递:

```text
finally:
  #runs.delete(runId)
  if (wantsReflect) #finalized.set(runId)   // 先更新本地状态机
  await send(run_done).catch(logDeliveryError) // 投递失败不回滚本地状态;reaper 仍能兜底
```

**修复思路:** 把"控制信号"与"业务副作用"在时间上解耦——信号先发、副作用后跑且各自隔离;把"本地状态机"与"IO 投递"解耦——状态先翻、投递后做且失败不回滚。两处都是"先确立运行时真相,再做可失败的对外动作"。

**影响文件:**
- `apps/backend/src/features/run/supervisor.ts`(`run_done` case:调换 `run_finalized` 与 `#onRunComplete` 顺序,listener 改 fire-and-forget + 各自 catch)。
- `packages/runner-daemon/src/runner-daemon.ts`(`#drive` 的 `finally`:`#finalized.set` 先于 `await send(run_done)`)。
- 测试:`apps/backend/src/features/run/supervisor.test.ts`、`packages/runner-daemon/src/runner-daemon.test.ts`(新增 hang/throw listener 与 send 抛错用例)。

### Patch P2 — sink 分级:账本 critical / 日志可观测(H3, L8)

**目标:** 账本投影失败不再被静默吞;EventLog 与投影的失败语义分离;锁释放调用约定统一。

- `supervisor.ts` `event` case 保持 `eventLog.append` 失败上抛(critical 不变);`onRunEvent` listener 维持 fire-and-forget,但把"账本投影"从 best-effort 路径里**摘出来单独定级**——见 P3 的拆分。
- `onRunComplete` 里的**账本终态写**升为 critical:用 `try/catch` 包裹,失败上抛或显式标记 run 为 `degraded`,绝不静默。
- 统一锁释放约定:reaper 路径与 `run_done` 路径都改为 `await convSvc.completeRun(...)`(或都 fire-and-forget,二选一),消除 L8 的双约定。推荐统一为 `await`(锁释放是 critical)。

**修复思路:** 给链路上每个 sink 贴一致性标签(§3.1)。账本写是 critical → 失败必须传播或显式降级(`degraded`),绝不被 `console.error` 吞掉;EventLog 与投影的失败语义彼此独立,不能一处失败连坐另一处;锁释放是 critical → 调用约定统一为 `await`,消灭 reaper 与 run_done 的双约定(事实 9)。

**影响文件:**
- `apps/backend/src/features/run/supervisor.ts`(`event` case 保持 `eventLog.append` 失败上抛;`onRunComplete` 账本终态写加 `try/catch` 升 critical;reaper 与 `run_done` 两处锁释放统一为 `await convSvc.completeRun(...)`)。
- `apps/backend/src/features/conversation/projection.ts`(账本终态写从无包裹改为 critical 包裹,与 best-effort 段分离——具体拆分见 P3)。
- 测试:`apps/backend/src/features/run/supervisor.test.ts`(账本写失败断言传播/degraded;EventLog 失败独立上抛)。

> H3 的 P2 是**治标**:在投影桥仍承载 assistant 消息的前提下,先把"账本投影失败被静默吞"堵住。**治本**见 P7——把 assistant 消息的权威入口收回 ledger,让投影桥彻底退出"承载业务事实"的角色。

### Patch P3 — 拆分混职责回调(M4, M6, L7, L9)

**目标:** 一个函数只承担一个一致性等级的职责。

- `projection.ts onRunComplete` 拆成三段,各自定级:
  1. `await` 投影链 + 写终态 revision + broadcast(**critical**,`try/catch`,失败传播)。
  2. `appendTodo` + `triggerMentionedAgents`(**best-effort**,fire-and-forget,各自 catch)。
  3. `completeRun` 释放锁(**critical**,移入 `finally` 或独立 await,保证无论 1/2 是否抛错都执行——当前它在 try 外但 1 段无 try 包裹,1 段抛错会跳过它)。
- `main.ts onRunEvent` 拆开:todo 累积(纯本地状态)、@mention 解析(只在终态做一次,不在每条 streaming revision 上重复正则)、投影编排各自独立。@mention 正则提前 `escapeRegExp` 缓存,避免热路径重复编译。
- `sse-watcher processEntry`(M6):`onSend` + `upsertMessageDelivery` 加幂等键(以 `messageId` + `revisionState` 为键),让重连重发覆盖 update 态而非追加;或把两步包成单事务。
- daemon `#onStart`(L7):把协议校验/工具注入/agent 构建与 preload 语义解耦——preload 失败(fold 后内容为空但 `hasPreloaded=true`)不应让 `continue()` 抛 "Cannot continue without a user message";空 preload 应降级为普通 run 或显式拒绝,而非走 continue 路径。

**修复思路:** 一个函数只跑一个一致性等级。把"critical 的事实写"与"best-effort 的副作用/通知"在同一回调里物理切开:critical 段用 `try/catch` 包裹并把锁释放挪进 `finally`(保证 1/2 抛错也能释放),best-effort 段 fire-and-forget 各自 catch。热路径(每条 streaming revision 都跑的 @mention 正则)挪到终态只跑一次并缓存编译。IO+记账两步用幂等键合并,消除重连重发(§3.4)。

**影响文件:**
- `apps/backend/src/features/conversation/projection.ts`(`onRunComplete` 拆三段定级,`completeRun` 进 `finally`)。
- `apps/backend/src/main.ts`(`onRunEvent` 拆:todo 累积 / @mention 只在终态解析一次 + `escapeRegExp` 缓存 / 投影编排)。
- `apps/lark-bot/src/sse-watcher.ts`(`processEntry`:`onSend` + `upsertMessageDelivery` 加幂等键或单事务)。
- `packages/runner-daemon/src/runner-daemon.ts`(`#onStart`:preload 语义与 continue 解耦,空 preload 降级)。
- 测试:`projection.test.ts`(第 1 段抛错锁仍释放)、`main.test.ts`(@mention K 步只解析一次)、`sse-watcher.test.ts`(重连不重发)。

### Patch P4 — 对话锁状态机收敛(H2)

**目标:** `activeConversations`/`pendingRuns` 的迁移规则集中到单一所有者。

抽出一个 `ConversationLock`(或 `RunGate`)小状态机,暴露 `acquire(cid, n)` / `release(cid)` / `isActive(cid)`,内部维护计数与集合。`forkAgentRuns`、`completeRun`、`triggerMentionedAgents` 不再各自读写两个容器,只调用状态机方法。fork 失败的减一、全失败的清理都走 `release`,与业务 `try/catch` 无关。

**修复思路:** 把分散在三个函数 `try/catch`/`finally` 分支里的并发状态迁移,收敛成一个有单一所有者、迁移入口明确的状态机(§3.3)。业务代码只调 `acquire`/`release`/`isActive`,不再直接摸 `activeConversations`/`pendingRuns` 两个容器;迁移正确性由状态机内部保证,与业务成败解耦。这是 P11 跨域合并的前置(P11 在此基础上把直发路径的线程锁也并进来)。

**影响文件:**
- `apps/backend/src/features/conversation/service.ts`(新增 `ConversationLock`;`forkAgentRuns`/`completeRun`/`triggerMentionedAgents` 改调状态机方法,删除对两容器的直接读写)。
- 测试:`apps/backend/src/features/conversation/service.test.ts`(全 fork 失败锁正确释放;并发 `triggerMentionedAgents` 与 `forkAgentRuns` 计数不串)。

**修复思路:** 包是依赖边界,不是文件抽屉(§3.5)。`agent-spec` 是单一消费者薄包,把它并回唯一消费者 `runner-daemon`、删包、改本地相对 import,既不损失任何类型安全,又少一层 workspace 协议/构建产物的固定成本。协议层的不透明边界(`z.record`)是有意为之,不在裁撤范围内。

### Patch P6 — AgentFsLike 契约下沉 core(包边界,事实 11/12)

**目标:** 文件系统契约只有一处规范定义,所有实现/消费方引用它,不再复刻同形声明。

**规范定义的归属(D4 决策):** `AgentFsLike` 是"Agent 能读写的文件系统长什么样"这一**领域原语契约**,与 `Tool`/`Message`/`ChatModel` 同级——它属于 L1。`agent-fs` 本身就是一个 **L1 级的文件系统原语**(只做路径解析、挂载分派、访问控制),不是 L2 框架概念。因此契约的规范定义应**下沉到 `core`**,而不是停留在 `tools-common`(L2 工具层)。

- 把 `AgentFsLike`(及 `pjoin`)从 `tools-common/src/agent-fs-like.ts` **移入 `core`**,作为唯一规范定义,与 `Tool`/`Message` 并列导出。
- `tools-common` 改为从 `core` 引用 `AgentFsLike`(它本就只依赖 `core`,方向不变);所有 plugin(`plugin-fs-memory`、`plugin-progressive-skill`)和 `afs-tools.ts` 的 import 路径相应改到 `core`(或经 `tools-common` re-export 过渡,二选一)。
- `agent-fs` 的 `AgentFS` 类显式 `implements AgentFsLike`:让 `agent-fs` 依赖 `core`,`import type { AgentFsLike }`,把今天的 duck-typing 兼容升级为编译期保证。这样 `harness/create-generic-agent.ts` 里 `workspace.fs` 传给 `AgentFsLike` 形参就有类型背书,任一方改签名会立即编译失败。
- `agent-fs/src/types.ts` 的 `ReadableBackend`/`WritableBackend` 是 backend 层(相对路径、单挂载点)契约,语义与 `AgentFsLike`(逻辑路径、多挂载聚合)**不同层**——保留,但加注释说明分层关系。
- 校验依赖方向无环:`core` 不依赖任何 workspace 包(最底层);`tools-common` 与 `agent-fs` 都只新增/保留对 `core` 的依赖,不成环。

> **为什么不是"`agent-fs` 依赖 `tools-common`"**:那会让一个 L1 原语反向依赖 L2 工具层,把分层倒置。契约下沉 `core` 让依赖方向始终指向更底层,`agent-fs` 与 `tools-common` 成为 `core` 的平行消费者,语义最干净。

**修复思路:** 同一契约只能有一处规范定义(§3.6),且它的归属由分层决定——`AgentFsLike` 是 L1 原语契约,应落在最底层 `core`,让所有实现/消费方都向下引用同一份,把今天的 duck-typing 兼容升级为编译期保证。依赖方向始终指向更底层,无环。

**影响文件:**
- 新增/移入:`packages/core`(承接 `AgentFsLike` + `pjoin`,与 `Tool`/`Message` 并列导出)。
- 修改:`packages/tools-common/src/agent-fs-like.ts`(改为从 `core` re-export 或直接删除)、`packages/tools-common/src/afs-tools.ts` 及 `plugin-fs-memory`/`plugin-progressive-skill` 的 import 路径;`packages/agent-fs/src/agent-fs.ts`(`AgentFS implements AgentFsLike`)、`packages/agent-fs/package.json`(新增对 `core` 的依赖)、`packages/agent-fs/src/types.ts`(给 `ReadableBackend`/`WritableBackend` 加分层注释)。
- 校验:`harness/create-generic-agent.ts` 处 `workspace.fs` 传参获得类型背书;依赖图无环。
- 测试:类型/编译断言——`AgentFS` 满足 `core` 的 `AgentFsLike`,故意删 `AgentFS.stat` 应使 `typecheck` 失败。

### Patch P7 — 对话账本单一权威入口(事实 14/15/16,治本 H3)

**目标:** assistant 消息的权威物理入口收回 ledger,消除双事实来源;`event_log` 回归"纯执行细节",投影桥降级为可观测扇出。

落实 §3.7"一条业务事实只有一个权威物理入口"。采用**方案 B**(ledger 为对话消息的唯一权威载体),不采用方案 A(把 ledger 整体降为 log 的纯投影)——因为 `facts-and-projections.md` 已立硬边界:event_log 不该承载"成员加入"等不属于任一次运行的对话事件,A 会破坏这条边界。

- **Supervisor 直接写 ledger**:assistant 消息(含 streaming revision 与终态)由 Supervisor/Conversation Service 经与人类消息**同一条入口**(`appendLedgerEntry`/`appendAndBroadcast`)直接写入 ledger,与 `postMessage` 对称。ledger 不再有"半截下游于另一机制"的消息。
- **`event_log` 收窄为纯执行细节**:只保留 `tool_start`/`tool_end`/`text_delta` 这类运行内部事件,不再作为"对话消息内容"的事实来源。`facts-and-projections.md` 中 `event_log` 对 assistant 消息的"事实来源"标记**撤销**,改标 ledger 为唯一来源。
- **投影桥降级为可观测扇出**:`main.ts onRunEvent` → `projection.ts` 不再"把 event_log 派生成 ledger 业务事实",而是从权威 ledger 写入点**扇出**给可观测/副作用消费者(broadcast 给前端、ops 记录),全部 best-effort(对齐 §3.1)。"投影"语义从"产生事实"变为"分发已成事实"。
- **保留会话投影的硬边界**:"只有会话投影这一处把运行可见结果变成对话消息"仍成立——只是它现在读的是 ledger 这一唯一源,而非 event_log。
- 兼容性:不为旧的"event_log → 派生 ledger"路径保留并行写;这是要拆除的债,一次切换。历史数据迁移在实施前评估(若已有派生 ledger 数据,确认其与 event_log 一致后即作为权威 ledger 内容,不回放)。

> P2 与 P7 的关系:P2 让坏掉时不再静默(短期止血,可独立先合);P7 拆掉"事实被派生"的结构根源(长期治本)。两者不冲突,P7 落地后 P2 的"账本投影失败"语义自然消失——因为投影不再承载事实。

**修复思路:** 让对话消息这一类事实只有一个权威物理入口(§3.7)。今天 assistant 消息"自产半截、下游半截"是因为它没有直写入口、只能经 event_log 派生进 ledger;修法是给 assistant 消息补上与人类消息对称的直写入口,让 ledger 成为唯一真相,event_log 退回纯执行细节,投影桥从"产生事实"降为"分发已成事实"。一次切换,不保留并行派生写。

**影响文件:**
- `apps/backend/src/features/run/supervisor.ts` / `apps/backend/src/features/conversation/service.ts`(assistant 消息经 `appendLedgerEntry`/`appendAndBroadcast` 直写 ledger,与 `postMessage` 对称)。
- `apps/backend/src/main.ts` + `apps/backend/src/features/conversation/projection.ts`(`onRunEvent` → 投影桥从"派生 ledger 事实"改为"从权威写入点扇出 broadcast/ops",全 best-effort)。
- `apps/backend/src/features/event-log/*`(`event_log` 收窄为 `tool_start`/`tool_end`/`text_delta` 等纯执行事件,不再承载对话消息内容)。
- `docs/architecture/foundations/facts-and-projections.md`(撤销 `event_log` 对 assistant 消息的"事实来源"标记,改标 ledger 唯一来源)。
- 迁移:历史"派生 ledger"数据在实施前评估,确认与 event_log 一致后作为权威 ledger 内容、不回放。
- 测试:终态 ledger 不依赖 event_log 回放即完整;投影桥关闭时 ledger 仍有完整 assistant 消息;`event_log` 不再含对话消息内容。

### Patch P8 — 领域契约归一:LedgerEntry / Member 单一本体(事实 17/18)

**目标:** 同一领域对象只有一处规范定义,其它模块引用而非复刻同形声明。落实 §3.8。

- **LedgerEntry**:以 `packages/conversation` 的 `LedgerEntry` zod 为唯一本体。backend `ports.ts` 的 `LedgerRow`、lark-bot `sse-watcher.ts` 的 `LedgerEntry` interface **全部删除**,改 `import type { LedgerEntry }`。`runId` 字段(今天 backend 有、lark 无)的归属一次性裁定:若它是消息领域的一部分 → 加进本体 schema(可选字段);若它只是 backend 存储行的附属 → 用 `LedgerEntry & { runId?: string }` 在 backend 局部扩展,不污染本体。消除"三份同形、字段已漂移"。
- **Member**:以 `packages/conversation` 的 `Member`(discriminated union)为唯一本体。backend `MemberRow` 作为**存储行**可保留(它是 DB 投影,不是领域模型),但其 `kind` 必须从本体的枚举派生;web `SenderRef` 的 `kind` 多出的 `"system"` 要么并入本体(若系统确实是一类成员),要么改用本体类型 + 单独的渲染态,不再自定义同形枚举。`packages/message` 的 `MessageAuthor`(`system|user|agent|tool`)是**消息作者角色**、与会话 `Member`(参与者身份)不同层——保留,但加注释厘清两者关系,避免下个 agent 再把它们混为一谈。
- 校验:删除复刻定义后全仓 `typecheck` 通过即证明引用方已对齐本体。

**修复思路:** 同一领域对象只有一个本体,模块边界不是领域边界(§3.8)。把 backend/lark/web 各自手抄的同形声明删掉,统一 `import` `packages/conversation` 的权威定义;漂移字段(`runId`)一次裁定归属——属领域则进本体 schema,属存储则在 backend 局部 `&` 扩展,不污染本体。删除复刻后全仓 `typecheck` 通过本身就是"引用方已对齐"的证明。

### Patch P9 — 机制下沉本体:MessageId / content codec / terminal 谓词(事实 19/20/21)

**目标:** id 生成、content 编解码、terminal 判定这三类机制各有唯一所有者,调用方一律引用。

- **MessageId**:在 `packages/message` 立 `messageId` 工厂的**唯一所有者**。`assistantMessageId` 已在此;把 `msg:`(人类)、`sys:`(系统通知)两条规则也收成 `humanMessageId(cid, sender)` / `systemMessageId(cid, tag)` 导出,`service.ts:252/280` 改为调用,消除散落字面量。所有 messageId 规则集中一处,便于审计前缀命名空间。
- **content codec**:把"序列化/反序列化 + 从 blocks 抽 text + legacy `{text}` 回退"收敛成 `packages/message`(或 `conversation`)的单一 codec 函数对。`projectForMember`/`formatContent`、`projection.ts`、web reducer、lark `safeJsonParse` 全部改调它,删除各层重复的 `JSON.parse` 与抽 text 分支。一处改 content 形状,所有 surface 同步。
- **terminal 谓词**:全仓裸 `state === "done" || state === "error"`、`state === "streaming"` 一律替换为 `isTerminalMessageState` / `isOpenMessageState`(`projection.ts:206/256`、web `isBusy`)。同时厘清 run 级 `status`(`succeeded`/`interrupted`)与 message 级 `state`(`done`/`error`/...)**两套术语**:在 `projection.ts` 写终态处显式标注"这是 run.status → message.state 的映射点",不再让两个状态机在同一行裸比较里交织。

**修复思路:** 机制(id 生成、content 编解码、terminal 判定)是领域本体的一部分,应下沉到本体所在的包统一导出,调用方一律引用(§3.8)。三类机制各立唯一所有者:messageId 规则集中到 `packages/message` 工厂;content 编解码收敛成单一 codec 函数对;terminal 判定用已有谓词替换全仓裸字符串比较,并显式标注 run.status → message.state 的映射点,不再让两套术语在裸比较里交织。

**影响文件:**
- `packages/message/src/helpers.ts`(新增 `humanMessageId`/`systemMessageId`,与已有 `assistantMessageId` 并列;content codec 与 terminal 谓词在此或 `packages/conversation` 统一导出)。
- `apps/backend/src/features/conversation/service.ts`(`:252`/`:280` 改调 messageId 工厂)。
- `packages/conversation/src/index.ts`(`projectForMember`/`formatContent` 改调统一 codec)、`apps/backend/src/features/conversation/projection.ts`(`:39/181` codec、`:206/256` terminal 谓词、`:257` 标注映射点)、`apps/web/src/lib/conversation-reducer.ts`(reducer 抽 text 改调 codec、`isBusy` 改谓词)、`apps/lark-bot/src/sse-watcher.ts`(`safeJsonParse` 改调 codec)。
- 测试:grep 守卫——`run:`/`msg:`/`sys:` 前缀只在工厂出现一次,裸 `state === "done"` 全部消失;codec 单测覆盖 blocks/legacy `{text}`/纯 string 三种输入。

### Patch P10 — reflect run 与 run 终态单一权威(事实 22/23,与 P7 同批)

**目标:** run 的"存在"与"完成"各只有一个权威入口,消除 daemon 内存先于 DB、reaper 与 run_done 两套"完成"语义。

- **reflect run 双本体**:reflect run 的权威 INSERT 应在 daemon 发 `run_started` 之前/同步完成,或把 daemon 侧 `#runs.set` 明确降级为"本地执行态缓存"(非权威),DB 行为唯一权威。`parentRunId` 不再 fallback `""`——缺父则显式拒绝或记 degraded,不写空串占位(空串会让血缘断链)。
- **run 终态两套语义归一**:reaper 路径与 run_done 路径走**同一个** `finalizeRun(runId, status)` 入口,内部统一决定"写 status + attempt + EventLog + 是否发 `run_finalized`"。reaper 不再各写一份 `interrupted`;`interrupted` 只是 `finalizeRun` 的一个 status 取值。`projection.ts` 扫 ledger 补写终态的逻辑,在 P7 落地后(assistant 消息直写 ledger)自然退化为纯校验/兜底,不再是第三个独立写入点。
- 与 P7 同属"事实归一"批:reflect run/终态属 run 域,assistant 消息属 conversation 域,但同根于"run 生命周期事实被多处各记一份",建议同批 review。

**修复思路:** run 的"存在"与"完成"各只能有一个权威入口。"存在":DB 权威行先于(或同步于)daemon 发 `run_started`,daemon 内存 `#runs` 明确降级为非权威缓存;`parentRunId` 缺父显式拒绝/记 degraded,不写空串断链。"完成":reaper 与 run_done 收口到同一个 `finalizeRun(runId, status)`,由它统一决定写 status/attempt/EventLog 与是否发 `run_finalized`,`interrupted` 只是一个 status 取值;P7 落地后 `projection.ts` 扫 ledger 补终态自然退化为校验兜底。

**影响文件:**
- `packages/runner-daemon/src/runner-daemon.ts`(`:278` reflect run 造 id/`#runs.set` 降级为非权威缓存)。
- `apps/backend/src/features/run/supervisor.ts`(`#registerDaemonStartedReflectRun` `:313` 权威 INSERT 前移/同步;`:447` `parentRunId` 去掉 `""` fallback;新增 `finalizeRun`,reaper `:128–191` 与 run_done `:508` 两路改调它)。
- `apps/backend/src/features/conversation/projection.ts`(`:252` 扫 ledger 补终态降级为校验/兜底)。
- 测试:reflect 权威行在 `run_started` 被消费前已存在;`parentRunId` 为空显式拒绝;reaper 与 run_done 同入口产生一致的 status+attempt+EventLog 写入集合。

### Patch P11 — 会话并发锁体系归一(事实 24,H2 的体系级延伸)

**目标:** "线程/会话能否再起 run"只有一套 busy 状态机。

- P4 抽出的 `ConversationLock`(`acquire`/`release`/`isActive`)上升为**唯一并发闸门**:HTTP 直发路径的 `threads: Set<string>` + `ThreadBusyError` 与会话 @ 触发路径的 `activeConversations`/`pendingRuns` 合并到它后面。两条入口路径(直发、@ 触发)调用同一闸门,busy 判定不再各算一套、互不感知。
- `ThreadBusyError` 保留为对外错误语义,但其判定来源改为统一闸门。
- 与 P4 关系:P4 是把会话锁的三处迁移收敛成一个状态机(单域内收敛);P11 是把**另一套**线程锁并进同一状态机(跨域合并)。P4 先落,P11 在其上扩展。

**修复思路:** "线程/会话能否再起 run"只能有一套 busy 状态机。把 HTTP 直发路径的 `threads` + `ThreadBusyError` 并进 P4 抽出的 `ConversationLock` 后面,让两条入口路径(直发、@ 触发)走同一闸门,busy 判定不再各算一套、互不感知;`ThreadBusyError` 作为对外错误语义保留,但判定来源改为统一闸门。

**影响文件:**
- `apps/backend/src/features/run/service.ts`(`:37/82/87` 删除 `threads: Set<string>` 自有判定,改调统一闸门;`ThreadBusyError` 抛出点来源切到闸门)。
- `apps/backend/src/features/conversation/service.ts`(P4 的 `ConversationLock` 上升为唯一并发闸门,接纳直发路径)。
- 测试:同一 thread 被直发与 @ 触发两路并发起 run 时,统一闸门正确拒绝第二个;`ThreadBusyError` 仍按预期抛出但来源是统一闸门。

## 10. 最终准则

> **链路上每个 sink 声明自己的一致性等级,控制信号永远跑在业务副作用之前;每条业务事实只有一个权威物理入口,每个领域对象只有一个本体,每个语义契约在仓库里只有一处规范定义,每个包的存在都对应一条真实的依赖边界。** 失败语义不互相绑架,事实不被派生成第二份真相,领域对象不被各模块各抄一份,契约不被复刻成多份,薄抽象不假装成独立单元。
