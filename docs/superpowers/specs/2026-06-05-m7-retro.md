# M7 Retro — AgentSpec 与 Runner（Backend ↔ Harness 跨进程契约）

## Actual vs Spec

### Matched

- `AgentSpecV1` 字段全量匹配：`schemaVersion: "1"` / `workspace` / `threadId` / `model { provider, model, baseURL? }` / `apiKey?` / `permissionMode?` / `maxSteps?` / `input`
- `agent-spec` 零工作区依赖（仅 `zod ^3.23.0`），架构文档 §12 §四 反向依赖原则
- `AgentEvent.error` variant 结构匹配：`{ type: "error"; payload: { message: string; stack? } }`
- framework `agent.run` 行为不变——**不主动 yield error event**，异常仍向上抛
- `runner-stdio` 依赖链匹配：agent-spec + harness + adapter-anthropic + framework（type-only），不反向依赖 backend
- `runEntry(io: EntryIO)` DI 函数化签名，返回 exit code（0/1），不调 `process.exit`/`process.stdout`/`process.stderr`
- `bin.ts` thin wrapper：构造 `EntryIO` → `runEntry` → `process.exit(code)`
- apiKey 优先级 `spec.apiKey > env.ANTHROPIC_API_KEY > error`（无第三 fallback）
- SIGTERM/SIGINT → AbortController.abort，传给 agent.run
- abort 后 exit code = 0（不是失败）
- runner 不主动 retry / 不持久化状态 / 不读 stdin
- NDJSON 协议：stdout 每行 `JSON.stringify(event) + "\n"`，stderr 人类可读 trace
- schemaVersion 错配 hard fail，不静默降级
- `bin` 名 `my-agent-runner`（全称无歧义）

### Differences

1. **`agent-protocol/` 空壳无需清理**。已不存在——M6 后已无人引用，目录从未在仓库中出现过。commit 1 无删除操作。

2. **测试数量超 spec 预估**。Spec 预估 ~21（1 + 8 + 12），实际 28（1 + 13 + 14）：
   - agent-spec: 13 vs 8。增补了 `workspace: ""`、`threadId: ""`、`maxSteps` 非整数、`CURRENT_SCHEMA_VERSION` 常量、全字段 optional 填满等 5 个 case
   - runner-stdio entry: 11 vs 10。增补了 `custom apiKeyEnv` 测试（spec 有 `apiKeyEnv` 字段但测试清单未列）
   - runner-stdio smoke: 3 vs 2。增补了 schema validation failure 的端到端 smoke case

3. **`EntryIO.createAgent` DI 字段**。Spec 描述为"可选 `createAgent: typeof createGenericAgent` 入参"，实现为 `EntryIO` 的 optional 字段而非 `runEntry` 第二参数。效果等价，且保持单参数函数签名更简洁。

4. **harness 动态 import**。`createAgent ?? (await import("@my-agent-team/harness")).createGenericAgent` —— spec 假设静态 import harness，实现用动态 import 实现"DI 注入时不加载 harness"。生产路径（bin.ts）不传 createAgent，动态 import 触发一次（ESM 缓存）。

5. **`baseURL` vs `baseUrl` 字段名映射**。AgentSpec 使用 `baseURL`（遵循 URL 缩写习惯），AnthropicChatModel 使用 `baseUrl`（camelCase）。entry.ts 在构造 ChatModel 时做映射：`baseUrl: spec.model.baseURL`。

6. **`asError()`/`asMessage()` 类型守卫**。Biome `noNonNullAssertion` 规则要求 `written[0]!` → `written[0]?.`，但 `?.` 阻止 TypeScript 窄化 union type。引入两个 helper 做显式类型窄化：`asError(ev)` / `asMessage(ev)`，既满足 linter 又保持类型安全。

7. **bash.ts formatting 顺带修复**。Commit 1 附带 `packages/tools-common/src/bash.ts` 的 biome 格式化变更（来自 `bun run format` 全局运行）。非 M7 功能范围，但保证 commit 1 CI 独立绿。

| ID | 决策 | 结果 |

| D1 | `EntryIO.createAgent` 为 optional 字段而非 `runEntry` 第二参数 | 好。单参数简化调用方；测试和生产 bin 路径清晰 |

## Lessons

1. **Linter 规则与类型窄化的交互是 TDD 中的隐藏复杂度**。Biome `noNonNullAssertion` 规则将 `written[0]!` 自动修正为 `written[0]?.`，但 `?.` 阻止 TypeScript 通过 `expect(type).toBe("error")` 窄化 discriminated union。`asError()` helper 是干净的解——它把"类型断言"从 lint-unfriendly 的 `!` 变成 lint-friendly 的函数调用，同时保持了运行时 safety（如果 type 不对会 throw）。**教训：当 linter 规则与类型系统冲突时，提取一个具名 helper 比 suppress comment 更有价值。**

2. **DI 不总是需要复杂的 IoC 容器**。`EntryIO.createAgent` 只是一个 optional 函数字段，但足够让 11 个 entry 测试全部用 mock agent 跑而不需要真实的 Anthropic API key。对比 M6 harness 测试需要真文件系统（tmpdir），M7 runner 测试完全在内存中运行。**简单的 DI pattern（optional factory function）在测试隔离性上的 ROI 远超其实现成本。**

3. **"填空而非设计"的 spec 仍然有设计空间**。M7 spec 自称为填空题，但实现中仍然做了多个设计决策：动态 import vs 静态 import、EntryIO 字段 vs 第二参数、smoke 测试数量。这些决策小但累积影响代码结构。**即使是"照 spec 实现"，也值得在 retro 中记录这些 micro-decision 的理由。**

4. **Schema 包的治理成本确实极低**。agent-spec 18 LOC 生产代码、零业务逻辑、零工作区依赖。增加这样一个包的成本（package.json + tsconfig × 2 + 13 tests）约 100 LOC boilerplate，但换来的是 backend 和 runner 之间的编译时 + 运行时双重契约保障。**这个 tradeoff 是值得的——100 LOC boilerplate 换来零 runtime drift 风险。**
