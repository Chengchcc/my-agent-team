## 二、实现要点

### 做对了的

1. **不建第二事实源。** 从头到尾 EventLog 是唯一写入点。`llm_call`/`tool_call` 是 AgentEvent 的新 variant,json_extract 可查,schema 零改动——正是 14-event-log §4.3 承诺的「加事件类型零 schema 改动」。

2. **runner-daemon 真的不用改。** delta 白名单是 `text_delta | tool_start | tool_end`,新增的 `llm_call | tool_call` 都不在其中,自动走 else 分支进 EventLog。这是 14-event-log 架构的红利:路由是反选(白名单=delta,其余=event),所以新增持久化事件不需要改路由。

3. **executeOne 的 step 参数化。** executeOne 本身是 `async function*`,被 runLoop 以 `yield*` 消费——所以在其内部 `yield { type: "tool_call" }` 能直达 EventLog,无需改 runLoop 的消费方式。step 作为显式参数传入,比用闭包捕获循环变量更清晰。

4. **collectStream 在非 stream 分支能捕获 stopReason。** collectStream 本身透出 stopReason(只是之前没人用),这次补上了。

### 踩到的坑

1. **pricing test 数量级错误。** 价格表是 USD/1M tokens,1M tokens 不是 $0.015 而是 $15。写 test 时心里想的「1M token 花几毛钱」其实是 1K token 的量级。修正是用 10K input + 1K output 这种真实场景的 token 数。

2. **toolBreakdown errorRate 硬编码 0。** 用 `...stats` 展开后覆盖 `errorRate: 0`,忘了重新计算。code review 发现并修正。

3. **非 stream 分支 stopReason 未捕获。** spec 原话说「collectStream 不透出」但实际 collectStream 是透出的——spec 错,实现跟着错。review 发现。

4. **bun test expect().toEqual() 对 undefined 字段宽容。** 适配器测试 mock 的 finalMessage.usage 不带 cacheCreate/cacheRead,实际 adapter 现在会补 `undefined` 值;toEqual 不检查 undefined 字段,所以 14 个老测试全绿,无需改。这是 bun test 的行为特性,不是 bug。

### spec 与实际实现的偏差

| spec 说 | 实际 |
|---------|------|
| collectStream 不透出 stopReason | collectStream 实际透出了(stream-utils.ts:7) |
| stopReason 仅 stream 分支可得 | 修正后两个分支都可得 |
| llm_call/tool_call 是汇总型,非 delta | 正确 — runner-daemon 白名单不含它们 |

## 三、架构判断验证

14-event-log 的铁律在本期经受住了考验:

1. **「EventLog 是唯一 trace 事实源」** — 新增 `llm_call`/`tool_call` 就是往 EventLog 多 append 两个 event type,schema 不动、索引不动、读路径不动。如果当初建了 `otel_span` 表,本期就要同时写两份,且 query 要 join 两张表——正是 §十一 ADR 警告的「制造第二事实源」。

2. **「采集只读不改控制流」** — runLoop 的改动全部是 `yield` 新事件 + 局部变量,没改任何 if/else 分支、没动 run status、没改 checkpointer 恢复。Proof:112 个框架测试全绿,一行测试断言都不用调(除了那个 rigid 的 `toEqual` snapshot)。

3. **「指标可丢可降级」** — `rt.model.id ?? "unknown"`、`usage?.input ?? 0`、cost 返回 null,UI 全程处理 null case。不是事后打的补丁,是设计时就考虑好的。

## 五、经验教训

1. **spec 也会错。** 「collectStream 不透出 stopReason」这句是 spec 写的,但代码不是。写完 spec 后应该先验证 spec 假设(尤其是「某函数不透出某字段」这种代码事实),再照着写实现。

2. **bun test 的 toEqual 对 undefined 宽松是双刃剑。** 好处是向后兼容(老测试不用改),坏处是它掩盖了字段变化——如果某个调用方真的依赖「没有 cacheCreate 字段」,toEqual 不会报。

3. **价格表是配置,不是代码逻辑。** 本期用代码常量起步是对的(够用、可单测),但如果未来模型频繁增减,应该移到 env/JSON 配置文件,让非工程人员也能改价格。

4. **shadcn/ui Charts 模板太长。** chart.tsx 298 行,其中 ~250 行是 shadcn 官方模板的 tooltip/legend 逻辑。如果能拆成独立 npm 包而不是 copy-in,文件会轻很多。
