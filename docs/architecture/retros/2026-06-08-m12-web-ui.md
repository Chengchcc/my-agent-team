## Actual vs Spec

### Matched

- C1–C10 全部 10 阶段主体功能
- BFF 透明代理（REST catch-all + SSE pipe）+ 同源 cookie 鉴权
- React Query 唯一数据源，SSE patch 缓存，无全局 store
- 历史快照 ⊕ 实时事件拼接，首尾相接
- 打字机回放（M12），M13 delta 流预留
- tool_use/tool_result 配对渲染，interrupt 审批卡
- Agent 人格/记忆只读展示
- 10 字段 AgentForm + 创建后 model 不可改 UI 限制
- 虚拟滚动 CSS content-visibility（D13 已知债）
- BFF proxy 的 signal/cookie/header 透传

### Differences

1. **登录方案：fetch → 原生 HTML form**。Spec §四预期 `fetch` + `router.push`，但浏览器在 HTTP localhost 环境下 `fetch` 设 `Set-Cookie` 不稳定（有时不存、有时被中间件跳过）。最终方案：纯 HTML `<form>` POST → 302 redirect，浏览器导航原生处理 cookie。**成本最低，可靠性最高。**

2. **shadcn 组件按需装，不退手抄**。Spec 建议"bunx 失败退 B"，实测 `bunx shadcn@latest` 工作稳定，全程 A。

3. **bun --watch 在 dev 环境不可靠**。后端加 `--watch` 后频繁被杀（与 Next.js 编译竞争？），回退到 `bun run`。文件监听功能推迟。

4. **Bun HTTP server 的 header 丢失 bug**。`Bun.serve` 在 `await` 异步操作后再读 `req.headers` 会丢失自定义 header（如 `x-auth-token`）。workaround：auth 检查放在所有 async 操作之前。**这是 Bun 运行时行为，不是 spec 预期的问题。**

5. **EventLog 写入路径双通道**。Spec 预期 runner 直写 EventLog DB（`spec.storage.eventLog`），但 `buildSpecJson` 最初没传此配置。修复后 runner 双写：primary → EventLog SQLite，fallback → stdout → supervisor → EventLog.append。**双通道确保事件不丢。**

6. **SSE subscribe 不检测 run 完成**。EventLog.subscribe 在 run 已完成后仍然 polling，导致 SSE 连接永不关闭。短期修复：前端在收到 done 后关闭 EventSource 并 refetch history。**EventLog 本身需要 done 检测——列入 M13 债。**

7. **resolveLegacyThreadRun 对 agent_thread 误 forward**。`backfillLegacyThreads` 为所有 thread 创建 conversation 表记录，导致新 agent_thread 的 POST /runs 被转发到 conversation handler，返回 `{seq, triggeredRuns}` 而非 `{runId, attemptId}`。修复：先检查 `thread.kind === "agent_thread"` 跳过 forward。

## Lessons

1. **SSE over BFF over Bun 的坑密度极高**。3 层（浏览器 EventSource → Next.js Route Handler → Bun.serve SSE）每层都有非标准行为：Bun 的 `idleTimeout` 默认杀长连接、Bun 的 async-before-headers header 丢失、Next.js 15 的 `params` Promise API、浏览器的 `Secure` cookie on HTTP localhost 拒绝。**教训：SSE 跨多层的集成测试必须是最早写的，不能放到最后。**

2. **`process.env` 错误缺乏 fail-fast**。`BACKEND_TOKEN` 未设 → BFF 发空 header → backend 401。没有加载期断言、没有请求级日志、没有类型级别的 `Some("dev-token")`。排查 401 花的时间最长（B1→B2→B3 三次误判）。**教训：启动期检查所有必选 env、BFF 侧加 token 注入日志。**

3. **"内容故意不处理" 的注释 = 未来 bug**。Supervisor 的 `// stdout consumed to prevent backpressure; content intentionally unused` 是事件丢失的根因。注释解释了意图（"durable path 才是正源"），但 durable path 根本没接。**教训：不处理数据的代码路径应至少加计数器/度量，或触发告警，不能静默丢弃。**

4. **React useEffect 的 timing 需要显式建模**。`live.reset()` 在 done 事件到达时立即清空消息，但 React 还在同一批次中渲染消息——用户看到闪现 0ms。加了 done→history refetch→300ms delay→reset 的流水线后才稳定。**教训：跨 effect 的状态转换（streaming → done → history-loaded）应该有明确的阶段机，而不是靠 useEffect 依赖数组的隐式时序。**

5. **`backfillLegacyThreads` 打破了 invariant**。"所有 conversation 表中的 thread 都是 multi-member conversation"——backfill 打破了这条，导致 agent_thread 被误 forward。**教训：backfill/migration 脚本必须考虑它们对现有代码 invariant 的影响，不能只求"**表里有数据"**。**

6. **30 个 fix commit → squash 到 4 个**。36 个原始 commit 中 30 个是修复。根源：E2E 集成测试放在最后（C10），而不是每个阶段后。如果 C2 结束时就有 BFF→backend 的集成验证，B1/B2/B3 都会早发现。**教训：每个 Phase 结束做端到端冒烟测试，不攒到 C10。**
